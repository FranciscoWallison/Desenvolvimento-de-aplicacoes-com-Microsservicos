# 🛡️ OWASP Top 10 — Guia de Estudo

## 📖 O que é a OWASP?

O **OWASP** — *Open Worldwide Application Security Project* (renomeado de *Open **Web** Application Security Project*) — é uma organização sem fins lucrativos focada em **aprimorar a segurança de software**. Um dos seus principais trabalhos é manter atualizado um relatório com o **Top 10 dos maiores riscos de segurança em aplicações web**.

Esse relatório é reconhecido mundialmente por desenvolvedores e profissionais de segurança como o **primeiro passo a ser analisado para uma codificação mais segura**.

> 🔗 Documento oficial: [OWASP Top 10 — 2021](https://owasp.org/www-project-top-ten/)

> 💡 **A lista evolui — segurança é um campo em constante mudança.** Comparada à edição de 2017, a versão **2021** trouxe **3 categorias novas**, todas presentes nesta lista:
> - **A04 — Insecure Design** (Design Inseguro)
> - **A08 — Software and Data Integrity Failures** (Falhas de Integridade de Software e Dados)
> - **A10 — Server-Side Request Forgery / SSRF**
>
> Por isso é essencial **manter-se atualizado** sobre as táticas e técnicas atuais dos agentes de ameaça. Visão de gestão de risco em [24.md](24.md).

## 1. [Broken Access Control](https://owasp.org/Top10/A01_2021-Broken_Access_Control/) — Controle de Acesso Quebrado

**O que é:** Falha no controle de acesso que permite a usuários não autorizados visualizar, modificar ou excluir dados que não deveriam.

**Exemplo real:**
```
GET /api/users/123/orders   ← usuário 123 acessa seus pedidos
GET /api/users/456/orders   ← mesmo usuário consegue ver pedidos de OUTRO ID
```
A aplicação não valida se o usuário autenticado é dono do recurso solicitado (**IDOR — Insecure Direct Object Reference**).

**Como prevenir:**
- Negar acesso por padrão (deny by default)
- Validar autorização em **cada requisição** no servidor (nunca confiar no front-end)
- Aplicar o princípio do **menor privilégio**

---

## 2. [Cryptographic Failures](https://owasp.org/Top10/A02_2021-Cryptographic_Failures/) — Falhas Criptográficas

**O que é:** Dados sensíveis expostos por falta de criptografia, uso de algoritmos fracos ou má gestão de chaves.

**Exemplo real:**
- Senhas armazenadas em **MD5** ou **SHA-1** (algoritmos quebrados)
- Tráfego de login em **HTTP** (sem TLS)
- Chaves privadas commitadas no Git

**Como prevenir:**
- Usar **TLS 1.2+** em todo o tráfego
- Hash de senhas com **bcrypt, Argon2 ou scrypt** (nunca MD5/SHA-1)
- Criptografar dados sensíveis em repouso (AES-256)
- Nunca versionar chaves/segredos — usar cofres (Vault, AWS KMS)

## 3. [Injection](https://owasp.org/Top10/A03_2021-Injection/) — Injeção

**O que é:** Inserção de scripts ou comandos maliciosos em campos de entrada que são interpretados pela aplicação. Inclui **SQL Injection, NoSQL Injection, Command Injection** e **XSS** (Cross-Site Scripting).

**Exemplo real (SQL Injection):**
```sql
-- Código vulnerável (concatenação):
"SELECT * FROM users WHERE email = '" + input + "'"

-- Entrada maliciosa:
' OR '1'='1

-- SQL resultante: retorna TODOS os usuários
SELECT * FROM users WHERE email = '' OR '1'='1'
```

**Como prevenir:**
- **Prepared Statements** (consultas parametrizadas)
- ORMs com binding automático (Sequelize, Hibernate, SQLAlchemy)
- Validação e sanitização de entrada
- Escapar saída em HTML para prevenir XSS

---

## 4. [Insecure Design](https://owasp.org/Top10/A04_2021-Insecure_Design/) — Design Inseguro

**O que é:** Falhas estruturais de **arquitetura e design** — não são bugs de código, são decisões erradas de projeto.

**Exemplo real:**
- Mensagem de erro: `"Senha incorreta para o usuário admin@empresa.com"` → revela que o e-mail existe
- Recuperação de senha por **pergunta secreta** ("nome do seu pet") — facilmente descoberta por engenharia social
- Sistema bancário sem **rate limiting** em transferências

**Como prevenir:**
- **Threat Modeling** desde o início do projeto
- Mensagens de erro genéricas (`"Credenciais inválidas"`)
- Aplicar **Secure by Design** e revisar fluxos críticos com a equipe de segurança

---

## 5. [Security Misconfiguration](https://owasp.org/Top10/A05_2021-Security_Misconfiguration/) — Configuração Insegura

**O que é:** Configurações padrão, incompletas ou erradas em servidores, frameworks, bancos de dados ou nuvem.

**Exemplo real:**
- Bucket **S3 público** com dados de clientes
- Console de admin do **Tomcat/phpMyAdmin** acessível com `admin/admin`
- Headers HTTP de segurança ausentes (`X-Frame-Options`, `CSP`)
- Mensagens de **stack trace** expostas em produção

**Como prevenir:**
- Hardening do servidor (remover serviços/portas desnecessários)
- Revisar permissões de nuvem (princípio do menor privilégio)
- Ambientes idênticos (dev = staging = prod) via IaC (Terraform, Ansible)
- Scanners automatizados de configuração (Trivy, kube-bench)

## 6. [Vulnerable and Outdated Components](https://owasp.org/Top10/A06_2021-Vulnerable_and_Outdated_Components/) — Componentes Vulneráveis e Desatualizados

**O que é:** Uso de bibliotecas, frameworks ou ferramentas com versões antigas ou sem suporte, contendo CVEs conhecidos.

**Exemplo real:**
- **Log4Shell (CVE-2021-44228)** — falha crítica na biblioteca Log4j que afetou milhões de sistemas em 2021
- Usar Node.js 12 (EOL) em produção
- jQuery 1.x com XSS conhecido

**Como prevenir:**
- Inventário de dependências (SBOM)
- Scanners contínuos: **`npm audit`**, **Dependabot**, **Snyk**, **OWASP Dependency-Check**
- Atualizações regulares e remoção de bibliotecas não usadas

## 7. [Identification and Authentication Failures](https://owasp.org/Top10/A07_2021-Identification_and_Authentication_Failures/) — Falhas de Identificação e Autenticação

**O que é:** Falhas que permitem ataques de **força bruta**, aceitação de **senhas fracas**, sessões mal gerenciadas ou ausência de MFA.

**Exemplo real:**
- Aceitar senha `123456`
- Não bloquear conta após 1000 tentativas erradas
- Token de sessão **sem expiração** ou exposto na URL
- Não invalidar sessão após logout

**Como prevenir:**
- **MFA** (autenticação multifator) obrigatória em contas críticas
- Política de senhas fortes + verificação em listas de senhas vazadas (haveibeenpwned)
- **Rate limiting** e bloqueio progressivo
- Tokens JWT com expiração curta + refresh tokens
- Cookies de sessão com `HttpOnly`, `Secure`, `SameSite`

## 8. [Software and Data Integrity Failures](https://owasp.org/Top10/A08_2021-Software_and_Data_Integrity_Failures/) — Falhas de Integridade de Software e Dados

**O que é:** Código ou infraestrutura que assume integridade sem verificação — especialmente em **pipelines CI/CD**, **atualizações automáticas** e **dependências externas**.

**Exemplo real:**
- Pipeline CI/CD que baixa scripts de URL pública sem verificar hash
- Atualização automática de software sem **assinatura digital**
- Caso real: **SolarWinds (2020)** — atacantes injetaram código malicioso no pipeline de build, distribuindo malware para 18.000 organizações

**Como prevenir:**
- Assinatura digital de pacotes (Sigstore, GPG)
- Verificação de hash/checksum em downloads
- Pipelines CI/CD isolados, com revisão de código obrigatória
- **Subresource Integrity (SRI)** para scripts externos no front-end

---

## 9. [Security Logging and Monitoring Failures](https://owasp.org/Top10/A09_2021-Security_Logging_and_Monitoring_Failures/) — Falhas de Log e Monitoramento

**O que é:** Ausência ou má qualidade de logs e monitoramento, impedindo a detecção de ataques em andamento ou a investigação pós-incidente.

**Exemplo real:**
- Aplicação **não loga tentativas de login falhas**
- Logs sem timestamp, sem ID de usuário ou sem contexto
- Equipe descobre invasão **6 meses depois** pelo cliente
- Logs armazenados sem proteção (atacante apaga rastros)

**Como prevenir:**
- Logar eventos críticos: login (sucesso/falha), alterações de permissão, acesso a dados sensíveis
- Centralizar com **SIEM** (Splunk, ELK, Datadog)
- Alertas automáticos para padrões suspeitos
- Logs imutáveis (write-once) e backup off-site

## 10. [Server-Side Request Forgery (SSRF)](https://owasp.org/Top10/A10_2021-Server-Side_Request_Forgery_%28SSRF%29/) — Falsificação de Requisição no Servidor

**O que é:** A aplicação web busca um recurso remoto **sem validar a URL** fornecida pelo usuário, permitindo que o atacante force o servidor a acessar recursos internos.

**Exemplo real:**
```
# Funcionalidade legítima: importar imagem por URL
POST /import-image
{ "url": "https://exemplo.com/foto.jpg" }

# Ataque SSRF — acessa metadados internos da AWS:
{ "url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/" }
```
O servidor faz a requisição e devolve credenciais da máquina ao atacante.

**Como prevenir:**
- **Allowlist** de domínios permitidos (não blocklist)
- Bloquear acesso a IPs internos (127.0.0.1, 169.254.x.x, 10.x.x.x)
- Desabilitar redirecionamentos automáticos
- Segmentação de rede (a aplicação não deveria acessar metadados da nuvem)

## 🧱 Princípios de Segurança da OWASP (Secure Design)

> Da leitura *Mais sobre os princípios de segurança da OWASP*. Enquanto o **Top 10** lista *o que dá errado*, estes princípios orientam *como projetar certo* — são a base do **A04 — Insecure Design** (acima).

| Princípio | O que significa |
|-----------|-----------------|
| **Minimizar a superfície de ataque** | Reduzir os pontos exploráveis — menos endpoints, recursos e features = menos vetores de ataque |
| **Privilégio mínimo** | Cada usuário só tem o acesso necessário à sua tarefa — ver [24.md](24.md) |
| **Defesa em profundidade** | Várias camadas de controle, nunca uma só — ver [57.md](57.md) |
| **Separação de tarefas** | Ações críticas dependem de mais de uma pessoa (cada uma com privilégio mínimo) — controle administrativo, ver [25.md](25.md) |
| **Mantenha a segurança simples** (KISS) | Evitar complexidade desnecessária; complexidade dificulta a segurança |
| **Corrija os problemas corretamente** | Achar a **causa raiz**, conter o impacto, identificar vulnerabilidades e **testar** a correção |
| **🆕 Estabelecer padrões seguros** (secure defaults) | O estado **padrão** já é o seguro; tornar o app inseguro deve exigir trabalho extra |
| **🆕 Falhar com segurança** (fail securely) | Ao falhar, o controle assume a opção **mais segura** (firewall que cai → bloqueia tudo, não libera tudo) |
| **🆕 Não confie nos serviços** | Não confiar cegamente em sistemas de terceiros/parceiros; validar antes de usar (ex.: conferir o saldo de pontos antes de exibir ao cliente) |
| **🆕 Evite a segurança por obscuridade** | A segurança **não pode** depender de manter o código/segredo escondido; deve vir de senhas fortes, defesa em profundidade, auditoria e controles de fraude |

> 📛 **Renomeação:** a sigla OWASP permanece, mas o nome mudou de *Open **Web** Application Security Project* para *Open **Worldwide** Application Security Project* — refletindo escopo além da web (ex.: [OWASP Mobile Top 10](https://owasp.org/www-project-mobile-top-10/)).
