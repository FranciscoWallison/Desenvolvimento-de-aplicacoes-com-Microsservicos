# 🧪 Laboratório Prático: Broken Access Control via Código-Fonte

> **Tema:** Exploração de funcionalidade de admin com URL imprevisível
> **Vulnerabilidade:** [A01 — Broken Access Control](47.md) + [A04 — Insecure Design](47.md)
> **Plataforma:** [PortSwigger Web Security Academy](https://portswigger.net/web-security)
> **Pré-requisitos:** Kali Linux com Firefox

---

## 🎯 Contexto

Estamos em uma **startup de pentest** explorando a aplicação web fictícia **"We Like to Shop"** (um e-commerce). Nossa missão: descobrir vulnerabilidades antes que atacantes reais o façam.

### Resumo do lab anterior

Na aula passada:
- Conseguimos acessar o **painel de administrador** sem estar logados
- A descoberta foi feita lendo o arquivo `/robots.txt`, que expunha a URL `/administrator-panel`
- O time de desenvolvimento foi notificado e supostamente corrigiu o problema

**Agora, nosso trabalho é validar a correção.** A página de admin realmente está protegida?

---

## 🔬 Etapa 1 — Validar que a Correção Antiga Funcionou

Antes de procurar uma nova falha, confirmamos que **as correções anteriores foram aplicadas**:

### Teste 1: URL antiga do admin

```
https://<lab-id>.web-security-academy.net/administrator-panel
```

**Resultado esperado:** `404 Not Found` ✅
A página foi renomeada — não está mais nessa URL.

### Teste 2: Arquivo robots.txt

```
https://<lab-id>.web-security-academy.net/robots.txt
```

**Resultado esperado:** `404 Not Found` ✅
O arquivo foi removido — não há mais "dica" sobre rotas privadas.

> 💡 **Lição:** o time fez a correção **mais óbvia** (renomear + remover robots.txt), mas isso é uma defesa **frágil**. Vamos ver por quê.

---

## 🔍 Etapa 2 — Inspecionar o Código-Fonte

Toda aplicação web entrega **HTML + CSS + JavaScript** ao navegador do usuário. Tudo isso é **público por definição** — qualquer pessoa pode ler.

### Como inspecionar

1. Na página principal, **botão direito** em qualquer área em branco
2. Clicar em **"Inspect"** (Inspecionar) — ou pressionar `F12`
3. Abrir a aba **Elements** / **Inspetor**

### O que encontramos

Dentro de uma tag `<script>` no HTML:

```javascript
var isAdmin = false;
if (isAdmin) {
  var topLinksTag = document.getElementsByClassName("top-links")[0];
  var adminPanelTag = document.createElement('a');
  adminPanelTag.setAttribute('href', '/admin-dk7fvh');  // 🚨 URL EXPOSTA
  adminPanelTag.innerText = 'Admin panel';
  topLinksTag.appendChild(adminPanelTag);

  var pTag = document.createElement('p');
  pTag.innerText = ' | ';
  topLinksTag.appendChild(pTag);
}
```

> ⚠️ O número após `/admin-` é gerado aleatoriamente por sessão de lab — no seu caso será diferente.

---

## 🚨 Etapa 3 — Identificar a Vulnerabilidade

### O que esse código faz?

1. Define `isAdmin = false`
2. **Se** o usuário fosse admin, criaria dinamicamente um link de menu apontando para `/admin-dk7fvh`
3. Como não somos admin, o link **não aparece na tela**...

### O problema (duas falhas combinadas)

| Falha | Categoria OWASP |
| ----- | --------------- |
| URL secreta hard-coded no JavaScript que vai para **todos os usuários** | A04 — Insecure Design |
| Não há autenticação/autorização real no servidor — só "esconder o link" | A01 — Broken Access Control |

> 🧠 **Princípio violado:** *Security through obscurity is not security.*
> Esconder o link **não** protege o recurso. A proteção precisa estar no **servidor**, validando se o usuário é admin **antes de responder**.

## 💥 Etapa 4 — Explorar a Falha

Sabendo a URL exposta no JavaScript, basta acessá-la diretamente:

```

```

**Resultado:** entramos no painel de administração 🎯

Clicando em **Delete** no usuário `carlos`, o lab é resolvido:

## 🧠 Análise da Vulnerabilidade

### Por que a correção do time foi insuficiente?

```
┌──────────────────────────────────────────────────────────┐
│ Correção aplicada:                                       │
│   ✗ Renomear /administrator-panel → /admin-dk7fvh        │
│   ✗ Remover robots.txt                                   │
│                                                          │
│ O que faltou:                                            │
│   ✓ Não enviar a URL secreta no JS público               │
│   ✓ Validar permissões no servidor em CADA requisição    │
│   ✓ Retornar 403/404 para não-admins, mesmo com URL correta │
└──────────────────────────────────────────────────────────┘
```

### Pirâmide da defesa correta

```
              ┌─────────────────────────┐
              │  Validação no SERVIDOR  │  ← defesa real (essencial)
              │  (auth + autorização)   │
              └─────────────────────────┘
                          ▲
              ┌─────────────────────────┐
              │   Token/cookie de sessão│  ← identidade
              └─────────────────────────┘
                          ▲
              ┌─────────────────────────┐
              │  Esconder UI do não-admin│  ← UX, NÃO segurança
              └─────────────────────────┘
```

Esconder o link é apenas **UX** (interface limpa). **Não substitui** validação real.

## 🛡️ Como Deveria Ter Sido Corrigido

### ❌ Errado (o que o time fez)

```javascript
// Cliente — exposto a qualquer um que abra o DevTools
var isAdmin = false;
if (isAdmin) {
  // Cria link para /admin-dk7fvh
}
```

### ✅ Certo (o que deveria ser feito)

**No servidor (exemplo em Node.js/Express):**

```javascript
app.get('/admin-panel', requireAuth, requireAdmin, (req, res) => {
  res.render('admin');
});

function requireAdmin(req, res, next) {
  if (!req.user || !req.user.isAdmin) {
    return res.status(403).send('Forbidden');
  }
  next();
}
```

**No cliente:**

```javascript
// Apenas faz a requisição — o servidor decide se responde
fetch('/admin-panel')
  .then(res => {
    if (res.status === 403) hideAdminUI();
    else showAdminUI();
  });
```

**Pontos-chave:**
- Nenhuma URL secreta no JS público
- Servidor **valida em cada requisição**
- Cliente confia no servidor, não controla acesso

## 📋 Checklist de Boas Práticas (Broken Access Control)

- [ ] Toda rota privada exige **autenticação** antes de qualquer lógica
- [ ] Toda rota privada exige **autorização** baseada em papel/permissão
- [ ] **Nunca** confiar em controle feito apenas no front-end
- [ ] **Não embutir** URLs/IDs secretos em JS, HTML ou CSS público
- [ ] Logs de acesso a recursos sensíveis (conecta com [A09](47.md))
- [ ] Testar com usuário **anônimo** e **comum** tentando acessar URLs de admin
- [ ] Política **deny by default** — só liberar o que foi explicitamente permitido

## 🎓 Conceitos Aprendidos

| Conceito | Definição |
| -------- | --------- |
| **Broken Access Control** | Falha que permite acesso indevido a recursos |
| **Security through Obscurity** | Tentar proteger algo apenas escondendo — anti-padrão |
| **Information Disclosure** | Vazamento de informação por código-fonte/comentários/robots.txt |
| **DevTools / Inspect Element** | Ferramenta nativa do navegador que expõe todo HTML/JS recebido |
| **Defense in Depth** | Múltiplas camadas de proteção; nunca depender de só uma |

## 💡 Reflexão Final

> "Toda informação enviada ao navegador é pública. **Sempre.**
> Se uma URL, ID, token ou regra de negócio precisa ser secreta — ela precisa viver no servidor, não no cliente."

Esse princípio único resolve a maior parte dos casos de **Broken Access Control** que você vai encontrar no dia a dia.
