## 🌐 Etapa 2 — O que é um Proxy?

Um **proxy** é um intermediário entre o cliente (navegador) e o servidor:

```
┌──────────┐    requisição    ┌─────────┐    requisição    ┌──────────┐
│ Usuário  │ ───────────────► │  PROXY  │ ───────────────► │ Servidor │
│ (browser)│ ◄─────────────── │ (Burp)  │ ◄─────────────── │          │
└──────────┘     resposta     └─────────┘     resposta     └──────────┘
                                  │
                                  ▼
                          🔍 Inspecionar
                          ✏️ Modificar
                          🛑 Bloquear
                          📋 Salvar
```

### Casos de uso de um proxy

- **Cache** (acelerar acessos repetidos)
- **Controle de acesso corporativo** (filtragem)
- **Anonimato** (VPN/proxy reverso)
- **🎯 Pentest** — **interceptar, inspecionar e modificar** requisições em trânsito

O Burp usa essa posição para **mostrar e editar tudo** que o navegador envia ao servidor.

## ⚙️ Etapa 3 — Configurando a Interceptação

### Botões essenciais da aba Proxy → Intercept

| Botão | Função |
| ----- | ------ |
| **Intercept is on/off** | Liga/desliga a captura |
| **Forward** | Envia a requisição (modificada ou não) para o servidor |
| **Drop** | Descarta a requisição |
| **Open Browser** | Abre Chromium já configurado com o proxy do Burp |

> ⚠️ **Atenção:** com **Intercept ON**, qualquer página fica carregando até você clicar em **Forward**. Para navegar normalmente, desligue.

## 🔬 Etapa 4 — Capturando uma Requisição Simples (Login)

Após clicar em **Open Browser**, acessamos `My Account` e fazemos login com `wiener:peter` com a interceptação **ligada**.

O Burp captura:

```http
POST /login HTTP/1.1
Host: <lab-id>.web-security-academy.net
Cookie: session=YkAslORaw46PZuLT82t8rLwZAnBAFeuS
Content-Length: 68

csrf=LiZxlebKjnvkpuEDoYisCCpFgbl9HmDq&username=wiener&password=peter
```

**O que aprendemos da requisição:**
- Método: `POST`
- Endpoint: `/login`
- **Cookie de sessão** já existia (sessão anônima)
- Corpo: `csrf` + `username` + `password` em form-urlencoded

## 🍪 Etapa 5 — A Descoberta-Chave: o Cookie `Admin`

Após o login, fazemos uma ação qualquer (mudar e-mail) e o Burp mostra:

```http
POST /my-account/change-email HTTP/1.1
Host: <lab-id>.web-security-academy.net
Cookie: Admin=false; session=ohc3KdUlUHONEbtnft4ktXCMlt1CcKDtnpM9a
Content-Type: application/x-www-form-urlencoded

email=email%40email.com&csrf=5vDl3N8lRmbrSCLXAhjpPKfMtOxrMl
```

🚨 **Olhem o cookie!**

```
Cookie: Admin=false; session=...
```

O servidor está mandando **um cookie que define quem é admin** — e o navegador devolve esse cookie a cada requisição. Isso é uma **bandeira vermelha** óbvia: se o cliente controla esse valor, basta mudá-lo.

---

## 💥 Etapa 6 — Explorando a Falha

### Passo 1: tentar acessar `/admin` com cookie modificado

Com interceptação **ligada**, digitamos `<lab-id>/admin` na URL e, ao capturar a requisição no Burp, **editamos o cookie**:

```diff
- Cookie: Admin=false; session=ohc3...
+ Cookie: Admin=true;  session=ohc3...
```

Clicamos em **Forward** → a página `/admin` carrega! ✅

### Passo 2: deletar o usuário `carlos`

Clicar em **Delete** dispara uma nova requisição. O Burp captura:

```http
GET /admin/delete?username=carlos HTTP/1.1
Cookie: Admin=false; session=...
```

O cookie **voltou a `false`** automaticamente (o servidor reseta a cada hop). Editamos novamente:

```
Cookie: Admin=true; session=...
```

**Forward** → usuário `carlos` deletado → lab resolvido. 🎯

> ✅ *Congratulations, you solved the lab!*

## 🧠 Análise da Vulnerabilidade

### Onde está a falha?

```
┌─────────────────────────────────────────────────────────┐
│  ❌ ERRADO — confiar em cookie controlado pelo cliente   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Servidor: "Recebi Cookie: Admin=true. Beleza, é admin."│
│  Cliente:  "Hahaha, eu mesmo escrevi esse cookie."      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### O cookie de privilégio:
1. É enviado pelo **cliente** em toda requisição
2. Pode ser visto e modificado por qualquer ferramenta (Burp, DevTools, extensão)
3. **Nunca deve** carregar decisões de autorização

### Princípio violado

> **Never trust the client.**
> O navegador é território hostil. Qualquer dado que o cliente possa modificar precisa ser **revalidado no servidor** com base em uma fonte confiável (sessão no servidor, JWT assinado, banco de dados).

## 🛡️ Como Deveria Ser Implementado

### ❌ Errado (o que o lab faz)

```javascript
// Pseudocódigo do servidor vulnerável
app.get('/admin', (req, res) => {
  if (req.cookies.Admin === 'true') {  // 🚨 confia no cliente
    return res.render('admin-panel');
  }
  return res.status(403).send('Forbidden');
});
```

### ✅ Certo

```javascript
// Sessão guardada NO SERVIDOR (ex: Redis, banco)
// O cookie é apenas um ID opaco que aponta para a sessão real

app.get('/admin', requireAuth, async (req, res) => {
  const user = await db.users.findById(req.session.userId);
  if (!user.isAdmin) {                  // ✅ verdade vem do banco
    return res.status(403).send('Forbidden');
  }
  return res.render('admin-panel');
});
```

### Diferenças-chave

| Errado | Certo |
| ------ | ----- |
| Cookie carrega o privilégio (`Admin=true`) | Cookie carrega só um ID de sessão opaco |
| Cliente decide a permissão | Servidor consulta o banco para decidir |
| Modificável com 1 clique no Burp | Modificar não muda a permissão real |
| Sem assinatura | Sessão server-side OU JWT assinado |

---

## 📋 Checklist — Como NÃO usar cookies

- [ ] **Nunca** colocar `Admin`, `Role`, `Permissions`, `IsAdmin`, `UserId` editável no cookie
- [ ] Cookies de sessão devem ser **IDs opacos aleatórios** (ex: 128 bits de entropia)
- [ ] Estado real (papel, permissões) fica no **servidor** (Redis, DB, memória)
- [ ] Se precisar de stateless → **JWT assinado** com `HS256`/`RS256`, validando a assinatura **em cada requisição**
- [ ] Flags obrigatórias: `HttpOnly`, `Secure`, `SameSite=Lax` (ou `Strict`)
- [ ] Expiração curta + refresh tokens
- [ ] Renovar (rotacionar) ID de sessão após login

## 🎓 Conceitos Aprendidos

### Burp Suite
| Termo | Definição |
| ----- | --------- |
| **Proxy interceptador** | Posicionado entre cliente e servidor para inspecionar/modificar tráfego |
| **Intercept** | Modo que congela cada requisição até decisão manual |
| **Forward** | Libera a requisição para o servidor |
| **Drop** | Descarta a requisição |

### Web/HTTP
| Termo | Definição |
| ----- | --------- |
| **Cookie** | Pequeno dado armazenado no navegador, enviado em toda requisição ao mesmo domínio |
| **Session cookie** | Cookie que identifica uma sessão de usuário (deve ser opaco) |
| **CSRF token** | Token anti-falsificação de requisição entre sites |
| **HTTP Method** | `GET`, `POST`, `PUT`, `DELETE`, etc. — define a intenção da requisição |

### Segurança
| Termo | Definição |
| ----- | --------- |
| **Privilege Escalation** | Ganhar privilégios maiores que os atribuídos |
| **Client-side Trust** | Anti-padrão de confiar em dados controlados pelo cliente |
| **Tampering** | Modificação maliciosa de dados em trânsito |

> A transcrição menciona: *"existe outra ferramenta para alterar o Cookie, porque o Burp precisa atualizar a cada requisição."*

Essa ferramenta é provavelmente:

- **Cookie Editor** (extensão do Chrome/Firefox) — edita cookies direto no navegador, persistente
- **Burp Repeater** — reenvia uma única requisição N vezes com modificações sem precisar interceptar tudo

## 💡 Reflexão Final

> *"Tudo que vem do cliente é dado, não verdade.*
> *Cookies, headers, parâmetros, JSON do body — qualquer um pode ser forjado.*
> *Autorização sempre acontece no servidor, com base em estado que o cliente não controla."*

Memorize esse mantra e você terá imunidade contra a maior parte das falhas A01 do OWASP Top 10.
