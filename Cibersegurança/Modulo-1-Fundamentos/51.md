# 🏛️ Os 8 Domínios de Segurança do CISSP — Guia de Estudo

## 📖 O que é o CISSP?

O **CISSP** (*Certified Information Systems Security Professional*) é uma das certificações mais respeitadas do mundo em cibersegurança, mantida pelo **(ISC)²**. Sua estrutura divide o conhecimento em **8 domínios** que cobrem **todas as áreas de atuação** de um profissional de segurança.

> 💡 **Por que estudar isso?** Os 8 domínios funcionam como um **mapa mental** da carreira. Cada vaga de segurança (analista SOC, pentester, arquiteto, GRC, DevSecOps) se encaixa em um ou mais desses domínios.

---

## 🗺️ Visão Geral dos 8 Domínios

| # | Domínio | Foco em uma frase |
|---|---------|-------------------|
| 1 | **Security and Risk Management** | Políticas, conformidade, riscos, ética |
| 2 | **Asset Security** | Proteção de dados e ativos (ciclo de vida) |
| 3 | **Security Architecture and Engineering** | Como projetar sistemas seguros |
| 4 | **Communication and Network Security** | Redes, comunicação, infraestrutura |
| 5 | **Identity and Access Management (IAM)** | Quem é você + ao que pode acessar |
| 6 | **Security Assessment and Testing** | Pentest, auditoria, validação |
| 7 | **Security Operations** | Resposta a incidentes, SOC, forense |
| 8 | **Software Development Security** | Segurança no SDLC, código seguro |

## 1️⃣ Security and Risk Management — Gerenciamento de Segurança e Riscos

### Foco
Definir a **postura de segurança** da organização: o quanto ela consegue se defender e reagir a mudanças.

### Componentes-chave
- 🎯 Objetivos e metas de segurança
- 📉 Processos de **redução de risco**
- ⚖️ **Compliance** (LGPD, GDPR, HIPAA, PCI-DSS)
- 🔄 **Planos de Continuidade de Negócios (BCP)** e Disaster Recovery
- 📜 Reguladores legais
- 🤝 Ética profissional e organizacional

### InfoSec — processos de design
- Resposta a incidentes
- Gerenciamento de vulnerabilidade
- Segurança de aplicativos
- Segurança em nuvem
- Segurança de infraestrutura

### Exemplo prático
> Uma equipe altera como **PII** (Personally Identifiable Information) é tratada para se adequar ao **GDPR** da União Europeia.

### Cargos típicos
**CISO**, GRC Analyst, Risk Manager, Compliance Officer

---

## 2️⃣ Asset Security — Segurança de Ativos

### Foco
Gerenciar o **ciclo de vida** dos dados e recursos da organização: armazenamento, manutenção, retenção e destruição.

### Componentes-chave
- 📊 Análise de impacto na segurança
- 💾 Plano de **recuperação de dados** (backups)
- 🗑️ Política de **destruição segura** de dados
- 🏷️ **Classificação de ativos** (público, interno, confidencial, restrito)

### Estados dos dados (sempre cobrado em prova)
| Estado | Definição | Exemplo de proteção |
|--------|-----------|---------------------|
| **At rest** (em repouso) | Armazenado em disco | Criptografia AES-256 |
| **In transit** (em trânsito) | Viajando pela rede | TLS 1.3, VPN |
| **In use** (em uso) | Sendo processado em RAM | Enclaves seguros, TEE |

### Exemplo prático
> Um analista cria **backups diários** de uma base de dados crítica e define política de **retenção de 7 anos** para atender exigência fiscal.

### Cargos típicos
Data Protection Officer (DPO), Data Custodian

---

## 3️⃣ Security Architecture and Engineering — Arquitetura e Engenharia de Segurança

### Foco
**Projetar** sistemas, ferramentas e processos seguros desde a base.

### Princípios de design (decorar!)
| Princípio | Explicação |
|-----------|------------|
| **Threat Modeling** | Mapear ameaças antes de construir |
| **Least Privilege** | Conceder o mínimo de acesso necessário |
| **Defense in Depth** | Múltiplas camadas de proteção |
| **Fail Secure** | Em caso de falha, o padrão é negar |
| **Separation of Duties** | Nenhum indivíduo tem poder total |
| **Keep It Simple** (KISS) | Complexidade gera bugs e brechas |
| **Zero Trust** | Nunca confiar, sempre verificar |
| **Trust but Verify** | Confiar baseado em evidência contínua |

### Conceito-chave: Responsabilidade Compartilhada
**Todos** os envolvidos no projeto de um sistema têm papel ativo na redução de risco — não é "trabalho só da equipe de segurança".

### Exemplo prático
> Usar um **SIEM** para monitorar tentativas de login incomuns que possam indicar um atacante tentando acessar dados privados.

### Cargos típicos
Security Architect, Security Engineer, Cloud Security Engineer

---

## 4️⃣ Communication and Network Security — Comunicação e Segurança de Rede

### Foco
Proteger redes físicas, sem fio, locais, remotas e em nuvem.

### Desafios típicos
- Trabalho **remoto** e **híbrido** — funcionário acessando de casa/café
- Conexões externas inseguras
- Ataques **man-in-the-middle** em redes públicas

### Controles principais
- 🔐 **VPN** para acesso remoto
- 🚧 **Firewalls** (stateful e stateless)
- 🛂 **Network segmentation** (DMZ, VLAN)
- 📡 Proteção Wi-Fi (WPA3)
- 🚦 **NAC** (Network Access Control)

### Exemplo prático
> Funcionário viajando acessa o sistema interno **via VPN corporativa**, com MFA, mantendo o tráfego criptografado mesmo em Wi-Fi público.

### Cargos típicos
Network Security Engineer, SOC Analyst (turno de rede)

---

## 5️⃣ Identity and Access Management (IAM) — Gerenciamento de Identidade e Acesso

### Foco
**Quem é o usuário** (autenticação) e **ao que ele tem direito** (autorização).

### Os 4 As do IAM (modelo AAAA)
| Componente | Pergunta que responde |
|------------|----------------------|
| **Authentication** | Você é quem diz ser? |
| **Authorization** | Você pode fazer isso? |
| **Accounting** | O que você fez? (logs) |
| **Auditing** | Quem fez o quê, quando? |

### Princípio central: Least Privilege
Conceder **somente** o acesso mínimo necessário para concluir uma tarefa — e **revogar** quando não for mais preciso.

### Exemplo prático
> Um atendente de suporte só vê o telefone do cliente **enquanto resolve o chamado**. Quando o ticket fecha, o acesso é automaticamente removido.

### Conexão com [OWASP Top 10](47.md)
- **A01 — Broken Access Control** ([49.md](49.md), [50.md](50.md))
- **A07 — Identification and Authentication Failures**

### Cargos típicos
IAM Engineer, Identity Architect

---

## 6️⃣ Security Assessment and Testing — Avaliação e Teste de Segurança

### Foco
**Validar** que as defesas funcionam. Identificar riscos, ameaças e vulnerabilidades antes que atacantes o façam.

### Atividades típicas
- 🎯 **Pentest** (teste de penetração)
- 📋 **Auditorias de segurança**
- 🔍 **Vulnerability Assessment**
- 🐛 **Bug Bounty** ([48.md](48.md))
- 👥 Red Team / Blue Team / Purple Team
- 📊 Coleta e análise de dados de segurança

### Exemplo prático
> Auditar **permissões dos usuários** trimestralmente para garantir que ninguém acumulou acessos indevidos ao longo do tempo (*privilege creep*).

### Cargos típicos
Pentester, Red Team Operator, Security Auditor, Bug Bounty Hunter

---

## 7️⃣ Security Operations — Operações de Segurança

### Foco
**Detectar, responder e recuperar** de incidentes. É a "linha de frente" da segurança.

### Ferramentas e processos
- 📚 **Treinamento e conscientização**
- 📝 Relatórios e documentação
- 🚨 **IDS/IPS** (detecção/prevenção de intrusão)
- 🔭 **SIEM** (Splunk, ELK, Chronicle)
- 📜 Gerenciamento de logs
- 🆘 **Gerenciamento de incidentes**
- 📖 **Playbooks** (runbooks de resposta)
- 🔬 **Análise forense pós-violação**
- 🎓 Reuniões de **lições aprendidas**

### NIST Incident Response Lifecycle
```
┌──────────────┐   ┌─────────────────┐   ┌──────────────────┐   ┌──────────────┐
│ Preparation  │ → │ Detection &     │ → │ Containment,     │ → │ Post-Incident│
│              │   │ Analysis        │   │ Eradication,     │   │ Activity     │
│              │   │                 │   │ Recovery         │   │              │
└──────────────┘   └─────────────────┘   └──────────────────┘   └──────────────┘
```

### Exemplo prático
> O SOC detecta **grande volume de dados** sendo exfiltrado às 3h da manhã. A equipe **isola** o host comprometido, **erradica** o malware e **restaura** o ambiente a partir do backup.

### Cargos típicos
SOC Analyst (Tier 1/2/3), Incident Responder, Forensic Analyst, Threat Hunter

## 8️⃣ Software Development Security — Segurança de Desenvolvimento de Software

### Foco
Embutir segurança em **cada etapa** do ciclo de vida de desenvolvimento (SDLC).

### Princípio central
> **A segurança não pode ser um pensamento tardio (*afterthought*).** Precisa estar no design, no código, nos testes e no deploy.

### O SDLC seguro
```
┌─────────┐   ┌────────┐   ┌─────────────┐   ┌────────┐   ┌────────┐   ┌──────────┐
│ Design  │ → │  Code  │ → │ Build/Test  │ → │ Deploy │ → │Operate │ → │ Retire   │
└────┬────┘   └───┬────┘   └──────┬──────┘   └────┬───┘   └────┬───┘   └────┬─────┘
     │            │               │                │            │            │
   Threat       Secure          SAST/DAST       IaC scan      Monitor      Secure
   Model        Coding          Pentest         Secrets       SIEM         Wipe
                Review          SCA                                        Decommission
```

### Atividades-chave
- 🧠 **Threat Modeling** (STRIDE, PASTA)
- ✍️ **Code Review** (manual + automatizado)
- 🔎 **SAST** (Static Application Security Testing)
- 🏃 **DAST** (Dynamic Application Security Testing)
- 📦 **SCA** (Software Composition Analysis — dependências)
- 🎯 **Pentest** antes do release
- ✅ **OWASP Top 10** como referência ([47.md](47.md))

### Exemplo prático
> Analista de uma farmacêutica valida que a **criptografia** está corretamente configurada em um novo dispositivo médico que armazenará dados privados de pacientes.

### Cargos típicos
AppSec Engineer, DevSecOps Engineer, Secure Code Reviewer

## 💡 Reflexão Final

> Os 8 domínios não são silos isolados — são **lentes diferentes** sobre o mesmo problema: proteger pessoas, dados e sistemas.
>
> Um incidente real normalmente atravessa vários domínios: começa em **8** (bug no código) → explora **5** (escalação de privilégio) → propaga em **4** (rede) → é detectado em **7** (SOC) → vira lição em **1** (políticas atualizadas).

Conhecer os 8 domínios te dá o **vocabulário** para conversar em qualquer mesa de cibersegurança — do CISO ao pentester.
