## 🔁 Etapa 4 — Enviando para o Repeater

Aqui está o pulo do gato:

1. **Clique direito** na requisição capturada
2. Selecione **Send to Repeater** (atalho: `Ctrl+R`)
3. Vá para a aba **Repeater** (no topo do Burp)
4. Clique em **Send** para enviar a requisição

### O que mudou?
- O Repeater mostra **Request** (esquerda) e **Response** (direita)
- Você pode **modificar e reenviar** quantas vezes quiser **sem precisar interceptar** novas requisições do navegador

### Por que o Repeater é melhor que o Intercept?

| Intercept | Repeater |
|-----------|----------|
| Captura **todas** as requisições | Trabalha com **uma** específica |
| Precisa do **navegador** para gerar a requisição | Reenvia direto, sem navegador |
| Trava a navegação enquanto está ON | Não afeta o navegador |
| Bom para **descobrir** uma requisição | Bom para **experimentar** com ela |

> 🧠 **Fluxo padrão de pentest:** `Intercept` (descobrir) → `Send to Repeater` (experimentar) → `Send to Intruder` (automatizar)

## 💥 Etapa 6 — A Exploração: Mass Assignment

Agora vem o ataque. Modificamos o **Request** (esquerda) adicionando o campo `roleid` no body — campo que **o formulário nunca enviou**:

### Antes (request original):

```http
POST /my-account/change-email HTTP/1.1
Content-Type: application/json

{
  "email": "novo@email.com"
}
```

### Depois (request modificada — ataque):

```http
POST /my-account/change-email HTTP/1.1
Content-Type: application/json

{
  "email": "novo@email.com",
  "roleid": 2
}
```

Clique em **Send** novamente.

**Resultado:** a Response mostra `"roleid": 2` 🎯

O servidor aceitou o campo extra e **promoveu o `wiener` a admin**.

## 🧠 O que é Mass Assignment?

> **Mass Assignment** acontece quando uma aplicação **aceita automaticamente** todos os campos do body do usuário e os atribui aos atributos internos do objeto — **sem filtrar** quais campos podem ser modificados.

### Exemplo do problema em código

#### ❌ Errado (vulnerável a Mass Assignment)

```javascript
// Express + Sequelize — anti-padrão
app.post('/my-account/change-email', async (req, res) => {
  const user = await User.findByPk(req.session.userId);
  await user.update(req.body);  // 🚨 atualiza QUALQUER campo enviado
  res.json(user);
});
```

Se o usuário enviar `{ "email": "...", "roleid": 2, "balance": 999999 }`, **tudo** é gravado.

#### ✅ Certo (allowlist explícita)

```javascript
app.post('/my-account/change-email', async (req, res) => {
  const user = await User.findByPk(req.session.userId);
  // Só permite atualizar campos da allowlist
  const { email } = req.body;
  await user.update({ email });
  res.json({ username: user.username, email: user.email });
});
```

Ou com DTO/schema validation (Zod, Joi, class-validator):

```javascript
const schema = z.object({ email: z.string().email() }).strict();
const data = schema.parse(req.body);  // rejeita campos extras
await user.update(data);
```

## 🛡️ Defesas Contra Mass Assignment

| Defesa | Como aplicar |
|--------|--------------|
| **Allowlist de campos** | Listar explicitamente quais campos podem ser atualizados |
| **DTO com validação estrita** | Schema que **rejeita** campos não declarados (`.strict()`) |
| **Separar endpoints** | `/change-email` só muda email; `/change-role` só muda role e exige auth de admin |
| **Princípio do menor privilégio** | Mesmo se o atacante mandar `roleid: 2`, o endpoint não deveria poder alterar role |
| **Code review** | Procurar por `update(req.body)`, `assign(req.body)`, `Object.assign(user, req.body)` |
| **Response filtering** | Não devolver campos sensíveis (`roleid`, `isAdmin`) na resposta |

---

## 📋 Checklist — Endpoints Seguros

- [ ] Cada endpoint atualiza **apenas** os campos pertinentes ao seu propósito
- [ ] Validação por schema com **modo estrito** (`additionalProperties: false`)
- [ ] Campos sensíveis (`role`, `permissions`, `isAdmin`) **nunca** atualizáveis via endpoint de usuário
- [ ] Endpoint de mudança de role exige autorização de admin **e** auditoria
- [ ] Não devolver campos internos na response (`roleid`, `passwordHash`, `apikey` interno)
- [ ] Testes automatizados que tentam injetar campos extras

## 🎓 Conceitos Aprendidos

| Termo | Definição |
|-------|-----------|
| **Mass Assignment** | Atualizar todos os campos do body sem filtrar — anti-padrão |
| **Privilege Escalation** | Ganhar privilégios maiores que os atribuídos (continuação de [50.md](50.md)) |
| **Burp Repeater** | Aba do Burp que reenvia uma requisição modificada N vezes |
| **Send to Repeater** | Ação que move a requisição do Intercept para o Repeater (Ctrl+R) |
| **Allowlist (whitelist)** | Lista explícita do que é **permitido** (oposto: blocklist) |
| **DTO** | Data Transfer Object — molde que define a forma esperada dos dados |
| **Information Leakage via Response** | Resposta expor campos sensíveis (como `roleid`) — facilita o ataque |

## 💡 Reflexão Final

> *"Cada campo extra que sua API aceita sem validar é uma porta para escalação de privilégio.*
> *Allowlist, não blocklist. DTO estrito, não 'update tudo que vier'."*

Mass Assignment foi a vulnerabilidade do **CVE-2012-2054** que comprometeu o GitHub em 2012 — um pesquisador conseguiu **adicionar sua chave pública** ao repositório do Rails enviando um campo extra. A lição não envelheceu: ainda é uma das falhas mais comuns em APIs modernas.
