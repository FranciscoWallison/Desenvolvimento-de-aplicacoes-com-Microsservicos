# 🎯 Quiz Consolidado: Acessando a Página de Administrador

> **Cenário:** Carol é pentester e recebeu uma aplicação para testar. Após a etapa de **varredura (reconnaissance)**, descobriu que existe uma página de administrador e deseja explorá-la mesmo sem ter acesso. O objetivo é validar se o **controle de acesso** está implementado corretamente.
>
> **Pergunta:** Existem diversas formas de tentar acessar a página de administrador. Quais cabem a ela testar? **(Selecione 3)**

## ✅ Alternativa A — Força Bruta de URLs (**CORRETA**)

> *"A aplicação pode ter nenhuma proteção de acesso, nesse caso, Carol pode trabalhar por meio de **força bruta**, onde ela pode testar uma lista de nomes relacionados à página de administrador na própria URL e tentar acessar."*

### 💡 O que é isso? — Directory Enumeration

A técnica se chama **directory busting** ou **directory enumeration**. Funciona assim:

1. Você tem uma lista de palavras comuns (`admin`, `administrator`, `dashboard`, `panel`, `console`, `backend`, `cp`, `manage`...)
2. Uma ferramenta automatiza requisições: `GET /admin`, `GET /administrator`, etc.
3. Analisa as respostas: `200 OK` = encontrou; `404 Not Found` = não existe
4. Compila uma lista de páginas **ocultas mas acessíveis**

### 🛠️ Ferramentas principais

| Ferramenta | Linguagem | Uso |
|------------|-----------|-----|
| **Gobuster** | Go | Mais rápido e moderno |
| **ffuf** | Go | Fuzzing flexível |
| **dirb** / **dirbuster** | C / Java | Clássicos, vêm no Kali |
| **wfuzz** | Python | Muito customizável |

### Exemplo prático — Gobuster

```bash
# Sintaxe básica
gobuster dir -u https://alvo.com -w /usr/share/wordlists/dirb/common.txt

# Com extensões e threads
gobuster dir \
  -u https://alvo.com \
  -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt \
  -x php,html,js \
  -t 50 \
  -o resultado.txt
```

**Saída esperada:**
```
/admin                (Status: 301) [Size: 178] [--> /admin/]
/login                (Status: 200) [Size: 4521]
/api                  (Status: 401) [Size: 32]
/backup.zip           (Status: 200) [Size: 1024]   ← achado importante!
/.git/HEAD            (Status: 200) [Size: 23]     ← achado crítico!
```

### 📚 Wordlists populares (já incluídas no Kali)

```
/usr/share/wordlists/
├── dirb/
│   ├── common.txt           ← rápido, 4.6k palavras
│   └── big.txt              ← mais completo
├── dirbuster/
│   ├── directory-list-2.3-small.txt
│   └── directory-list-2.3-medium.txt   ← padrão profissional
└── SecLists/                ← coleção massiva (instalar com apt)
    └── Discovery/Web-Content/
```

### ⚠️ Quando usar (e quando NÃO)

✅ **Use quando:**
- Está em pentest **autorizado** com escopo definido
- Está em CTF / laboratório (PortSwigger, HackTheBox, TryHackMe)
- Está validando sua própria aplicação

❌ **NÃO use:**
- Em sistemas sem autorização (é crime — Lei 12.737/2012 no Brasil)
- Sem rate limiting (pode derrubar o servidor — vira DoS acidental)

## ❌ Alternativa B — Interrogar Funcionários (**ERRADA**)

> *"Investigar os funcionários da empresa que desenvolveu a aplicação para deduzir quem deles é o administrador e interrogá-lo sobre a página e funções de administrador."*

### 💡 Por que está errada?

**Múltiplos problemas:**

1. **Foge do escopo técnico** — pentest de aplicação web testa a **aplicação**, não pessoas
2. **"Interrogar" é abusivo** — funcionários não devem nem podem ser pressionados
3. **Engenharia social tem regras** — se for parte do contrato (red team engagement), exige autorização **explícita** e **escopo definido**
4. **Pode violar leis** — assédio, coação, LGPD (dados pessoais)

### 🎓 Como seria a versão CORRETA (engenharia social legítima)

Se o escopo do pentest **incluir** engenharia social:
- **Phishing simulado autorizado** (com consentimento da direção)
- **OSINT passivo** — pesquisar perfis públicos no LinkedIn
- **Pretexting documentado** — com playbook aprovado pela empresa

> 🔗 Conexão com [10.md](10.md), [15.md](15.md), [46.md](46.md) — phishing/engenharia social como ataques **éticos** quando autorizados.

## ✅ Alternativa C — Modificar Cookies / IDs (**CORRETA**)

> *"Algumas aplicações fazem o controle de acesso por meio de funções do usuário ao fazer login e armazenam informações em IDs em arquivos JSON ou cookies. Carol pode cadastrar uma conta normal, fazer login e modificar seus cookies ou ID para administradora."*

### 💡 Por que está correta?

Essa é exatamente a técnica que vimos nos labs:

| Local da falha | Lab |
|----------------|-----|
| Cookie `Admin=true/false` no header | [50.md](50.md) |
| Campo `roleid` no body JSON (Mass Assignment) | [54.md](54.md) |

### 🛠️ Ferramentas

- **Burp Suite Proxy** — intercepta e modifica em tempo real ([50.md](50.md))
- **Burp Repeater** — reenvia requisições modificadas ([54.md](54.md))
- **Cookie Editor** (extensão de navegador) — edita cookies persistentes
- **DevTools** — vê e altera cookies localmente

### Vulnerabilidades OWASP ligadas

- **A01** — Broken Access Control
- **A04** — Insecure Design (quando o sistema **foi projetado** para confiar no cliente)

## ✅ Alternativa D — Inspecionar Código-Fonte (**CORRETA**)

> *"Carol pode inspecionar a aplicação no próprio navegador e buscar no script da página a URL em códigos."*

Exemplo dado:

```html
<script>
  var isAdmin = false;
  if (isAdmin) {
    var adminPanelTag = document.createElement('a');
    adminPanelTag.setAttribute('https://example.com/admin-interface');
    adminPanelTag.innerText = 'Admin interface';
  }
</script>
```

### 💡 Por que está correta?

Tudo que é enviado ao navegador é **público por definição** — incluindo JavaScript com URLs sensíveis.

Já vimos isso em ação em [49.md](49.md), onde encontramos a URL `/admin-dk7fvh` escondida em um `<script>`.

### 🔍 O que procurar no código-fonte

| Padrão | Significado |
|--------|-------------|
| `var isAdmin = false` + `if (isAdmin)` | Controle de UI escondendo links sensíveis |
| `fetch('/api/admin/...')` | Endpoints internos referenciados |
| URLs hard-coded em arrays/objetos | Mapa de rotas vazado |
| Comentários `// TODO: remove before prod` | Código de debug esquecido |
| API keys, tokens, chaves | Credenciais expostas |
| Arquivos `.map` (source maps) | Código-fonte original reconstruível |

### Onde inspecionar

1. **DevTools → Sources / Debugger** — ver todo o JS carregado
2. **View Source** (`Ctrl+U`) — HTML completo
3. **DevTools → Network → JS** — todos os scripts baixados
4. **Burp → Target → Site map** — mapa completo de tudo capturado

## ⚖️ Ética e Legalidade (sempre relevante)

> Toda técnica desta página é **legítima em pentest autorizado**.
> Sem autorização, é **crime** no Brasil (Lei nº 12.737/2012 — "Lei Carolina Dieckmann").

**Antes de qualquer teste, garanta:**
- [ ] **Autorização escrita** do dono da aplicação
- [ ] **Escopo definido** (IPs, domínios, horários)
- [ ] **Contrato/NDA** assinado
- [ ] **Rules of Engagement** documentadas

## 💡 Reflexão Final

> *"O atacante explora cada camada onde a aplicação confiou demais:*
> *- Confiou no `robots.txt` para 'esconder' rotas → leu robots.txt*
> *- Confiou no front-end para esconder links → leu o JavaScript*
> *- Confiou no cookie do cliente para autenticar → editou o cookie*
> *- Confiou no body para atualizar só os campos do formulário → injetou roleid*
> *- Confiou que ninguém adivinharia URLs → fez fuzzing com Gobuster"*
>
> **Cada técnica deste quiz ataca uma confiança mal colocada. A defesa começa onde a confiança em dados do cliente termina.**
