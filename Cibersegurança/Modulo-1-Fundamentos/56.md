## 📖 As 5 Fases do Pentest

Antes de entrar nas ferramentas, vale fixar onde elas se encaixam:

```
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐
│ 1. Reconhecimento│ → │ 2. Varredura     │ → │ 3. Exploração    │ → │ 4. Manter Acesso │ → │ 5. Cobrir Rastros│
│                  │   │                  │   │                  │   │                  │   │                  │
│ Coletar info     │   │ Scanear ativos,  │   │ Explorar         │   │ Backdoors        │   │ Limpar logs      │
│ passivamente     │   │ portas, serviços │   │ vulnerabilidades │   │ persistência     │   │ (em red team)    │
└──────────────────┘   └──────────────────┘   └──────────────────┘   └──────────────────┘   └──────────────────┘
        ▲                       ▲
        │                       │
   Wappalyzer              Wapiti, Gobuster
   OSINT                   Burp Scanner, Nikto
```

> 💡 As **fases 1 e 2** são onde 70% do trabalho real acontece. Quanto melhor o reconhecimento, mais fácil a exploração.

## 🔍 Fase 1: Reconhecimento (Reconnaissance)

### Wappalyzer — Fingerprinting de Tecnologias

> **O que é:** extensão de navegador que identifica **quais tecnologias** uma aplicação web usa (linguagem, framework, CMS, CDN, analytics, etc.).

### Por que importa?

A primeira etapa de qualquer pentest é **coletar o máximo de informação** sobre a aplicação. Saber a stack tecnológica permite focar em **vulnerabilidades específicas** daquele ecossistema.

**Exemplo real:**
> Se Wappalyzer mostra que o site usa **Java**, você imediatamente pensa em:
> - **Log4Shell** (CVE-2021-44228) — vulnerabilidade crítica do Log4j
> - **Spring4Shell** (CVE-2022-22965) — RCE no Spring Framework
> - **Struts** (CVE-2017-5638) — RCE histórico do Apache Struts
>
> Se mostra **WordPress**, foca em:
> - Plugins desatualizados
> - Arquivos `wp-config.php` expostos
> - Endpoint `xmlrpc.php`

### Instalação

| Navegador | Link |
|-----------|------|
| Chrome | [Chrome Web Store](https://chrome.google.com/webstore/detail/wappalyzer-technology-pro/gppongmhjkpfnbhagpmjfkannfbllamg) |
| Firefox | [Mozilla Add-ons](https://addons.mozilla.org/pt-BR/firefox/addon/wappalyzer/) |
| Edge | [Microsoft Add-ons](https://microsoftedge.microsoft.com/addons/detail/wappalyzer-technology-p/mnbndgmknlpdjdnjfmfcdjoegcckoikn) |

### O que ele detecta

| Categoria | Exemplos |
|-----------|----------|
| **Linguagem** | PHP, Python, Ruby, Java, Node.js |
| **Framework** | Django, Rails, Spring, Laravel, Express |
| **CMS** | WordPress, Drupal, Joomla, Shopify |
| **JavaScript** | React, Vue, Angular, jQuery |
| **Web Server** | Nginx, Apache, IIS, Cloudflare |
| **CDN** | Cloudflare, Akamai, Fastly |
| **Analytics** | Google Analytics, Hotjar, Mixpanel |
| **Banco de dados** | MySQL, PostgreSQL, MongoDB (quando exposto) |

### ⚠️ Limitação

> Em algumas aplicações o Wappalyzer **não consegue identificar tudo** — especialmente se há WAF/CDN escondendo o backend (Cloudflare mascara muito).

### Alternativas ao Wappalyzer

- **BuiltWith** ([builtwith.com](https://builtwith.com)) — versão web
- **WhatRuns** — extensão similar
- **Nikto** (CLI) — scanner com fingerprinting
- **whatweb** (CLI, vem no Kali) — `whatweb https://alvo.com`

## 🔎 Fase 2: Varredura (Scanning)

### Wapiti — Scanner de Vulnerabilidades Web

> **O que é:** scanner automatizado **open source** que faz "auditoria caixa-preta" de aplicações web — testa por vulnerabilidades comuns sem precisar do código-fonte.

### Por que usar?

Quando você pega uma aplicação inicialmente, **dificilmente sabe por onde começar**. O Wapiti faz a triagem inicial: escaneia todas as URLs e dispara payloads conhecidos para identificar falhas.

### Instalação

✅ Já vem **pré-instalado no Kali Linux** — basta abrir o terminal.

Para instalar em outras distros:
```bash
pip install wapiti3
```

### Uso básico

```bash
# Varredura simples
wapiti -u https://alvo.com

# Varredura com módulos específicos
wapiti -u https://alvo.com -m xss,sql,exec,file

# Salvar relatório em HTML
wapiti -u https://alvo.com -f html -o relatorio.html

# Autenticação por cookie
wapiti -u https://alvo.com -c "session=ABC123"

# Profundidade da varredura
wapiti -u https://alvo.com -d 3
```

### Módulos de detecção do Wapiti

| Módulo | Detecta |
|--------|---------|
| `xss` | Cross-Site Scripting (refletido e armazenado) |
| `sql` | SQL Injection |
| `exec` | Command Injection / RCE |
| `file` | Path Traversal / LFI / RFI |
| `xxe` | XML External Entity |
| `ssrf` | Server-Side Request Forgery |
| `redirect` | Open Redirect |
| `permanentxss` | XSS armazenado |
| `csrf` | Cross-Site Request Forgery |
| `htaccess` | Bypass de htaccess |
| `backup` | Arquivos de backup expostos (`.bak`, `.old`) |

### Exemplo de saída

```
[+] Launching module xss
[!] Cross Site Scripting found in https://alvo.com/search.php
  Parameter: q
  Payload: <script>alert(1)</script>

[+] Launching module sql
[!] SQL Injection found in https://alvo.com/products.php
  Parameter: id
  Database: MySQL >= 5.0
```

### ⚠️ Cuidados ao usar scanners automáticos

- **Falsos positivos** — sempre validar manualmente
- **Falsos negativos** — scanner não encontra vulnerabilidades de lógica de negócio
- **Pode derrubar a aplicação** — use `--scope` e `--max-files-per-dir` para limitar
- **Gera muito tráfego** — pode acionar WAF/rate limit

### Alternativas ao Wapiti

| Scanner | Característica |
|---------|---------------|
| **Burp Suite Scanner** (Pro) | Mais preciso, integrado ao proxy |
| **OWASP ZAP** | Open source, alternativa ao Burp |
| **Nikto** | Foco em config errada e arquivos expostos |
| **Nuclei** | Templates customizáveis (YAML), super rápido |
| **Acunetix** | Comercial, muito completo |

## 🍪 Cookie Manager — Editor Persistente de Cookies

> **O que é:** extensão de navegador que permite **visualizar, editar, excluir e exportar** cookies sem precisar abrir DevTools ou proxy.

### Quando usar?

Já vimos em [50.md](50.md) como manipular cookies via **Burp Suite**. O problema do Burp é que ele **reseta** o cookie em cada requisição (precisa reinterceptar sempre).

O **Cookie Manager** resolve isso:
- Edita o cookie **diretamente no navegador**
- A mudança **persiste** entre requisições
- Não precisa de proxy intermediário
- Mais ágil para testes rápidos

### Comparação: Burp vs Cookie Manager

| Característica | Burp Suite | Cookie Manager |
|----------------|-----------|----------------|
| Editar cookie | ✅ (a cada requisição) | ✅ (persistente) |
| Velocidade | Mais lento | Mais rápido |
| Ver requisição completa | ✅ | ❌ |
| Modificar headers além de cookie | ✅ | ❌ |
| Aprender o que está acontecendo | ✅ excelente | ⚠️ caixa-preta |
| Workflow profissional | Padrão de pentest | Quick win |

> 💡 **Fluxo ideal:** use **Burp** para entender a aplicação e mapear cookies. Depois, use **Cookie Manager** para testar variações rápidas sem proxy.

### Instalação

| Navegador | Link |
|-----------|------|
| Chrome | [CookieManager — Cookie Editor](https://chrome.google.com/webstore/detail/cookiemanager-cookie-edit/hdhngoamekjhmnpenphenpaiindoinpo) |
| Firefox | [A Cookie Manager](https://addons.mozilla.org/pt-BR/firefox/addon/a-cookie-manager/) |
| Opera | [Cookie Editor](https://addons.opera.com/pt-br/extensions/details/cookie-editor-2/) |

### Funcionalidades principais

- 👁️ **Visualizar** todos os cookies do site atual
- ✏️ **Editar** valor, expiração, domínio, path
- ➕ **Adicionar** cookies manualmente
- ❌ **Excluir** cookies específicos ou todos
- 📥 **Exportar/Importar** (útil para passar sessão a colegas em time)
- 🔍 **Buscar** cookies por nome/valor

## 🧠 Conceitos Aprendidos

| Termo | Definição |
|-------|-----------|
| **Reconhecimento** (Recon) | Fase de **coletar informação** sobre o alvo |
| **Varredura** (Scanning) | Fase de **enumerar ativos** e procurar vulnerabilidades |
| **Fingerprinting** | Identificar tecnologias/versões pelo "rastro digital" |
| **Caixa-preta** | Teste sem acesso ao código-fonte |
| **Caixa-branca** | Teste com acesso ao código (white-box) |
| **Caixa-cinza** | Híbrido (algum acesso, mas limitado) |
| **WAF** | Web Application Firewall — bloqueia ataques na frente da aplicação |
| **Scanner** | Ferramenta automatizada que testa vulnerabilidades conhecidas |

## ✅ Quiz Mental

1. Você pega um site novo para pentestar. Qual a **primeira** coisa a fazer?
2. Wappalyzer mostra que o site usa Java. Que vulnerabilidade famosa você pesquisa primeiro?
3. Você suspeita de SQL Injection em vários formulários. Como triar rápido?
4. Quer testar 5 valores de `roleid` diferentes sem reabrir o Burp toda vez. Qual ferramenta?
5. Qual a diferença entre **Reconhecimento** e **Varredura**?

<details>
<summary>Ver respostas</summary>

1. **Reconhecimento** — Wappalyzer (tecnologias) + Gobuster (URLs) + inspecionar código-fonte
2. **Log4Shell (CVE-2021-44228)** — Log4j vulnerável a RCE
3. **Wapiti** com módulo `sql`: `wapiti -u https://alvo.com -m sql`
4. **Cookie Manager** (mais rápido) ou **Burp Repeater** (mais controle)
5. **Reconhecimento** é **passivo** (só observar — Wappalyzer, OSINT). **Varredura** é **ativo** (interage com o alvo — Wapiti, Gobuster, Nikto)

</details>

## ⚖️ Lembrete Ético

> Todas essas ferramentas, sem exceção, exigem **autorização** para uso contra alvos que não são seus.
>
> - Rodar Wapiti num site qualquer = ataque ativo (e ilegal)
> - Wappalyzer em sites aleatórios = OK (só lê o que já é público)
> - Gobuster sem permissão = invasão de sistema (Lei 12.737/2012)

Use em **CTFs, labs (PortSwigger, HTB, THM), sistemas próprios** ou em **pentest contratado com escopo formal**.
