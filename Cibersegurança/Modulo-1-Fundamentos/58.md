## 🎯 Contexto: Por Onde Estamos?

Em [57.md](57.md) vimos o **login bypass** — usar SQLi para pular autenticação. Agora subimos um degrau:

> **Missão:** Extrair dados de **outras tabelas** do banco (não só da que está sendo consultada).

Para isso, vamos usar **`UNION SELECT`** — uma técnica que "anexa" o resultado de uma segunda query à query original. Mas antes disso, precisamos descobrir **uma informação crítica**: **quantas colunas** a query original retorna.

---

## 🧠 Por Que Descobrir Colunas? — A Regra do UNION

O operador `UNION` no SQL tem uma **regra rígida**:

> 🔑 **As duas queries unidas devem ter exatamente o mesmo número de colunas.**

```sql
SELECT id, name FROM products              -- 2 colunas ✅
UNION
SELECT username, password FROM users       -- 2 colunas ✅
-- FUNCIONA: ambas têm 2 colunas
```

```sql
SELECT id, name FROM products              -- 2 colunas
UNION
SELECT username, password, email FROM users  -- 3 colunas ❌
-- ERRO: "The used SELECT statements have a different number of columns"
```

**Conclusão:** antes de fazer UNION para extrair dados, **precisamos saber a "largura" da query original**. Daí o lab de hoje.


## 🔍 Etapa 2 — A Técnica `ORDER BY N`

### Como funciona?

O `ORDER BY` aceita **número da coluna** em vez do nome:

```sql
SELECT name, price FROM products ORDER BY 1   -- ordena pela 1ª coluna (name)
SELECT name, price FROM products ORDER BY 2   -- ordena pela 2ª coluna (price)
SELECT name, price FROM products ORDER BY 3   -- ❌ ERRO: só existem 2 colunas
```

**A pegadinha do pentester:**
- Se `ORDER BY N` funciona → existem **pelo menos N colunas**
- Se `ORDER BY N` dá erro → existem **menos que N colunas**

Aumentando `N` progressivamente, encontramos o número exato.

**Resultado:** ✅ página carrega normalmente → **temos pelo menos 1 coluna**

URL:


**Resultado:** ✅ página carrega normalmente → **temos pelo menos 2 colunas**

URL:

```
https://<lab-id>.web-security-academy.net/filter?category=Pets' ORDER BY 3--
```

## ✅ Conclusão: A Query Tem **2 Colunas**

```
ORDER BY 1 → OK ✅
ORDER BY 2 → OK ✅
ORDER BY 3 → ERRO ❌
            ↑
       parou aqui, então o limite é 2
```

> 💡 **Sempre que `ORDER BY N` falha, o número de colunas é `N - 1`.**

---

## 🎓 Alternativa: `UNION SELECT NULL` (técnica equivalente)

Outra forma de descobrir o número de colunas — usando o próprio UNION com `NULL`:

```sql
' UNION SELECT NULL--                  -- testa 1 coluna
' UNION SELECT NULL, NULL--            -- testa 2 colunas
' UNION SELECT NULL, NULL, NULL--      -- testa 3 colunas
```

`NULL` funciona com **qualquer tipo de dado** (string, int, date), evitando outro erro: "operand types must match".

| Técnica | Vantagem | Desvantagem |
|---------|----------|-------------|
| **`ORDER BY N`** | Simples, incremental | Precisa testar 1, 2, 3, 4... |
| **`UNION SELECT NULL`** | Direto ao ponto | Mais "barulhento" nos logs |

Pentesters experientes usam **`ORDER BY`** primeiro (mais discreto) e depois confirmam com **`UNION SELECT NULL`**.

---

## 🧩 Por Que `--` no Final?

O `--` comenta o resto da query original. Sem ele, a query ficaria malformada:

### Sem `--` (errado):

```sql
SELECT name, description FROM products WHERE category = 'Pets' ORDER BY 2' 
                                                                          ↑
                                                          apóstrofo órfão da query original
```

### Com `--` (correto):

```sql
SELECT name, description FROM products WHERE category = 'Pets' ORDER BY 2--' 
                                                                          ↑
                                                              tudo após o -- vira comentário
```

## 🗺️ O Que Vem Pelo Caminho: Anatomia de um UNION Attack

Descobrir as colunas é **a primeira etapa**. O ataque UNION completo segue 4 passos:

```
┌──────────────────────────────────────────────────────────────────┐
│ 1. Descobrir número de colunas (este lab)                        │
│    → ORDER BY 1, 2, 3... até dar erro                            │
│                                                                  │
│ 2. Descobrir quais colunas aceitam string                        │
│    → ' UNION SELECT 'a',NULL--                                   │
│    → ' UNION SELECT NULL,'a'--                                   │
│                                                                  │
│ 3. Descobrir nomes de tabelas e colunas                          │
│    → ' UNION SELECT table_name,NULL                              │
│         FROM information_schema.tables--                         │
│                                                                  │
│ 4. EXTRAIR dados sensíveis                                       │
│    → ' UNION SELECT username,password FROM users--               │
└──────────────────────────────────────────────────────────────────┘
```

> 📌 **Spoiler:** o próximo lab vai cobrir os passos 2-4 (extrair efetivamente os dados das outras tabelas).

## 🧠 Por Que Funciona Esta Vulnerabilidade?

A aplicação **concatena** o parâmetro `category` direto na query:

```javascript
// ❌ Anti-padrão
const query = `SELECT name, description FROM products WHERE category = '${category}'`;
db.execute(query);
```

Resultado quando `category = "Pets' ORDER BY 2--"`:

```sql
SELECT name, description FROM products WHERE category = 'Pets' ORDER BY 2--'
                                                                ↑
                                            input do usuário modificou a query
```

> 🔑 **A causa raiz é sempre a mesma:** concatenação em vez de prepared statement.

---

## 🛡️ Como Prevenir UNION-Based SQL Injection

### ❌ Errado

```javascript
const query = `SELECT name, description FROM products WHERE category = '${category}'`;
```

### ✅ Certo — Prepared Statement

```javascript
const [rows] = await db.execute(
  'SELECT name, description FROM products WHERE category = ?',
  [category]
);
```

Com prepared statement, `Pets' ORDER BY 2--` é tratado como **um único valor literal**. O banco procura por uma categoria com esse nome exato (não acha nada) e devolve lista vazia. **Não dá erro 500, não revela colunas, não permite UNION.**

### Defesas adicionais

| Camada | Prática |
|--------|---------|
| **Validação de input** | Categoria deve estar em allowlist (`Pets`, `Gifts`, etc.) |
| **Permissões do banco** | Conta da aplicação só lê tabela `products`, não tem acesso a `users` |
| **Ocultar erros** | Não retornar 500 detalhado em produção — usar 400 genérico |
| **WAF** | Bloquear padrões como `UNION SELECT`, `ORDER BY \d+--`, `' OR` |

---

## 📋 Checklist do Atacante (e do Defensor)

### Sinais de que a aplicação é vulnerável

- [ ] Parâmetros que aparecem na URL (`?category=`, `?id=`)
- [ ] Apóstrofo único (`'`) causa erro 500
- [ ] `' --` retorna a mesma página que sem o payload
- [ ] `' ORDER BY 1--` funciona
- [ ] `' ORDER BY 999--` dá erro

### O que o defensor verifica

- [ ] Todas as queries usam prepared statements?
- [ ] Há validação de tipo/formato no servidor?
- [ ] Erros do banco são **logados** mas **não expostos** ao usuário?
- [ ] WAF bloqueia padrões de SQLi conhecidos?
- [ ] Conta do banco tem **privilégios mínimos**?

## 🎓 Conceitos Aprendidos

| Termo | Definição |
|-------|-----------|
| **UNION** | Operador SQL que combina resultados de duas queries em uma só |
| **ORDER BY N** | Ordena por número da coluna (1-indexed) |
| **`information_schema`** | Schema padrão que descreve a estrutura do banco (tabelas, colunas) — alvo do passo 3 |
| **Column enumeration** | Processo de descobrir quantas colunas a query original retorna |
| **`NULL` placeholder** | Valor universal aceito em qualquer tipo de coluna |
| **Type juggling** | Adaptar tipos para que o UNION funcione (string vs número vs data) |

## ✅ Quiz Mental

1. Por que precisamos saber o número de colunas **antes** de fazer um UNION attack?
2. Se `ORDER BY 5` funciona mas `ORDER BY 6` dá erro, quantas colunas a query tem?
3. Por que `NULL` é melhor que valores reais no `UNION SELECT` inicial?
4. Qual é o motivo do `--` no final do payload?
5. Prepared statements impedem este ataque. Por quê?

<details>
<summary>Ver respostas</summary>

1. Porque o operador **UNION exige que ambas as queries tenham o mesmo número de colunas**. Sem isso, dá erro de sintaxe
2. **5 colunas** — quando `ORDER BY N` falha, a resposta é `N - 1`
3. Porque `NULL` é compatível com **qualquer tipo** (string, int, date). Valores reais podem dar erro de "tipo incompatível" e atrasar a enumeração
4. Para **comentar** o restante da query original que viria depois do input, evitando sintaxe malformada (apóstrofo órfão)
5. Porque o input passa como **dado**, não como **template**. `Pets' ORDER BY 2--` vira o valor literal de busca, não código SQL executável

## 🎯 Caso Prático — O TCC do Samuel

> **Cenário:** Samuel está fazendo seu **TCC** na área de dados, usando **MySQL**. Ele está estudando SQL e se deparou com o comando `ORDER BY` mas não sabe suas funcionalidades. Quais alternativas você indicaria para ele?
>
> *(Selecione 2)*

### ✅ Gabarito

| Alt | Status | Texto | Por quê |
|-----|--------|-------|---------|
| **A** | ❌ | "ORDER BY... força uma consulta e imprime dados" | Texto confuso; ORDER BY **só ordena** o resultado de uma consulta já existente |
| **B** | ❌ | "ORDER BY permite manipular o banco sem permissão" | ORDER BY **não escala privilégios** — só ordena dados que você já tem permissão de ler |
| **C** | ✅ | "ORDER BY organiza resultados por coluna em ordem crescente/decrescente/alfabética" | **Função legítima** do operador SQL |
| **D** | ✅ | "Por meio do ORDER BY, descobrir quantas colunas via SQL Injection" | **Função ofensiva** — exatamente este lab |

### 💡 A grande sacada do quiz

Este exercício reforça **a essência da segurança ofensiva**:

> **A mesma ferramenta tem usos legítimos (C) e ofensivos (D).**
> **O atacante usa SQL "normal" — só que aplicado num contexto onde a aplicação não esperava.**

### 📊 Comparação: ORDER BY em desenvolvimento vs pentest

| Uso | Quem usa | Para quê |
|-----|----------|----------|
| **Listar produtos por preço crescente** | Dev (Samuel no TCC) | UX — usuário quer ver do mais barato ao mais caro |
| **Listar usuários por data de cadastro** | Dev | Relatório, dashboard |
| **`ORDER BY 1, 2, 3...` em loop** | Pentester | Enumerar colunas para preparar UNION attack |

A **sintaxe** é idêntica. O **contexto** é que define se é desenvolvimento ou ataque.

### 🔄 Outros exemplos do mesmo princípio

| Recurso SQL | Uso legítimo | Uso ofensivo |
|-------------|--------------|--------------|
| **`ORDER BY`** | Ordenar resultados | Enumerar colunas (este lab) |
| **`UNION SELECT`** | Combinar tabelas relacionadas | Extrair dados de outras tabelas |
| **`information_schema`** | Documentação do banco | Mapear estrutura do alvo |
| **Comentários `--` / `#`** | Documentar queries | Anular cláusulas (login bypass) |
| **`SLEEP()` / `BENCHMARK()`** | Performance testing | Blind time-based SQLi |
| **`LOAD_FILE()`** | ETL, importação | Ler `/etc/passwd` se permissão sobrar |

> 🧠 **Princípio do pentester:** *toda função do banco que existe para um propósito legítimo pode ser usada como vetor de ataque quando combinada com input não validado*.

### ❌ Por que A e B estão erradas

#### A — "Forçar uma consulta"

O ORDER BY **não força nada** — ele é uma **cláusula** que vem **dentro** de um `SELECT` para ordenar resultado. Sozinho não faz nada:

```sql
ORDER BY 1     -- ❌ erro de sintaxe; ORDER BY não roda sozinho
SELECT * FROM products ORDER BY 1    -- ✅ correto
```

#### B — "Manipular sem permissão"

Esta é a alternativa mais **perigosa de interpretar errado**. ORDER BY **não dá privilégios**:

- Se você **pode ler** uma tabela → pode ordenar o resultado
- Se você **não pode ler** uma tabela → ORDER BY não muda isso, dará "permission denied"

O que o ORDER BY **revela** numa vulnerabilidade SQLi é informação **estrutural** (quantas colunas) — não acesso a dados que você não tinha. A extração real de dados vem com `UNION SELECT` ou outras técnicas no **próximo passo** do ataque.

### 🎓 Para Samuel (e para você que estuda)

Se você está estudando SQL para desenvolvimento (como o Samuel no TCC), **decore a função C**:

```sql
-- Ordenação crescente (padrão)
SELECT name, price FROM products ORDER BY price;

-- Ordenação decrescente
SELECT name, price FROM products ORDER BY price DESC;

-- Múltiplas colunas
SELECT name, category, price FROM products
ORDER BY category ASC, price DESC;

-- Por número da coluna (cuidado: dificulta manutenção)
SELECT name, price FROM products ORDER BY 2;  -- ordena por price
```

Se você estuda **segurança**, lembre da função D — mas saiba que **ela só funciona porque a aplicação está mal construída**. O ORDER BY em si é inocente.

## 💡 Reflexão Final

> *"Antes de extrair, você enumera.*
> *Antes de enumerar, você sonda.*
> *Antes de sondar, você observa.*
>
> *Em SQL Injection, paciência metódica vale mais que payload genial."*

O `ORDER BY N` é um exemplo perfeito disso: **um operador SQL absolutamente normal** vira ferramenta de reconhecimento quando você sabe **como ler a resposta da aplicação**. A diferença entre "página carrega" e "erro 500" é, nesse contexto, **informação tão valiosa quanto os dados que você quer roubar**.
