## 🛠️ Etapa 1 — O Payload Mágico

Continue no lab da PortSwigger e, com a categoria `Pets` selecionada, modifique a URL para:

```
https://<lab-id>.web-security-academy.net/filter?category=Pets' UNION SELECT * FROM users--
```

### A query final no servidor:

```sql
SELECT * FROM products WHERE category = 'Pets' UNION SELECT * FROM users--'
```

### Por que isso funciona?

```
┌──────────────────────────────────────┐   ┌──────────────────────────────────────┐
│ Query 1 (original)                   │   │ Query 2 (injetada)                   │
│                                      │ + │                                      │
│ SELECT * FROM products               │   │ SELECT * FROM users                  │
│ WHERE category = 'Pets'              │   │                                      │
│                                      │   │                                      │
│ Retorna: 2 colunas (name, desc)      │   │ Retorna: 2 colunas (username, pass)  │
└──────────────────────────────────────┘   └──────────────────────────────────────┘
                    │                                       │
                    └───────────────  UNION  ───────────────┘
                                       │
                                       ▼
                       Resultado único combinado:
                       ┌─────────────────────────────┐
                       │ name        | description   │ ← produtos
                       │ administrator | s3cr3tP@ss  │ ← usuário leakado!
                       │ wiener        | peter       │ ← outro usuário
                       └─────────────────────────────┘
```

## 💥 Etapa 2 — Capturar a Senha do Admin

Ao acessar a URL, a página de produtos agora mostra:
- Os produtos da categoria `Pets` (resultado normal da query 1)
- **No fim da lista**, aparecem registros estranhos: `administrator | s3cr3tP@ssw0rd`, `wiener | peter`, etc.

| Condição | Estado |

| Tabela `users` também tem **2 colunas** | ✅ (sorte/conhecimento prévio) |

## 🧠 Por Que Funcionou? — Anatomia Completa

### A vulnerabilidade

```javascript
// ❌ Servidor vulnerável (anti-padrão clássico)
app.get('/filter', async (req, res) => {
  const { category } = req.query;
  const query = `SELECT * FROM products WHERE category = '${category}'`;
  const results = await db.execute(query);
  res.render('product-list', { products: results });
});
```

### O que o atacante fez

```
Input do usuário:  Pets' UNION SELECT * FROM users--
                      ↓
URL:               /filter?category=Pets'%20UNION%20SELECT...
                      ↓
Backend constrói:  SELECT * FROM products WHERE category = 'Pets' UNION SELECT * FROM users--'
                                                                  ↑                          ↑
                                                            input injetado            comentário descarta
                                                                                       o "' " final
                      ↓
Banco executa:     duas queries unidas → resultado misturado
                      ↓
Aplicação renderiza: products + users na mesma lista
```

### Os 4 "bugs" que conspiraram

| # | Bug | Onde |
|---|-----|------|
| 1 | **Concatenação de string** em vez de prepared statement | Código do servidor |
| 2 | **`SELECT *`** revela todos os dados, mesmo os sensíveis | Query original |
| 3 | **Senhas em texto plano** na tabela `users` | Design do banco |
| 4 | **Conta de aplicação com permissão de ler `users`** | Configuração de banco |

> 💡 Cada um sozinho não seria desastroso. Combinados, viram um vazamento total de credenciais.

## 🛡️ Como Prevenir — Defesa em 4 Camadas

### ❌ Errado

```javascript
const query = `SELECT * FROM products WHERE category = '${category}'`;
```

### ✅ Camada 1: Prepared Statement

```javascript
const [rows] = await db.execute(
  'SELECT id, name, description, price, image_url FROM products WHERE category = ?',
  [category]
);
```

**Por que melhora:**
- `category` é tratado como **dado literal**, nunca como código SQL
- O ataque vira: "procurando pela categoria literal `Pets' UNION SELECT...`" → nenhum produto encontrado, lista vazia

### ✅ Camada 2: Evitar `SELECT *`

```javascript
// ❌ Ruim — devolve tudo, inclusive senha se a tabela for users
SELECT * FROM products...

// ✅ Bom — só os campos que a tela precisa
SELECT id, name, description, price, image_url FROM products...
```

**Por que importa:**
- Mesmo se houver SQLi, o atacante só pode injetar com a mesma forma de colunas
- Reduz o "raio de exposição" caso outras tabelas sejam acessadas

### ✅ Camada 3: Hash de Senhas

```javascript
// ❌ NUNCA armazenar senhas em texto plano
INSERT INTO users (username, password) VALUES ('admin', 's3cr3t');

// ✅ Sempre com hash + salt
const hash = await bcrypt.hash(password, 12);
INSERT INTO users (username, password_hash) VALUES ('admin', '$2b$12$...');
```

**Por que isso é crítico aqui:**
- Se o admin tinha senha **`s3cr3tP@ssw0rd`** em texto plano, o atacante usa direto
- Se tivesse `$2b$12$xxx...` (bcrypt), o atacante teria que **quebrar o hash** — computacionalmente inviável para senha forte
- Conecta com **A02 — Cryptographic Failures** ([47.md](47.md))

### ✅ Camada 4: Princípio do Menor Privilégio no Banco

```sql
-- ❌ Conta do app pode ler TUDO
GRANT ALL ON db.* TO 'app_user';

-- ✅ Conta do app só vê o que precisa
GRANT SELECT (id, name, description, price, image_url) ON db.products TO 'app_user';
-- (note: NEM cria SELECT em db.users)
```

**Por que importa:**
- Mesmo com SQLi explorável, o `UNION SELECT * FROM users` **falharia** com "permission denied"
- A SQLi continua sendo um bug — mas o impacto fica contido

> 🔑 **Defense in Depth:** uma falha em uma camada não compromete o sistema todo. Aqui foram 4 camadas falhando ao mesmo tempo.

## 🎓 Conceitos Aprendidos

| Termo | Definição |
|-------|-----------|
| **UNION SELECT** | Combina resultado de uma segunda query à query original |
| **Information disclosure** | Vazamento não-intencional de dados |
| **Credential leak** | Vazamento específico de credenciais (login + senha) |
| **Reúso de senha** | Usar a mesma senha em vários serviços — risco amplificado por leaks |
| **Defense in Depth** | Múltiplas camadas de defesa independentes |
| **Menor privilégio (DB)** | Conta da aplicação só tem permissão do que **precisa** ler/escrever |

## ✅ Quiz Mental

1. Por que precisávamos saber o número de colunas **antes** de tentar `UNION SELECT * FROM users`?
2. Se a tabela `users` tivesse 5 colunas e a tabela `products` tivesse 2, o que aconteceria?
3. Como o **bcrypt** teria mitigado o impacto deste vazamento?
4. Por que `SELECT id, name FROM products` é **melhor** que `SELECT * FROM products` do ponto de vista de segurança?
5. Cite **3 camadas de defesa** que teriam evitado esse ataque.

<details>
<summary>Ver respostas</summary>

1. Porque o `UNION` exige **número idêntico de colunas** entre as queries unidas. Sem isso, dá erro de sintaxe
2. Erro: *"The used SELECT statements have a different number of columns"*. Teríamos que reduzir as colunas selecionadas: `UNION SELECT username, password FROM users--`
3. As senhas estariam em **hash** (`$2b$12$...`). O atacante extrairia o hash, mas precisaria fazer **força bruta offline** para descobrir a senha original — inviável para senhas fortes
4. Porque reduz a **superfície de exposição**. Mesmo com UNION injection, o atacante só vê os campos `id` e `name` — não `password`, `cpf`, `email` etc. que `*` revelaria
5. Possíveis respostas: **prepared statements** (impede a injection), **hash de senha** (mitiga o leak), **menor privilégio no banco** (conta da app sem `SELECT` em `users`), **WAF** (bloqueia payloads conhecidos), **logs e alertas** (detectar tentativas)

</details>

## 💡 Reflexão Final

> *"Em SQL Injection, o `UNION SELECT` é o canivete-suíço:*
> *- Sabe o nome da tabela? Extrai os dados direto.*
> *- Não sabe? Pergunta ao `information_schema`.*
> *- A tabela tem mais colunas que a sua? Usa `NULL` para preencher.*
> *- Os tipos não batem? `CAST()` resolve.*
>
> *Mas tudo isso pressupõe uma única falha: a aplicação concatenar input do usuário em SQL.*
> *Conserte a raiz e nenhuma das técnicas funciona."*

A raiz é sempre a mesma. A solução também: **prepared statements + menor privilégio + hash de senha**.
