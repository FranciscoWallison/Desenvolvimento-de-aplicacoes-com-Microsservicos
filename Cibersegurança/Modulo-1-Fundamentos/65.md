## 🎯 Contexto: Fechando a Tríade XSS

Já vimos os 2 tipos onde o **servidor** processa o payload:

| Tipo | Onde processa | Onde armazena | Vimos em |
|------|---------------|---------------|----------|
| **Reflected** | Servidor reflete na resposta | Não armazena | [62.md](62.md) |
| **Stored** | Servidor salva no banco | Banco de dados | [63.md](63.md) |
| **DOM-based** | **Cliente** (JavaScript do navegador) | **Nem servidor, nem banco** | ***este lab*** |

> 🔑 **A grande diferença:** em DOM XSS o ataque acontece **inteiramente no navegador do cliente** — o servidor **não tem ideia** do que está acontecendo. Isso significa: WAFs do lado do servidor **não detectam**, logs do servidor **não registram**, e a defesa precisa estar no **código JavaScript** que roda no browser.

---

## 📖 Conceitos-Chave do DOM XSS

### O que é o DOM?

**DOM (Document Object Model)** é a representação da página HTML **dentro do navegador** — uma árvore de objetos que o JavaScript pode ler e modificar. É o que permite páginas dinâmicas (mudar conteúdo sem recarregar).

### Sources e Sinks (vocabulário essencial)

| Termo | Definição | Exemplos |
|-------|-----------|----------|
| **Source** | Lugar de onde o input "entra" no JavaScript | `location.search`, `location.hash`, `document.URL`, `document.referrer`, `window.name` |
| **Sink** | Função/propriedade que **executa** ou **renderiza** o input | `document.write()`, `innerHTML`, `eval()`, `setTimeout()`, `location.href` |

> 🎯 **DOM XSS acontece quando dados de uma *source* fluem para um *sink* sem sanitização.**

## 🔍 Etapa 2 — Entender o que o JavaScript faz

Antes do payload, vamos **investigar** como o JS injeta o termo de busca.

1. Faça uma busca normal (ex: `teste`)
2. Abra o **DevTools** (F12) → aba **Elements**
3. Procure por uma tag `<img>` referenciando o termo:

```html
<img src="/resources/images/tracker.gif?searchTerms=teste">
```

O JavaScript da página faz algo equivalente a:

```javascript
document.write('<img src="/resources/images/tracker.gif?searchTerms=' + searchTerm + '">');
```

Note os 3 elementos:
- **Source:** `searchTerm` vem de `location.search` (URL)
- **Sink:** `document.write()` insere direto no HTML
- **Sem escape:** o input vai cru pro DOM

## 💥 Etapa 3 — O Payload

No campo de busca, digite:

```
"><svg onload=alert(1)> //
```

Clique em buscar.

🎯 **Boom!** Uma caixa de alerta com `1` aparece.

> ✅ *"Congratulations, you solved the lab!"*

---

## 🧠 Por Que Funciona — Anatomia do Payload

Vamos decompor `"><svg onload=alert(1)> //`:

### Antes do payload (HTML gerado normalmente)

```html
<img src="/resources/images/tracker.gif?searchTerms=teste">
```

### Após o payload (`"><svg onload=alert(1)> //`)

```html
<img src="/resources/images/tracker.gif?searchTerms="><svg onload=alert(1)> //">
```

### Quebra visual do que aconteceu

```
<img src="/resources/images/tracker.gif?searchTerms=
                                                    ↑
                                              ASPAS FECHA o atributo src

"><svg onload=alert(1)>
 ↑↑                  ↑
 || > FECHA a tag <img>
 |
 nova tag <svg> com onload=alert(1)

 //">
 ↑
 // é comentário JS (neutraliza o lixo restante)
```

### Resultado final no DOM

```html
<img src="/resources/images/tracker.gif?searchTerms=">  ← tag img fechada
<svg onload=alert(1)>                                    ← nova tag SVG injetada!
//">                                                     ← lixo comentado
```

A tag `<svg>` é **carregada pelo browser**, dispara o evento `onload`, e o `alert(1)` executa.

### Por que `<svg>` e não `<script>`?

`<script>` injetado dinamicamente via `document.write()` **não executa** após a página carregar. Tags como `<svg>`, `<img>`, `<iframe>` com handlers (`onload`, `onerror`) executam **imediatamente** ao serem inseridas no DOM.

**Outros payloads equivalentes:**

```html
"><img src=x onerror=alert(1)>           <!-- imagem inválida → onerror dispara -->
"><iframe src=javascript:alert(1)>        <!-- src com pseudo-protocolo JS -->
"><body onload=alert(1)>                  <!-- só funciona em alguns contextos -->
"><details open ontoggle=alert(1)>        <!-- HTML5 details com toggle -->
```

## 🔬 O Que Torna Isso DOM XSS (e Não Reflected)

Olhar **onde** o payload é processado é o que diferencia:

| | **Reflected XSS** ([62.md](62.md)) | **DOM-based XSS** (este lab) |
|--|-----------------------------------|------------------------------|
| **Servidor recebe o payload?** | ✅ Sim (na URL `?q=...`) | ✅ Sim (na URL) |
| **Servidor processa/reflete?** | ✅ Sim (devolve HTML com payload) | ❌ **Não** (devolve HTML genérico) |
| **Onde o payload vira HTML?** | No template do servidor | No **JavaScript do cliente** |
| **WAF do servidor detecta?** | ✅ Pode detectar | ❌ Geralmente **não** |
| **Log do servidor registra?** | ✅ Tem entrada no log | ⚠️ Só registra a URL, **não o ataque** |

### Como confirmar que é DOM XSS

**Teste prático:** olhe a **resposta HTTP** do servidor (DevTools → Network → ver Response):

- **Se** o payload aparece no HTML cru da resposta → é **Reflected** (servidor refletiu)
- **Se** o payload **NÃO aparece** na resposta crua, mas SIM no DOM após JS executar → é **DOM-based**

> 💡 **No nosso lab:** se você olhar a resposta do servidor, o payload **não está lá**. Ele só aparece **depois** do JavaScript da página construir o `<img>` com `document.write()`.

## 🗂️ Os Sinks Perigosos Mais Comuns

Lugares no JavaScript onde input pode virar execução de código:

| Sink | O que faz | Risco |
|------|-----------|-------|
| `document.write()` | Insere HTML cru no documento | 🔴 Alto |
| `element.innerHTML` | Substitui HTML interno do elemento | 🔴 Alto |
| `element.outerHTML` | Substitui o elemento inteiro | 🔴 Alto |
| `element.insertAdjacentHTML()` | Insere HTML em posição relativa | 🔴 Alto |
| `eval()` | Executa string como JavaScript | 🔴 Crítico |
| `Function()` constructor | Cria função dinâmica | 🔴 Crítico |
| `setTimeout(string, ...)` | Executa string após timer | 🔴 Alto |
| `setInterval(string, ...)` | Executa string em loop | 🔴 Alto |
| `location.href = ...` | Permite `javascript:` URLs | 🟡 Médio |
| `element.src = ...` | Permite `javascript:` em alguns elementos | 🟡 Médio |
| `jQuery $()`/`.html()` | Insere HTML (similar a innerHTML) | 🔴 Alto |

### Os Sources mais comuns

Lugares de onde dados controlados pelo atacante chegam:

| Source | Conteúdo |
|--------|----------|
| `location.search` | Query string da URL (`?q=...`) |
| `location.hash` | Fragmento da URL (`#...`) — **não vai pro servidor!** |
| `location.pathname` | Caminho da URL |
| `document.URL` | URL completa |
| `document.referrer` | URL da página anterior |
| `window.name` | Nome da janela (controlável por outra página!) |
| `localStorage` / `sessionStorage` | Armazenamento local (se outro XSS já escreveu nele) |
| `postMessage` | Mensagens entre janelas/iframes |

> 🔥 **Curiosidade perigosa:** `location.hash` **nunca é enviado ao servidor**. Isso significa que **logs do servidor nunca veem** o payload — ataque 100% invisível para o backend.

## 🎭 Por Que DOM XSS É Especialmente Perigoso

### 1. Invisível para o servidor
Logs, SIEM ([64.md](64.md)), WAF — nenhum vê o ataque acontecer. A descoberta vem só quando a vítima reclama.

### 2. Difícil de detectar em pentest passivo
Scanners tradicionais que só leem a resposta HTTP **não pegam** DOM XSS — precisam **executar o JavaScript** da página.

### 3. Comum em SPAs modernos
React, Vue, Angular usam **muito** DOM manipulation. Frameworks têm proteções, mas APIs como `dangerouslySetInnerHTML`, `v-html`, `[innerHTML]` quebram a proteção e abrem porta para DOM XSS.

### 4. `location.hash` é "stealth mode"
Como `#...` não vai pro servidor, ataques via hash são **invisíveis** em qualquer log de tráfego HTTP.

## 🛡️ Como Prevenir DOM XSS

A defesa é **diferente** dos outros tipos — precisa estar no **código JavaScript do cliente**, não no servidor.

### ❌ Errado — confiar em escape do servidor

Se o servidor escapa input antes de devolver, isso **não ajuda** com DOM XSS — o servidor nem vê o payload (ele veio via JS).

### ✅ Certo 1: Não usar sinks perigosos

```javascript
// ❌ Vulnerável
document.write('<img src="?q=' + searchTerm + '">');
element.innerHTML = userInput;

// ✅ Seguro — usa textContent (não interpreta HTML)
element.textContent = userInput;

// ✅ Seguro — cria elemento e define atributo via API
const img = document.createElement('img');
img.src = '/resources/images/tracker.gif';
img.dataset.searchTerm = searchTerm;  // atributo data-, não HTML cru
container.appendChild(img);
```

### ✅ Certo 2: Sanitizar com DOMPurify

Se **precisar** renderizar HTML (ex: editor WYSIWYG), use **DOMPurify**:

```javascript
import DOMPurify from 'dompurify';

const dirty = userInput;  // pode ter <script>alert(1)</script>
const clean = DOMPurify.sanitize(dirty);
element.innerHTML = clean;  // agora é seguro
```

### ✅ Certo 3: Encode no contexto correto

```javascript
// Para inserir em HTML
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#39;');
}

document.write(escapeHtml(searchTerm));  // agora seguro
```

### ✅ Certo 4: Frameworks modernos com cuidado

```jsx
// React — seguro por padrão
function Search({ term }) {
  return <div>Você buscou por: {term}</div>;  // {} escapa automaticamente
}

// ❌ EVITAR — quebra a proteção
function Search({ term }) {
  return <div dangerouslySetInnerHTML={{__html: term}} />;
}
```

```vue
<!-- Vue — seguro por padrão -->
<div>{{ term }}</div>

<!-- ❌ EVITAR — v-html quebra a proteção -->
<div v-html="term"></div>
```

### ✅ Certo 5: Content Security Policy (CSP)

CSP estrita bloqueia execução de scripts inline e de origens não autorizadas — mesmo se DOM XSS injetar `<script>`, o browser **recusa** executar:

```http
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'
```

> 🔗 Detalhes de CSP em [62.md](62.md).

### ✅ Certo 6: Bibliotecas Anti-XSS por Linguagem

A maioria das linguagens tem **bibliotecas oficiais** que fazem o trabalho pesado de escape/sanitização. Não reinvente a roda:

| Linguagem | Biblioteca | Site/Repo | Uso típico |
|-----------|------------|-----------|-----------|
| **JavaScript** | **DOMPurify** | [github.com/cure53/DOMPurify](https://github.com/cure53/DOMPurify) | Sanitização de HTML rico no cliente |
| **PHP** | **HTMLPurifier** | [htmlpurifier.org](http://htmlpurifier.org/) | Sanitização robusta com allowlist |
| **.NET** | **Microsoft AntiXSS Library** | [docs.microsoft.com](https://docs.microsoft.com/pt-br/dotnet/api/system.web.security.antixss) | Encoders por contexto (HTML, JS, URL, CSS) |
| **Java** | **OWASP Java Encoder** | [github.com/OWASP/owasp-java-encoder](https://github.com/OWASP/owasp-java-encoder) | Encoding por contexto |
| **Java** | **OWASP AntiSamy** | [github.com/nahsra/antisamy](https://github.com/nahsra/antisamy) | Sanitização com políticas XML |
| **Python** | **bleach** | [github.com/mozilla/bleach](https://github.com/mozilla/bleach) | Sanitização inspirada em DOMPurify |
| **Ruby** | **sanitize** (gem) | [github.com/rgrove/sanitize](https://github.com/rgrove/sanitize) | Sanitização com whitelist |
| **Go** | **bluemonday** | [github.com/microcosm-cc/bluemonday](https://github.com/microcosm-cc/bluemonday) | Inspirado no AntiSamy |

#### Quando usar **encoder** vs **sanitizer**

- **Encoder** (escape): transforma `<` em `&lt;` etc. — para quando o input deveria ser **texto literal**
- **Sanitizer**: remove tags/atributos perigosos, mantém HTML válido — para quando o input pode ser **HTML rico** (editor WYSIWYG, comentários com formatação)

> 💡 **Regra prática:** se você não tem certeza, use **encoding** (mais simples e seguro). Só use sanitizer quando precisar **mesmo** permitir HTML formatado.

### ⚠️ X-XSS-Protection — Header DEPRECATED

Você pode encontrar este header em tutoriais antigos:

```http
X-XSS-Protection: 1; mode=block
```

**Era** um filtro automático contra Reflected XSS implementado em IE, Chrome e Safari antigos. **Hoje está DEPRECATED:**

| Browser | Status atual |
|---------|--------------|
| **Chrome** (Blink) | ❌ Removido — XSS Auditor desabilitado por padrão e removido na v78 |
| **Firefox** | ❌ Nunca implementou |
| **Edge** (Chromium) | ❌ Removido junto com Chrome |
| **Safari** | ⚠️ Ainda suporta, mas desencorajado |
| **MDN recomenda** | 🔴 **Não usar** — preferir CSP estrita |

#### Por que foi descontinuado

1. Causava **falsos positivos** que quebravam páginas legítimas
2. Existiam **bypasses conhecidos** que invalidavam a proteção
3. **CSP** cobre o mesmo caso com mais robustez

#### Recomendação atual da OWASP/MDN

```http
# Em vez de X-XSS-Protection, use:
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'

# Se você ainda precisa do header por compatibilidade legada (raro):
X-XSS-Protection: 0    # Desabilita o filtro buggy, deixa CSP cuidar
```

> 🔗 [MDN: X-XSS-Protection (deprecated)](https://developer.mozilla.org/pt-BR/docs/Web/HTTP/Headers/X-XSS-Protection)

### 🔠 Encoding × Validation — A Confusão Mais Comum

Termos parecidos mas com papéis diferentes:

| Termo | O que faz | Quando aplicar |
|-------|-----------|----------------|
| **Encoding** (escape) | Transforma chars perigosos em entidades seguras (`<` → `&lt;`) | **Saída** — antes de inserir em HTML/JS/URL |
| **Validation** | Verifica se input atende **regras de formato** (e-mail válido, idade numérica) | **Entrada** — antes de aceitar o dado |
| **Sanitization** | Remove/transforma partes perigosas, mantém o resto | **Entrada** (para Stored) ou **saída** (para HTML rico) |

#### Por que só encoding não basta

A leitura original cita um caso importante: se a aplicação faz **encoding cego** mas a aplicação injeta HTML, o atacante pode contornar usando **entidades HTML**:

```
Atacante envia:    &lt;script&gt;
                          ↓
App aplica encoding:  &amp;lt;script&amp;gt;
                          ↓
HTML renderizado:     &lt;script&gt;
                          ↓
Browser interpreta:   <script>      ← BOOM!
```

Por isso é **essencial combinar**:
- ✅ **Validation** na entrada (rejeita o que não bate com formato esperado)
- ✅ **Encoding** na saída (no contexto certo: HTML, JS, URL, CSS)
- ✅ **CSP** como defesa em profundidade

## 📋 Checklist Anti-DOM-XSS

- [ ] Auditar todo uso de `innerHTML`, `outerHTML`, `document.write`, `eval`, `setTimeout(string)`
- [ ] Trocar `innerHTML` por `textContent` quando possível
- [ ] Usar `createElement` + `setAttribute` em vez de string concatenation
- [ ] Aplicar **DOMPurify** se HTML rico for inevitável
- [ ] Validar todos os **sources** (`location.*`, `document.referrer`, `postMessage`)
- [ ] Configurar **CSP** estrita como defesa em profundidade
- [ ] Code review específico para `dangerouslySetInnerHTML` / `v-html` / `[innerHTML]`
- [ ] Usar linters: **eslint-plugin-security**, **eslint-plugin-no-unsanitized**

## ✅ Quiz Mental

1. Qual a diferença **fundamental** entre Reflected XSS e DOM XSS, se ambos vêm da URL?
2. Por que `<script>alert(1)</script>` muitas vezes **não funciona** em DOM XSS via `document.write`?
3. Por que `location.hash` é considerado um source "stealth" para ataques?
4. Se uma aplicação usa **React** com `{interpolation}` normal, ela está vulnerável a DOM XSS?
5. WAF do servidor protege contra DOM XSS?

<details>
<summary>Ver respostas</summary>

1. **Reflected:** o servidor processa o payload e reflete na resposta HTML. **DOM-based:** o servidor entrega HTML genérico, e o **JavaScript do cliente** é quem insere o payload no DOM. A URL é a mesma, mas o ponto de execução é diferente
2. Porque `<script>` inserido dinamicamente via `document.write()` ou `innerHTML` **não executa** após o carregamento inicial da página. Por isso usamos tags com handlers de evento (`<svg onload>`, `<img onerror>`) que **disparam ao serem inseridos**
3. Porque o fragmento de URL (depois do `#`) **nunca é enviado ao servidor**. Logs HTTP, WAF, SIEM — nada registra o conteúdo do hash. O ataque é completamente invisível no backend
4. **Geralmente não.** React escapa `{interpolation}` por padrão. Vira vulnerável **somente** se usar `dangerouslySetInnerHTML`, ou se usar bibliotecas que manipulam DOM manualmente
5. **Não.** WAFs do servidor inspecionam requisições/respostas HTTP. DOM XSS acontece **depois** do servidor — no JavaScript que executa no browser. A defesa precisa estar no código cliente ou via **CSP**

</details>

## 💡 Reflexão Final

> *"Reflected XSS atravessa o servidor.*
> *Stored XSS dorme no banco.*
> *DOM XSS acontece debaixo do nariz de todo mundo — invisível para o servidor, invisível para os logs, invisível para o WAF.*
>
> *É o tipo mais difícil de detectar e o mais negligenciado nas defesas tradicionais.*
> *Quando você o entende, entende por que CSP existe."*

### A grande lição da tríade XSS

| Tipo | Onde "vive" | Onde defender |
|------|-------------|---------------|
| **Reflected** | URL → servidor → HTML | Escape no template do **servidor** |
| **Stored** | Banco → HTML | Escape no template + sanitização na **entrada** |
| **DOM-based** | JavaScript do cliente | Defesa no **código JS** + **CSP** |

A doutrina é a mesma — **nunca inserir input cru em contexto sensível** — mas o **local da defesa muda**:

> 🔑 **Servidor escapa para Reflected/Stored. Cliente escapa para DOM. CSP cobre os 3.**
