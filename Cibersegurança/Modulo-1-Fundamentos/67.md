# 🛡️ WAF — Web Application Firewall

> **Tema:** Firewall especializado em **aplicações web** — filtra requisições HTTP/HTTPS antes de chegarem ao servidor
> **Curso fonte:** *Segurança de Rede — Firewall, WAF e SIEM*
> **Relacionado a:** [47.md](47.md) (OWASP Top 10), [64.md](64.md) (SIEM), [66.md](66.md) (Redes)

---

## 📖 O que é um WAF?

> **WAF (Web Application Firewall)** é um firewall **especializado em tráfego web (HTTP/HTTPS)** que opera na **camada de aplicação (camada 7 do modelo OSI)** — filtrando requisições maliciosas antes que cheguem ao servidor web.

### Diferença para um firewall tradicional

Um **firewall tradicional** (visto no quiz do Samuel — alternativa A) inspeciona pacotes IP/TCP — funciona nas **camadas 3 e 4** do modelo OSI. Vê de onde vem, para onde vai, qual porta. Mas **não entende o conteúdo HTTP**.

Um **WAF** entende:
- 📍 Cabeçalhos HTTP (`User-Agent`, `Cookie`, `Referer`)
- 📍 Métodos (`GET`, `POST`, `PUT`, `DELETE`)
- 📍 Parâmetros de URL (`?id=1`)
- 📍 Corpo da requisição (JSON, form-data, XML)
- 📍 Padrões de payload (SQL Injection, XSS, path traversal)

### Comparação visual

```
┌─────────────────────────────────────────────────────────────┐
│                  Modelo OSI — Camadas                       │
├─────────────────────────────────────────────────────────────┤
│  7 — Aplicação      HTTP, HTTPS, FTP, DNS    ◄── WAF aqui   │
│  6 — Apresentação   SSL/TLS                                 │
│  5 — Sessão                                                 │
│  4 — Transporte     TCP, UDP                                │
│  3 — Rede           IP, ICMP                  ◄── FW tradicional  │
│  2 — Enlace         Ethernet, MAC                           │
│  1 — Física         Cabos, Wi-Fi                            │
└─────────────────────────────────────────────────────────────┘
```

> 🔑 **A grande sacada:** firewall tradicional pergunta *"essa origem pode chegar nessa porta?"*. WAF pergunta *"essa requisição HTTP é maliciosa?"* — coisas totalmente diferentes.

## ⚙️ Como Funciona um WAF (Proxy Reverso)

O WAF é tipicamente implementado como **proxy reverso** entre a internet e o servidor:

```
                  ┌─────────────────────────────┐
                  │  USUÁRIO (browser/cliente)  │
                  └────────────┬────────────────┘
                               │ Requisição HTTPS
                               ▼
                  ┌─────────────────────────────┐
                  │  🛡️ WAF (proxy reverso)     │
                  │                             │
                  │  1. Inspeciona requisição   │
                  │  2. Aplica regras           │
                  │  3. Permite OU bloqueia     │
                  └────────────┬────────────────┘
                               │ Requisição filtrada
                               ▼
                  ┌─────────────────────────────┐
                  │  SERVIDOR DE APLICAÇÃO      │
                  │  (a aplicação web real)     │
                  └─────────────────────────────┘
```

### Fluxo de decisão

```
Requisição chega ao WAF
        ↓
    Bate em alguma regra de bloqueio? ────┐ Sim → 403 Forbidden / Captcha / Bloqueia IP
        ↓ Não                              │
    Bate em padrão suspeito? ──────────────┤ Talvez → Loga, alerta, mas passa (modo "Detection")
        ↓ Não                              │
    Encaminha ao servidor ─────────────────┘
```

## 🗂️ Os 3 Tipos de WAF

Existem **3 modelos de implantação** principais — cada um com tradeoffs claros:

### 1️⃣ WAF de Rede (Network-Based / Hardware)

> **O que é:** **Aparelho físico** (appliance) instalado **dentro do data center** da empresa, na fronteira da rede.

| Característica | Detalhe |
|----------------|---------|
| **Tipo** | Hardware dedicado (appliance) |
| **Localização** | Data center on-premises |
| **Performance** | 🚀 Alta (hardware otimizado) |
| **Latência** | 🟢 Mínima (próximo ao servidor) |
| **Custo inicial** | 🔴 Alto (compra de hardware) |
| **Custo recorrente** | 🟢 Baixo (energia + manutenção) |
| **Manutenção** | Por equipe interna ou contrato com vendor |
| **Exemplos de vendors** | [Barracuda](https://www.barracuda.com/products/webapplicationfirewall), F5 BIG-IP ASM, Imperva SecureSphere, Fortinet FortiWeb |

**Quando faz sentido:** empresas grandes com servidores próprios, alta carga, requisitos de compliance que exigem hardware controlado (bancos, governo).

**Desvantagens:**
- ❌ Capex alto
- ❌ Não escala elasticamente
- ❌ Difícil em arquiteturas cloud-native

---

### 2️⃣ WAF de Host (Host-Based / Software)

> **O que é:** WAF **baseado em software** instalado **diretamente no servidor da aplicação** ou rodando como módulo do web server.

| Característica | Detalhe |
|----------------|---------|
| **Tipo** | Software local |
| **Localização** | No próprio servidor da aplicação |
| **Performance** | ⚠️ Consome recursos do servidor |
| **Latência** | 🟢 Mínima (mesmo processo) |
| **Custo inicial** | 🟢 Baixo (geralmente open source) |
| **Custo recorrente** | 🟡 Médio (operação + tuning) |
| **Personalização** | 🟢 Máxima (você controla tudo) |
| **Complexidade** | 🔴 Alta (precisa expertise) |
| **Exemplos** | **ModSecurity** (Apache/Nginx/IIS), **NAXSI** (Nginx), **Coraza** (Go) |

**Quando faz sentido:** equipes técnicas que querem **controle total**, projetos open source, integração customizada, ambientes legados.

**Desvantagens:**
- ❌ Consome CPU/RAM do servidor de aplicação
- ❌ Manutenção das regras é trabalhosa
- ❌ Curva de aprendizado íngreme

> 💡 **ModSecurity** é o WAF open source mais conhecido — e usa o **OWASP Core Rule Set (CRS)** como base de regras prontas contra OWASP Top 10.

---

### 3️⃣ WAF na Nuvem (Cloud-Based / WAFaaS)

> **O que é:** WAF **gerenciado por um provedor de nuvem** — você só configura via dashboard, o provedor faz todo o resto.

| Característica | Detalhe |
|----------------|---------|
| **Tipo** | Serviço gerenciado (SaaS) |
| **Localização** | Distribuído globalmente (edge network) |
| **Performance** | 🚀 Alta (CDN global) |
| **Latência** | 🟢 Pode até reduzir (edge caching) |
| **Custo inicial** | 🟢 Mínimo (sem hardware) |
| **Custo recorrente** | 🟡 Por assinatura/uso |
| **Manutenção** | Pelo **provedor** (você só configura) |
| **Escalabilidade** | 🚀 Elástica automática |
| **Exemplos** | **Cloudflare**, **AWS WAF**, **Azure WAF**, **Google Cloud Armor**, **Akamai Kona**, **Fastly Next-Gen WAF** |

**Quando faz sentido:** **maioria dos casos modernos** — startups, apps cloud-native, e-commerce, qualquer aplicação que escala dinamicamente.

**Desvantagens:**
- ❌ **Você delega a segurança a um terceiro**
- ❌ Custos podem escalar com tráfego
- ❌ Menos flexibilidade que ModSecurity

> 💡 **Cloudflare** é o nome mais conhecido — protege mais de 20% da web atualmente. Tem plano grátis que já inclui WAF básico.

## 🎯 O Que um WAF Bloqueia?

Os ataques mais comuns que um WAF moderno detecta e bloqueia (todos do [OWASP Top 10](47.md)):

| Ataque | Como WAF detecta | Visto em |
|--------|------------------|----------|
| **SQL Injection** | Padrões como `' OR 1=1`, `UNION SELECT`, `DROP TABLE` | [57.md](57.md), [58.md](58.md), [59.md](59.md) |
| **XSS** | Tags `<script>`, `<svg onload>`, `javascript:` URLs | [62.md](62.md), [63.md](63.md), [65.md](65.md) |
| **Path Traversal** | `../`, `..\\`, encoded `%2e%2e/` | — |
| **Command Injection** | `;`, `\|`, `&&`, `$()` em campos esperando texto | — |
| **XXE** | `<!DOCTYPE>`, `<!ENTITY>` em XML | — |
| **SSRF** | URLs apontando para IPs internos (`127.0.0.1`, `169.254.169.254`) | [47.md](47.md) (A10) |
| **Force Brute** | Múltiplas tentativas de login em curto intervalo | — |
| **DDoS L7** | Picos anômalos de requisições | — |
| **Bots maliciosos** | User-Agents conhecidos, padrões de scraping | — |

### Modos de operação

Todo WAF profissional opera em pelo menos 2 modos:

| Modo | O que faz |
|------|-----------|
| **Detection / Monitor** | Apenas **loga** os alertas — não bloqueia. Bom para tuning inicial |
| **Prevention / Block** | **Bloqueia** ativamente os ataques detectados |
| **Challenge** | Apresenta **CAPTCHA** ou JS challenge para validar humano |
| **Learning** | Observa tráfego legítimo para criar baseline (WAFs com ML) |

## 🏷️ Tipos de Firewall — Classificação Tradicional

Antes de falar de WAF×IPS, vale entender a **classificação clássica** dos firewalls. Existem **3 categorias** principais + uma "categoria guarda-chuva" muito usada na prática (UTM).

> 💡 [66.md](66.md) já mostra a comparação rápida **stateless × stateful × NGFW**. Aqui aprofundamos com a taxonomia tradicional usada em literatura de segurança.

### 1️⃣ Packet Filtering (Filtragem de Pacotes)

> **O que é:** o tipo mais antigo e simples. Cria uma **lista de regras** e o firewall analisa **cada pacote** decidindo se passa ou bloqueia, com base em IP, porta e protocolo.

Existe em **2 modos**:

#### 🔹 Estático (Stateless)

```
Pacote chega → Compara com regras → Aceita ou bloqueia
                                          ↑
                              (sem memória de pacotes anteriores)
```

- **Analisa cada pacote isoladamente** — sem conhecimento da conexão
- **Rápido** (não mantém estado em memória)
- **Vulnerável a spoofing** — atacante pode forjar pacotes como se viessem de uma conexão legítima

> ⚠️ **Spoofing (IP spoofing)** é a falsificação do IP de origem. Como o stateless **não sabe** se aquela conexão foi iniciada de verdade, um atacante pode enviar pacotes "como se" fizessem parte de uma sessão em andamento.

#### 🔹 Dinâmico (Stateful)

```
Pacote chega → Verifica a TABELA DE ESTADO de conexões ativas
            → Se faz parte de uma conexão legítima → permite
            → Se não faz → aplica regras estáticas
```

- Surgiu para **corrigir** as fraquezas do stateless contra **spoofing**
- Mantém **tabela de estado** das conexões TCP ativas (SYN → SYN-ACK → ACK)
- Sabe **distinguir** um pacote "resposta legítima" de um pacote forjado
- Mais lento (precisa rastrear estado) mas **muito mais seguro**

> 🔑 **Hoje, todo firewall sério é stateful.** O pfSense que estamos usando ([69.md](69.md), [70.md](70.md)) é stateful por padrão.

### 2️⃣ Proxy Services (Firewall de Aplicação)

> **O que é:** firewall que **intermedia** completamente a conexão entre cliente e servidor — opera na **camada de aplicação** (camada 7 OSI).

```
Cliente → [ Proxy Firewall ] → Servidor
            ↑
   - Encerra a conexão do cliente
   - Inspeciona o conteúdo da requisição
   - Cria uma NOVA conexão com o servidor
   - Devolve a resposta processada
```

**Funções extras** que o packet filtering simples não faz:

- ✅ **Inspeção de conteúdo** (não só headers — vê dentro do payload)
- ✅ **Cache** de respostas (acelera o acesso)
- ✅ **Autenticação** antes de liberar acesso
- ✅ **Log detalhado** de tráfego de aplicação

> 💡 **O WAF (visto neste documento) é um caso especial de proxy services** — especializado em HTTP/HTTPS.

Exemplos reais:
- **Squid** (proxy de cache clássico)
- **HAProxy** (proxy reverso para load balancing)
- **Nginx** (web server que também funciona como proxy reverso)

### 3️⃣ Stateful Inspection (Inspeção de Estado)

> **O que é:** **evolução** do stateful packet filtering — não só verifica o estado da conexão, mas **inspeciona o comportamento** comparando com o que **deveria** estar acontecendo no protocolo.

```
Pacote chega → "É TCP. Estado da conexão: ESTABLISHED."
            → "Mas espera... esse pacote é um SYN no meio de
               uma conexão estabelecida? Isso não é normal."
            → BLOQUEIA (potencial ataque)
```

- **Compara o que está acontecendo × o que se espera** que aconteça
- Detecta **anomalias de protocolo** (SYN flood, ataques de sequência TCP)
- É mais "inteligente" que stateful filtering puro

> 🔑 **Diferença prática:**
> - **Stateful packet filtering** → "esse pacote pertence à conexão X?"
> - **Stateful inspection** → "esse pacote pertence à conexão X **E** está se comportando como o protocolo manda?"

### 🧰 UTM — Unified Threat Management

> **O que é:** **não é um tipo técnico**, é uma **categoria comercial** — firewall "tudo em um" que combina várias funções em um único produto.

```
┌─────────────────────────────────────────────┐
│           FIREWALL UTM                      │
│                                             │
│  ✓ Packet Filtering (stateful)              │
│  ✓ Proxy Services                           │
│  ✓ Stateful Inspection                      │
│  ✓ IDS/IPS                                  │
│  ✓ VPN                                      │
│  ✓ Antivírus de gateway                     │
│  ✓ Filtro de conteúdo web                   │
│  ✓ Anti-spam                                │
│  ✓ Logs centralizados                       │
└─────────────────────────────────────────────┘
```

> 🎯 **pfSense é UTM!** O firewall que usamos nos labs é exatamente isso — combina filtragem de pacotes stateful, NAT, VPN, IDS/IPS (via Snort/Suricata), proxy, logs.

| Característica | UTM |
|----------------|-----|
| **Filosofia** | "Tudo em um aparelho" |
| **Vantagem** | Custo menor, gerenciamento centralizado, alinhamento entre features |
| **Desvantagem** | Performance pode sofrer (uma máquina fazendo tudo) |
| **Usado em** | Pequenas e médias empresas — onde não compensa ter aparelhos dedicados |
| **Exemplos** | pfSense, OPNsense, Fortinet FortiGate, Sophos UTM, WatchGuard |

### 🔬 NGFW × UTM — Diferença Sutil

| | **UTM** | **NGFW** (Next-Generation Firewall) |
|--|---------|--------------------------------------|
| **Foco** | "Tudo em um" para SMB | Inspeção profunda + consciência de aplicação para enterprise |
| **Inspeção** | Stateful + IPS + antivírus | **Deep Packet Inspection** + reconhece app pelo conteúdo (Facebook × LinkedIn × Skype) |
| **Mercado-alvo** | PMEs, escritórios médios | Grandes empresas, data centers |
| **Exemplos** | pfSense, Sophos UTM | Palo Alto, Cisco Firepower, Check Point Quantum |

> 💡 Há **muita sobreposição** entre UTM e NGFW. Em 2026, a fronteira é fluida — vendors usam os termos quase intercambiavelmente. A regra prática: **NGFW** é mais "inteligente" sobre **camada 7 (aplicação)**, **UTM** é mais sobre "muitos recursos integrados".

### 📊 Tabela Consolidada das 3 Classificações Tradicionais

| Tipo | Camada OSI | Inspeção | Performance | Caso clássico |
|------|:----------:|----------|:-----------:|---------------|
| **Packet Filtering Stateless** | 3-4 | Pacote isolado, sem contexto | 🚀🚀🚀 | ACLs em roteadores antigos |
| **Packet Filtering Stateful** | 3-4 | Pacote + tabela de estado | 🚀🚀 | Firewall padrão moderno (pfSense, iptables) |
| **Proxy Services** | 7 | Conteúdo completo da aplicação | 🚀 | Squid, WAF |
| **Stateful Inspection** | 3-7 | Estado + comportamento de protocolo | 🚀🚀 | Cisco ASA, Check Point |

Como mencionado, **pfSense é UTM** — combina vários tipos:

```
┌──────────────────────────────────────────────────┐
│  pfSense (em [69.md], [70.md])                   │
│                                                  │
│  Por padrão: Packet Filtering Stateful           │
│  + Pacotes opcionais:                            │
│    - Snort/Suricata (IDS/IPS)                    │
```

## 🆚 WAF × Firewall × IDS × IPS — Não Confunda!

Erro clássico em provas e entrevistas:

| Ferramenta | Camada OSI | O que faz | Onde fica |
|------------|:----------:|-----------|-----------|
| **Firewall tradicional** | 3-4 | Filtra IPs/portas/protocolos | Borda de rede |
| **WAF** | 7 | Filtra requisições HTTP maliciosas | Antes do servidor web |
| **IDS** (Intrusion Detection) | 3-7 | **Detecta** intrusões (só alerta) | Espelhamento de tráfego |
| **IPS** (Intrusion Prevention) | 3-7 | Detecta **e bloqueia** intrusões | Inline no tráfego |

> 💡 **Mnemônico:**
> - **Firewall** = "quem pode entrar?"
> - **WAF** = "essa requisição HTTP é segura?"
> - **IDS** = "vejo algo estranho!" (alerta)
> - **IPS** = "vejo algo estranho — bloqueado!" (ação)

## 🔄 WAF × IPS — Por Que Eles Trabalham Juntos

Uma confusão **muito comum** em provas e entrevistas: *"Se eu já tenho WAF, preciso de IPS? Eles fazem a mesma coisa?"*

**Resposta curta:** não fazem a mesma coisa — são **complementares**, não substitutos.

### O que cada um faz, em uma frase

| | **WAF** | **IPS** |
|--|---------|---------|
| **Foco** | Aplicações **web** (HTTP/HTTPS) | **Todo tráfego de rede** |
| **Camada OSI** | 7 (aplicação) | 3-7 (rede até aplicação) |
| **Análise** | Profunda em **requisições HTTP** | Mais ampla mas mais **superficial** |
| **Detecta** | SQLi, XSS, padrões de OWASP Top 10 | Worms, malware, exploits de protocolo |
| **Posição** | Antes do servidor web | Inline no tráfego de rede |
| **"Robustez"** | 🟢 Robusto para web | 🟡 Generalista |

### Visualização: as camadas de defesa

```
┌─────────────────────────────────────────────────────────────┐
│                       INTERNET                              │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
                  ┌─────────────────────┐
                  │   🔥 Firewall       │ ← Camadas 3-4
                  │   (IPs/portas)      │   "Quem pode entrar?"
                  └──────────┬──────────┘
                             │
                             ▼
                  ┌─────────────────────┐
                  │   🛡️ IPS            │ ← Camadas 3-7
                  │   (intrusões        │   "Detectei e bloqueei
                  │    em geral)        │    um exploit conhecido"
                  └──────────┬──────────┘
                             │
                             ▼
                  ┌─────────────────────┐
                  │   🌐 WAF            │ ← Camada 7
                  │   (ataques HTTP)    │   "Esse SQL Injection
                  │                     │    no parâmetro id?"
                  └──────────┬──────────┘
                             │
                             ▼
                  ┌─────────────────────┐
                  │   🖥️ APLICAÇÃO     │
                  └─────────────────────┘
```

### Por que WAF é considerado "mais robusto" para web

WAF foi **construído especificamente** para entender HTTP/HTTPS:
- Sabe distinguir um `' OR 1=1` malicioso de um nome legítimo com apóstrofo
- Entende sessão, cookies, autenticação web
- Analisa **comportamento** (não só assinatura): rate limiting, bot detection, anomalia de navegação
- Conhece o **contexto** da aplicação web

### Por que IPS é "menos robusto" (mas necessário)

IPS é **generalista** — protege tudo, não só web:
- Detecta exploits de **SMB, RDP, SSH, DNS** (não só HTTP)
- Bloqueia **worms** se espalhando pela rede
- Identifica **malware** conhecido por assinatura
- Funciona para **qualquer protocolo**, não só HTTP

### O ponto-chave: complementaridade

```
┌─────────────────────────────────────────────────────────────┐
│  Sem WAF, IPS sozinho:                                      │
│   ✅ Bloqueia exploit de SMB (porta 445)                    │
│   ❌ Não pega SQLi sutil em /api/login                      │
│                                                             │
│  Sem IPS, WAF sozinho:                                      │
│   ✅ Bloqueia <script> em formulário                        │
│   ❌ Não pega worm tentando RDP no servidor                 │
│                                                             │
│  WAF + IPS = cobertura completa                             │
└─────────────────────────────────────────────────────────────┘
```

### Quando você precisa de cada um?

| Cenário | WAF? | IPS? |
|---------|:----:|:----:|
| API REST pública | ✅ Sim | 🟡 Bom ter |
| Servidor SSH expostos | 🔵 Não ajuda | ✅ Essencial |
| E-commerce | ✅ Essencial | ✅ Recomendado |
| Servidor de arquivos interno (SMB) | 🔵 Não ajuda | ✅ Essencial |
| App web + microsserviços | ✅ Essencial | ✅ Recomendado |
| DNS server | 🔵 Não ajuda | ✅ Essencial |

### Hierarquia de decisões num ataque típico

Imagine um atacante tentando explorar uma aplicação web:

```
1. Atacante manda payload de SQLi via HTTPS
        ↓
2. Firewall: "Vem da porta 443? Pode passar."     ✅ libera (camada 4)
        ↓
3. IPS: "Não bate com nenhuma assinatura de       ✅ libera (não é exploit
        worm/exploit conhecido."                       conhecido)
        ↓
4. WAF: "Detectei 'UNION SELECT' no parâmetro!"    🚫 BLOQUEIA (camada 7)
        ↓
5. Aplicação: ... nem recebe a requisição.
```

> 💡 **A grande sacada:** o WAF é quem pega esse tipo de ataque. Mas se o atacante mudar para tentar um exploit de **Log4Shell** no header `User-Agent`, é o **IPS** que reconhece a assinatura primeiro.

### Conexão com SIEM ([64.md](64.md))

Tanto WAF quanto IPS **enviam alertas para o SIEM** — o SIEM correlaciona:

```yaml
rule: "Possível atacante persistente"
when:
  - count(ips.alerts.from_ip == X) > 5  WITHIN_LAST 10min
  - AND count(waf.alerts.from_ip == X) > 3 WITHIN_LAST 10min
then:
  - severity: CRITICAL
  - action: block_ip(X)
  - alert: SOC_TIER2
```

> 🔗 Como funciona correlação de eventos: [64.md](64.md).

## 🏛️ DMZ — Zona Desmilitarizada (Segmentação de Rede)

> **DMZ (*Demilitarized Zone*)** é uma **sub-rede intermediária** entre a internet (zona não confiável) e a rede interna corporativa (zona confiável). Hospeda serviços que precisam ser **acessíveis ao público** sem expor a rede interna.

O nome vem do termo militar — uma "zona desmilitarizada" é uma área tampão entre dois territórios potencialmente hostis. Em redes, é o mesmo conceito: uma **camada intermediária**.

### Topologia clássica de uma DMZ

```
┌─────────────────────────────────────────────────────────────┐
│                      INTERNET                               │
│                  (zona não confiável)                       │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
                  ┌─────────────────────┐
                  │  🔥 Firewall Externo│  ← regras frouxas (libera 80/443)
                  └──────────┬──────────┘
                             │
              ┌──────────────┴──────────────┐
              │                             │
              ▼                             ▼
  ┌─────────────────────────┐   ┌─────────────────────────┐
  │  📍 DMZ                 │   │  💻 REDE INTERNA        │
  │  (zona controlada)      │   │  (zona confiável)       │
  │                         │   │                         │
  │  • Servidor web         │   │  • Estações de trabalho │
  │  • Servidor de e-mail   │   │  • Banco de dados       │
  │  • Servidor DNS público │   │  • File server          │
  │  • WiFi de visitantes   │   │  • ERP, RH, etc.        │
  └────────────┬────────────┘   └────────────┬────────────┘
               │                             │
               │ pode acessar a interna?     │
               │ Não, sem regra explícita.   │
               └──────────────┬──────────────┘
                              ▼
                  ┌─────────────────────┐
                  │  🔥 Firewall Interno│  ← regras rígidas (bloqueia tudo, libera só específico)
                  └─────────────────────┘
```

### Por que DMZ existe — o cenário da Andriely

A Andriely precisa **liberar acesso para visitantes** (potencialmente máquinas comprometidas) **sem expor a rede interna**. Soluções:

| Abordagem | Problema |
|-----------|----------|
| ❌ Liberar acesso na rede interna | Máquina visitante comprometida → ataca tudo |
| ❌ Negar acesso a visitantes | Reuniões executivas prejudicadas |
| ✅ **Criar DMZ para visitantes** | Visitantes acessam serviços expostos, mas **não veem** a rede interna |

### Os 3 elementos clássicos de uma DMZ

1. **Firewall externo** — entre internet e DMZ (regras mais frouxas, permite tráfego web)
2. **Firewall interno** — entre DMZ e rede interna (regras rígidas, **deny by default**)
3. **Servidores** na DMZ — só os que **precisam** ser acessíveis externamente

### Tipos de implementação

#### Single Firewall (3-Legged DMZ) — mais simples e barato

```
       Internet
          │
          ▼
     [ Firewall ]
      ├──► DMZ (interface 1)
      └──► LAN interna (interface 2)
```

O **mesmo firewall** tem 3 interfaces de rede (Internet, DMZ, LAN). Configurações de regras controlam o que passa entre cada zona.

> ✅ **Vantagem:** custo menor (só 1 firewall)
> ❌ **Desvantagem:** se o firewall cai, **toda a proteção cai**

#### Dual Firewall — mais robusto e padrão enterprise

```
   Internet → [Firewall Externo] → DMZ → [Firewall Interno] → LAN
```

**Dois firewalls** em série, ideal de **fornecedores diferentes** (defense in depth real).

> ✅ **Vantagem:** se um firewall é comprometido, o outro ainda protege
> ✅ **Vantagem extra:** firewalls de vendors diferentes (Cisco + Palo Alto) → atacante precisa de **2 exploits diferentes**
> ❌ **Desvantagem:** custo dobrado, manutenção dobrada

### O que tipicamente vai na DMZ

| Recurso | Por que fica na DMZ |
|---------|---------------------|
| **Servidor web** (Apache/Nginx) | Precisa ser acessível pela internet |
| **Servidor de e-mail** (SMTP, IMAP) | Recebe e-mail externo |
| **DNS público** | Resolve nomes para clientes externos |
| **Servidor FTP/SFTP público** | Trocas de arquivo com parceiros |
| **WAF** ([67.md](67.md) seção 1) | Inspeciona tráfego antes de chegar ao app |
| **Reverse proxy** | Proteção e cache para apps internos |
| **WiFi de visitantes** | Como no caso da Andriely |

### O que **NÃO** vai na DMZ

| Recurso | Por que fica na rede interna |
|---------|------------------------------|
| Banco de dados de produção | Acesso só pela aplicação (DMZ → DB via regra específica) |
| Active Directory / LDAP | Identidade central, alvo de alto valor |
| File server interno | Documentos confidenciais |
| Backups | Não devem ser acessíveis externamente |
| ERP, RH, sistemas internos | Sem necessidade de exposição |

### DMZ × VLAN × Subnet — não confunda

| Conceito | O que é |
|----------|---------|
| **DMZ** | **Conceito de segurança** — sub-rede entre zonas de confiança diferentes |
| **VLAN** | **Tecnologia** que segmenta logicamente uma rede física (camada 2) |
| **Subnet** | **Divisão lógica** de uma rede IP (camada 3) |

> 💡 Uma DMZ pode **ser implementada como uma VLAN** ou subnet separada. Os termos não são mutuamente exclusivos — são **camadas diferentes** da mesma ideia.

### Conexão com WAF e outras defesas

Em arquiteturas modernas:

```
Internet → Firewall Externo → DMZ {WAF → Servidor Web} → Firewall Interno → DB Server
```

O **WAF mora na DMZ** porque precisa estar **na frente do servidor web** (que também está na DMZ), mas **antes** dos sistemas internos.

> 🔗 Sobre WAF, ver seção anterior deste documento.

### DMZ em ambiente cloud (AWS/Azure/GCP)

O conceito tradicional de DMZ evoluiu na cloud:

| Conceito on-premises | Equivalente cloud |
|----------------------|-------------------|
| Firewall externo | **Security Group** (AWS) / **NSG** (Azure) |
| DMZ | **Public Subnet** |
| Rede interna | **Private Subnet** |
| Firewall interno | **NACL** ou Security Group entre subnets |
| Reverse proxy | **API Gateway** / **Application Load Balancer** |

Mesma ideia, abstração diferente.

### Para o cenário da Andriely especificamente

```
┌────────────────────────────────────────────────────────────┐
│  Servidores de domínio web + Switches WiFi                 │
│  ─────────────────────────────────────────────             │
│  Visitantes conectam ao WiFi → recebem IP de uma           │
│  sub-rede DMZ → podem acessar apps da empresa via web      │
│  → MAS não veem a rede interna corporativa.                │
│                                                            │
│  Se uma máquina visitante estiver comprometida →           │
│  fica contida na DMZ, sem alcançar Active Directory,       │
│  banco de dados ou file servers.                           │
└────────────────────────────────────────────────────────────┘
```

> 🔑 **Princípio de defesa em profundidade aplicado:** ao isolar visitantes em uma DMZ, a Andriely reduz drasticamente o "raio de explosão" de um eventual incidente.

## 🚧 Limitações do WAF (Importante!)

Um WAF **não é solução mágica**. Limitações importantes:

### 1. Não é substituto para código seguro
> 🔑 **WAF é defesa em profundidade, não defesa principal.** Prepared statements ([57.md](57.md)), escape de HTML ([62.md](62.md)) e validação de input continuam essenciais.

### 2. Pode causar falsos positivos
Regras agressivas bloqueiam tráfego legítimo. Exemplo: um usuário chamado **`Larispudim's account`** ([57.md](57.md)) pode ser bloqueado por conter `'` mesmo sendo legítimo.

### 3. Pode ser contornado (bypass)
Existem técnicas conhecidas para evadir WAFs:
- **Encoding** (URL, Unicode, HTML entity)
- **Comentários inline** em SQL (`/*!50000UNION*/`)
- **Case mixing** (`UnIoN SeLeCt`)
- **Tamper scripts** do sqlmap ([60.md](60.md))
- **HTTP Parameter Pollution** (`?id=1&id=' OR 1=1`)

### 4. Não vê HTTPS sem decriptar
WAF precisa **terminar SSL/TLS** para inspecionar o conteúdo. Isso significa:
- Você precisa instalar certificados no WAF
- O tráfego do WAF até o servidor pode ser HTTP (interno) ou HTTPS de novo

### 5. Não previne DOM XSS
Como visto em [65.md](65.md), **DOM XSS acontece no navegador do cliente** — o WAF nem vê o ataque acontecer.

## 🛠️ Como Configurar um WAF (Conceitual)

Independente do tipo, todo WAF tem 4 elementos básicos:

```yaml
# Pseudo-configuração de WAF

regras_base:
  - OWASP Core Rule Set (CRS)  # regras prontas contra Top 10
  - Custom rules da sua app

exceções:
  - IPs de parceiros: [203.0.113.1, 198.51.100.5]  # whitelist
  - User agents internos: ["MonitoringBot/1.0"]

modo_operação:
  - Production: BLOCK    # ativo
  - Staging: DETECT      # só loga

resposta_a_ataque:
  - Bloqueio HTTP: 403 Forbidden
  - Rate limit: 100 req/min por IP
  - Notificação: SIEM (ver 64.md)
  - Bloqueio progressivo: 1ª vez → CAPTCHA, 2ª vez → IP banido 24h
```

| Termo | Definição |

| **Detection mode** | WAF só observa e loga, não bloqueia |
| **Prevention mode** | WAF bloqueia ativamente |

3. Em qual modo o WAF deve ser configurado **na primeira semana** após instalação? Por quê?
4. Por que WAF **não protege** contra DOM XSS?
5. Cite 3 técnicas de **bypass de WAF**.

<details>
<summary>Ver respostas</summary>

1. Porque firewall tradicional opera nas **camadas 3-4** (IP/porta/protocolo) e **não entende o conteúdo HTTP**. Um ataque SQLi via porta 443 (HTTPS) passa sem problemas pelo firewall — só o WAF detecta o `' OR 1=1` no parâmetro
2. **Startup SaaS:** WAF **cloud** (Cloudflare/AWS WAF) — escala automática, baixo custo inicial, manutenção pelo provedor. **Banco:** WAF de **rede (hardware)** — controle total, alta performance, compliance regulatório (BACEN, PCI-DSS)
3. **Detection / Monitor mode** — para identificar **falsos positivos** antes de bloquear tráfego legítimo. Bloquear de cara pode derrubar usuários reais
4. Porque DOM XSS acontece **inteiramente no navegador do cliente** (ver [65.md](65.md)) — o servidor entrega HTML genérico e o JavaScript do cliente é quem insere o payload no DOM. O WAF nem vê o ataque acontecer. A defesa é **CSP** + escape no JavaScript
5. **Encoding** (URL/Unicode), **comentários inline em SQL** (`/*!UNION*/`), **case mixing** (`UnIoN SeLeCt`), **tamper scripts** do sqlmap, **HTTP Parameter Pollution**, **fragmentação de payload**, **chunked encoding**

## 💡 Reflexão Final

> *"Firewall tradicional protege a porta da casa.*
> *WAF protege as conversas que entram pela porta.*
> *Um sem o outro deixa metade da casa aberta — mas WAF sem código seguro é mura ao redor de uma casa de papelão."*

### A regra de ouro do WAF

> 🔑 **WAF é defesa em profundidade — não defesa principal.**
>
> - **Primária:** código seguro (prepared statements, escape, validação)
> - **Secundária:** WAF bloqueando padrões conhecidos
> - **Terciária:** SIEM detectando o que escapou ([64.md](64.md))
> - **Quaternária:** Resposta a incidentes ([64.md](64.md))

Confiar **apenas no WAF** é como instalar trancas extras numa porta de papel — atacantes criativos sempre encontram um jeito. Mas **junto com código bem escrito**, o WAF eleva muito a barreira de entrada.

### Para o desenvolvedor

A pergunta a se fazer ao desenhar qualquer endpoint público:

> 🤔 *"Se um atacante mandasse `' OR 1=1--` ou `<script>alert(1)</script>` aqui, o que aconteceria?"*

- Se a resposta for **"nada — meus prepared statements ignoram"** → ✅ código seguro
- Se a resposta for **"meu WAF bloquearia antes de chegar"** → ⚠️ defesa frágil
- Se a resposta for **"acho que executaria"** → 🔴 código vulnerável

O WAF é a **última linha de defesa**, não a primeira.
