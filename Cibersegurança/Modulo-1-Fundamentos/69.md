# 🧪 Laboratório Prático: pfSense — Interface DMZEXT + Regra de Block Geral

> **Tema:** Configurar interface de rede e regra **deny-by-default** no firewall pfSense
> **Pré-requisitos:** [68.md](68.md) (ambiente VirtualBox montado com firewall + WAF + server)
> **Conceitos base:** [67.md](67.md) (DMZ, WAF, firewall)

---

## 🎯 Objetivo do Lab

Neste laboratório vamos:

1. **Acessar** o pfSense pela primeira vez (via interface host-only)
2. **Adicionar** uma nova interface de rede chamada **DMZEXT** (para a zona do WAF)
3. **Criar a primeira regra**: um **block geral** que **nega tudo por padrão**

> 🔑 **Por que começar bloqueando tudo?** Princípio fundamental de firewall: **deny by default**. Depois liberamos só o que é necessário. Mais seguro que o inverso.

---

## 📖 Antes de Começar — O Que é pfSense?

> **pfSense** é uma distribuição **open source** de **firewall/roteador** baseada em FreeBSD. É o "firewall enterprise" que cabe na sua casa.

| Característica | Detalhe |
|----------------|---------|
| **Licença** | Open source (Apache 2.0) |
| **Base** | FreeBSD |
| **Interface** | Web (HTTPS) — você configura tudo pelo navegador |
| **Recursos** | Firewall, NAT, VPN, IDS/IPS (com Snort/Suricata), captive portal, balanceador |
| **Usado por** | PMEs, labs, infraestruturas críticas, ISPs pequenos |
| **No mundo real** | Equivale a Cisco ASA, Fortinet FortiGate, Palo Alto |

> 💡 **Por que pfSense no curso?** É **gratuito, profissional e open source** — você pode ter exatamente o que grandes empresas usam, em uma VM no seu PC.

## 🛠️ Etapa 1 — Acessar o pfSense

Com a VM `firewall` ligada (do setup do [68.md](68.md)):

### 1.1 — Abrir o painel web

No seu navegador (do **host**, não da VM), acesse:

```
https://192.168.56.10
```

> ⚠️ **Aviso de certificado:** o pfSense usa certificado **autoassinado**. O navegador vai reclamar — clique em **"Avançado" → "Continuar mesmo assim"**. Em lab, isso é esperado.

### 1.2 — Login

| Campo | Valor |
|-------|-------|
| **Username** | `admin` |
| **Password** | `qwerty` |

> 🔴 **Em produção, NUNCA use credenciais padrão.** A primeira coisa a fazer num pfSense real é trocar a senha admin imediatamente. Mas em lab, mantemos para simplicidade.

### Por que `192.168.56.10`?

Esse é o IP da interface **host-only** do firewall (configurada no [68.md](68.md), Adaptador 2). O VirtualBox cria essa rede `vboxnet0` no range `192.168.56.0/24` por padrão — e a VM firewall foi pré-configurada para responder em `.10`.

## 🌐 Etapa 2 — Adicionar a Interface DMZEXT

O pfSense, ao iniciar, já reconhece algumas das interfaces (WAN, LAN, DMZ). Mas a **DMZEXT** (onde fica o WAF) ainda precisa ser ativada.

### 2.1 — Acessar Interface Assignments

No menu superior:

```
Interface → Assignments
```

### 2.2 — Adicionar a porta disponível

Na seção **"Available network ports"**:

1. Há uma interface disponível (geralmente `em3` ou `vmx3` — corresponde ao Adaptador 4 da VM)
2. Clique no botão verde **"+ Add"** ao lado
3. Ela aparece como **OPT2** (porta opcional 2)

### 2.3 — Configurar a OPT2 como DMZEXT

Clique no link **OPT2** (na tabela superior) para editar e preencha:

```yaml
Enable: ✓ check
Description: DMZEXT

IPv4 Configuration Type: Static IPv4
IPv6 Configuration Type: None

MAC Address:    <vazio>
MTU:            <vazio>
MSS:            <vazio>
Speed and Duplex: <vazio>

IPv4 Address: 172.100.2.10 / 24
IPv4 Upstream gateway: None

Block private networks and loopback addresses: <vazio>
Block bogon networks: <vazio>
```

### 2.4 — Salvar e aplicar

- **Save** (botão azul no rodapé)
- **Apply Changes** (botão verde no topo, aparece após salvar)

> ✅ **Validação:** vai aparecer **DMZEXT** no menu Interface, com IP `172.100.2.10/24`.

### Por que esses valores?

| Campo | Valor | Por quê |
|-------|-------|---------|
| **Enable** | ✓ | Interface começa desabilitada — precisamos ativar |
| **Description: DMZEXT** | texto | Nome amigável que aparece em menus e regras |
| **Static IPv4** | sim | Em lab queremos IP fixo (não DHCP), para estabilidade |
| **IP `172.100.2.10/24`** | range privado | Rede `172.100.2.0/24` — separada das demais para isolar |
| **Upstream gateway: None** | sem gateway | Esta interface **não tem rota** para fora — é DMZ interna |
| **Block private networks** | vazio | Em interface interna, **queremos** aceitar IPs privados |
| **Block bogon networks** | vazio | Em interface interna, não precisa filtrar IPs reservados |

> 💡 As duas últimas opções (`Block private/bogon`) só fazem sentido em interfaces **WAN** (de cara para internet) — não em DMZ interna.

## 🚧 Etapa 3 — Criar a Regra de Block Geral

Agora a parte conceitualmente mais importante: definir que **por padrão, nada passa**.

### 3.1 — Acessar Firewall Rules

No menu superior:

```
Firewall → Rules → DMZEXT
```

> 💡 Cada interface tem suas próprias regras. Estamos definindo as regras que se aplicam ao tráfego **entrando na interface DMZEXT** (vindo do WAF).

### 3.2 — Adicionar nova regra

Logo abaixo da tabela vazia, clique no botão verde **"↑ Add"** (com a seta para cima — isso significa "adicionar no topo da lista"). Configure:

```yaml
Action: Block
Disabled: <vazio>
Interface: DMZEXT
Address Family: IPV4
Protocol: Any

Source: any
Destination: any

Log: ✓ check
Description: BLOCK GERAL
```

### 3.3 — Salvar e aplicar

- **Save** (botão azul no rodapé)
- **Apply Changes** (botão verde no topo)

> ✅ Vai aparecer a regra **"BLOCK GERAL"** com ícone de proibido (🚫) na tabela.

### O que cada campo significa

| Campo | Valor | Significado |
|-------|-------|-------------|
| **Action: Block** | Bloqueia e descarta silenciosamente | Diferente de **Reject** (que avisa o remetente) |
| **Interface: DMZEXT** | A regra se aplica nessa interface | Tráfego entrando pela DMZEXT |
| **Protocol: Any** | TCP, UDP, ICMP — qualquer um | Não discrimina protocolo |
| **Source: any** | De qualquer IP | Não importa de onde vem |
| **Destination: any** | Para qualquer IP | Não importa para onde vai |
| **Log: ✓** | Registra a tentativa nos logs | Essencial para auditoria/troubleshooting |
| **Description** | Texto livre | Documenta para você lembrar depois |

---

## 🧠 Por Que "Block Geral" é a Primeira Regra?

### O princípio: **Deny by Default**

Existem dois modelos de firewall:

| Modelo | Padrão | Filosofia |
|--------|--------|-----------|
| **Allow by default** | Permite tudo, bloqueia o ruim conhecido | Blocklist — ❌ ruim |
| **Deny by default** | **Bloqueia tudo**, libera só o necessário | Allowlist — ✅ bom |

> 🔑 **Em segurança, sempre allowlist > blocklist** ([55.md](55.md) discute isso no contexto de SQL Injection).

### Como funciona na prática

```
Regra 1 (topo):    PERMITIR tráfego HTTP do WAF para o server     ← exceção específica
Regra 2:           PERMITIR ping do firewall para o WAF             ← outra exceção
Regra 3 (fundo):   BLOCK GERAL (any → any)                          ← captura tudo o resto
```

O pfSense processa as regras **de cima para baixo**. A primeira que casar com o pacote é aplicada. Se nada casar antes, o **BLOCK GERAL** captura.

> 💡 **Em pfSense, regras adicionadas com seta "↑ Add" vão para o TOPO; com "↓ Add" vão para o FIM.** Como queremos o block geral como **última** regra (após exceções), o ideal é usar "↓ Add" — mas em lab vazio, tanto faz.

### O caso da Andriely revisitado

Lembra do cenário em [67.md](67.md), onde a Andriely queria isolar visitantes na DMZ? É **exatamente isso** que estamos fazendo aqui — **construindo** as regras que isolam zonas.

## ⚙️ Etapa 4 — Validação

Como confirmar que tudo funciona?

### 4.1 — Status da Interface

Menu: **Status → Interfaces**

DMZEXT deve aparecer como **"up"** com IP `172.100.2.10/24`.

### 4.2 — Regra ativa

Em **Firewall → Rules → DMZEXT** deve aparecer:

```
[Ícone Block] | * | * | * | * | none | BLOCK GERAL
```

### 4.3 — Testar o bloqueio

Da VM **WAF** (que está na rede DMZEXT), tente pingar o firewall:

```bash
ping 172.100.2.10
```

**Resultado esperado:** **timeout** (sem resposta). A regra está bloqueando o ICMP de WAF para firewall.

### 4.4 — Confirmar no log

No pfSense: **Status → System Logs → Firewall**

Vai aparecer entradas como:

```
Block | DMZEXT | 172.100.2.20 → 172.100.2.10 | ICMP
```

Cada tentativa bloqueada gera uma linha. **Isso é ouro para forense** ([64.md](64.md) explica como SIEM correlaciona esses logs).

## 🛡️ Princípios de Regras de Firewall

Algumas regras (literais) sobre regras de firewall:

### 1. Ordem importa
Pfsense lê **de cima para baixo**, primeira regra que casa ganha. Coloque exceções **acima** do block geral.

### 2. Block ≠ Reject
- **Block**: descarta silenciosamente — atacante não sabe se o IP existe
- **Reject**: responde com **RST/ICMP unreachable** — confirma que algo está lá

Em geral, **Block** é mais seguro (não vaza informação).

### 3. Logar bloqueios é essencial
Sem logs, você não vê o que está sendo bloqueado — não pode ajustar regras, nem detectar ataques persistentes.

### 4. Regras específicas → regras genéricas
Coloque regras detalhadas (porta específica, IP específico) **antes** das gerais.

### 5. Tudo que não é permitido é negado
Versão simplificada do **deny by default**. Se não há regra explícita liberando, **block geral** captura.

## 🎯 Caso Prático — As 4 Regras da Andriely

> **Cenário:** Andriely configurou um firewall que lê regras **de cima para baixo**. A lista atual:
>
> 1. **Accept** tráfego de entrada na **porta 80 e 443** (HTTP/HTTPS) vindo de qualquer lugar
> 2. **Reject** tráfego de entrada na **porta 22 (SSH)** vindo de IPs de **pessoas não-técnicas** do escritório
> 3. **Accept** tráfego de entrada na **porta 22 (SSH)** vindo de IPs de **pessoas técnicas** do escritório
> 4. **Block** todo o tráfego (block geral)
>
> Como colega da Andriely, escolha as 3 afirmações **corretas** para o relatório.

### ✅ Gabarito: A + C + D

| Alt | Status | Por quê |
|-----|--------|---------|
| **A** | ✅ | Regra 1 aceita HTTP/HTTPS de qualquer origem |
| **B** | ❌ | **Erro proposital:** regra 2 trata de SSH, não HTTP. Funcionário de contabilidade acessando HTTP passa pela regra 1 |
| **C** | ✅ | Contabilidade = não-técnica + SSH = casa com regra 2 (Reject). Regras 3 e 4 não chegam a ser avaliadas |
| **D** | ✅ | Admin = técnico + SSH. Regras 1 e 2 não casam. Regra 3 aceita |

### 🔍 Análise de cada cenário, passo a passo

#### Cenário 1: Contabilidade tentando HTTP (a alternativa B)

```
Pacote: HTTP de IP da contabilidade
   ↓
Regra 1 (HTTP/HTTPS qualquer origem) → CASA → ACCEPT ✅
   ↓
Regras 2, 3, 4 → nunca avaliadas
```

> ⚠️ **Por isso B está errada:** funcionário de contabilidade **consegue sim** acessar HTTP. A regra 2 nem chega a ser verificada.

#### Cenário 2: Contabilidade tentando SSH (a alternativa C)

```
Pacote: SSH (porta 22) de IP da contabilidade
   ↓
Regra 1 (HTTP/HTTPS) → não casa (porta 22 ≠ 80/443)
   ↓
Regra 2 (Reject SSH não-técnicos) → CASA → REJECT ✋
   ↓
Regras 3, 4 → nunca avaliadas
```

✅ **C está correta:** bloqueio acontece exatamente pela regra 2.

#### Cenário 3: Admin tentando SSH (a alternativa D)

```
Pacote: SSH (porta 22) de IP do admin (técnico)
   ↓
Regra 1 (HTTP/HTTPS) → não casa
   ↓
Regra 2 (Reject não-técnicos) → não casa (admin é técnico)
   ↓
Regra 3 (Accept SSH técnicos) → CASA → ACCEPT ✅
   ↓
Regra 4 → nunca avaliada
```

✅ **D está correta:** o admin chega na regra 3 e é aceito.

#### Cenário 4: Qualquer outro tráfego (não testado, mas importante)

```
Pacote: FTP (porta 21) vindo de qualquer lugar
   ↓
Regra 1 (HTTP/HTTPS) → não casa
   ↓
Regra 2 (SSH não-técnicos) → não casa (não é SSH)
   ↓
Regra 3 (SSH técnicos) → não casa
   ↓
Regra 4 (BLOCK GERAL) → CASA → BLOCK ✋
```

Esse é o papel da regra 4: a **rede de segurança final**, capturando tudo que não foi previsto.

### 💡 As 3 lições do caso da Andriely

#### 1. Ordem das regras é tudo
Trocar regra 2 e 3 de posição **muda completamente** o comportamento. Se a regra "Accept SSH técnicos" viesse antes da "Reject SSH não-técnicos", apenas técnicos seriam aceitos (não-técnicos seriam bloqueados pela regra 4 — mesmo resultado funcionalmente, mas a lógica fica menos clara).

#### 2. Regras específicas antes de regras gerais
A ordem ideal é **mais específica → menos específica**:
- Regras com IP específico, porta específica → no topo
- Regras genéricas → no meio
- **Block geral** → sempre no fim

#### 3. A regra 4 (block geral) é o "safety net"

Se você esquecer de listar um protocolo no firewall, ele simplesmente **não passa**. Sem ela, qualquer protocolo não previsto seria liberado (allow by default — anti-padrão).

### 🔄 Reescrevendo de forma mais limpa

A configuração da Andriely está **correta**, mas pode ser tornada mais robusta:

```
1. ACCEPT   HTTP/HTTPS    ← públicos
2. ACCEPT   SSH from IPs_tecnicos  ← exceção positiva primeiro
3. BLOCK    SSH from any   ← regra de bloqueio mais geral (redundante mas explícita)
4. BLOCK    qualquer       ← deny by default
```

Essa estrutura **deixa a intenção mais clara**: "aceitamos HTTP/HTTPS e SSH só de técnicos. Resto bloqueia."

> 🔑 **Princípio de pentest:** *quem lê suas regras de firewall 6 meses depois (incluindo você mesmo) precisa entender a intenção sem precisar deduzir.*

### 🔗 Conexão com o conceito de Default Action

A regra 4 ("BLOCK qualquer") é **explícita**, mas em pfSense ela é o **comportamento padrão implícito** ([69.md](69.md) seção "Por que Block Geral é a primeira regra"). Em muitos firewalls (iptables, Cisco ASA), você precisa **declará-la explicitamente** — é uma boa prática mantê-la mesmo quando o firewall já faz isso por padrão.

## ✅ Quiz Mental

1. Por que acessamos pfSense em `192.168.56.10` e não em outro IP?
2. Por que **deny by default** é melhor que **allow by default**?
3. Se eu colocar o "BLOCK GERAL" **acima** das regras de exceção, o que acontece?
4. Por que ativamos **Log: ✓** na regra de block?
5. Qual a diferença entre **Block** e **Reject**?

<details>
<summary>Ver respostas</summary>

1. Porque é o IP da **interface host-only** do firewall — a única rede que o seu PC e a VM compartilham diretamente. As outras interfaces (WAN, DMZ, DMZEXT) não são acessíveis do host
2. Porque **allowlist > blocklist** sempre — você consegue antecipar o que precisa permitir, mas não consegue antecipar **todos** os ataques possíveis. Block default cobre os ataques desconhecidos automaticamente
3. **Nenhuma exceção funciona** — block geral pega primeiro tudo. É um erro clássico que derruba serviços. Lembre: ordem importa, do mais específico para o mais genérico
4. Para ter **rastreabilidade** — sem logs, você não vê tentativas de ataque, não pode ajustar regras incorretas, não pode encaminhar pro SIEM ([64.md](64.md))
5. **Block** descarta silenciosamente (atacante não sabe se IP existe). **Reject** responde com erro explícito (mais transparente, mas vaza informação). Em fronteira pública, **Block** é preferível

</details>

## 💡 Reflexão Final

> *"Toda regra de firewall que você adiciona é uma decisão de segurança documentada.*
> *Em produção, daqui a 6 meses, alguém vai olhar essas regras tentando entender por que algo está bloqueado.*
> *Por isso, **Description** é tão importante quanto a regra em si.*
>
> *Firewall sem documentação vira mistério — e mistério em segurança é dívida técnica explosiva."*

### A grande lição

> 🔑 **Deny by default + log everything + document everything.**
>
> Três princípios que diferenciam um firewall **bem configurado** de um firewall **que ninguém entende**.

E lembra do mantra de toda a família "Segurança de Rede":

> **WAF é especialista em HTTP. IPS é generalista em redes. Firewall é o porteiro. SIEM é o detetive.**
>
> Cada um faz sua parte — e o lab que você acabou de construir é a versão miniatura dessa orquestra.
