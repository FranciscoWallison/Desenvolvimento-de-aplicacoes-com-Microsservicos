## 🎯 Objetivo do Lab

Em [69.md](69.md) configuramos o firewall com **block geral** em todas as interfaces — ou seja, nada passa. Agora vamos **criar a primeira exceção controlada**: permitir tráfego **ICMP (ping)** dentro da rede DMZ, para podermos diagnosticar conectividade.

> 🔑 **Por que começar pelo ping?** Ping é a ferramenta de diagnóstico mais básica. Se você não consegue pingar, **nada vai funcionar** — APIs, websockets, banco de dados, todos dependem do mesmo caminho de rede. Permitir ping primeiro garante uma base mínima testável.

---

## 📖 Conceitos Novos Neste Lab

### 1. Pass × Block — as duas faces da moeda

| Action | O que faz | Quando usar |
|--------|-----------|-------------|
| **Pass** | Permite o tráfego passar | Regra de **exceção** explícita |
| **Block** | Descarta silenciosamente | Bloqueio default ou específico |
| **Reject** | Descarta e avisa o remetente | Quando quer ser transparente |
| **Match** | Marca o tráfego sem bloquear/passar | Para limitar banda, logs |

### 2. Aliases do pfSense — `DMZ net` × `DMZ address`

Quando você cria uma interface no pfSense, ele cria **automaticamente** dois aliases especiais:

| Alias | Significado | Exemplo (nossa DMZ) |
|-------|-------------|---------------------|
| **DMZ net** | A **subnet inteira** (todos os IPs daquela rede) | `172.100.1.0/24` |
| **DMZ address** | Apenas o **IP da interface** do firewall na DMZ | `172.100.1.10/32` |

> 💡 Isso evita você ter que digitar IPs hardcoded — se mudar a configuração da interface, as regras se ajustam sozinhas. **Boa prática.**

### 3. ICMP Subtypes — Echo Request × Echo Reply

ICMP tem vários "subtipos" (códigos de mensagem). Os 2 que importam para ping:

| Subtype | Quem envia | O que diz |
|---------|------------|-----------|
| **Echo request** | Quem **está pingando** | "Olá? Está aí?" |
| **Echo reply** | Quem **responde ao ping** | "Sim, estou aqui!" |

> 🔑 **Para ping funcionar, ambos precisam passar.** Se libera só request, o ping sai mas a resposta volta bloqueada → timeout. Sempre selecionar **os dois** (Ctrl+clique para multi-seleção).

Firewall → Rules → DMZ

Você verá a regra **BLOCK GERAL** já criada (do [69.md](69.md)) bloqueando tudo. Vamos adicionar uma **exceção acima dela**.

---

## ⚙️ Etapa 2 — Criar a Regra de Pass para ICMP

### 2.1 — Botão Add

Logo abaixo da tabela, clique no botão verde **↑ Add** (a seta para cima significa "adicionar no topo" — exatamente o que queremos, **acima** do block geral).

### 2.2 — Configurar a regra

Preencha conforme abaixo:

```yaml
Action: Pass
Disabled: <vazio>
Interface: DMZ
Address Family: IPV4
Protocol: ICMP
ICMP Subtypes: Echo reply + Echo request
                (segurar Ctrl para selecionar os dois)

Source: DMZ net
Destination: DMZ address

Log: ✓ check
Description: Permite ping na interface DMZ
```

### 2.3 — Salvar e aplicar

- **Save** (botão azul no rodapé)
- **Apply Changes** (botão verde no topo)

> ✅ A regra aparece **acima** do BLOCK GERAL na tabela, com ícone de check ✅ verde.

---

## 🧠 Por Que Cada Campo Foi Escolhido Assim

Vamos destrinchar a configuração — **cada decisão tem propósito**:

### `Action: Pass`

Permite o tráfego. Sem isso, o pacote sequer chegaria na regra (block geral pegaria).

### `Interface: DMZ`

Regra aplicada **apenas** ao tráfego que entra pela interface DMZ. Não afeta WAN, LAN, DMZEXT.

### `Protocol: ICMP`

Restringe a regra **somente** ao ICMP. Não libera TCP/UDP — quem quiser SSH ou HTTP cai no block geral.

### `ICMP Subtypes: Echo request + Echo reply`

Os dois subtipos do ping. Sem **echo reply**, a resposta volta bloqueada e o ping nunca completa.

### `Source: DMZ net`

Apenas **dispositivos da própria DMZ** (`172.100.1.0/24`) podem usar essa regra. Tráfego vindo de fora (LAN, WAN, DMZEXT) **não usa** essa regra.

### `Destination: DMZ address`

Destino é **apenas o IP do firewall na DMZ** (`172.100.1.10`). Não libera ping entre **outros** hosts da DMZ — só ping **para o firewall**.

### `Log: ✓`

Cada ping bem-sucedido gera log. Importante para:
- Auditar quem está testando conectividade
- Detectar **flood de ping** (ataques DoS via ICMP)
- Alimentar SIEM ([64.md](64.md))

## ✅ Etapa 3 — Validando a Regra

Hora de testar! Saímos do pfSense e vamos para a VM **server** (que está na rede DMZ).

### 3.1 — Login no server

No VirtualBox, abra a VM **server** e faça login:

```
Username: root
Password: qwerty
```

> 🔴 **Lembrete:** credenciais padrão são para **lab**. Em produção, **NUNCA** use `root/qwerty`.

### 3.2 — Pingar o firewall

No terminal do server, execute:

```bash
ping 172.100.1.10
```

### 3.3 — Resultado esperado

```
PING 172.100.1.10 (172.100.1.10) 56(84) bytes of data.
64 bytes from 172.100.1.10: icmp_seq=1 ttl=64 time=0.523 ms
64 bytes from 172.100.1.10: icmp_seq=2 ttl=64 time=0.412 ms
64 bytes from 172.100.1.10: icmp_seq=3 ttl=64 time=0.387 ms
...
```

✅ **Funciona!** Pacotes vão e voltam.

### 3.4 — Confirmar no log do pfSense

Voltando ao pfSense: **Status → System Logs → Firewall**

Vai aparecer entradas tipo:

```
Pass | DMZ | 172.100.1.20 → 172.100.1.10 | ICMP
```

(O `172.100.1.20` é o IP do server na rede DMZ; `172.100.1.10` é o firewall.)

### 3.5 — Confirmar que ainda funciona o bloqueio

Para garantir que **só ICMP** foi liberado (não tudo), tente um SSH:

```bash
ssh 172.100.1.10
```

**Resultado esperado:** timeout. SSH continua bloqueado pelo block geral. ✅ Comportamento correto.

## 🧠 Validando o Princípio "Ordem Importa"

Antes desse lab, a regra única na DMZ era:

```
1. BLOCK GERAL (any → any)   ← capturava TUDO
```

Agora ficou:

```
1. PASS ICMP (DMZ net → DMZ address)   ← exceção específica primeiro
2. BLOCK GERAL (any → any)              ← captura o resto
```

Quando o ping do server chega:

```
Pacote: ICMP echo request de 172.100.1.20 → 172.100.1.10
   ↓
Regra 1 (Pass ICMP) → CASA → PASS ✅
   ↓
Regra 2 (Block geral) → nunca avaliada
```

Quando o SSH do server tenta:

```
Pacote: TCP porta 22 de 172.100.1.20 → 172.100.1.10
   ↓
Regra 1 (Pass ICMP) → não casa (TCP ≠ ICMP)
   ↓
Regra 2 (Block geral) → CASA → BLOCK ✋
```

**É exatamente** a "ordem das regras" que vimos no caso da Andriely em [69.md](69.md).

## 🛡️ Princípios Reforçados Neste Lab

### 1. Allowlist explícita
Em vez de "permitir tudo, bloquear o ruim" (allow by default), fizemos **"bloquear tudo, permitir só o que preciso"** (deny by default). É **muito mais seguro**.

### 2. Específico antes de genérico
Pass ICMP (específica) **acima** de Block geral (genérica). Essa é a regra de ouro de qualquer firewall.

### 3. Mínimo necessário
Não liberamos "qualquer ICMP de qualquer lugar". Restringimos:
- Source = só da DMZ
- Destination = só para o firewall
- Subtypes = só echo request/reply

Quanto mais restrita a regra, **menor o risco** se algo vazar.

### 4. Logar permite e bloqueia
Logamos tanto a regra de Pass (`Log: ✓`) quanto a de Block (no [69.md](69.md)). Sem logs, não há como auditar nem detectar abuso.

## ⚠️ Pegadinhas Comuns

### "Liberei o ping mas não funciona"

**Causas possíveis:**
1. Selecionou só **Echo request** (faltou **Echo reply**) → resposta volta bloqueada
2. Source/Destination invertidos
3. Esqueceu de clicar em **Apply Changes** após Save
4. Regra adicionada **abaixo** do block geral (block pega antes)

### "Quero permitir ping de qualquer lugar"

Mude `Source` para `any` e `Destination` para `any`. **Mas cuidado:** isso pode permitir **DDoS por ICMP flood**. Em produção, sempre restrinja.

### "ICMP é só para ping?"

**Não.** ICMP tem dezenas de tipos:
- Echo request/reply (ping)
- Destination unreachable
- Time exceeded (usado por `traceroute`)
- Redirect
- Source quench

Por isso o pfSense permite escolher subtypes — geralmente queremos só ping.

## 💡 Reflexão Final

> *"Cada regra Pass é uma **decisão explícita** que diz: 'Eu confio neste tráfego'.*
> *Cada regra Block é uma **negação** que diz: 'Isso não deveria estar acontecendo'.*
>
> *Um firewall bem desenhado tem **poucas Pass** (mínimas, específicas) e **muitas Block** implícitas via deny by default.*
>
> *Se sua lista de Pass está crescendo demais — você está perdendo o controle."*

### O ciclo virtuoso do firewall

```
1. Tudo bloqueado por padrão       ← Block geral
2. Adicionar Pass mínima           ← "preciso disso"
3. Testar funcionamento            ← validar
4. Logar para auditar              ← rastreabilidade
5. Revisar trimestralmente         ← remover Pass não usadas
```

A maior fraqueza de um firewall **não é a complexidade técnica** — é a **fadiga de regras**: ninguém revisa, regras antigas acumulam, e em 2 anos você tem 500 Pass que ninguém mais entende.

> 🔑 **Princípio de ouro:** *firewall não é "configurar e esquecer". É infraestrutura viva que precisa de manutenção contínua.*
