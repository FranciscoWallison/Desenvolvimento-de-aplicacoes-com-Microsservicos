> **Tema:** Encontrar, filtrar e interpretar logs gerados pelo firewall
> **Pré-requisitos:** [69.md](69.md) (Block Geral) + [70.md](70.md) (Pass ICMP) — precisamos das regras com `Log: ✓`
> **Conexão crítica:** [47.md](47.md) — OWASP A09 (Security Logging and Monitoring Failures)

## 🎯 Objetivo do Lab

Em [69.md](69.md) e [70.md](70.md) criamos regras com a flag **Log: ✓** marcada. Agora vamos:

1. **Encontrar** onde os logs ficam no pfSense
2. **Usar a Dynamic View** para acompanhar tráfego **em tempo real**
3. **Filtrar logs** por IP, porta, ação — encontrar o que importa em meio ao ruído
4. **Interpretar** cada campo de um log do firewall
5. Conectar com o **OWASP A09** ([47.md](47.md)) na prática

> 🔑 **Sem análise de log, segurança vira fé.** Você "acha" que está bloqueando. Sem logs, não tem como confirmar.

---

## 📖 Por Que Logs São Críticos — A9 do OWASP Top 10

O **OWASP Top 10 2021** lista como **A09 — Security Logging and Monitoring Failures**: a ausência ou má qualidade de logs é uma das **10 maiores vulnerabilidades** de aplicações web.

> 🔗 Conceito completo em [47.md](47.md). Cenário prático de uso de logs com SIEM em [64.md](64.md).

### O que os logs do firewall te dão

| Capacidade | Sem log | Com log |
|-----------|:-------:|:-------:|
| Confirmar que bloqueios funcionam | ❌ Fé | ✅ Evidência |
| Detectar ataques persistentes | ❌ Cego | ✅ Padrão visível |
| Auditar conformidade (LGPD, PCI-DSS) | ❌ Reprova | ✅ Atende |
| Forense pós-incidente | ❌ Impossível | ✅ Timeline reconstruída |
| Ajustar regras (tuning) | ❌ Chute | ✅ Baseado em dados |
| Alimentar SIEM | ❌ Sem fonte | ✅ Pipeline funcional |

> 💡 **A grande sacada:** firewall sem log é cego. Você sabe **o que ele deveria fazer** (regras), mas não **o que ele está fazendo** (realidade).

## 📊 Etapa 2 — Acessar os Logs do Firewall

No pfSense, navegue:

```
Status → System Logs → Firewall
```

Você verá um painel com **abas** no topo:

| Aba | O que mostra |
|-----|--------------|
| **Normal View** | Logs em formato tabela tradicional (estático) |
| **Dynamic View** ⭐ | Logs **em tempo real** — novos eventos aparecem automaticamente |
| **Summary View** | Estatísticas agregadas (top IPs, top portas) |
| **Settings** | Configurações de log (rotação, remote logging) |

> 🎯 Vamos focar no **Dynamic View** — é o mais útil para análise ativa.

## 🔍 Etapa 3 — Dynamic View — Logs em Tempo Real

Clique em **Dynamic View**. A tela mostra logs sendo atualizados **automaticamente** a cada poucos segundos.

### Anatomia de uma linha de log

```
Time      | Interface | Rule | Action | Source IP:Port → Destination IP:Port | Protocol
21:34:52  | DMZ       | 3    | Pass   | 172.100.1.20 → 172.100.1.10           | ICMP
21:34:53  | DMZ       | 1    | Block  | 172.100.1.20:54321 → 172.100.1.10:22  | TCP:S
```

Cada campo importa:

| Campo | Significado |
|-------|-------------|
| **Time** | Quando aconteceu — para ordenar e correlacionar |
| **Interface** | Em qual interface (DMZ, DMZEXT, WAN, LAN) |
| **Rule** | Número da regra que casou (clique para editar!) |
| **Action** | **Pass** (✅ verde) / **Block** (🚫 vermelho) / **Reject** |
| **Source IP:Port** | Quem mandou o pacote |
| **→** | Direção do fluxo |
| **Destination IP:Port** | Para onde ia |
| **Protocol** | TCP, UDP, ICMP, etc. + flags TCP (`S` = SYN, `A` = ACK) |

### Códigos visuais comuns

| Símbolo | Significado |
|---------|-------------|
| ✅ Verde | **Pass** (regra permitiu) |
| 🚫 Vermelho | **Block** (regra bloqueou) |
| 🟡 Amarelo | **Reject** (bloqueou e avisou o remetente) |

## 🎯 Etapa 4 — Filtrar Logs Para Achar o Que Importa

Em um ambiente real, o firewall gera **milhares de linhas por minuto**. Filtrar é essencial.

### 4.1 — Abrir o painel de filtro

No canto superior direito do **Dynamic View**, há um ícone de funil (☰ ou similar) → clique para abrir o **filtro avançado**.

### 4.2 — Filtros disponíveis

| Campo | O que filtra |
|-------|--------------|
| **Source IP Address** | Origem do tráfego |
| **Destination IP Address** | Destino |
| **Source Port** | Porta de origem |
| **Destination Port** | Porta de destino (ex: 22 = SSH, 80 = HTTP) |
| **Protocol** | TCP, UDP, ICMP, etc. |
| **Action** | Pass, Block, Reject |
| **Interface** | DMZ, DMZEXT, WAN, LAN |
| **Time** | Período específico |

### 4.3 — Aplicar o filtro

Para encontrar o ping do server ao firewall (`70.md`):

```yaml
Destination IP Address: 172.100.1.10
```

Clique em **Apply Filter**. A lista é reduzida só ao tráfego com destino `172.100.1.10`.

### 4.4 — O log do ping aparece assim

```
21:34:52 | DMZ | 3 | Pass | 172.100.1.20 → 172.100.1.10 | ICMP echo request
21:34:52 | DMZ | 3 | Pass | 172.100.1.20 → 172.100.1.10 | ICMP echo reply
21:34:53 | DMZ | 3 | Pass | 172.100.1.20 → 172.100.1.10 | ICMP echo request
21:34:53 | DMZ | 3 | Pass | 172.100.1.20 → 172.100.1.10 | ICMP echo reply
```

✅ **Confirmação:** a regra **Pass ICMP** do [70.md](70.md) está funcionando, e cada ping deixa rastro.

E o SSH bloqueado:

```
21:35:10 | DMZ | 1 | Block | 172.100.1.20:48392 → 172.100.1.10:22 | TCP:S
```

✅ **Confirmação:** o **Block Geral** do [69.md](69.md) bloqueou o SSH. A flag `S` indica SYN — o atacante tentando iniciar uma conexão TCP.

## 🧪 Etapa 5 — Cenários Reais de Análise

### 🔍 Cenário 1: "Por que minha aplicação não conecta?"

**Sintoma:** server tenta acessar serviço X, dá timeout.

**Investigação no log:**
```yaml
Filtro:
  Source IP: <IP do server>
  Destination Port: <porta do serviço>
```

Se aparecer **Block** → regra bloqueando legitimamente, falta exceção.
Se **não aparece nada** → tráfego nem chegou ao firewall (problema de roteamento).

### 🔍 Cenário 2: "Tem ataque acontecendo?"

**Sintoma:** muitos pacotes block vindos do mesmo IP.

**Investigação:**
```yaml
Filtro:
  Action: Block
  Source IP: <IP suspeito>
```

Se aparecer **dezenas de tentativas em portas diferentes** → port scan.
Se **mesma porta + muitas conexões** → brute force.

### 🔍 Cenário 3: "Estou sendo escaneado?"

**Sintoma:** vários IPs distintos tentando portas comuns.

**Investigação:**
```yaml
Filtro:
  Action: Block
  Destination Port: 22 (ou 3389, 445, etc.)
  Tempo: últimas 24h
```

Se aparecer **muitos source IPs únicos** → varredura distribuída.

### 🔍 Cenário 4: "Meu site está sob DDoS?"

**Sintoma:** lentidão, queda de serviço.

**Investigação:**
```yaml
Filtro:
  Destination IP: <IP do server público>
  Destination Port: 80 ou 443
```

Se aparecer **milhares de conexões/min** → confirmar DDoS, considerar Cloudflare/AWS Shield.

## 🚀 Etapa 6 — Próximo Nível: Encaminhar para SIEM

Logs locais do pfSense são **bons para investigação pontual**, mas têm limites:

- ❌ Disco da VM é finito (rotação descarta logs antigos)
- ❌ Não consegue correlacionar com **outros sistemas** (WAF, servidor, IDS)
- ❌ Sem alertas automáticos
- ❌ Sem dashboards executivos

**Solução:** encaminhar logs do pfSense para um **SIEM** (Splunk, Graylog, Elastic, Wazuh).

### Como fazer em alto nível

No pfSense:

```
Status → System Logs → Settings → Remote Logging Options
```

Configure:
- **IP do servidor SIEM**
- **Porta** (geralmente 514 para syslog)
- **Quais logs enviar** (Firewall, System, DHCP, etc.)

Resultado: cada linha que aparece no Dynamic View **também** vai para o SIEM, onde:
- Fica armazenada por anos
- Pode ser correlacionada com logs de outros sistemas
- Dispara alertas automáticos (regras tipo *"se 100 blocks do mesmo IP em 1min → notificar SOC"*)

> 🔗 Cenário completo de SIEM em [64.md](64.md) — usando exatamente esse tipo de pipeline.

## 🧠 Princípios de Análise de Log

### 1. Não tente ler tudo — filtre
Em produção, é impossível ler log linha a linha. **Sempre filtre por IP, porta ou janela de tempo**.

### 2. Cruze tempo com contexto
Um Block às 14h não diz nada. Um Block às **3h da manhã de domingo** de um IP do **leste europeu** é um sinal completamente diferente — conecta com [64.md](64.md) (cenário SIEM).

### 3. Conte, não classifique
Em vez de classificar cada linha, **conte padrões**:
- "5 blocks de IP X em 1 min" → suspeito
- "1000 blocks de IP X em 1 min" → ataque ativo
- "5000 IPs únicos bloqueados na porta 22 em 1h" → varredura coordenada

### 4. Logue tudo, alerte pouco
Logar **muito** é bom — alertar **muito** é ruim (alert fatigue).
A regra ideal: log everything, alert on patterns.

### 5. Documente por que está logando
Cada regra com `Log: ✓` deve ter uma **Description** clara — em produção, alguém vai perguntar 6 meses depois por que estamos logando aquilo.

## ⚠️ Pegadinhas Comuns

### "Não vejo logs aparecendo"

**Causas possíveis:**
1. A regra que disparou **não tem `Log: ✓` marcado**
2. Tráfego nem chegou na interface — problema de roteamento
3. O filtro está muito restritivo
4. Dynamic View pausou (clique em **Pause/Resume**)

### "Vejo blocks de IPs internos"

**Investigar:**
- Pode ser tráfego legítimo que esqueceu de liberar (criar Pass)
- Pode ser scanner interno autorizado (whitelist)
- Pode ser malware lateral (tentativas anômalas)

### "Logs sumiram após reinicializar a VM"

**Causa:** logs estão em armazenamento volátil ou foram rotacionados.
**Solução:** configurar **Remote Logging** para um SIEM externo.

### "Sistema lento com Log: ✓ em todas as regras"

**Causa:** logar tudo em alto volume consome CPU/disco.
**Solução:** logar apenas as regras importantes (block geral, regras críticas).

<details>
<summary>Ver respostas</summary>

1. **Normal View** é estático (você precisa atualizar a página); **Dynamic View** se atualiza sozinho a cada poucos segundos. Dynamic é melhor para acompanhar tráfego em tempo real, Normal é melhor para investigar período passado
2. **TCP com flag SYN** — o pacote está tentando **iniciar uma conexão** TCP. Se for Block, alguém tentou abrir uma conexão e foi negado (típico de port scan / brute force)

5. Logs locais: capacidade finita (rotação), sem correlação entre sistemas, sem alertas automáticos, sem dashboards. SIEM resolve tudo isso — armazena anos, correlaciona com WAF/server/IDS, dispara alertas, gera relatórios para auditoria

## 🎯 Caso Prático — A Andriely e o A09 da OWASP

> **Cenário:** Andriely está estudando segurança da informação e notou que o **A09 — Security Logging and Monitoring Failures** está em **9º lugar** do OWASP Top 10 2021. Ficou curiosa: **que tipos de falhas** são essas? Como colega dela, selecione **3 alternativas verdadeiras**.

### ✅ Gabarito: A + B + C

| Alt | Status | Por quê |
|-----|--------|---------|
| **A** | ✅ | Perda de logs armazenados **apenas localmente** |
| **B** | ✅ | Logs **inexistentes ou pouco claros** |
| **C** | ✅ | Monitoramento **incapaz de detectar/alertar em tempo real** |
| **D** | ❌ | **Pegadinha** — pentest **não é falha**, é ferramenta para encontrar falhas |

### 📚 As falhas A09 oficiais da OWASP

A documentação da OWASP lista os seguintes **exemplos** de falhas categorizadas como A09:

| Tipo de falha | Por que é problema | Mitigação (deste curso) |
|---------------|--------------------|-------------------------|
| **Eventos auditáveis não são logados** | Login, falhas de auth, alterações de privilégio não geram log | Marcar `Log: ✓` em todas as regras críticas ([69.md](69.md), [70.md](70.md)) |
| **Logs sem contexto** | Falta IP origem, user-id, timestamp preciso | Configurar formato detalhado no pfSense (`Level of detail: full`) |
| **Logs armazenados só localmente** | Atacante apaga rastros, disco enche, hardware falha | **Remote Logging** para SIEM ([64.md](64.md)) |
| **Sem monitoramento ativo** | Logs existem mas ninguém olha — pilha cresce sem ação | Dashboard Dynamic View ([este doc](#etapa-3--dynamic-view--logs-em-tempo-real)) + alertas SIEM |
| **Sem alertas em tempo real** | Detecção média: ~200 dias entre invasão e descoberta | SIEM com regras de correlação ([64.md](64.md)) |
| **Sem resposta a incidentes** | Mesmo detectado, ninguém age | Playbooks ([40.md](40.md)) + IR Lifecycle ([51.md](51.md)) |

### ❌ Por que D está errada

> *"Testes de intrusão, feitos para descobrir as vulnerabilidades existentes, podem quebrar o gerenciador de logs."*

**3 erros conceituais nessa alternativa:**

#### 1. Pentest não é uma falha
Pentest é **ferramenta de descoberta de falhas**. Confundir "atividade que revela problemas" com "o problema em si" é como dizer que "termômetros causam febre".

#### 2. Pentest gerar volume de logs é esperado
Um pentest **deve** gerar muitos logs — scanners, brute force, varreduras tudo deixa rastro. Isso é **desejável** para validar que o pipeline de logging funciona sob carga.

#### 3. Se um pentest "quebra" o sistema de logs, o problema é o sistema
Se o gerenciador de logs **não aguenta** o volume de um pentest, ele já **estaria falhando** em produção sob ataque real. O pentest **revelou** a fragilidade — não causou.

### 🧠 Padrão para identificar uma falha A09

Pergunta-chave para qualquer cenário:

> 🔑 **"Sem esse problema, eu conseguiria detectar/investigar um incidente?"**

- ✅ **Se não conseguiria** → é falha A09
- ❌ **Se conseguiria mesmo assim** → não é A09

**Aplicando aos alternativas:**

| Alt | Sem essa falha, eu conseguiria? | É A09? |
|-----|--------------------------------|--------|
| **A** Logs perdidos | ❌ Não (sem evidência = sem investigação) | ✅ Sim |
| **B** Logs pouco claros | ❌ Não (não dá pra reconstruir timeline) | ✅ Sim |
| **C** Sem detecção em tempo real | ❌ Não (descobre tarde demais) | ✅ Sim |
| **D** Pentest derrubando logs | ✅ Sim (problema é capacidade, não logging em si) | ❌ Não — é outro tipo de problema (capacity/resilience) |

### ⏰ Casos famosos de A09 que viraram desastre

| Empresa | Ano | O que aconteceu |
|---------|-----|-----------------|
| **Equifax** | 2017 | Vazamento de 147M de pessoas — logs **existiam** mas certificado SSL do sistema de monitoramento estava expirado por 10 meses. Sem decifrar o tráfego, atacantes ficaram invisíveis |
| **Target** | 2013 | 40M de cartões vazados — alertas do FireEye **dispararam**, mas equipe **ignorou** (falha de processo, não tecnológica) |
| **Marriott** | 2018 | 500M de hóspedes vazados — atacante esteve na rede por **4 anos** sem ser detectado. Logging mal configurado |
| **SolarWinds** | 2020 | 18000 organizações comprometidas — backdoor passou em update legítimo, e logs de build não foram auditados |

> 🔑 Em todos os casos: **os logs existiam ou poderiam existir**. O que faltou foi **monitorar, alertar e agir** em cima deles.

### 💭 Frase-mantra do A09

> *"Logar tudo é metade da batalha. Olhar para os logs é a outra metade."*

Ou, dito de outra forma:

> 🔑 **Log sem análise = ruído.**
> **Análise sem log = adivinhação.**
> **Os dois juntos = capacidade de defesa real.**
