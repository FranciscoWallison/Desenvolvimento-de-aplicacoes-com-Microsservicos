> **Tema:** Capturar e analisar **pacotes brutos** no nível de rede usando o pfSense
> **Pré-requisitos:** [70.md](70.md) (Pass ICMP), [71.md](71.md) (análise de logs)

## 🎯 Objetivo do Lab

Em [71.md](71.md) vimos como **logs** mostram **as decisões do firewall** (Pass/Block) — ótimo para auditoria.
Mas e quando você precisa ver **o pacote em si** — MAC de origem, headers IP, payload, tráfego que **nem casou com regra alguma**?

Para isso existe o **Packet Capture** no pfSense — um **sniffer integrado** ao firewall.

Neste lab vamos:
1. Capturar pacotes ICMP gerados pelo ping
2. Interpretar **endereços MAC**, **IPs**, **ethertype**, **flags**
3. Entender quando usar **Packet Capture** vs **logs**
4. Ver tráfego que **não aparece nos logs** (porque nem chega às regras)

---

## 🆚 Logs × Packet Capture — Qual a Diferença?

Confusão muito comum. Vamos esclarecer:

| | **Logs do Firewall** ([71.md](71.md)) | **Packet Capture** (este doc) |
|--|---------------------------------------|-------------------------------|
| **Granularidade** | 1 linha por decisão | 1 entrada por pacote (vai e volta) |
| **Mostra** | Pass/Block, IP/porta, regra que casou | **Pacote completo** (MAC, headers, payload) |
| **Quando aparece** | Quando a regra tem `Log: ✓` | Sempre que pacote passa pela interface |
| **Captura tráfego não regrado?** | ❌ Não | ✅ Sim |
| **Mostra MAC?** | ❌ Não | ✅ Sim |
| **Performance** | Baixo impacto | Alto impacto (consome CPU/disco) |
| **Uso típico** | Auditoria, alertas, SIEM | Debug pontual, forense profunda |
| **Por baixo dos panos** | Logs estruturados do pf | **tcpdump** (libpcap) |

> 🔑 **Regra prática:**
> - **Logs** = você quer saber "o que está sendo permitido/bloqueado?"
> - **Packet Capture** = você quer saber "o que está nesse pacote exatamente?"

## 📖 Por Baixo dos Panos: tcpdump

O **Packet Capture do pfSense é uma interface gráfica para o `tcpdump`** ([66.md](66.md)). O FreeBSD (sistema base do pfSense) tem o tcpdump nativo e o pfSense apenas oferece um formulário para configurá-lo.

Tudo que você fizer pela GUI **gera um comando tcpdump** por trás. Exemplo:

```bash
# GUI: Interface DMZ + Protocol ICMP + Count 100
# Equivalente CLI:
tcpdump -i em2 -nn -v -c 100 icmp
```

> 💡 Saber isso ajuda no dia a dia — se você cair num pfSense sem GUI ou num servidor Linux qualquer, **tcpdump funciona igual**.

## 🛠️ Etapa 1 — Iniciar a Captura

### 1.1 — Acessar o painel

No pfSense (`https://192.168.56.10`):

```
Diagnostics → Packet Capture
```

### 1.2 — Configurar a captura

Preencha o formulário conforme:

```yaml
Interface: DMZ
Promiscuous: <vazio>
Address Family: Any
Protocol: ICMP
Host Address: <vazio>
Port: <vazio>
Packet Length: 0
Count: 100
Level of detail: full
Reverse DNS start: <vazio>
```

### 1.3 — O que cada campo significa

| Campo | Valor | Significado |
|-------|-------|-------------|
| **Interface** | DMZ | Captura tráfego que entra/sai dessa interface |
| **Promiscuous** | vazio | Modo promíscuo capta **tudo** na rede (até o que não é para o firewall) — em VM virtual, geralmente sem efeito |
| **Address Family** | Any | IPv4, IPv6 ou ambos |
| **Protocol** | ICMP | Filtra apenas pacotes ICMP (ping). Pode ser TCP, UDP, ARP, etc. |
| **Host Address** | vazio | Filtraria pacotes envolvendo um IP específico — vazio = qualquer |
| **Port** | vazio | Filtraria por porta — vazio = qualquer |
| **Packet Length** | 0 | Quantos **bytes** capturar de cada pacote (0 = pacote inteiro) |
| **Count** | 100 | Para automaticamente após 100 pacotes |
| **Level of detail** | full | Quanta informação mostrar (Normal, Medium, High, Full) |

### 1.4 — Botão Start

Clique em **Start** ao final da página. A página fica "carregando" — significa que a captura **está acontecendo**.

## 🌊 Etapa 2 — Gerar Tráfego

Sem fechar a aba do pfSense, abra a VM **server** e execute:

```bash
ping -c 3 172.100.1.10
```

O `-c 3` envia exatamente **3 pacotes** e para. Isso é melhor que `ping` infinito porque controla o volume capturado.

### 2.1 — Parar a captura

Volte ao pfSense e clique em **Stop** (botão amarelo no rodapé).

## 🔬 Etapa 3 — Interpretando a Captura

O pfSense vai exibir algo como:

```
17:35:31.805055 08:00:27:0e:5c:05 > 08:00:27:7a:2b:8d, ethertype IPv4 (0x0800),
length 98: (tos 0x0, ttl 64, id 30578, offset 0, flags [DF], proto ICMP (1), length 84)
    172.100.1.100 > 172.100.1.10: ICMP echo request, id 48707, seq 1308, length 64

17:35:31.805096 08:00:27:7a:2b:8d > 08:00:27:0e:5c:05, ethertype IPv4 (0x0800),
length 98: (tos 0x0, ttl 64, id 0, offset 0, flags [DF], proto ICMP (1), length 84)
    172.100.1.10 > 172.100.1.100: ICMP echo reply, id 48707, seq 1308, length 64
```

Parece intimidador no início. Mas tem estrutura clara — vamos destrinchar.

### 3.1 — Anatomia de uma linha

```
17:35:31.805055   ← TIMESTAMP (alta precisão, microssegundos)
08:00:27:0e:5c:05 > 08:00:27:7a:2b:8d   ← MAC origem → MAC destino (camada 2)
ethertype IPv4 (0x0800)                  ← Tipo de pacote na camada 2
length 98                                ← Tamanho total em bytes
(tos 0x0, ttl 64, id 30578, ...)         ← Headers IP (camada 3)
172.100.1.100 > 172.100.1.10             ← IP origem → IP destino
ICMP echo request, id 48707, seq 1308    ← Payload (camada de aplicação)
```

### 3.2 — Camadas que aparecem

```
┌───────────────────────────────────────────────┐
│ Camada 2 (Enlace)                             │
│   MAC origem → MAC destino + ethertype        │
├───────────────────────────────────────────────┤
│ Camada 3 (Rede)                               │
│   IP origem → IP destino + tos/ttl/id/flags   │
├───────────────────────────────────────────────┤
│ Camada 4+ (Transporte/Aplicação)              │
│   ICMP echo request/reply + id + seq          │
└───────────────────────────────────────────────┘
```

Isso mapeia diretamente o **modelo TCP/IP** que vimos em [66.md](66.md).

### 3.3 — Campos importantes para forense

| Campo | Por que importa |
|-------|----------------|
| **MAC origem/destino** | Identifica o **dispositivo físico** — pode revelar dispositivo desconhecido na rede |
| **ttl** | Time to live — quantos roteadores o pacote ainda pode atravessar. Útil para detectar **traceroute** ou **spoofing** |
| **id (IP)** | Identificador único do pacote — útil para detectar fragmentação ou pacotes duplicados |
| **flags [DF]** | "Don't Fragment" — pacote não pode ser dividido. Pacotes com DF dão MTU discovery |
| **ICMP id/seq** | Identifica qual processo ping está enviando — útil para confirmar se é a mesma sessão |

## 🕵️ Etapa 4 — Vendo Tráfego que NÃO Aparece nos Logs

Esse é o "superpoder" do Packet Capture. Vamos provar.

### 4.1 — Iniciar nova captura

Volte em `Diagnostics → Packet Capture` e configure:

```yaml
Interface: DMZ
Protocol: any           ← Importante: sem filtro de protocolo
Count: 100
Level of detail: full
```

Clique em **Start**.

### 4.2 — Gerar tráfego não regrado

Na VM **server**:

```bash
# ARP request — descobrir MAC do firewall
arping 172.100.1.10

# Ou simplesmente esperar — o sistema gera tráfego ARP periodicamente
```

### 4.3 — Parar e observar

Você vai ver algo como:

```
17:42:12.123456 08:00:27:0e:5c:05 > Broadcast, ethertype ARP (0x0806), length 60:
    Request who-has 172.100.1.10 tell 172.100.1.100, length 46

17:42:12.124789 08:00:27:7a:2b:8d > 08:00:27:0e:5c:05, ethertype ARP (0x0806), length 60:
    Reply 172.100.1.10 is-at 08:00:27:7a:2b:8d, length 46
```

🎯 **Esse tráfego ARP não apareceu nos logs do firewall.** Porque ARP é **camada 2** — não passa pelas regras do firewall (que são camada 3/4).

### 4.4 — Por que isso importa

Vários ataques acontecem em **camada 2**:

| Ataque | O que faz | Por que firewall não vê |
|--------|-----------|-------------------------|
| **ARP Spoofing** | Atacante diz "eu sou o gateway" → recebe tráfego de todos | É ARP, não IP |
| **MAC Flooding** | Saturar tabela do switch para forçar broadcast | Camada 2 pura |
| **DHCP Starvation** | Esgotar pool de IPs do servidor DHCP | Antes do tráfego IP normal |
| **Rogue Access Point** | Wi-Fi falso atraindo conexões | Não passa pelo firewall na origem |

> 🔑 **Lição:** firewall + logs são **necessários, mas não suficientes**. Para defesa completa, você precisa **também** olhar a camada de enlace com sniffer.

## 📊 Etapa 5 — Filtros Avançados

Em produção, capturas amplas geram **gigabytes** de dados. Filtros essenciais:

### Filtrar por host específico

```yaml
Host Address: 172.100.1.20    ← captura só pacotes envolvendo este IP
```

Equivalente tcpdump: `tcpdump -i em2 host 172.100.1.20`

### Filtrar por porta

```yaml
Protocol: TCP
Port: 22                       ← só SSH
```

Equivalente tcpdump: `tcpdump -i em2 tcp port 22`

### Filtrar por subnet

```yaml
Host Address: 172.100.1.0/24
```

Equivalente: `tcpdump -i em2 net 172.100.1.0/24`

### Capturar para análise externa (Wireshark)

O Packet Capture do pfSense permite **baixar o `.pcap`** após parar a captura. Esse arquivo abre direto no **Wireshark** — que tem GUI completa para análise profunda.

> 💡 **Workflow profissional:** captura no pfSense → baixa `.pcap` → abre no Wireshark → análise visual com filtros gráficos.

## 🚨 Cuidados em Produção

### 1. Performance

Capturar **tudo** em uma interface de alta velocidade pode **derrubar** o pfSense por:
- CPU 100% (compactando/escrevendo pacotes)
- Disco cheio (cada GB de tráfego = GB de captura)
- Memória esgotada

**Solução:** sempre limitar com `Count`, `Host`, `Port`.

### 2. Privacidade / LGPD

Captura crua pode conter:
- Senhas em texto claro (se HTTP, não HTTPS)
- Conteúdo de e-mails
- Dados pessoais (CPF, dados bancários)

> 🔴 **Em produção:** capture **com permissão**, **por tempo limitado**, **com motivo documentado**, e **delete depois**. LGPD/GDPR se aplicam.

### 3. Não capture indefinidamente

Capturas longas viram **dívida técnica de privacidade**. Sempre defina prazo.

## 🎓 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Packet Capture** | Captura de pacotes brutos passando por uma interface |
| **tcpdump** | Sniffer CLI clássico (Unix/Linux) — base do pfSense Packet Capture |
| **Wireshark** | Sniffer GUI — abre arquivos `.pcap` |
| **libpcap** | Biblioteca usada por tcpdump/Wireshark para captura |
| **Promiscuous mode** | Captura tráfego que **não é para você** (em redes Wi-Fi/hub) |
| **Ethertype** | Identifica o tipo de pacote na camada 2 (0x0800 = IPv4, 0x0806 = ARP, 0x86DD = IPv6) |
| **TTL** | Time to live — número de roteadores que o pacote pode atravessar |
| **MTU / DF** | Tamanho máximo de pacote / Don't Fragment |
| **`.pcap`** | Formato padrão de arquivo de captura de pacotes |

## ⚠️ Pegadinhas Comuns

### "Não vejo nada na captura"

**Causas possíveis:**
1. Filtros restritivos demais (protocol + host + port)
2. Tráfego está em outra interface (escolheu DMZ mas tráfego é DMZEXT)
3. Modo promíscuo não funciona em VM em "NAT" (use bridge ou internal)

### "Capturei mas não consigo interpretar"

**Solução:** baixe o `.pcap` e abra no **Wireshark**. A GUI ajuda muito.

### "Capturei 10GB sem querer"

**Solução:** sempre definir `Count` ou `Packet Length` para evitar crescimento descontrolado.

### "Não vejo o conteúdo HTTPS"

**Esperado.** HTTPS é criptografado. Capture vê os pacotes (IPs, portas, tamanhos) mas **não o conteúdo**. Para decriptar precisa da chave privada do servidor (legal só se você é dono).

## ✅ Quiz Mental

1. Quando você usa **logs** vs **Packet Capture**?
2. Por que ARP **não aparece nos logs do firewall**?
3. O que significa **ethertype IPv4 (0x0800)** em uma captura?
4. Por que **`-c 3`** no ping é importante quando você está capturando?
5. Você capturou tráfego HTTPS. Por que não vê senhas em texto claro?

<details>
<summary>Ver respostas</summary>

1. **Logs** = quando você quer auditar decisões do firewall (Pass/Block, regras, padrões). **Packet Capture** = quando você precisa ver **o pacote real** (MAC, headers, payload) — para debugar problemas pontuais, fazer forense ou observar tráfego que nem passa pelas regras
2. Porque **ARP é camada 2 (enlace)** — opera com **MAC addresses**, antes do IP. As regras do firewall são camada 3-4 (IP/porta). Firewall simplesmente não tem o que decidir sobre ARP — ele "passa" sem ser avaliado
3. **Ethertype** identifica o tipo de pacote na camada 2. O valor **0x0800** é o código padronizado para **IPv4** (0x0806 = ARP, 0x86DD = IPv6). É como o firewall sabe "ah, esse pacote contém IPv4 lá dentro"
4. Sem `-c 3`, o ping fica **infinito** — vai gerando pacotes até você matar manualmente. Com a captura ativa, isso significa **logs crescendo sem parar** + risco de encher disco. Sempre limite o tráfego de teste
5. HTTPS usa **TLS** que criptografa o payload. A captura vê **IPs, portas, tamanhos de pacote, timing** — mas o conteúdo (senhas, URLs específicas, body) está cifrado. Para decifrar precisa da **chave privada do servidor** — possível só se você é dono (forense legítima) ou está fazendo MITM (ataque)

</details>

> *"Logs te contam **o que o firewall decidiu**.*

### A pirâmide da observabilidade de rede

```
       ┌───────────────────┐
       │ Alerta / SIEM     │ ← O que merece atenção?
       └─────────┬─────────┘
                 │
       ┌─────────┴─────────┐
       │ Logs do Firewall  │ ← O que decidi?
       └─────────┬─────────┘
                 │
       ┌─────────┴─────────┐
       │ Packet Capture    │ ← O que está acontecendo?
       └───────────────────┘
```

Quanto mais embaixo na pirâmide, **mais detalhe** mas **menos contexto**.
Quanto mais em cima, **mais síntese** mas **menos profundidade**.

> 🔑 **Profissional sênior sabe alternar entre as camadas.** Quando o alerta do SIEM não te diz o suficiente, você desce para os logs. Quando os logs não bastam, você ativa packet capture.

### As 3 camadas de observação

Saí desses dois labs com uma estrutura mental clara:

```
        ┌───────────────────┐
        │ Alerta / SIEM     │ ← O que merece atenção?  (64.md)
        └─────────┬─────────┘
                  │
        ┌─────────┴─────────┐
        │ Logs do Firewall  │ ← O que decidi?  (71.md)
        └─────────┬─────────┘
                  │
        ┌─────────┴─────────┐
        │ Packet Capture    │ ← O que está realmente passando?  (este doc)
        └───────────────────┘
```

Quanto mais para **baixo**, mais **detalhe** mas menos **contexto**. Quanto mais para **cima**, mais **síntese** mas menos **profundidade**.

> 🔑 **Profissional sênior sabe alternar entre as camadas.** Começa pelo alerta, desce para os logs se precisar de mais detalhe, ativa packet capture para forense pontual.

### A grande sacada da dupla logs + packet capture

A diferença essencial que finalmente entendi:

| Pergunta | Onde responder |
|----------|---------------|
| *"Quais conexões foram permitidas/bloqueadas?"* | **Logs** ([71.md](71.md)) |
| *"Qual regra causou esse bloqueio?"* | **Logs** |
| *"O que tem dentro deste pacote — MAC, headers, payload?"* | **Packet Capture** |
| *"Tem tráfego ARP/camada 2 acontecendo na rede?"* | **Packet Capture** (logs não veem) |
| *"Qual ataque está em curso agora?"* | **Logs filtrados** + **Packet Capture** focado |

### ⚠️ Sobre a inversão do enunciado original

A descrição da Alura diz que *"tráfego mostra só conexões com match em regras e logs mostram todas as conexões"* — mas a prática mostra **o contrário**:

- **Logs** registram **apenas** conexões que casaram com regras (com `Log: ✓`)
- **Packet Capture** captura **tudo** que passa pela interface — incluindo tráfego que **nem chega às regras** (como ARP em camada 2)

Foi exatamente isso que vi: capturei tráfego ARP que **não apareceu nos logs** porque opera em **camada 2** (MAC), antes do firewall que avalia camada 3-4 (IP/porta).

### As 3 lições que levo para o dia a dia

#### 1. Logs sem análise = ruído
Por melhores que sejam os logs, sem alguém olhando, viram **planilha gigantesca no fim do trimestre**. A `Dynamic View` mostra que análise pode ser **proativa** — não esperar incidente.

#### 2. Packet Capture é remédio, não vitamina
Captura crua **consome recursos** e tem **implicações de privacidade/LGPD**. Use **com motivo, por tempo limitado, e delete depois**. Não é monitoramento contínuo.

#### 3. ARP e outros ataques camada 2 são pontos cegos
Firewall + logs não veem ataques de camada 2 (ARP spoofing, MAC flooding, rogue AP). Para defesa completa, precisa de **outros controles**:
- 802.1X (autenticação por porta de switch)
- Dynamic ARP Inspection
- DHCP Snooping
- Port Security
