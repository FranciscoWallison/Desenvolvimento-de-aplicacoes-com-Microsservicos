## 🎯 Objetivo do Lab

Em [71.md](71.md) vimos que **logs locais têm limites**: capacidade finita, sem correlação entre sistemas, sem alertas automáticos. A solução é encaminhar logs para um **SIEM externo**.

Neste lab vamos:
1. **Importar** a 4ª VM do laboratório — o **Graylog**
2. Aprender 3 conceitos avançados do pfSense: **NAT 1:1**, **Virtual IPs**, **Interface INTRANET**
3. Configurar **acesso** do host (seu PC) à interface web do Graylog
4. Ter o **SIEM funcional** pronto para receber logs (próximo lab)

> 🔑 **Por que isso fecha o OWASP A09:** sem SIEM externo, perda de logs locais = perda de evidências. Sem correlação, ataques sutis passam despercebidos. Este lab elimina essas duas fraquezas.

## 📖 O Que é o Graylog?

> **Graylog** é uma plataforma **open source** de gerenciamento e análise de logs — categoria SIEM ([64.md](64.md)).

| Característica | Detalhe |
|----------------|---------|
| **Licença** | Open source (SSPL) — gratuito para uso interno |
| **Base** | Java + MongoDB (metadata) + Elasticsearch/OpenSearch (logs) |
| **Interface** | Web (HTTP, porta 9000) |
| **Recursos** | Coleta, parsing, dashboards, alertas, busca full-text |
| **Concorrentes** | Splunk (comercial), Elastic Stack/ELK, Wazuh, Chronicle |
| **Quando usar** | PMEs/labs/projetos que precisam de SIEM sem pagar licença |

### O que ele faz

```
┌──────────────┐    syslog/GELF    ┌──────────────┐
│  pfSense     │ ──────────────►   │              │
│              │                   │              │
│  (envia logs)│                   │   GRAYLOG    │
└──────────────┘                   │              │
                                   │  - Armazena  │
┌──────────────┐                   │  - Indexa    │
│  WAF         │ ──────────────►   │  - Pesquisa  │
└──────────────┘                   │  - Alerta    │
                                   │  - Dashboard │
┌──────────────┐                   │              │
│  Servidores  │ ──────────────►   │              │
└──────────────┘                   └──────────────┘
```

> 💡 **Ponto único de visibilidade** sobre logs de **vários sistemas** — exatamente o que falta nos labs anteriores.

## 🔧 Etapa 2 — Configurar NAT 1:1 no pfSense

Aqui entra um conceito **novo e importante**: **NAT 1:1** (One-to-One NAT).

### 2.1 — O que é NAT 1:1?

| | NAT comum (PAT) | **NAT 1:1** |
|--|-----------------|-------------|
| **Mapeamento** | N IPs internos → 1 IP externo (porta-based) | 1 IP interno ↔ 1 IP externo |
| **Direção** | Apenas saída | **Bidirecional** (entra e sai) |
| **Caso de uso** | Vários usuários acessando internet | Expor um servidor interno |
| **Analogia** | "Vários funcionários compartilhando 1 telefone" | "Cada funcionário tem sua própria linha direta" |

> 🔑 **Por que NAT 1:1 aqui?** O Graylog está em `172.100.1.101` (rede interna). Queremos que ele seja "visto" como `192.168.56.11` no host. NAT 1:1 faz exatamente essa tradução bidirecional.

### 2.2 — Acessar e adicionar

No pfSense (`https://192.168.56.10`):

```
Firewall → NAT → 1:1
```

Clique no botão verde **Add** e configure:

```yaml
Disabled: <vazio>
No BINAT (NOT): <vazio>
Interface: INTRANET
Address Family: IPv4
External subnet IP: Single host - 192.168.56.11
Internal IP: Single host - 172.100.1.101
Destination: Any
Description: Graylog nateando para intranet
NAT reflection: Use system default
```

### 2.3 — O que cada campo significa

| Campo | Significado |
|-------|-------------|
| **Interface: INTRANET** | A NAT acontece quando entra/sai por essa interface |
| **External subnet IP: 192.168.56.11** | O IP "público" (lado de fora) — como o host vai chamar o Graylog |
| **Internal IP: 172.100.1.101** | O IP real do Graylog na rede interna |
| **Destination: Any** | Qualquer destino — não restringe |
| **NAT reflection: system default** | Permite acessar de dentro da própria rede (avançado) |

### 2.4 — Salvar e aplicar

- **Save** (rodapé)
- **Apply Changes** (topo)

## 🌐 Etapa 3 — Criar Virtual IP

Agora precisamos dizer ao pfSense que ele "responde também" pelo IP `192.168.56.11` (do lado de fora). Isso é função do **Virtual IP**.

### 3.1 — O que é Virtual IP?

> **Virtual IP** é um **IP adicional** que uma interface "responde" — sem ter sido configurada como IP principal.

Casos de uso:
- **IP Alias** — adicionar um IP secundário (caso aqui)
- **CARP** — alta disponibilidade (failover)
- **Proxy ARP** — responder ARP por IPs não locais

### 3.2 — Configurar

No pfSense:

```
Firewall → Virtual IPs
```

Botão verde **Add**:

```yaml
Type: IP Alias
Interface: INTRANET
Address(es): 192.168.56.11/32
Description: Graylog na intranet
```

> 💡 **`/32`** significa "apenas este IP único" — não uma subnet inteira.

## 🔓 Etapa 4 — Criar Regra de Acesso

Lembra do **Block Geral** ([69.md](69.md))? Por causa dele, mesmo com NAT e Virtual IP configurados, o tráfego ainda é bloqueado. Precisamos criar uma **regra de Pass** para a porta 9000 do Graylog.

### 4.1 — Acessar regras da INTRANET

```
Firewall → Rules → INTRANET
```

Botão verde **↑ Add** (topo) — para a regra ficar **acima** do block geral:

### 4.2 — Configurar a regra

```yaml
Action: Pass
Disabled: <vazio>
Interface: INTRANET
Address Family: IPV4
Protocol: TCP

Source: INTRANET net
Destination: Single host or alias - 172.100.1.101
Destination Port Range: from 9000 to 9000

Log: ✓ check
Description: Acesso da INTRANET ao Graylog
```

### 4.3 — Por que esses valores?

| Campo | Por quê |
|-------|---------|
| **Action: Pass** | Estamos liberando, não bloqueando |
| **Protocol: TCP** | Graylog usa HTTP (TCP), não UDP nem ICMP |
| **Source: INTRANET net** | Apenas dispositivos da própria INTRANET podem acessar |
| **Destination: 172.100.1.101** | IP real do Graylog (não o IP externo do NAT) |
| **Port: 9000-9000** | Porta única da interface web do Graylog |
| **Log: ✓** | Auditar quem acessa o SIEM (boas práticas!) |

### 4.4 — Salvar e aplicar

## ✅ Etapa 5 — Validação

### 5.1 — Acessar a interface web

No navegador do **host** (seu PC):

```
http://192.168.56.11:9000
```

A tela de login do Graylog deve aparecer.

### 5.2 — Por que `192.168.56.11`?

Porque é o **External IP** que configuramos no NAT 1:1. O pfSense recebe a requisição em `192.168.56.11:9000`, traduz para `172.100.1.101:9000` (IP real do Graylog), e a regra de Pass autoriza a conexão.

## 🐞 Troubleshooting Comum

### "Página não carrega — timeout"

**Possíveis causas:**

#### 1. Porta 9000 não está aberta no Graylog
Confirme no terminal da VM Graylog:
```bash
ss -nlt
```

Se a porta 9000 não aparecer, **reinicie a VM**:
```bash
reboot
```

> 💡 Essa é uma dica conhecida do fórum da Alura — após reboot as portas costumam abrir corretamente.

#### 2. Block Geral pegando antes da regra Pass
Verifique se a regra Pass está **acima** do Block Geral em `Firewall → Rules → INTRANET`.

#### 3. NAT 1:1 mal configurado
Confirme em `Firewall → NAT → 1:1` que os IPs External (`192.168.56.11`) e Internal (`172.100.1.101`) estão corretos.

#### 4. Virtual IP não criado
Verifique em `Firewall → Virtual IPs` se o `192.168.56.11/32` está listado.

### "Página carrega mas é HTTPS recusada"

Graylog está em **HTTP**, não HTTPS. Use `http://...:9000` (não `https://`).

### "Não consigo nem chegar no firewall"

Verifique se a VM **firewall** está ligada e respondendo em `https://192.168.56.10`.

## 🧠 Conceitos Aprendidos (Avançados!)

| Conceito | Definição |
|----------|-----------|
| **NAT 1:1** | Mapeamento **bidirecional** 1-para-1 entre IP interno e externo |
| **NAT comum (PAT)** | Mapeamento N-para-1 com tradução de portas |
| **Virtual IP** | IP secundário que uma interface "responde" sem ser o principal |
| **IP Alias** | Tipo de Virtual IP — um IP extra na mesma interface |
| **CARP** | Tipo de Virtual IP para **alta disponibilidade** (failover) |
| **NAT reflection** | Permite acessar o IP NATeado de dentro da própria rede |
| **Interface INTRANET** | Nova interface de rede para isolar tráfego administrativo |
| **Graylog** | Plataforma open source de SIEM/log management |
| **GELF** | Graylog Extended Log Format — formato nativo de envio |

## ✅ Quiz Mental

1. Por que precisamos de **NAT 1:1** em vez de NAT comum (PAT)?
2. O que o **Virtual IP** faz, exatamente?
3. Por que a regra Pass usa `172.100.1.101` como destino, em vez do `192.168.56.11`?
4. Se eu não criasse a regra Pass, o que aconteceria?
5. Como isso resolve concretamente a categoria **A09 da OWASP**?

<details>
<summary>Ver respostas</summary>

1. NAT comum traduz N→1 com **portas diferentes** — funciona para saída (vários usuários acessando internet). Mas **não funciona para entrada** (host não sabe qual porta usar para chegar ao Graylog específico). NAT 1:1 cria um **mapeamento dedicado e bidirecional** — o host fala `192.168.56.11`, vai parar em `172.100.1.101`
2. Faz a interface INTRANET **responder também** pelo IP `192.168.56.11`, mesmo sem ele ser o IP principal. É como dar **um apelido extra** para a interface, que aceita conexões nesse IP adicional
3. Porque a regra Pass é avaliada **depois** do NAT. Quando a requisição chega no firewall em `192.168.56.11`, o NAT 1:1 já a traduziu para `172.100.1.101`. Então a regra Pass precisa autorizar a entrega para o **IP real**, não o externo
4. **Tudo continuaria bloqueado** — o NAT 1:1 e Virtual IP só dizem **"como traduzir"**, mas o **Block Geral** continua decidindo **"não passa"**. Precisamos da regra Pass para **autorizar explicitamente**
5. Centralizando logs num SIEM externo (Graylog), você **não perde** logs em falhas locais, ganha **correlação** entre sistemas, pode configurar **alertas em tempo real**, tem **retenção** longa para auditoria, e centraliza visibilidade — exatamente os 5 pontos do A09

</details>

## 💡 Reflexão Final

> *"Logs locais te contam o que aconteceu no firewall.*
> *Logs centralizados num SIEM te contam o que aconteceu na **organização inteira**.*
>
> *A diferença não é técnica — é de **visão**.*"

### O salto de visibilidade

Antes deste lab, cada sistema tinha sua própria caixinha de logs:
- pfSense → System Logs
- WAF → logs locais
- Server → /var/log/...
- Cada um isolado, sem correlação

Agora temos um **ponto único de entrada** para todos os logs do laboratório. Esse é o **modelo profissional**: um SIEM consolidando visibilidade sobre **toda a infraestrutura**.

### O mantra completo da família

> **Deny by default + log everything + capture when needed + document everything + analyze constantly + centralize for correlation.**

Adicionamos **"centralize for correlation"** — porque o lab provou que análise sem centralização vê apenas **fragmentos**, nunca o **todo**.

A orquestra de segurança de rede agora tem o **maestro** que faltava. 🎼🛡️
