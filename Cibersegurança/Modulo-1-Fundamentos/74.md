## 🎯 Objetivo do Lab

Em [71.md](71.md) vimos os logs **localmente** no pfSense (Dynamic View). Em [73.md](73.md) montamos o Graylog. Agora vamos **conectar os dois**: cada log que aparece no pfSense **também** vai para o Graylog.

Isso fecha definitivamente o ciclo do **OWASP A09** ([47.md](47.md)):

```
Antes deste lab:
   pfSense → logs LOCAIS → discos finitos → perda em incidentes ❌

Depois deste lab:
   pfSense → logs LOCAIS  ✅
          ↓
        SYSLOG (rede)
          ↓
       Graylog → armazenamento longo + correlação + alertas ✅
```

> 🔑 **Por que esse é o lab mais importante da família até agora:** sem este passo, ter Graylog não serve para nada. Logs precisam **chegar** lá.

## 📖 Conceitos-Chave

### O que é syslog?

> **syslog** é um **protocolo padrão** (RFC 5424/3164) para **transmissão de mensagens de log** entre sistemas.

Existe desde os anos 80 — é a "língua universal" do logging em Unix/Linux/network appliances.

```
┌──────────────┐                    ┌──────────────┐
│              │   syslog message   │              │
│  pfSense     │ ─────────────────► │  Graylog     │
│  (emissor)   │   UDP/TCP 514/1514 │  (receptor)  │
│              │                    │              │
└──────────────┘                    └──────────────┘
```

### Anatomia de uma mensagem syslog

```
<134>Jun 11 12:34:56 pfSense filterlog: 1,,,1000000103,em2,match,block,in,4,0x0,,64,12345,0,DF,1,icmp,84,172.100.1.20,172.100.1.10
 ↑       ↑           ↑       ↑          ↑
 PRI    timestamp   hostname process    payload (campos do filterlog)
```

| Campo | Significado |
|-------|-------------|
| **PRI** `<134>` | Facility + Severity codificados |
| **timestamp** | Quando aconteceu |
| **hostname** | Quem mandou (pfSense) |
| **process** | Qual processo gerou (filterlog, dhcpd, ssh, etc.) |
| **payload** | Conteúdo específico do log (no caso, dados do firewall) |

### Por que UDP 514 historicamente, e 1514 no nosso lab?

| Porta | Característica |
|-------|---------------|
| **514** | Porta **padrão** (well-known) de syslog. **Privilegiada** (< 1024) — requer root |
| **1514** | Porta **alta** (não-privilegiada). Pode ser usada por processos comuns |

> 💡 O Graylog VM provavelmente roda em **modo não-root** por segurança, então escuta em 1514 (não pode escutar em 514 sem privilégio).

### Por que isso resolve A09 do OWASP?

| Falha A09 | Como Remote Logging resolve |
|-----------|----------------------------|
| **Logs locais perdidos** em falhas | Logs já estão **fora** do sistema afetado |
| **Atacante apagando rastros** | Mesmo se invadir pfSense, os logs já saíram |
| **Sem retenção longa** | Graylog tem disco dedicado, sem rotação agressiva |
| **Sem correlação** | Graylog combina logs de pfSense + WAF + outros |
| **Sem alertas em tempo real** | Configuráveis no Graylog |

## 🛠️ Etapa 1 — Configurar Remote Logging no pfSense

### 1.1 — Acessar as configurações

No pfSense (`https://192.168.56.10`):

```
Status → System Logs → Settings
```

Role até a seção **Remote Logging Options**.

### 1.2 — Configurar

```yaml
Enable Remote Logging: ✓ check
Source Address: DMZ
IP Protocol: IPv4
Remote log servers: 172.100.1.101:1514
Remote Syslog Contents:
  Firewall Events: ✓ check
```

### 1.3 — Salvar

Clique em **Save** no final da página.

---

## 🧠 Por Que Cada Campo Foi Escolhido Assim

### `Enable Remote Logging: ✓`

Sem isso, nada é enviado. Óbvio mas crítico.

### `Source Address: DMZ`

O pfSense tem **várias interfaces** (WAN, LAN, DMZ, DMZEXT, INTRANET). Esse campo diz **qual IP de origem** aparece nos pacotes syslog enviados.

**Por que DMZ?**
Porque o Graylog (`172.100.1.101`) está na rede DMZ. Para o pacote chegar nele sem precisar atravessar **outras** interfaces, o pfSense usa seu IP da própria DMZ como origem.

> 💡 Se escolhêssemos LAN ou INTRANET, o pacote precisaria de **rota** para chegar à DMZ — pode dar timeout ou ser bloqueado.

### `IP Protocol: IPv4`

Toda nossa rede é IPv4. Se fosse dual-stack, escolheríamos IPv4+IPv6.

### `Remote log servers: 172.100.1.101:1514`

- **172.100.1.101**: IP real do Graylog na rede DMZ (não o `192.168.56.11` do NAT — esse é só para acesso do host)
- **:1514**: porta onde o Graylog está escutando para receber syslog

> ⚠️ **Pegadinha comum:** usar `192.168.56.11` aqui dá erro. Esse IP só existe via NAT 1:1 que configuramos em [73.md](73.md). Internamente, o pfSense precisa falar com o IP **real**.

### `Firewall Events: ✓`

Categoriza **quais tipos de log** mandar. Outras opções:

| Categoria | O que envia |
|-----------|-------------|
| **System Events** | Eventos do sistema (reboot, configuração) |
| **Firewall Events** | ✓ Decisões Pass/Block — **o que queremos!** |
| **DNS Events** | Consultas DNS |
| **DHCP Events** | Negociação DHCP |
| **PPP Events** | PPP/PPPoE |
| **VPN Events** | OpenVPN, IPsec |
| **Auth Events** | Login/logout |
| **All** | Tudo de uma vez (cuidado: volume alto) |

> 💡 Marcar **Firewall Events** envia exatamente o que vimos no [71.md](71.md) (Pass/Block). É o que precisamos para começar.

## ✅ Etapa 2 — Validar no Graylog

### 2.1 — Acessar o Graylog

No navegador do host:
```
http://192.168.56.11:9000
```

(A porta 9000 foi liberada em [73.md](73.md).)

### 2.2 — Confirmar que o Input está rodando

No Graylog: **System → Inputs**

Deve haver um **Syslog UDP Input** escutando na porta **1514** (essa parte foi pré-configurada na VM do curso, mas é importante saber que existe).

> 💡 Se não houver, precisa criar: **System → Inputs → Select Input: Syslog UDP → Launch new input**.

### 2.3 — Gerar tráfego para logar

No server (VM da DMZ), faça:

```bash
ping -c 3 172.100.1.10
ssh 172.100.1.10
```

Cada pacote vai bater nas regras do pfSense, gerar log local (`Log: ✓` que configuramos em [70.md](70.md) e [69.md](69.md)) e **agora também** vai sair via syslog.

### 2.4 — Ver no Graylog

No Graylog: **Search** (aba superior).

Você deve ver entradas chegando em tempo real. Cada linha mostra:
- **Timestamp**
- **Source** (será `pfSense`)
- **Message** (o conteúdo bruto do syslog)

> 🎯 **Pronto!** Os logs do pfSense agora vivem **fora** do firewall. Mesmo se o pfSense for invadido e logs locais apagados, **o Graylog tem cópia segura**.

## 🐞 Troubleshooting Comum

### "Não chegam logs no Graylog"

**Verifique na ordem:**

#### 1. Remote Logging está habilitado?
Confira em `Status → System Logs → Settings` se `Enable Remote Logging` está marcado.

#### 2. Source Address está certo?
Deve ser **DMZ** (mesma rede do Graylog). Se for LAN ou INTRANET, pacote pode não chegar.

#### 3. Graylog está escutando na 1514?
Na VM do Graylog, terminal:
```bash
ss -nlu | grep 1514
```
Se nada retornar, o input não está rodando. Recriar pelo painel web.

#### 4. Firewall bloqueando?
Embora syslog de pfSense PARA Graylog seja **outbound**, regras podem afetar. Verifique em `Status → System Logs → Firewall`.

### "Vejo logs no Graylog mas estão truncados"

**Causa:** UDP tem MTU limite (~1500 bytes). Logs muito longos são cortados.
**Solução:** usar **GELF TCP** em vez de UDP syslog (próximo lab).

### "Vejo logs antigos repetindo"

**Causa:** Graylog está fazendo replay ou cache.
**Solução:** `System → Indices → Recalculate index ranges`.

### "pfSense parou de mandar do nada"

**Causa:** disk full no pfSense (raro, mas acontece se tem outros logs ativos).
**Solução:** verificar `Status → Monitoring → Disk` e limpar.

## 🛡️ Princípios de Log Forwarding

### 1. Envie cedo, envie sempre
Quanto antes o log sai do sistema-fonte, **menos chance** de ser perdido ou adulterado.

### 2. Use TCP para logs críticos
**UDP** é mais rápido, mas **perde pacotes** sob carga. Para logs críticos (login, falhas de auth), use TCP ou **GELF TCP** (que garante entrega).

### 3. Não envie senhas em texto claro
Syslog padrão é **texto plano**. Use **TLS** (rsyslog TLS, syslog-ng TLS) em produção.

### 4. Tenha um plano de retenção
Logs precisam de **prazo claro**: 30 dias? 1 ano? 7 anos? Depende de:
- Requisitos legais (LGPD = direito ao esquecimento)
- Setor (financeiro = mais tempo)
- Tipo de log (login = mais; debug = menos)

### 5. Buffering local também
Se a rede para o SIEM cai, **o sistema-fonte deve guardar logs em fila local** e reenviar depois. pfSense faz isso por padrão.

## 🎓 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **syslog** | Protocolo padrão para enviar logs entre sistemas |
| **Remote Logging** | Configuração no pfSense para mandar logs por rede |
| **PRI** | Campo do syslog que codifica facility + severity |
| **Facility** | Categoria do log (kernel, mail, daemon, security) |
| **Severity** | Gravidade (0=Emergency, 7=Debug) |
| **GELF** | Graylog Extended Log Format — alternativa nativa ao syslog |
| **Porta 514** | Padrão histórico syslog UDP |
| **Porta 1514** | Não-privilegiada — usada em ambientes sem root |
| **Source Address** | IP de origem dos pacotes syslog (escolhido pela interface) |
| **Buffering** | Guardar logs localmente quando o destino está fora |

## ✅ Quiz Mental

1. Por que `Source Address: DMZ` em vez de LAN ou INTRANET?
2. Por que o IP é `172.100.1.101` e não `192.168.56.11` na configuração?
3. Por que usamos UDP 1514 em vez de 514?
4. O que aconteceria se o Graylog ficasse offline por 1 hora?
5. Como isso resolve concretamente a falha "logs apagados por atacante" do A09?

<details>
<summary>Ver respostas</summary>

1. Porque o Graylog está **fisicamente** na DMZ — usar Source Address DMZ é o caminho **direto** sem precisar de rota extra. Se fosse LAN/INTRANET, o pacote teria que ser roteado e pode bloquear por regras
2. Porque `192.168.56.11` é um **IP "fake"** criado pelo NAT 1:1 do [73.md](73.md) — só serve para acesso de FORA (do host). Internamente, o pfSense precisa falar com o **IP real** do Graylog na DMZ, que é `172.100.1.101`
3. Porque a porta **514 é privilegiada** (< 1024) e só processos rodando como **root** podem escutar nela. Para segurança, o Graylog roda como usuário não-root, então escuta em **1514** (porta alta, não-privilegiada)
4. O pfSense **bufferiza localmente** (guarda na fila interna) enquanto não consegue alcançar o Graylog. Quando o Graylog volta, o pfSense **reenvia** os logs acumulados. Sem este buffer, perderíamos esse 1h
5. O atacante pode apagar **logs locais** no pfSense — mas **os logs já estão no Graylog**, em outro sistema, com **outra credencial**. Para apagar de lá, o atacante precisaria comprometer o Graylog **também**, dobrando o trabalho e dando tempo para detecção

</details>

## 💡 Reflexão Final

> *"Log na máquina local é como anotar nome de quem entrou na casa **em um caderno na própria sala de entrada**.*
> *Se o ladrão entrar, ele leva o caderno também.*
>
> *Log remoto é como mandar o nome **para um cofre na casa do vizinho**.*
> *Para o ladrão sumir com o registro, precisa invadir a casa do vizinho também.*"

### O salto qualitativo

Antes deste lab, os logs eram **vulneráveis à mesma ameaça que estavam tentando detectar**. Agora eles vivem em **outro sistema, com outras credenciais, em outra zona de rede**.

Esse é o princípio de **separação de evidências** — fundamental em forense digital e exigido pela maioria das normas (LGPD, ISO 27001, PCI-DSS).

### O que mudou na visão sistêmica

Antes:
```
pfSense (logs aqui) ← atacante
```

Agora:
```
pfSense → syslog → Graylog (logs aqui) ← atacante teria que invadir 2 coisas
```

Conceito simples. **Diferença gigantesca**.

### Mantra atualizado

> **Deny by default + log everything + capture when needed + document everything + analyze constantly + centralize for correlation + ship logs immediately.**

Adicionamos **"ship logs immediately"** — porque o lab acabou de provar que **manter logs onde o ataque acontece é apostar contra si mesmo**.

A orquestra está afinando. Próximo movimento: dashboards e alertas. 🎼🛡️
