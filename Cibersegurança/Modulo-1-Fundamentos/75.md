# 🧪 Laboratório Prático: WAF → Graylog (Encaminhando Logs do nginx)

> **Tema:** Configurar o **WAF (nginx)** para enviar logs ao Graylog via syslog + liberar a regra correspondente no pfSense
> **Pré-requisitos diretos:** [73.md](73.md) (Graylog instalado), [74.md](74.md) (pfSense → Graylog funcionando)
> **Conceitos base:** [67.md](67.md) (o que é WAF)

## 🎯 Objetivo do Lab

Em [74.md](74.md) configuramos o **pfSense** para enviar logs ao Graylog. Agora vamos fazer o mesmo com o **WAF**.

Quando isso estiver feito, o Graylog terá **duas fontes** alimentando ele:
- 🔥 Logs do firewall (Pass/Block de rede)
- 🛡️ Logs do WAF (requisições HTTP, ataques bloqueados por padrões OWASP)

Essa é a configuração **mínima viável de SIEM real** — pelo menos duas fontes correlacionáveis. É exatamente o que falava o conceito de "**correlação multi-fonte**" do [64.md](64.md).

---

## 📖 O WAF é o nginx (com ModSecurity)

Antes de mexer nele, vale alinhar: o **WAF do nosso lab é o nginx** rodando **ModSecurity** como módulo. Vimos a explicação conceitual em [67.md](67.md):

> **nginx** é um web server / proxy reverso. **ModSecurity** é o motor de regras WAF (OWASP Core Rule Set). Juntos formam um WAF profissional, open source, gratuito.

Por isso, "configurar o WAF" neste lab significa **editar a configuração do nginx**.

### Logs do nginx — os dois arquivos clássicos

| Arquivo | O que registra |
|---------|----------------|
| `/var/log/nginx/access.log` | Cada **requisição HTTP** recebida (200, 404, 503) |
| `/var/log/nginx/error.log` | Erros + alertas (incluindo **bloqueios do ModSecurity**) |

Nosso objetivo: redirecionar esses dois para syslog → Graylog.

## 🛠️ Etapa 1 — Editar o `nginx.conf`

### 1.1 — Acessar a VM WAF

No VirtualBox, abra a VM **WAF** e faça login:

```
Username: root
Password: qwerty
```

> 🔴 Credencial padrão **só em lab**. Em produção, nunca.

### 1.2 — Abrir o nginx.conf

```bash
nano /etc/nginx/nginx.conf
```

### 1.3 — Localizar a linha do error_log

Procure a linha já existente:

```nginx
error_log /var/log/nginx/error.log;
```

### 1.4 — Adicionar as configurações de remote logging

**Logo abaixo** dessa linha, adicione:

```nginx
# Remote logging

access_log syslog:server=172.100.1.101:1514;
error_log syslog:server=172.100.1.101:1514;
```

### 1.5 — Salvar e sair

- **Ctrl + X** (sair)
- **S** + **Enter** (salvar)

## 🧠 Por Que Esses Comandos Especificamente

### A diretiva `access_log syslog:server=...`

Tradução: *"em vez de gravar requisições em arquivo, mande via syslog para o servidor X."*

Sintaxe completa:
```nginx
access_log syslog:server=<IP>:<porta>[,facility=<>][,tag=<>][,severity=<>];
```

| Parâmetro | Significado |
|-----------|-------------|
| `server=172.100.1.101:1514` | Endereço do Graylog na **rede DMZ** + porta syslog |
| `facility=` (opcional) | Categoria syslog (default: `local7`) |
| `tag=` (opcional) | Tag que aparece no log (default: `nginx`) |
| `severity=` (opcional) | Gravidade (default: `info`) |

### Por que `172.100.1.101` (IP interno)?

Mesma lógica do [74.md](74.md): o IP `192.168.56.11` é só **fachada** criada pelo NAT 1:1 para acesso do host. **Internamente**, o Graylog responde em `172.100.1.101` (rede DMZ).

> ⚠️ Tentar mandar para `192.168.56.11` daqui dá timeout.

### Por que `:1514`?

Porta **não-privilegiada** onde o Graylog escuta — explicado em detalhes no [74.md](74.md).

### Por que duas linhas (access_log + error_log)?

Porque o nginx tem **dois canais separados**:

| Canal | Tipo de conteúdo |
|-------|------------------|
| **access_log** | "Cliente X pediu URL Y, retornei status Z" |
| **error_log** | "Erro: regra ModSecurity #1234 bloqueou SQL Injection" |

Ambos são valiosos para o SIEM — bloqueios do WAF aparecem no **error_log**, então sem ele perdemos exatamente o que mais importa.

> 🔑 **Os bloqueios reais de ataque ficam no error_log**, não no access_log. Esquecer dele = SIEM cego para WAF.

## ✅ Etapa 2 — Validar a Configuração do nginx

Antes de aplicar, **sempre valide**. nginx tem um modo "dry-run":

### 2.1 — Testar sintaxe

```bash
nginx -t
```

**Resultado esperado:**
```
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
nginx: configuration file /etc/nginx/nginx.conf test successful
```

Se aparecer erro, **NÃO recarregue** — corrija primeiro o nginx.conf.

> 🔑 **Esse `nginx -t` é regra de ouro em produção.** Recarregar configuração com erro **derruba o servidor web**. O `-t` valida sem aplicar.

### 2.2 — Recarregar configuração

```bash
nginx -s reload
```

`-s reload` significa: *"recarregar configuração sem matar conexões ativas"*. Diferente de `systemctl restart` que **derruba** o serviço.

> 💡 **Reload × Restart** — em produção, **sempre prefira reload**. Restart causa downtime; reload é zero-downtime.

## 🔥 Etapa 3 — Criar Regra de Pass no Firewall

Como vimos em vários labs anteriores, o **Block Geral** do pfSense bloqueia tudo por padrão. Precisamos criar uma exceção para o tráfego **WAF → Graylog**.

### 3.1 — Acessar regras da DMZEXT

No pfSense (`https://192.168.56.10`):

```
Firewall → Rules → DMZEXT
```

> 💡 **Por que DMZEXT e não DMZ?** Porque o **WAF está na rede DMZEXT** (lembrando do setup do [68.md](68.md)). O pacote sai pela interface DMZEXT do firewall.

### 3.2 — Adicionar regra

Botão verde **↑ Add** (acima do Block Geral):

```yaml
Action: Pass
Disabled: <vazio>
Interface: DMZEXT
Address Family: IPV4
Protocol: UDP

Source: single host or alias - 172.100.2.100
Destination: single host or alias - 172.100.1.101
Destination Port Range: from 1514 to 1514

Log: ✓ check
Description: Envio de logs do WAF para o Graylog
```

### 3.3 — Por que cada valor

| Campo | Significado |
|-------|-------------|
| **Action: Pass** | Permite o tráfego (exceção ao Block Geral) |
| **Interface: DMZEXT** | Onde está o WAF |
| **Protocol: UDP** | Syslog padrão usa UDP (não TCP) |
| **Source: 172.100.2.100** | IP do WAF na rede DMZEXT — só ele pode usar essa regra |
| **Destination: 172.100.1.101** | IP do Graylog (DMZ) — destino exato |
| **Port 1514** | Porta de escuta do Graylog |
| **Log: ✓** | Auditar — quantos logs estão sendo encaminhados? |

### 3.4 — Salvar e aplicar

- **Save** (rodapé)
- **Apply Changes** (topo)

## 🐞 Troubleshooting Comum

### "nginx -t deu erro de sintaxe"

**Possíveis causas:**
1. Esqueceu ponto-e-vírgula no fim da linha
2. Faltou aspa em alguma config
3. Indentação errada (nginx é tolerante mas alguns parsers reclamam)

**Solução:** revise o trecho que adicionou. Compare com:

```nginx
# Remote logging
access_log syslog:server=172.100.1.101:1514;
error_log syslog:server=172.100.1.101:1514;
```

### "nginx -s reload deu erro"

Mesmo após `nginx -t` validar, o reload pode falhar se:
- O processo nginx **não está rodando** → use `systemctl start nginx`
- Não tem **permissão** → confirme que está como `root`

### "Não chegam logs do WAF no Graylog"

**Verifique na ordem:**

1. **Regra de Pass está ativa?** `Firewall → Rules → DMZEXT`
2. **WAF consegue alcançar o Graylog?** Da VM WAF: `ping 172.100.1.101`
3. **Graylog está escutando em 1514?** `ss -nlu | grep 1514` na VM Graylog
4. **nginx.conf foi salvo?** Reabra com `nano /etc/nginx/nginx.conf` e confirme as 2 linhas
5. **Configuração foi recarregada?** Rode `nginx -s reload` de novo

### "Logs do nginx pararam de aparecer no arquivo local"

**Causa esperada:** quando você usa `access_log syslog:...`, o nginx **substitui** o destino padrão.

**Solução:** se quiser **AMBOS** (arquivo local **e** syslog), adicione **duas linhas** de access_log:

```nginx
access_log /var/log/nginx/access.log;       # local
access_log syslog:server=172.100.1.101:1514; # remoto
```

## 🛡️ Princípios de Logging em WAF

### 1. Bloqueios são mais importantes que sucessos
Em logs de WAF, o **error_log** (bloqueios) tem prioridade maior que o **access_log** (requisições normais). Se precisar economizar, mande **só o error**.

### 2. Tagueie os logs por origem
Em produção, tagueie no nginx:

```nginx
access_log syslog:server=192.168.1.10:514,tag=waf-prod-01;
```

Assim no Graylog você filtra por `tag:waf-prod-01` — útil quando tem **vários WAFs**.

### 3. Não logue dados sensíveis no payload
Por padrão, nginx loga URL completa — incluindo `?cpf=12345678901`. Em produção, **sanitize**:

```nginx
log_format sanitized '$remote_addr - $request_uri_safe';
```

E configure `$request_uri_safe` pra mascarar campos sensíveis.

### 4. Reload, nunca restart
Já dissemos, mas vale reforçar: `nginx -s reload` mantém conexões ativas; `systemctl restart nginx` derruba.

### 5. Teste sempre com `-t` antes
Especialmente em produção. Já houve **outages de horas** por reload com sintaxe quebrada.

## ✅ Quiz Mental

1. Por que precisamos do **error_log** indo para syslog também, e não só do access_log?
2. O que `nginx -t` faz, exatamente? Por que ele é crítico em produção?
3. Diferença entre `nginx -s reload` e `systemctl restart nginx`?
4. Por que a regra de Pass usa o **IP específico** `172.100.2.100` em vez de `any`?
5. Se eu quiser **manter** o arquivo `/var/log/nginx/access.log` E mandar para syslog, como faço?

<details>
<summary>Ver respostas</summary>

1. Porque os **bloqueios do ModSecurity** (os ataques que o WAF efetivamente parou — SQLi, XSS, etc.) são registrados no **error_log**, não no access_log. Sem mandar o error_log, o SIEM fica **cego para o trabalho real do WAF**
2. `nginx -t` valida a sintaxe do arquivo **sem aplicar** a configuração. É crítico em produção porque um `reload` com config quebrada **derruba o nginx** — e usuários vendo "502 Bad Gateway" começa em segundos
3. **`nginx -s reload`**: envia sinal pro processo nginx ler a nova config **mantendo conexões ativas** = zero-downtime. **`systemctl restart`**: mata o processo nginx e abre de novo = **derruba serviço por alguns segundos**. Em produção, sempre prefira reload
4. **Princípio do menor privilégio**: a regra deve permitir **apenas** o que precisa — o WAF mandando log para o Graylog. Se permitirmos `any → 172.100.1.101:1514`, qualquer máquina comprometida na DMZEXT poderia enviar **logs falsos** para o SIEM (log poisoning). Com IP específico, só o WAF de verdade alimenta o Graylog
5. Adicionar **duas linhas** de access_log no nginx.conf: `access_log /var/log/nginx/access.log;` (mantém arquivo local) + `access_log syslog:server=172.100.1.101:1514;` (manda também via syslog). O nginx aceita múltiplos destinos pela mesma diretiva

</details>

## 💡 Reflexão Final

> *"O firewall te diz quem **tentou entrar**.*
> *O WAF te diz quem tentou **te enganar**.*
> *Junte os dois no mesmo SIEM, e você começa a ver o **rosto do atacante**.*"

### O salto de visibilidade

Antes deste lab, tinhamos **uma fonte** no SIEM (pfSense). Agora temos **duas** (pfSense + WAF). Pode parecer pouco — mas é a **diferença qualitativa** entre:

| Antes | Depois |
|-------|--------|
| "IP X foi bloqueado no firewall" | "IP X foi bloqueado no firewall **+** já tinha tentado SQLi 5 vezes no WAF antes" |
| Detecção: rede só | Detecção: rede **+** aplicação |
| Cobertura: camadas 3-4 | Cobertura: camadas 3-4 **+** 7 |

Quando o **server** também começar a mandar logs (próximos labs), teremos **3 perspectivas** sobre o mesmo evento. **Isso é defesa em profundidade real, com observabilidade.**

### Mantra atualizado

> **Deny by default + log everything + capture when needed + document everything + analyze constantly + centralize for correlation + ship logs immediately + multi-source for context.**

Adicionamos **"multi-source for context"** — porque o lab provou que **uma fonte só conta meio do filme**. Múltiplas fontes correlacionadas contam **a história inteira**.

A próxima fronteira é abrir o **server** para acesso público (porta 80) e começar a **simular ataques reais** — para finalmente ver o WAF bloqueando em tempo real, com tudo aparecendo no SIEM.
