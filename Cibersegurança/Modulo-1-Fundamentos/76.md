# 🧪 Laboratório Prático: Publicando o Server na INTERNET (NAT 1:1 + acesso ao Nginx para gerar logs)

> **Tema:** Expor o **WAF/server (nginx)** para a rede real via **NAT 1:1 + Virtual IP + regra Pass** na interface **INTERNET**, acessá-lo pelo navegador e **gerar logs** que chegam ao Graylog
> **Pré-requisitos diretos:** [73.md](73.md) (NAT 1:1 e Virtual IP — conceitos), [75.md](75.md) (WAF → Graylog já encaminhando logs)
> **Conceitos base:** [67.md](67.md) (o que é WAF), [68.md](68.md) (topologia + interface INTERNET/Bridge), [69.md](69.md) (Block Geral)
> **OWASP A09:** [47.md](47.md), [71.md](71.md)

---

## 🎯 Objetivo do Lab

Nos labs anteriores montamos o pipeline de logs: [74.md](74.md) mandou os logs do **firewall** ao Graylog e [75.md](75.md) mandou os logs do **WAF (nginx)** ao Graylog. Mas até agora **ninguém acessou o server de fora** — então o `access.log` do nginx estava praticamente vazio.

Neste lab vamos **fechar o ciclo**:

1. **Publicar** o WAF/server na rede **INTERNET** (a rede física real, via adaptador Bridge) usando **NAT 1:1** + **Virtual IP** + **regra Pass**
2. **Acessar** a página de boas-vindas do **Nginx** pelo navegador do seu PC físico
3. **Gerar logs de acesso HTTP** que — graças ao [75.md](75.md) — aparecem **automaticamente no Graylog**

> 🔑 **Por que isso importa:** um SIEM só tem valor com **tráfego real**. Este lab produz o primeiro acesso legítimo ao server, provando ponta a ponta que **acesso → nginx → syslog → Graylog** funciona. É o pré-requisito para os próximos labs (simular ataques e ver o WAF bloqueando no SIEM).

---

## 🧭 Por que precisamos de NAT aqui?

O server/WAF vive em `172.100.2.100`, numa **rede interna isolada** (DMZEXT — veja [68.md](68.md)). O seu PC físico está na sua **LAN doméstica** (algo como `192.168.100.x`). São **redes diferentes** — seu navegador **não tem rota** até `172.100.2.100`.

A solução é a mesma do [73.md](73.md): **NAT 1:1** cria um "espelho" do IP interno num IP da rede que você **alcança**. A diferença é a interface:

| | [73.md](73.md) — Graylog | **Este lab — WAF/server** |
|--|--------------------------|----------------------------|
| **Alvo interno** | `172.100.1.101` (Graylog) | `172.100.2.100` (WAF/nginx) |
| **Interface** | `INTRANET` (host-only, `192.168.56.x`) | **`INTERNET`** (Bridge, sua LAN real `192.168.100.x`) |
| **Quem acessa** | Só o seu host (rede host-only) | Qualquer dispositivo da **LAN real** |
| **Porta publicada** | `9000` (interface web do Graylog) | **`80` / `PORT_WEB`** (HTTP do nginx) |

> 💡 Publicar na **INTERNET** (Bridge) é o cenário mais próximo de "expor um servidor à internet de verdade". Por isso a regra de acesso será mais permissiva (`Source: any`) do que as regras internas do [75.md](75.md).

## 🔎 Etapa 0 — Descobrir um IP livre na sua rede

Como o IP externo tem que estar na **sua LAN real** (a que o adaptador Bridge enxerga), primeiro descobrimos qual rede é essa e um IP **livre** nela.

### 0.1 — Ver o IP do seu adaptador

- **Windows:** abra o `cmd` e rode `ipconfig`
- **Linux / macOS:** abra o terminal e rode `ifconfig` (ou `ip addr`)

Anote o IP do adaptador pelo qual você está conectado — ex.: `192.168.100.15`, máscara `255.255.255.0`. Isso indica a rede `192.168.100.0/24`.

### 0.2 — Achar um IP que ninguém está usando

Escolha um IP dessa faixa e faça `ping`:

```bash
ping 192.168.100.192
```

Interprete o resultado assim:

| Resultado do ping | Significado | Ação |
|-------------------|-------------|------|
| ❌ **Falha / timeout** (não responde) | Ninguém está usando esse IP | ✅ **Anote-o** — vamos usá-lo |
| ✅ **Sucesso** (respostas chegam) | O IP **já está em uso** | Teste **outro** IP |

> ⚠️ **Nota técnica:** ausência de resposta ao ping **geralmente** significa IP livre, mas um host pode estar ativo com **ICMP bloqueado**. Em lab é uma heurística boa o suficiente; em produção, reserve o IP via DHCP/IPAM em vez de "adivinhar por ping".

> 🔑 **Guarde o IP descoberto.** Nos exemplos abaixo usamos `192.168.100.192`, mas **o seu será diferente** — substitua em todos os campos.

## 🔧 Etapa 1 — NAT 1:1 (Firewall → NAT → 1:1)

Acesse o pfSense em `https://192.168.56.10` e vá em:

```
Firewall → NAT → 1:1
```

Botão verde **Add** e configure:

```yaml
Disabled: <vazio>
No BINAT (NOT): <vazio>
Interface: INTERNET
Address Family: IPv4
External subnet IP: Single host - 192.168.100.192   # ← o IP livre que você descobriu no ping
Internal IP: Single host - 172.100.2.100            # ← o WAF/server (fixo)
Destination: Any
Description: WAF na INTERNET
NAT reflection: Use system default
```

### O que cada campo faz

| Campo | Significado |
|-------|-------------|
| **Interface: INTERNET** | O NAT ocorre no adaptador Bridge — o que "toca" a sua LAN real |
| **External subnet IP: 192.168.100.192** | O IP "de fachada" que sua LAN vê — **descoberto pelo ping** (varia por máquina) |
| **Internal IP: 172.100.2.100** | O IP **real** do WAF/nginx na DMZEXT |
| **Destination: Any** | Não restringe a origem do tráfego |
| **NAT reflection: system default** | Comportamento padrão de reflexão (ver [73.md](73.md)) |

**Save** (rodapé) → **Apply Changes** (topo).

> 🧠 Recapitulando o [73.md](73.md): NAT 1:1 é **bidirecional** — pacote chegando em `192.168.100.192` é traduzido para `172.100.2.100`, e a resposta faz o caminho inverso. É isso que dá "endereço público" a um host interno.

## 🌐 Etapa 2 — Virtual IP (Firewall → Virtual IPs)

O NAT sozinho não basta: o pfSense precisa **responder pelo ARP** desse novo IP na rede INTERNET. Isso é papel do **Virtual IP** (tipo **IP Alias** — veja [73.md](73.md)).

```
Firewall → Virtual IPs
```

Botão verde **Add**:

```yaml
Type: IP Alias
Interface: INTERNET
Address(es): 192.168.100.192/32   # ← mesmo IP do ping; /32 = só este endereço
Description: WAF na INTERNET
```

**Save** → **Apply Changes**.

> 💡 **`/32`** = "apenas este IP único", não uma subnet. Sem o Virtual IP, a sua LAN pergunta *"quem tem 192.168.100.192?"* e **ninguém responde** — a conexão nem começa.

---

## 🔓 Etapa 3 — Regra Pass (Firewall → Rules → INTERNET)

Mesmo com NAT e Virtual IP prontos, o **Block Geral** ([69.md](69.md)) ainda derruba o tráfego. Precisamos de uma **exceção Pass** para a porta web na interface INTERNET.

```
Firewall → Rules → INTERNET
```

Botão verde **↑ Add** (para a regra ficar **acima** do Block Geral):

```yaml
Action: Pass
Disabled: <vazio>
Interface: INTERNET
Address Family: IPV4
Protocol: TCP

Source: any
Destination: Single host or alias - 172.100.2.100
Destination Port Range: from PORT_WEB to PORT_WEB    # PORT_WEB = 80 (HTTP)

Log: ✓ check
Description: Publicação do WAF na INTERNET
```

**Save** → **Apply Changes**.

### Por que cada valor

| Campo | Por quê |
|-------|---------|
| **Action: Pass** | Estamos **liberando** o acesso público (exceção ao Block Geral) |
| **Protocol: TCP** | HTTP roda sobre TCP (diferente do syslog UDP do [75.md](75.md)) |
| **Source: any** | É uma **publicação pública** — qualquer cliente da INTERNET pode acessar |
| **Destination: 172.100.2.100** | O **IP real** do WAF (a regra é avaliada **depois** do NAT — mesma lógica do [73.md](73.md)) |
| **Port: PORT_WEB (80)** | A porta HTTP do nginx (alias `PORT_WEB` definido no lab) |
| **Log: ✓** | Registrar **cada acesso** — é justamente o log que queremos ver no SIEM |

> ⚠️ **Contraste com o [75.md](75.md):** lá usamos `Source: 172.100.2.100` (só o WAF) por **menor privilégio**. Aqui `Source: any` é **intencional** — o objetivo é **expor** o serviço. Publicar à internet é sempre a regra mais arriscada: é exatamente ela que os próximos labs vão "atacar".

## ✅ Etapa 4 — Acessar o Nginx e ver o log no Graylog

### 4.1 — Abrir no navegador

No navegador do seu **PC físico**, acesse (com **o seu** IP descoberto):

```
http://192.168.100.192
```

Você deve ver a **página de boas-vindas do Nginx** ("Welcome to nginx!").

> 💡 Use **`http://`**, não `https://` — a publicação é na porta **80**.

### 4.2 — Confirmar o log no Graylog

Como o [75.md](75.md) já configurou o nginx para mandar o `access_log` via syslog, esse acesso gera um evento que chega ao Graylog.

1. Abra o Graylog: `http://192.168.56.11:9000` (do [73.md](73.md))
2. Vá em **Search**
3. Procure pela requisição — filtre por algo como `source:waf` ou pelo seu IP de origem, ou apenas recarregue a página e veja o evento mais recente aparecer

> 🔑 **É a prova ponta a ponta:** navegador → nginx (`172.100.2.100`) → `access_log syslog` → Graylog. A partir daqui, **todo acesso ao server é auditável no SIEM**.

## 🐞 Troubleshooting Comum

### "A página não abre / timeout"

Verifique **na ordem**:

1. **NAT 1:1** — `Firewall → NAT → 1:1`: External = IP do ping, Internal = `172.100.2.100`, Interface = INTERNET
2. **Virtual IP** — `Firewall → Virtual IPs`: `192.168.100.192/32` listado, Interface INTERNET
3. **Regra Pass acima do Block Geral** — `Firewall → Rules → INTERNET`
4. **Porta** — a regra libera `PORT_WEB` (80) e você acessou com `http://` (não `https://`)
5. **O nginx está no ar?** Na VM WAF: `nginx -t` e `ss -nlt | grep :80`

### "Escolhi um IP e o acesso vai para outro dispositivo da rede"

O IP externo **não estava livre** — outro host na LAN já o usa. Volte à **Etapa 0**, faça `ping` e escolha um IP que **não responda**.

### "Abre, mas não aparece nada no Graylog"

O caminho de logs é do [75.md](75.md). Confira:
- `access_log syslog:server=172.100.1.101:1514;` presente no `nginx.conf` (e `nginx -s reload` feito)
- A regra Pass **WAF → Graylog** (UDP 1514) ativa na DMZEXT
- O Graylog escutando: `ss -nlu | grep 1514`

### "HTTPS recusado / ERR_SSL"

A publicação é HTTP puro na porta 80. Force `http://` no navegador (ele pode estar tentando HTTPS automaticamente).

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Publicar um serviço** | Expor um host interno para uma rede externa via NAT 1:1 + Virtual IP + regra Pass |
| **Interface INTERNET** | O adaptador **Bridge** do firewall — conectado à LAN física real ([68.md](68.md)) |
| **Descoberta de IP livre por ping** | Heurística: IP que **não responde** ao ping costuma estar livre |
| **NAT 1:1 (recap)** | Mapeamento bidirecional 1↔1 entre IP interno e externo ([73.md](73.md)) |
| **Virtual IP / IP Alias (recap)** | IP secundário que a interface passa a "responder" ([73.md](73.md)) |
| **`Source: any`** | Regra de publicação pública — aceita clientes de qualquer origem |
| **PORT_WEB** | Alias para a porta HTTP (80) do nginx |
| **Loop de observabilidade** | Acesso real → nginx → syslog → SIEM, tudo auditável |

## ✅ Quiz Mental

1. Por que o server em `172.100.2.100` não é acessível direto do seu PC, e como o NAT 1:1 resolve isso?
2. Qual a diferença entre publicar na interface **INTERNET** (este lab) e na **INTRANET** ([73.md](73.md))?
3. Por que a regra Pass usa `172.100.2.100` como **Destination**, e não o IP externo `192.168.100.192`?
4. Por que aqui usamos `Source: any`, enquanto no [75.md](75.md) usamos um IP específico como Source?
5. O acesso à página do Nginx aparece no Graylog **por causa de que configuração feita antes**?

<details>
<summary>Ver respostas</summary>

1. Seu PC e o server estão em **redes diferentes** (sua LAN × DMZEXT isolada), então não há rota. O **NAT 1:1** cria um espelho do server num IP da **sua** rede (`192.168.100.192`); ao acessar esse IP, o pfSense traduz **bidirecionalmente** para `172.100.2.100`
2. **INTERNET** é o adaptador **Bridge** ligado à sua **LAN física real** — expõe o serviço para qualquer dispositivo dessa rede (cenário "internet de verdade"). **INTRANET** é host-only (`192.168.56.x`), acessível **só pelo seu host** — usado para administração (Graylog)
3. Porque a regra é avaliada **depois** do NAT. Quando o pacote chega, o NAT 1:1 **já traduziu** o destino para `172.100.2.100`; a regra Pass precisa autorizar a entrega ao **IP real**, não à fachada externa
4. Porque o **objetivo é oposto**: no [75.md](75.md) era tráfego **interno** (WAF → Graylog), onde vale o **menor privilégio** (`Source` = só o WAF, para evitar log poisoning). Aqui é uma **publicação pública** — queremos que **qualquer cliente** alcance o serviço, então `Source: any`
5. Pela configuração do **[75.md](75.md)**: o `access_log syslog:server=172.100.1.101:1514;` no `nginx.conf` faz o nginx enviar cada requisição HTTP ao Graylog. Sem ele, o acesso abriria a página mas **não geraria log no SIEM**

</details>

## 📝 Quiz Oficial do Curso — Função do NAT

> **Cenário:** Você e Andriely precisam instalar um SIEM no sistema da empresa. Para o firewall reconhecê-lo na rede, é preciso fazer um **NAT** do IP do SIEM. Andriely ficou na dúvida: **qual é a função do NAT?**

| | Alternativa | Veredito |
|--|-------------|----------|
| A | O NAT faz a inversão do proxy reverso do WAF | ❌ |
| B | O NAT é outro nome usado para VLANs, delimitando um range de IPs da VLAN | ❌ |
| C | O NAT é a forma utilizada para realizar roteamento em switches | ❌ |
| **D** | **O NAT traduz endereços IP privados (válidos só na rede interna) e também pode traduzir IP privado → IP público, usado para acessar a internet** | ✅ |

<details>
<summary>Ver gabarito comentado</summary>

**Resposta correta: D.**

**Por quê:** NAT (*Network Address Translation*) reescreve o endereço IP nos cabeçalhos dos pacotes — tipicamente traduzindo **IPs privados** ([RFC 1918: `10/8`, `172.16/12`, `192.168/16`], que não roteiam na internet) para um **IP roteável/público** (e vice-versa). No lab, é exatamente isso que o **NAT 1:1** faz: mapeia o IP interno do SIEM/WAF (`172.100.x`) para um IP que o firewall e a rede externa **enxergam** — permitindo a conexão. Ver [73.md](73.md) (NAT 1:1 conceitual) e a Etapa 1 deste doc.

**Por que as outras estão erradas:**
- **A** — "inversão do proxy reverso" não existe como função; proxy reverso é papel do **WAF/nginx** ([67.md](67.md)), camada 7, sem relação com tradução de IP (camada 3).
- **B** — **VLAN** segmenta a rede em **camada 2** (domínios de broadcast por `802.1Q`); NAT reescreve **IP em camada 3**. Conceitos distintos.
- **C** — roteamento em switches (L3 switching) **encaminha** pacotes por rota; **não reescreve** o endereço. NAT altera o endereço, não é roteamento.

</details>

## 💡 Reflexão Final

> *"Um SIEM sem tráfego é um palco vazio.*
> *O primeiro acesso legítimo ao server é a **primeira nota** da música que os próximos labs vão tocar."*

Até aqui construímos toda a **infraestrutura de observação** — firewall, WAF e SIEM conectados. Mas observação só faz sentido quando **há o que observar**. Este lab produziu o **primeiro evento real**: um GET na página do Nginx, atravessando NAT, firewall e WAF, e pousando no Graylog.

### O que mudou

| Antes | Depois |
|-------|--------|
| Server inalcançável de fora | Server **publicado** na LAN real |
| `access.log` do nginx vazio | Cada acesso vira **evento no SIEM** |
| SIEM configurado, mas "no escuro" | SIEM **enxergando tráfego real** |

### O que vem agora

Com o server exposto, a próxima fronteira é **atacá-lo de propósito** — SQLi, XSS, path traversal — para ver o **ModSecurity bloqueando** e comparar o SIEM **com e sem WAF**. Agora que a porta está aberta e o log está fluindo, o palco está pronto para o confronto. 🎼🛡️
