# 🧪 Laboratório Prático: Instalando o Snort (IDS/IPS) no pfSense

> **Tema:** Instalar e configurar o **Snort** como **IDS/IPS** na interface **INTERNET**, com bloqueio ativo (Block Offenders) e alertas para o syslog
> **Pré-requisitos diretos:** [68.md](68.md) (VMs + interface INTERNET), [76.md](76.md) (server publicado, gerando tráfego)
> **Conceitos base:** [67.md](67.md) (IDS × IPS, pfSense como UTM, Snort/Suricata)
> **SIEM:** [74.md](74.md) (pfSense → Graylog por syslog), [64.md](64.md)

---

## 🎯 Objetivo do Lab

Já temos **firewall** (camadas 3-4) e **WAF** (camada 7). Falta a peça do meio da imagem do [67.md](67.md): o **IPS** (camadas 3-7), que detecta e bloqueia **intrusões em qualquer protocolo** — não só HTTP.

Neste lab vamos:
1. **Instalar** o pacote **Snort** no pfSense (via Package Manager)
2. **Configurar** o Snort na interface **INTERNET** — em modo **IPS** (bloqueando ofensores)
3. **Habilitar feeds de regras** gratuitos (GPLv2, ET Open, OpenAppID, FEODO)
4. **Baixar as assinaturas** para o Snort ficar operacional

> 🔑 **IDS vs IPS (recap do [67.md](67.md)):**
> - **IDS** = "vejo algo estranho!" → só **alerta**
> - **IPS** = "vejo algo estranho — **bloqueado**!" → alerta **e age**
>
> O que transforma o Snort de IDS em IPS é **uma única opção**: `Block Offenders`. Sem ela, o Snort só observa.

---

## 📖 Onde o IPS se encaixa

Lembra da cadeia de defesa em profundidade do [67.md](67.md)?

```
INTERNET → 🔥 Firewall (3-4) → 🛡️ IPS/Snort (3-7) → 🌐 WAF/nginx (7) → 🖥️ Server
```

Colocamos o Snort na **INTERNET** — a interface Bridge que "toca" o mundo externo ([68.md](68.md)). É a **primeira linha** de inspeção profunda: worms, exploits de SMB/RDP/SSH/DNS, scans e malware por assinatura são pegos aqui, **antes** de chegarem ao WAF ou ao server.

> 💡 **pfSense é UTM** ([67.md](67.md)): o Snort é o módulo que adiciona a capacidade **IDS/IPS** ao nosso firewall — tudo na mesma caixa.

## 🛠️ Etapa 1 — Instalar o pacote Snort

Com **todas as VMs rodando**, acesse o pfSense em `https://192.168.56.10`.

### 1.1 — Abrir o Package Manager

```
System → Package Manager → Available Packages
```

### 1.2 — Instalar

1. No campo **Search**, busque `Snort`
2. Clique no botão verde **Install** (à direita)
3. Clique em **Confirm**

### 1.3 — Verificar a instalação

Ao terminar, confira em:

```
System → Package Manager → Installed Packages
```

O **Snort** deve aparecer na lista.

> 💡 Alternativa comum ao Snort: **Suricata** (multithread, mesmo formato de regras). O curso usa Snort; os conceitos valem para os dois.

---

## 🔧 Etapa 2 — Configurar a interface INTERNET

```
Services → Snort → Interfaces
```

Clique no botão verde **Add** (ao final da tabela vazia, canto inferior direito) e configure:

```yaml
Enable: <check>
Interface: INTERNET (em0)
Description: INTERNET
Snap Length: 1518

# --- Logging ---
Send Alerts to System Log: <check>
System Log Facility: LOG_AUTH
System Log Priority: LOG_ALERT
Enable Packet Captures: <vazio>
Packet Capture File Size: <vazio>
Enable Unified2 Logging: <vazio>

# --- Modo IPS (bloqueio) ---
Block Offenders: <check>
IPS Mode: Legacy Mode
Kill States: <check>
Which IP to Block: BOTH

# --- Performance / detecção ---
Search Method: AC-BNFA
Split ANY-ANY: <vazio>
Search Optimize: <vazio>
Stream Inserts: <vazio>
Checksum Check Disable: <vazio>

# --- Redes ---
Home Net: default
External Net: default
Pass List: default

Alert Suppression and Filtering: <vazio>
Advanced Configuration Pass-Through: <vazio>
```

Clique em **Save** (azul, ao final da página).

### O que significam os campos-chave

| Campo | Significado |
|-------|-------------|
| **Interface: INTERNET (em0)** | Onde o Snort inspeciona — a borda externa |
| **Snap Length: 1518** | Bytes capturados por pacote = frame Ethernet completo (MTU 1500 + cabeçalhos) |
| **Send Alerts to System Log** | Manda os alertas para o **syslog** — é o que permite encaminhá-los ao Graylog ([74.md](74.md)) |
| **LOG_AUTH / LOG_ALERT** | Categoria + prioridade do evento no syslog (facilita filtrar no SIEM) |
| **Block Offenders** ✅ | **A chave IDS→IPS.** Bloqueia ativamente o IP ofensor (senão, só alerta) |
| **IPS Mode: Legacy Mode** | Snort inspeciona uma **cópia** e bloqueia inserindo o IP numa tabela do pf. Mais compatível que o **Inline** (netmap) |
| **Kill States** ✅ | Ao bloquear, **derruba conexões ativas** do ofensor (não espera expirar) |
| **Which IP to Block: BOTH** | Bloqueia com base no IP de **origem e destino** do pacote ofensor |
| **Search Method: AC-BNFA** | Motor de pattern matching (Aho-Corasick BNFA) — bom equilíbrio memória × velocidade |
| **Home Net / External Net: default** | `HOME_NET` = redes que protegemos; `EXTERNAL_NET` = o resto. Regras usam essas variáveis para saber a **direção** do ataque |
| **Pass List: default** | Lista de IPs que **nunca** são bloqueados (evita se auto-trancar) |

> ⚠️ **Legacy × Inline:** no **Legacy Mode**, o pacote ofensor pode passar **uma vez** antes do IP ser bloqueado (o Snort age sobre uma cópia). O **Inline Mode** dropa o próprio pacote na hora, mas exige suporte a `netmap` na NIC. Para lab, **Legacy** é o mais seguro.

## ⚙️ Etapa 3 — Global Settings (feeds de regras)

Um IPS sem regras não detecta nada. Aqui escolhemos **quais bases de assinaturas** o Snort vai usar.

```
Services → Snort → Global Settings
```

```yaml
Enable Snort VRT: <vazio>          # exige Oinkcode (registro/pago) — pulamos
Enable Snort GPLv2: <check>        # regras da comunidade, gratuitas
Enable ET Open: <check>            # Emerging Threats Open (gratuito)
Enable ET Pro: <vazio>             # pago
Enable OpenAppID: <check>          # identificação de aplicações
Enable AppID Open Text Rules: <vazio>
Enable FEODO Tracker Botnet C2 IP Rules: <check>   # IPs de botnets (abuse.ch)

Update Interval: 1 DAY
Update Start Time: 12:00
Hide Deprecated Rules Categories: <vazio>
Disable SSL Peer Verification: <vazio>

Remove Blocked Hosts Interval: 12 HOURS
Remove Blocked Hosts After Deinstall: <check>
Keep Snort Settings After Deinstall: <check>
Startup/Shutdown Logging: <vazio>
```

Clique em **Save**.

### Os feeds de regras, explicados

| Feed | O que é | Custo |
|------|---------|-------|
| **Snort VRT (Talos)** | Regras oficiais da Cisco/Talos | Requer **Oinkcode** (grátis c/ atraso de 30 dias, ou pago) |
| **Snort GPLv2** | Regras da comunidade | 🟢 Gratuito, sem registro |
| **ET Open** | Emerging Threats Open (Proofpoint) | 🟢 Gratuito |
| **ET Pro** | Versão premium do ET | 🔴 Pago |
| **OpenAppID** | Detecta **aplicações** (Facebook, Skype…) no tráfego | 🟢 Gratuito |
| **FEODO Tracker** | Lista de IPs de **C2 de botnets** (Feodo/Emotet/Dridex) — abuse.ch | 🟢 Gratuito |

> 🔑 **Por que VRT fica desmarcado:** ele exige um **Oinkcode** (código de conta Snort). Sem conta, usamos as bases gratuitas — **GPLv2 + ET Open** já dão cobertura sólida para o lab.

| Campo | Significado |
|-------|-------------|
| **Update Interval / Start Time** | Quando o Snort baixa novas assinaturas automaticamente (ameaças mudam todo dia) |
| **Remove Blocked Hosts Interval: 12 HOURS** | Após 12h, um IP bloqueado é **liberado** (evita bloqueio permanente por falso-positivo) |
| **Keep Snort Settings After Deinstall** | Preserva a config se você reinstalar o pacote |

## ⬇️ Etapa 4 — Baixar as regras

As bases foram **selecionadas**, mas ainda não **baixadas**. Force o download:

```
Services → Snort → Updates
```

Clique em **Update Rules** (azul, logo após a lista). Aguarde o download e a instalação das assinaturas.

> 💡 Sem esse passo, o Snort inicia **sem regras** e não detecta nada. É o passo mais esquecido do lab.

Depois disso, volte em `Services → Snort → Interfaces` e confirme que a INTERNET está com o ▶️ **verde** (Snort rodando naquela interface).

## 🔗 Conectando ao SIEM

Marcamos **Send Alerts to System Log** com facility `LOG_AUTH`. Isso significa que os alertas do Snort entram no **system log do pfSense** — que já é encaminhado ao **Graylog** via remote syslog ([74.md](74.md)).

```
Snort detecta exploit → alerta no system log (LOG_AUTH) → syslog → Graylog
```

> 🔑 Agora o SIEM recebe **três perspectivas** do mesmo evento: **firewall** (Pass/Block de rede, [74.md](74.md)) + **WAF** (ataques HTTP, [75.md](75.md)) + **IPS/Snort** (intrusões em qualquer protocolo). É **defesa em profundidade com observabilidade** de verdade.

## 📚 Para Saber Mais: Tipos de IDS (HIDS × NIDS)

O [67.md](67.md) diferencia **IDS × IPS** (detectar × bloquear). Mas os IDS ainda se dividem em **dois tipos**, conforme **onde** observam:

### 🖥️ HIDS — Host-based IDS

**Instalado em cada máquina** a ser monitorada. Analisa **eventos e logs locais** (arquivos, integridade de sistema, chamadas, acessos) e atua como **última linha de defesa** — pega o que **já passou** pelo firewall/rede.

- **Vê:** o que acontece **dentro** do host (alterações de arquivo, escalonamento de privilégio, logins suspeitos)
- **Não vê:** tráfego que não chega àquela máquina
- **Exemplos:** Wazuh, OSSEC, Tripwire, EDRs de host

### 🌐 NIDS — Network-based IDS

**Monitora a rede toda** a partir da interface em que está. Analisa **todo o tráfego** combinando **inspeção de protocolos + regras** e detecta ataques por **comportamento anormal** do tráfego.

- **Vê:** pacotes de **vários hosts** trafegando pela rede (scans, exploits, worms)
- **Não vê:** o que ocorre **dentro** de um host sem gerar tráfego de rede
- **Exemplos:** **Snort**, Suricata, Zeek

> 🔑 **O Snort do nosso lab é um NIDS** — ele inspeciona o tráfego da interface **INTERNET** (Etapa 2), não os arquivos internos de uma máquina.

### Comparação

| | **HIDS** | **NIDS** |
|--|----------|----------|
| **Onde fica** | Em **cada host** | Na **rede** (numa interface) |
| **O que analisa** | Logs/arquivos/eventos do host | **Todo o tráfego** da interface |
| **Escopo** | Uma máquina | Muitas máquinas de uma vez |
| **Papel** | Última linha (pós-firewall) | Perímetro / detecção ampla |
| **Ponto cego** | Tráfego que não chega ao host | O que não vira tráfego de rede |
| **Exemplos** | Wazuh, OSSEC | **Snort**, Suricata, Zeek |

> 💡 **Complementares, não concorrentes:** NIDS pega o ataque **chegando pela rede**; HIDS pega o que **já entrou** no host. Num SIEM ([64.md](64.md)), correlacionar **ambos** dá o quadro completo — o NIDS vê o exploit voando, o HIDS confirma se ele **executou** no alvo.

## 🐞 Troubleshooting Comum

### "O Snort inicia e para sozinho (não fica verde)"

Quase sempre é **falta de regras**. Rode `Services → Snort → Updates → Update Rules` e confira se baixou algo. Sem assinaturas, o Snort não sobe.

### "Fiquei bloqueado / perdi acesso ao pfSense"

O `Block Offenders` pode bloquear um IP legítimo (falso-positivo). Soluções:
- Adicione a rede de gestão à **Pass List** (`Services → Snort → Pass Lists`)
- Remova o bloqueio em `Services → Snort → Blocked`
- Espere o `Remove Blocked Hosts Interval` (12h) expirar

### "Instalei mas não vejo Services → Snort"

Confirme em `System → Package Manager → Installed Packages` que o Snort **terminou** de instalar (a instalação baixa binários e pode demorar).

### "Update Rules falha com erro de SSL"

Em alguns ambientes, marque **Disable SSL Peer Verification** temporariamente (só em lab). Em produção, corrija a data/hora e os certificados do pfSense.

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Snort** | IDS/IPS open source baseado em **assinaturas** (regras) — do tipo **NIDS** |
| **IDS × IPS** | IDS **detecta/alerta**; IPS **detecta e bloqueia** ([67.md](67.md)) |
| **HIDS** | IDS **baseado em host** — instalado em cada máquina, analisa logs/eventos locais |
| **NIDS** | IDS **baseado em rede** — inspeciona todo o tráfego de uma interface (é o caso do Snort) |
| **Block Offenders** | Opção que transforma o Snort de IDS em **IPS** |
| **Legacy × Inline Mode** | Legacy bloqueia via tabela do pf (cópia); Inline dropa o pacote (netmap) |
| **Kill States** | Derruba conexões ativas do IP bloqueado |
| **HOME_NET / EXTERNAL_NET** | Variáveis que definem "nós" × "o mundo" para direção das regras |
| **Pass List** | IPs que nunca são bloqueados (whitelist do IPS) |
| **Feeds de regras** | GPLv2, ET Open, VRT, OpenAppID, FEODO — bases de assinaturas |
| **Oinkcode** | Código de conta Snort exigido para as regras VRT |
| **UTM** | Firewall all-in-one (pfSense) que incorpora o IPS ([67.md](67.md)) |

## ✅ Quiz Mental

1. Qual **única** configuração transforma o Snort de IDS em IPS?
2. Por que instalamos o Snort na interface **INTERNET** e não na DMZ?
3. Diferença entre **Legacy Mode** e **Inline Mode**?
4. Por que deixamos **Snort VRT** desmarcado e marcamos **GPLv2 + ET Open**?
5. Como os alertas do Snort chegam ao Graylog?
6. O Snort é **HIDS** ou **NIDS**? Qual a diferença?

<details>
<summary>Ver respostas</summary>

1. **Block Offenders.** Sem ela, o Snort apenas **alerta** (comportamento de IDS). Com ela marcada, ele **bloqueia** o IP ofensor (IPS)
2. Porque a **INTERNET** é a **borda externa** (Bridge) — queremos inspecionar o tráfego **assim que ele chega**, antes de atravessar o firewall/WAF até o server. É a primeira linha de defesa em profundidade ([67.md](67.md))
3. **Legacy Mode**: o Snort analisa uma **cópia** do tráfego e bloqueia o ofensor inserindo o IP numa **tabela do pf** — simples e compatível, mas o 1º pacote pode passar. **Inline Mode**: o Snort fica **no caminho** do pacote (via `netmap`) e **dropa o próprio pacote** ofensor na hora — mais eficaz, mas exige suporte de driver
4. Porque **VRT** exige um **Oinkcode** (conta Snort — grátis com atraso de 30 dias ou pago). **GPLv2** (comunidade) e **ET Open** (Emerging Threats) são **gratuitos e sem registro**, já oferecendo cobertura sólida para o lab
5. Marcamos **Send Alerts to System Log**; os alertas entram no **system log do pfSense**, que é encaminhado por **remote syslog ao Graylog** ([74.md](74.md)). Assim o SIEM ganha a fonte **IPS**, somada a firewall e WAF
6. **NIDS** (Network-based). Ele inspeciona **todo o tráfego** da interface INTERNET — vê pacotes de vários hosts. Um **HIDS** seria instalado **dentro** de cada máquina, analisando logs/arquivos locais (última linha, pós-firewall). São **complementares**: NIDS pega o ataque chegando pela rede, HIDS confirma se ele executou no host

</details>

## 📝 Quiz Oficial do Curso — Mecanismo de Segurança (IDS × IPS)

> **Cenário:** Ao monitorar o tráfego de rede, uma ferramenta **detectou um ataque** e, em seguida, **enviou ao firewall uma requisição para bloquear** o endereço de origem. Que mecanismo de segurança essa ferramenta representa?

| | Alternativa | Veredito |
|--|-------------|----------|
| A | É um **WAF** à frente do firewall detectando/notificando algo suspeito | ❌ |
| B | É um **IPS**, pois **apenas detectou** a anomalia | ❌ |
| **C** | É um **IDS/IPS**: o IDS **detectou** a anomalia e o IPS **preveniu** o ataque pedindo ao firewall para **bloquear** a conexão | ✅ |
| D | É um **IDS**, pois **preveniu** o ataque ao solicitar o bloqueio ao firewall | ❌ |

<details>
<summary>Ver gabarito comentado</summary>

**Resposta correta: C.**

**Por quê:** a ferramenta faz **as duas coisas** — **detecta** (papel de **IDS**) *e* **age**, solicitando ao firewall o bloqueio da origem (papel de **IPS**). Essa é exatamente a mecânica do **Snort com `Block Offenders`** no **Legacy Mode** (Etapa 2 deste doc): ele inspeciona o tráfego, gera o alerta e **insere o IP ofensor na tabela de bloqueio do pf** — ou seja, "pede ao firewall para bloquear". Monitorar **tráfego de rede** (não HTTP) confirma que é um **NIDS/IPS**, não um WAF.

**Por que as outras estão erradas:**
- **A** — **WAF** atua na **camada 7 (HTTP)**, à frente do servidor **web** ([67.md](67.md)); aqui a ferramenta monitora **tráfego de rede** genérico e conversa com o firewall — comportamento de IPS de rede, não de WAF.
- **B** — **contradição**: se "apenas detectou", seria **IDS**, não IPS. IPS **age**; detectar sozinho não caracteriza IPS.
- **D** — **contradição inversa**: **prevenir/bloquear** é papel do **IPS**, não do IDS. IDS **só detecta/alerta** ([67.md](67.md)).

> 🔑 Mnemônico ([67.md](67.md)): **IDS** = "vejo algo estranho!"; **IPS** = "vejo algo estranho — **bloqueado**!". A ferramenta do cenário faz **as duas** → **IDS/IPS**.

</details>

## 💡 Reflexão Final

> *"O firewall diz **quem** pode entrar.*
> *O WAF diz se a **requisição HTTP** é segura.*
> *O IPS diz: **‘esse pacote carrega um exploit conhecido — barrado’** — em qualquer protocolo."*

Com o Snort no ar, fechamos o **tripé de defesa** do [67.md](67.md): firewall (3-4), IPS (3-7) e WAF (7). Cada um cobre um ângulo, e os **três** agora despejam alertas no mesmo SIEM.

### O que mudou

| Antes | Depois |
|-------|--------|
| Defesa em camadas 3-4 (firewall) + 7 (WAF) | Camadas **3-7** cobertas com o IPS no meio |
| Detecção só de HTTP (WAF) e IP/porta (firewall) | Detecção de **exploits, worms, malware e botnets** por assinatura |
| SIEM com 2 fontes | SIEM com **3 fontes** correlacionáveis |

### O que vem agora

Com o IPS armado e as regras baixadas, a próxima etapa é **atacar de propósito** — e ver o Snort e o WAF **bloqueando em tempo real**, com tudo pousando no Graylog. O palco está montado; agora entra o adversário. 🎼🛡️
