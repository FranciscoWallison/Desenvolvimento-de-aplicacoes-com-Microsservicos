# 🧪 Laboratório Prático: Gerando Alertas com uma Regra Snort Customizada

> **Tema:** Criar uma **regra customizada** do Snort, dispará-la com **ICMP (`ping`)** e **telnet**, e ver os alertas no **Snort** e no **Graylog**
> **Pré-requisito direto:** [77.md](77.md) (Snort instalado + regras baixadas)
> **Relacionados:** [74.md](74.md) (pfSense → Graylog / System Events), [73.md](73.md) (acesso ao Graylog), [76.md](76.md) (server publicado — alvo do telnet)
> **Conceitos base:** [67.md](67.md) (IDS/IPS), [70.md](70.md) (ICMP/ping)

---

## 🎯 Objetivo do Lab

O Snort já está armado ([77.md](77.md)), mas usando só as regras **prontas** (GPLv2/ET Open). Agora vamos **escrever nossa própria regra** para entender a **anatomia de uma assinatura** e ver o ciclo completo:

```
regra customizada → tráfego de teste (ping/telnet) → alerta no Snort → syslog → Graylog
```

> ⚠️ **Isto é um teste, não um ataque.** Criamos uma regra genérica só para **provar o pipeline de alertas**. Ataques reais (SQLi, XSS) virão nos próximos labs.

> 📌 **Por que ICMP no exemplo?** Muitos ataques de **DoS** abusam de pacotes **ICMP** para saturar a rede — por isso é um alvo didático clássico. Lembrando que o mesmo ICMP é o que o `ping` usa para testar conectividade ([70.md](70.md)).

---

## 🛠️ Etapa 1 — Criar a regra customizada

Acesse o pfSense em `https://192.168.56.10` (com **todas as VMs rodando**).

### 1.1 — Abrir as regras da interface

```
Services → Snort → Interface Settings → INTERNET Rules
```

> 💡 A aba aparece como **"INTER Rules"** / **"INTERNET Rules"** — são as regras específicas da interface INTERNET que configuramos no [77.md](77.md).

### 1.2 — Selecionar a categoria custom

Em **Available Rule Categories**, selecione **`custom.rules`** — é o arquivo onde ficam as regras que **você** escreve (separado das regras baixadas dos feeds).

### 1.3 — Colar a regra

No campo **Defined Custom Rules**, cole:

```snort
alert icmp any any -> any any (msg:"TESTE ICMP LABORATORIO"; classtype:icmp-event; sid:1000000100;)
```

Clique em **Save**.

---

## 🔬 Anatomia da regra Snort

Toda regra tem duas partes: o **cabeçalho** (o que casar) e as **opções** (entre parênteses).

```
alert   icmp   any any   ->   any any   ( msg:"..."; classtype:...; sid:...; )
└─ação  └─proto └─origem  └─dir └─destino  └────────── opções ──────────┘
```

### Cabeçalho

| Campo | Valor | Significado |
|-------|-------|-------------|
| **Ação** | `alert` | **Gera alerta** (não bloqueia). Outras: `log`, `pass`, `drop`, `reject` |
| **Protocolo** | `icmp` | Protocolo a inspecionar (`icmp`/`tcp`/`udp`/`ip`) |
| **Origem** | `any any` | IP de origem + porta (`any` = qualquer) |
| **Direção** | `->` | Sentido do tráfego (`->` unidirecional; `<>` bidirecional) |
| **Destino** | `any any` | IP de destino + porta |

### Opções (entre parênteses)

| Opção | Valor | Significado |
|-------|-------|-------------|
| **`msg`** | `"TESTE ICMP LABORATORIO"` | Texto que aparece no alerta — como você vai reconhecê-lo |
| **`classtype`** | `icmp-event` | Classificação do evento (define prioridade padrão) |
| **`sid`** | `1000000100` | **Snort ID** — identificador único da regra |

> 🔑 **Por que `sid` começa em 1000000?** A faixa **≥ 1.000.000** é **reservada para regras locais/customizadas**. SIDs abaixo disso pertencem aos feeds oficiais (VRT/ET) — usar um deles causaria **conflito**. Por isso `1000000100`.

> ⚠️ **Ação `alert` × `Block Offenders`:** a regra usa `alert`, ou seja, **só alerta**. Mas lembre que no [77.md](77.md) marcamos **Block Offenders** — no **Legacy Mode** do pfSense, o Snort pode **bloquear o IP** que disparar *qualquer* alerta. A **Pass List** (default) protege gateways/redes locais de se auto-trancarem. Se um host de teste for bloqueado, remova-o em `Services → Snort → Blocked`.

---

## 🔄 Etapa 2 — Recarregar o Snort

A regra foi salva, mas o Snort ainda roda com a config antiga. Recarregue:

```
Services → Snort → Interfaces
```

Na **primeira linha**, coluna **Snort Status**, clique na seta **Reload** (em *Interface Settings Overview*). O Snort relê as regras **sem reiniciar** a interface.

> 💡 Sem o reload, sua regra nova **não está ativa** — é o passo mais esquecido.

---

## 🧪 Etapa 3 — Disparar o alerta

### 3.1 — Teste ICMP (camada de rede)

Do seu PC, faça `ping` no IP **do server publicado** ([76.md](76.md)):

```bash
ping 192.168.100.192        # use o SEU IP publicado (varia por máquina)
```

Cada echo request/reply é um pacote ICMP → casa com `alert icmp any any -> any any` → **gera alerta**.

### 3.2 — Teste telnet (camada de aplicação)

Para demonstrar que o SIEM também flagra **tentativas de conexão**, teste com `telnet` no server web (Nginx):

```bash
telnet 192.168.100.192
```

> 🪟 **No Windows**, o cliente telnet vem **desativado**. Busque **"Ativar ou desativar recursos do Windows"** no menu Iniciar e marque **Cliente Telnet** (em *.NET Framework 4.8 Advanced Services → Cliente Telnet*). No Linux/macOS, use o terminal direto.

> 💡 **Por que telnet?** Ele abre uma conexão TCP "crua" na porta — útil para simular uma **tentativa de acesso não autorizado** que o SIEM deve registrar. (Telnet em si é inseguro; aqui é só ferramenta de teste.)

---

## 👀 Etapa 4 — Ver os alertas no Snort

```
Services → Snort → Alerts
```

Você verá o alerta **`TESTE ICMP LABORATORIO`** disparado pelo `ping`.

> ℹ️ **Vai aparecer MAIS alertas do que você gerou.** Isso é normal: como o firewall está em **modo Bridge**, os gateways (firewall ↔ seu roteador) **conversam o tempo todo**, e parte desse tráfego ICMP também casa com a regra genérica `any any -> any any`. É um bom lembrete de que **regras amplas geram ruído** — em produção, restrinja origem/destino.

---

## 📡 Etapa 5 — Encaminhar os alertas ao Graylog

No [77.md](77.md) marcamos **Send Alerts to System Log** (facility `LOG_AUTH`). Ou seja, os alertas do Snort viram **System Events** no pfSense. Para que cheguem ao Graylog, o **remote logging** precisa incluir **System Events**.

```
Status → System Logs → Settings → Remote Logging Options
```

```yaml
Enable Remote Logging: <check>
Source Address: DMZ
IP Protocol: IPv4
Remote log servers: 172.100.1.101:1514
Remote Syslog Contents:
  System Events: <check>     # ← ESSENCIAL para os alertas do Snort
  Firewall Events: <check>
```

Clique em **Save**.

> 🔑 **O detalhe que fecha o lab:** no [74.md](74.md) encaminhamos os **Firewall Events**. Aqui adicionamos **System Events** — que é a categoria onde os alertas do Snort (via `LOG_AUTH`) se encaixam. Sem marcar isso, o Snort alerta localmente mas **nada aparece no SIEM**.

### Ver no Graylog

```
http://192.168.56.11        # interface web do Graylog (porta 9000 — ver 73.md)
```

Faça a busca e você verá os alertas **`TESTE ICMP LABORATORIO`** gerados pelo Snort, agora **centralizados no SIEM**.

## 🐞 Troubleshooting Comum

### "Meu alerta não aparece em Services → Snort → Alerts"

1. **Salvou** a regra em `custom.rules`? (Etapa 1.3)
2. **Recarregou** o Snort? (Etapa 2 — o passo mais esquecido)
3. O Snort está **verde/rodando** na interface INTERNET?
4. O `ping`/`telnet` foi para um IP que **passa** pela interface INTERNET?

### "Fui bloqueado depois do teste"

`Block Offenders` ([77.md](77.md)) + regra genérica pode bloquear o IP de teste. Remova em `Services → Snort → Blocked` e/ou adicione o IP à **Pass List**.

### "Vejo no Snort mas não no Graylog"

- **System Events** marcado em `Status → System Logs → Settings`? (Etapa 5 — a causa nº 1)
- O `Remote log servers` está `172.100.1.101:1514`?
- O Graylog está recebendo? Confira o input UDP 1514 ([74.md](74.md))

### "Muitos alertas que eu não gerei"

Esperado — modo **Bridge** faz os gateways trocarem ICMP constantemente. Regras `any any -> any any` capturam tudo. Restrinja a regra (ex.: `alert icmp any any -> 172.100.2.100 any`) para reduzir ruído.

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Regra Snort** | Assinatura: cabeçalho (ação/proto/origem→destino) + opções `(...)` |
| **`custom.rules`** | Arquivo das regras que **você** escreve (separado dos feeds) |
| **Ação `alert`** | Gera alerta sem bloquear (≠ `drop`/`reject`) |
| **Operador de direção** | `->` unidirecional; `<>` bidirecional |
| **`msg`** | Texto identificador do alerta |
| **`classtype`** | Classificação do evento → define prioridade |
| **`sid`** | ID único da regra; **≥ 1.000.000** para regras locais |
| **Reload do Snort** | Aplica novas regras sem reiniciar a interface |
| **System Events (syslog)** | Categoria onde caem os alertas do Snort (`LOG_AUTH`) |
| **Ruído de regra ampla** | `any any` gera muitos falsos/irrelevantes (ex.: bridge gateways) |

## ✅ Quiz Mental

1. O que cada parte de `alert icmp any any -> any any` significa?
2. Por que o `sid` precisa ser **≥ 1.000.000**?
3. A regra usa `alert`. Ela **bloqueia** o tráfego? E com `Block Offenders` ligado?
4. Por que, mesmo só dando `ping` uma vez, aparecem vários alertas?
5. Qual opção do **remote logging** é essencial para os alertas do Snort chegarem ao Graylog, e por quê?

<details>
<summary>Ver respostas</summary>

1. `alert` = ação (gerar alerta); `icmp` = protocolo; primeiro `any any` = IP+porta de **origem**; `->` = direção (só origem→destino); segundo `any any` = IP+porta de **destino**. As opções `(msg/classtype/sid)` descrevem/identificam o alerta
2. Porque a faixa **≥ 1.000.000** é **reservada para regras locais/customizadas**. SIDs abaixo pertencem aos feeds oficiais (VRT/ET) — reutilizar um causaria **conflito de identificador**
3. Sozinha, `alert` **só alerta** (não bloqueia — seria `drop`/`reject`). Mas com **Block Offenders** no **Legacy Mode**, o pfSense pode **bloquear o IP** que dispara *qualquer* alerta; a **Pass List** default protege gateways/redes locais
4. Porque o firewall está em modo **Bridge**: firewall e roteador (gateways) **trocam ICMP constantemente**, e esse tráfego também casa com a regra genérica `any any -> any any`. Regra ampla = muito ruído
5. **System Events** (em `Status → System Logs → Settings → Remote Logging Options`). Os alertas do Snort são enviados ao system log com facility `LOG_AUTH` ([77.md](77.md)), logo entram como **System Events** — sem marcar essa categoria, eles não são encaminhados ao Graylog

</details>

## 💡 Reflexão Final

> *"Uma regra é uma frase que descreve um ataque.*
> *`alert icmp any any -> any any` diz: **‘qualquer ICMP, me avise’**.*
> *Escrever regras é ensinar o sensor a reconhecer o que **importa para você**."*

Pela primeira vez saímos do **consumo** de regras prontas para a **autoria**. Entender a anatomia — ação, protocolo, direção, `sid`, `msg` — é o que separa quem **usa** um IDS de quem **customiza** um IDS para a própria realidade.

### O que mudou

| Antes | Depois |
|-------|--------|
| Snort com regras genéricas dos feeds | Regra **sua**, em `custom.rules` |
| Alertas ficavam no pfSense | Alertas **centralizados no Graylog** (System Events) |
| SIEM com firewall + WAF | SIEM também com **alertas de IPS autorais** |

### A lição do ruído

O lab também ensinou algo sutil: `any any -> any any` gerou **muito mais** do que o esperado. Em segurança, **especificidade é sinal, generalidade é ruído**. A próxima evolução é escrever regras **precisas** — e, enfim, dispará-las com **ataques reais** para ver o IPS e o WAF bloqueando em tempo real. 🎼🛡️
