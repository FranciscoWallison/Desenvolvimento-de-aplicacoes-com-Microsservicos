> **Tema:** Ativar o **SSH** no firewall pfSense para administração remota via terminal (porta 22)

## 🎯 Objetivo do Lab

Até aqui administramos o pfSense pela **interface web** (`https://192.168.56.10`). Mas boa parte do trabalho de segurança acontece no **terminal** — editar arquivos, rodar comandos, automatizar. Para isso vamos habilitar o **SSH (Secure Shell)** no firewall.

Neste lab vamos:
1. **Ativar** o servidor SSH no pfSense (`Enable Secure Shell`)
2. **Liberar a porta 22** via um **alias de portas** (`PORT_ADM`)
3. **Conectar** pelo terminal e cair no **shell** do pfSense

> 🔑 **Este acesso será usado em várias etapas do curso** (proxy reverso, DNS…). É a base para tudo que vem a seguir.

## 📖 O que é SSH — e por que não telnet

No [78.md](78.md) usamos o `telnet` — mas só como **ferramenta de teste**, porque ele é **inseguro** (trafega tudo em **texto claro**). Para administração real, usamos **SSH**.

| | **Telnet** | **SSH** |
|--|-----------|---------|
| **Porta** | TCP **23** | TCP **22** |
| **Criptografia** | ❌ Texto claro | ✅ Criptografado (AES) |
| **Uso** | Legado / teste | **Administração remota segura** |

> 🔑 Do [66.md](66.md): *"SSH (porta 22) = acesso remoto **seguro**, substitui o Telnet"*. Se alguém interceptar a conexão SSH, vê apenas **cifrado** — senha e comandos ficam protegidos.

## 🛠️ Etapa 1 — Ativar o Secure Shell

Com **todas as VMs rodando**, acesse o pfSense em `https://192.168.56.10`.

```
System → Advanced
```

Na seção **Secure Shell**:

1. Marque **Enable Secure Shell**
2. Clique em **Save** (botão azul, ao final da página)

Isso liga o **servidor SSH (`sshd`)** no firewall — mas ainda falta o firewall **permitir** a porta 22.

> 💡 Aqui dá para configurar também **autenticação por chave** (mais seguro que senha) e **trocar a porta**. No lab ficamos no padrão; em produção, veja a seção de segurança abaixo.

---

## 🔓 Etapa 2 — Liberar a porta 22 (alias PORT_ADM)

O pfSense usa **Aliases** — atalhos nomeados para IPs ou portas, reutilizados nas regras ([70.md](70.md)). Vamos ajustar o alias de portas para incluir o **22 (SSH)**.

```
Firewall → Aliases → Ports
```

Clique no **lápis (Editar)** do alias **`PORT_WEB`** e deixe assim:

```makefile
Name: PORT_ADM
Description: Portas Administração
Type: Port(s)
Port:
  80  - HTTP
  443 - HTTPS
  22  - SSH
```

Clique em **Save** → **Apply Changes**.

> ⚠️ **Atenção — isto renomeia o alias `PORT_WEB`:** aquele alias que no [76.md](76.md) publicava o WAF na porta **80** agora vira **`PORT_ADM`** e passa a agrupar **80, 443 e 22**. Confira em `Firewall → Rules` se as regras que usavam `PORT_WEB` continuam apontando para o alias renomeado (o pfSense costuma atualizar a referência, mas vale verificar).

### Por que usar um alias?

| Sem alias | Com alias `PORT_ADM` |
|-----------|----------------------|
| Repetir `80,443,22` em cada regra | Um nome só, reutilizável |
| Mudar porta = editar N regras | Edita **o alias**, todas as regras acompanham |
| Difícil de ler | Regra fala "portas de administração" |

## 💻 Etapa 3 — Conectar via SSH

Abra o **cmd** (Windows) ou o **terminal** (Linux/macOS):

```bash
ssh root@192.168.56.10
```

Quando pedir, informe:

```
Senha: qwerty
```

Você cai no **menu do console do pfSense**. Digite a opção:

```
8
```

A opção **8) Shell** te leva ao **shell** propriamente dito, onde você roda comandos livremente.

> 🔴 **`root` / `qwerty` é credencial padrão — SÓ em lab.** Em produção isso é um convite a invasão (o [77.md](77.md) mostrou que scanners procuram exatamente **SSH exposto**).

> 💡 **Por que `192.168.56.10`?** É o IP da interface **host-only** (gestão) — a mesma que usamos para a interface web. O SSH entra pela mesma "porta de administração".

---

## ⏱️ Observação — a conexão pode cair

Vamos reutilizar esse acesso em vários labs. Se você deixar a sessão **ociosa** por muito tempo, ela pode **cair** (timeout). Se isso acontecer, **repita o passo a passo** (basta reconectar com `ssh root@192.168.56.10`).

> 💡 **Por quê?** Servidores SSH derrubam sessões inativas por segurança. Dá para suavizar com **keepalive** no cliente:
> ```bash
> ssh -o ServerAliveInterval=60 root@192.168.56.10
> ```
> O `ServerAliveInterval=60` manda um "ping" a cada 60s, mantendo a sessão viva.

## 📚 Para Saber Mais: Formas de Realizar uma Conexão SSH

O SSH é um protocolo **criptográfico**, e há **mais de uma forma** de usá-lo além do que fizemos no lab.

### 1. SSH com senha (o que usamos)

Foi o método deste lab. A sintaxe geral é:

```bash
ssh -p [PORTA] [USUARIO]@[IP-DO-SERVIDOR]
```

- **`-p [PORTA]`** — a porta. O **padrão é 22**; como usamos a 22, nem precisamos do `-p` (`ssh root@192.168.56.10` já assume 22).
- Depois, digita-se a **senha do usuário** (no lab, `qwerty`).

> ⚠️ Simples, mas a senha pode ser **força-bruteada** — por isso não é o método mais seguro.

### 2. SSH com chaves (o mais seguro)

Em vez de senha, usa-se um **par de chaves** (pública + privada). O administrador **gera a chave**, guarda a **pública** no servidor e entrega a **privada** ao usuário. A conexão usa **o mesmo comando** `ssh -p [PORTA] [USUARIO]@[IP]`, mas autentica pela **chave** em vez de senha.

> 🔑 **Por que é mais seguro / flexível:** não se compartilha senha, e o administrador pode **conceder acesso temporário** — basta **deletar a chave pública** do servidor para **revogar** o acesso daquela pessoa, na hora. (Ferramentas de hosting como **cPanel/WHM** geram e distribuem essas chaves.)

### 3. SFTP — SSH File Transfer Protocol

O **SFTP** roda **sobre o SSH** (mesma porta 22, mesma criptografia), mas com foco em **manipular arquivos**: criar, mover, editar, copiar, apagar, compactar/descompactar. Clientes gráficos como **FileZilla** ou **Classic FTP** dão uma **interface visual** para arrastar arquivos do/para o servidor, com os privilégios do usuário logado.

> 💡 **Não confundir com FTP puro:** o **FTP** clássico é **inseguro** (texto claro, como o telnet do [78.md](78.md)). O **SFTP** é o FTP "dentro" do túnel SSH — **cifrado**. Sempre que puder, **SFTP > FTP**.

### Comparando

| Forma | Autenticação | Foco | Segurança |
|-------|--------------|------|-----------|
| **SSH + senha** | Senha do usuário | Shell remoto | 🟡 Média (força-bruteável) |
| **SSH + chave** | Par de chaves | Shell remoto | 🟢 Alta (revogável, sem senha) |
| **SFTP** | Senha **ou** chave | Transferência de arquivos | 🟢 Alta (sobre SSH) |

## 🔒 Boas Práticas de SSH (além do lab)

O lab usa o caminho mais simples. Em **produção**, endureça o SSH:

| Prática | Por quê |
|---------|---------|
| **Autenticação por chave** (desabilitar senha) | Senha é força-bruteável; chave não |
| **Desabilitar login de `root`** | Use um usuário comum + `sudo` (menor privilégio) |
| **Trocar a porta 22** | Reduz ruído de bots que varrem a 22 (não é segurança real, mas ajuda) |
| **Restringir origem** (Source na regra) | Só a rede de gestão acessa o SSH — não `any` |
| **Fail2ban / rate limit** | Bloqueia após N tentativas falhas |
| **Log/auditoria** | Mandar os logins SSH ao SIEM ([74.md](74.md)) |

> 🔑 SSH é **poder total** sobre a máquina. Protegê-lo é prioridade máxima — um SSH fraco anula todo o resto da defesa.

## 🐞 Troubleshooting Comum

### "Connection refused / timeout"

1. **Enable Secure Shell** está marcado em `System → Advanced`? (Etapa 1)
2. A **porta 22** está liberada? Confira o alias `PORT_ADM` e a regra que o usa
3. Você está acessando o IP certo (`192.168.56.10`, host-only)?

### "Permission denied (senha recusada)"

Confirme usuário `root` e senha `qwerty` (padrão do lab). Cuidado com teclado (layout ABNT × US pode trocar caracteres).

### "A sessão caiu no meio do trabalho"

Timeout de inatividade — reconecte (`ssh root@192.168.56.10`) e, se incomodar, use `-o ServerAliveInterval=60`.

### "Host key verification failed" (após reinstalar a VM)

A chave do host mudou. Remova a antiga:
```bash
ssh-keygen -R 192.168.56.10
```

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **SSH (Secure Shell)** | Protocolo de acesso remoto **criptografado**, porta **22** |
| **Telnet × SSH** | Telnet = texto claro (inseguro); SSH = cifrado (substitui o Telnet) |
| **`sshd`** | Servidor SSH que o `Enable Secure Shell` liga no pfSense |
| **Alias de portas** | Atalho nomeado (`PORT_ADM`) agrupando portas reutilizáveis em regras ([70.md](70.md)) |
| **Console do pfSense** | Menu numerado pós-login; **opção 8 = Shell** |
| **Autenticação por chave** | Alternativa segura à senha (par pública/privada); acesso **revogável** deletando a chave |
| **SFTP** | Transferência de arquivos **sobre SSH** (porta 22, cifrado) — ≠ FTP puro (inseguro) |
| **`ssh -p [PORTA]`** | Especifica a porta (padrão 22, omitível se for 22) |
| **ServerAliveInterval** | Keepalive do cliente SSH contra timeout de inatividade |

## 📝 Quiz Oficial do Curso — Protocolo SSH

> **Cenário:** Samuel teve dúvidas sobre **o que é** a conexão SSH e **qual porta** ela usa. Qual alternativa responde corretamente?

| | Alternativa | Veredito |
|--|-------------|----------|
| A | SSH conecta por criptografia usando a porta **23** | ❌ |
| B | SSH é protocolo criptográfico que conecta pela porta **25** | ❌ |
| **C** | SSH permite **acesso remoto** a uma máquina e conecta pela porta **22** | ✅ |
| D | SSH é conexão feita por **cabo de rede** que dá privilégios de root | ❌ |

<details>
<summary>Ver gabarito comentado</summary>

**Resposta correta: C.**

**Por quê:** o **SSH (Secure Shell)** é um protocolo de rede **criptográfico** para **acesso remoto** a uma máquina, e usa a **porta TCP 22** ([66.md](66.md)). Foi exatamente o que fizemos na Etapa 3 (`ssh root@192.168.56.10`, porta 22 por padrão).

**Por que as outras estão erradas:**
- **A** — a porta **23** é do **Telnet** (acesso remoto **inseguro**, texto claro — ver [78.md](78.md)), não do SSH.
- **B** — a porta **25** é do **SMTP** (envio de e-mail), sem relação com SSH.
- **D** — SSH é uma conexão **de rede** (lógica, sobre TCP/IP) — **não** um "cabo de rede". E, embora você **possa** logar como root, isso é uma questão de **credencial/permissão**, não a definição do protocolo.

> 🔑 Fixando as portas ([66.md](66.md)): **SSH = 22**, **Telnet = 23** (inseguro), **SMTP = 25**, **DNS = 53**, **HTTP = 80**, **HTTPS = 443**.

</details>
