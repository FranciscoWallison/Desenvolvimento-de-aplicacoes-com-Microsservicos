# 🧪 Laboratório Prático: Regras de Acesso para o Graylog Atualizar (Update + DNS)

> **Tema:** Criar **duas regras** na DMZ para o Graylog **acessar a internet** (atualizar pacotes) e **resolver nomes (DNS)**, depois rodar `apt update` e instalar o `iptables-persistent`
> **Pré-requisito direto:** [79.md](79.md) (SSH habilitado + alias `PORT_ADM`)
> **Conceitos base:** [69.md](69.md) (Block Geral / deny by default), [70.md](70.md) (aliases `DMZ net`), [66.md](66.md) (DNS porta 53)
> **Curso 2 — Proxy Reverso, SSH e DNS**

---

## 🎯 Objetivo do Lab

Por causa do **Block Geral** ([69.md](69.md)), o Graylog (`172.100.1.101`, na **DMZ**) **não consegue sair para a internet** — nem para baixar atualizações, nem para resolver nomes. Vamos abrir **exatamente o necessário**:

1. Uma regra para o Graylog **acessar repositórios** de pacotes (HTTP/HTTPS)
2. Uma regra para o Graylog **fazer consultas DNS** (resolver nomes → IPs)
3. Conectar via **SSH** e rodar `apt update` + instalar `iptables-persistent`

> 🔑 **Por que DUAS regras?** `apt update` precisa de **duas coisas**: (a) falar HTTP/HTTPS com o repositório e (b) **descobrir o IP** do repositório via **DNS**. Liberar só o HTTP/HTTPS **não basta** — sem DNS, o nome do repositório nunca vira IP e o update falha com "*Temporary failure resolving...*".

---

## 🧭 Direção das regras (recap)

As regras ficam na interface **DMZ** porque, no pfSense, a regra é avaliada **na interface onde o tráfego entra** ([69.md](69.md)/[70.md](70.md)). O Graylog está na DMZ, então o tráfego que ele manda **para fora** entra no firewall **pela interface DMZ** — é ali que autorizamos a **saída**.

```
Graylog (172.100.1.101 / DMZ) ──► [interface DMZ do firewall] ──► INTERNET
                                    ↑ as regras deste lab moram aqui
```

---

## 🛠️ Etapa 1 — Regra de acesso para atualização

Acesse o pfSense (`https://192.168.56.10`) e vá em:

```
Firewall → Rules → DMZ
```

Clique em **↑ Add** (verde, o primeiro após a tabela — para ficar **acima** do Block Geral) e configure:

```yaml
Action: Pass
Disabled: <vazio>
Interface: DMZ
Address Family: IPv4
Protocol: TCP

Source: Single host - 172.100.1.101        # ← só o Graylog
Destination: any
Destination Port Range: From PORT_ADM to PORT_ADM   # 80/443/22 (alias do 79.md)

Log: ✓ check
Description: Acesso para atualização
```

**Save**.

### Por que cada valor

| Campo | Por quê |
|-------|---------|
| **Protocol: TCP** | HTTP/HTTPS (repositórios `apt`) rodam sobre TCP |
| **Source: 172.100.1.101** | **Menor privilégio** — só o Graylog sai para atualizar, não a DMZ inteira |
| **Destination: any** | Repositórios ficam em servidores variados na internet |
| **Port: PORT_ADM** | Alias do [79.md](79.md) = **80 (HTTP) + 443 (HTTPS)** + 22. Reaproveitamos o alias existente |
| **Log: ✓** | Auditar o acesso de saída do Graylog |

> 💡 O `PORT_ADM` também inclui a **22 (SSH)** — mais amplo do que o estritamente necessário para `apt` (80/443). Aqui reaproveitamos o alias; em produção, um alias dedicado só com 80/443 seria mais "limpo" (menor privilégio).

---

## 🌐 Etapa 2 — Regra de consulta DNS

Clique em **↑ Add** de novo e configure:

```yaml
Action: Pass
Disabled: <vazio>
Interface: DMZ
Address Family: IPv4
Protocol: UDP

Source: DMZ net                             # toda a subnet da DMZ (70.md)
Destination: any
Destination Port Range: From DNS(53) to DNS(53)

Log: ✓ check
Description: Consulta DNS
```

**Save** → **Apply Changes**.

### Por que cada valor

| Campo | Por quê |
|-------|---------|
| **Protocol: UDP** | Consultas DNS usam **UDP/53** (TCP/53 só p/ respostas grandes e zone transfer) |
| **Source: DMZ net** | Toda a DMZ pode resolver nomes ([70.md](70.md) = `172.100.1.0/24`) — DNS é útil a todos |
| **Port: DNS(53)** | Porta padrão do DNS ([66.md](66.md)) |

> 🔑 **A diferença sutil das duas regras:** a de **update** é restrita a **um host** (`172.100.1.101`) — menor privilégio para sair à internet. A de **DNS** libera a **DMZ net** inteira — resolver nomes é uma necessidade geral e de baixo risco. Repare como o **escopo do Source** é uma decisão de segurança consciente.

---

## 💻 Etapa 3 — Atualizar o Graylog via SSH

Com as regras no ar, conecte no Graylog e atualize:

```bash
ssh user@172.100.1.101
# senha:
qwerty

su -            # virar root
# senha:
qwerty

apt update      # atualiza a lista de pacotes
```

> ✅ **Se o `apt update` funcionar, as duas regras estão corretas** — ele só consegue baixar a lista se conseguir **resolver o nome** (DNS) **e** falar **HTTP/HTTPS** (update). É o teste de validação natural do lab.

### Instalar o iptables-persistent

```bash
apt install iptables-persistent
```

> 💡 **O que é `iptables-persistent`?** O `iptables` é o firewall **do próprio host** (nível Linux). Por padrão, as regras do iptables **somem no reboot**. O pacote `iptables-persistent` **salva e restaura** essas regras automaticamente ao reiniciar — prepara o Graylog para ter um firewall local **que sobrevive a reinícios** (assunto dos próximos labs).

> 🧠 **Defesa em profundidade:** já temos o firewall de **rede** (pfSense). O `iptables` adiciona um firewall **no host** — duas camadas. Se um cair/for mal configurado, o outro ainda protege.

## 🐞 Troubleshooting Comum

### "`apt update` → Temporary failure resolving 'deb.debian.org'"

É **DNS**. Confirme a **regra de Consulta DNS** (Etapa 2): Protocol **UDP**, porta **53**, Source **DMZ net**, e **Apply Changes** feito.

### "Resolve o nome mas trava no download (0%/timeout)"

É a regra de **update** (Etapa 1). Confirme Protocol **TCP**, porta **PORT_ADM** (80/443), Source `172.100.1.101`.

### "As regras existem mas nada passa"

Elas precisam estar **acima do Block Geral** ([69.md](69.md)) em `Firewall → Rules → DMZ`. Regra de Pass abaixo do block não vale.

### "SSH no Graylog recusa conexão"

Diferente do [79.md](79.md) (SSH no **firewall**), aqui é SSH no **Graylog** (`172.100.1.101`). Confirme que o `sshd` do Graylog está ativo e que você usa `user` (não `root`) + `su -` depois.

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Regra de saída (DMZ)** | Autoriza tráfego que **sai** de hosts da DMZ (avaliada na interface de entrada) |
| **Update precisa de 2 permissões** | HTTP/HTTPS (baixar) **+** DNS (resolver o nome do repositório) |
| **DNS / porta 53 / UDP** | Resolve **nomes → IPs**; consultas via UDP 53 ([66.md](66.md)) |
| **`DMZ net`** | Alias da subnet inteira da DMZ (`172.100.1.0/24`) ([70.md](70.md)) |
| **Escopo do Source** | Decisão de segurança: 1 host (update) × subnet (DNS) |
| **`apt update`** | Atualiza a **lista** de pacotes disponíveis |
| **`su -`** | Assume o usuário **root** (com ambiente de login) |
| **`iptables`** | Firewall **do host** Linux (nível kernel/netfilter) |
| **`iptables-persistent`** | Salva/restaura regras do iptables **entre reboots** |

## ✅ Quiz Mental

1. Por que precisamos de **duas** regras para o Graylog atualizar, e não só uma de HTTP/HTTPS?
2. Por que a regra de DNS usa **UDP** e porta **53**?
3. Por que a regra de **update** tem Source `172.100.1.101`, mas a de **DNS** usa `DMZ net`?
4. O `apt update` funcionar prova o quê sobre as regras?
5. Para que serve o **`iptables-persistent`**, e por que ele importa para defesa em profundidade?

<details>
<summary>Ver respostas</summary>

1. Porque `apt update` faz **duas** operações de rede: **resolver o nome** do repositório (DNS) e **baixar** os índices via **HTTP/HTTPS**. Sem a regra de **DNS**, o nome nunca vira IP e o update falha em "resolving..." — mesmo com o HTTP liberado
2. Porque consultas DNS usam **UDP/53** por padrão (rápidas, sem handshake). TCP/53 fica para respostas grandes e transferência de zona. A porta **53** é o padrão do protocolo ([66.md](66.md))
3. **Menor privilégio na saída à internet:** só o Graylog (`172.100.1.101`) precisa **atualizar**, então a regra de update é restrita a esse host. Já **resolver nomes** (DNS) é uma necessidade geral e de baixo risco, então liberamos a **DMZ net** inteira. O escopo do Source é escolhido conscientemente
4. Prova que **ambas** as regras estão corretas: o update só completa se o Graylog conseguir **resolver o nome** (DNS OK) **e** **baixar** os índices (HTTP/HTTPS OK). É a validação ponta a ponta do lab
5. O `iptables` é o firewall **do host** (Linux), mas suas regras **somem no reboot**; o `iptables-persistent` as **salva e restaura** ao reiniciar. Importa porque adiciona uma **segunda camada** de firewall (host) além do pfSense (rede) — **defesa em profundidade**: se uma falhar, a outra ainda protege

</details>

## 📝 Quiz Oficial do Curso — DROP × REJECT (iptables)

> **Cenário:** Samuel rodou `iptables -nvL` e achou duas regras. Qual alternativa diferencia corretamente?
>
> ```bash
> iptables -A OUTPUT -j DROP     # regra 1
> iptables -A INPUT  -j REJECT   # regra 2
> ```

| | Alternativa | Veredito |
|--|-------------|----------|
| A | Ambas fazem **bloqueio de saída** do pacote | ❌ |
| B | Ambas impedem o pacote de chegar ao destino, mas o **DROP** envia erro à origem | ❌ |
| C | Ambas **garantem** que o pacote chegue ao destino | ❌ |
| **D** | Ambas impedem o pacote de chegar ao destino, mas o **REJECT** envia um pacote de erro à origem | ✅ |

<details>
<summary>Ver gabarito comentado</summary>

**Resposta correta: D.**

**Por quê:** as duas ações **bloqueiam** o pacote, mas se comportam diferente com a **origem**:

| Ação | O que faz | Origem percebe? |
|------|-----------|-----------------|
| **DROP** | Descarta o pacote **em silêncio** (nenhuma resposta) | ❌ Fica no escuro (timeout) |
| **REJECT** | Descarta **e responde** com erro (ex.: ICMP *port unreachable* ou TCP *RST*) | ✅ Sabe que foi recusado |

É **exatamente** a distinção **Block × Reject** que vimos no pfSense ([69.md](69.md)): **Block/DROP = descarta silencioso**; **Reject/REJECT = responde com erro**.

**Por que as outras estão erradas:**
- **A** — as regras nem são as duas de "saída": a 1 é `OUTPUT` (saída), mas a 2 é `INPUT` (**entrada**). E "bloqueio de saída" não descreve a diferença entre DROP e REJECT.
- **B** — **invertido**: quem envia o pacote de erro é o **REJECT**, não o DROP (o DROP é silencioso).
- **C** — o oposto do que acontece: ambas **impedem** o pacote, não garantem a entrega.

> 🔑 **Quando usar cada um** ([69.md](69.md)): em **fronteira pública**, prefira **DROP/Block** — o silêncio não confirma ao atacante que o host existe (não "vaza" informação). **REJECT** é mais "educado" (o cliente sabe na hora que foi recusado, sem esperar timeout) — bom para **redes internas**, onde transparência ajuda no diagnóstico.

</details>

## 💡 Reflexão Final

> *"‘Deny by default’ é bonito no papel — até o dia em que o seu próprio servidor não consegue nem se atualizar.*
> *A arte não é **liberar** ou **bloquear**; é liberar **exatamente o necessário**, e nada além."*

Este lab mostrou o **outro lado** do firewall: até agora bloqueávamos e liberávamos **entrada**; aqui liberamos **saída** — de forma **cirúrgica**. Duas regras, dois protocolos (TCP para o conteúdo, UDP para os nomes), dois escopos de Source (um host × subnet). Cada escolha é uma decisão de **risco × necessidade**.

### O que mudou

| Antes | Depois |
|-------|--------|
| Graylog isolado, sem internet | Graylog **atualiza** e **resolve nomes** — só isso |
| Só firewall de rede (pfSense) | Preparando firewall **de host** (iptables) |
| Regras de **entrada** | Primeiras regras de **saída** cirúrgicas |

### O que vem agora

Com o `iptables-persistent` instalado, o próximo passo é **configurar o firewall local** do Graylog — levar o "deny by default" para **dentro** do host. Camada sobre camada: é assim que se constrói defesa em profundidade de verdade. 🎼🛡️
