# 🧪 Laboratório Prático: Criando Regras de Saída no iptables (Graylog)

> **Tema:** Configurar um **firewall de host** no Graylog com **3 regras de OUTPUT** (egress) via `iptables`, entendendo **ordem** e **estado de conexão**
> **Pré-requisito direto:** [80.md](80.md) (`iptables-persistent` instalado + DROP × REJECT)
> **Conceitos base:** [79.md](79.md) (SSH), [70.md](70.md) (gateway = `172.100.1.10`), [69.md](69.md) (deny by default / ordem das regras), [66.md](66.md) (conexão stateful)
> **Curso 2 — Proxy Reverso, SSH e DNS**

---

## 🎯 Objetivo do Lab

No [80.md](80.md) instalamos o `iptables-persistent`. Agora vamos **usar** o iptables para dar ao **Graylog** um firewall **do próprio host**, controlando o que ele pode **enviar** (OUTPUT). A política é a de sempre: **deny by default**, liberando só o necessário.

Vamos criar **3 regras de saída**:
1. Permitir saída de conexões **já estabelecidas / relacionadas** (stateful)
2. Permitir saída para o **gateway** da rede (`172.100.1.10`)
3. **Negar** a saída de todo o resto para a DMZ — possível atacante/host desconhecido

> 🔑 **Por que filtrar a SAÍDA de um SIEM?** Se o Graylog for **comprometido**, um atacante tentaria **pivotar** dele para outros hosts da DMZ. Travar o egress limita esse **movimento lateral** — o Graylog continua funcionando (fala com o gateway + mantém conexões), mas **não inicia** conexões novas para vizinhos da DMZ.

---

## 🧭 Chains do iptables (recap rápido)

O `iptables` filtra o tráfego do **próprio host** em três chains principais:

| Chain | Filtra… |
|-------|---------|
| **INPUT** | Pacotes **chegando** neste host |
| **OUTPUT** | Pacotes **saindo** deste host ← *foco deste lab* |
| **FORWARD** | Pacotes **passando** através (roteador) |

Como o Graylog é um **endpoint** (não roteia), o que importa é o **OUTPUT** — o que ele consegue **mandar para fora**.

## 🔑 Etapa 1 — Acessar o Graylog (via firewall = jump host)

O Graylog (`172.100.1.101`) está na **DMZ interna** e seu PC não o alcança direto. Então usamos o **firewall como bastion / jump host**: primeiro SSH no firewall, depois SSH no Graylog.

```bash
# 1) SSH no firewall (bastion)
ssh root@192.168.56.10
# senha:
qwerty
# menu do pfSense:
8                       # opção 8 = Shell

# 2) do firewall, SSH no Graylog (DMZ interna)
ssh user@172.100.1.101
# senha:
qwerty
su -                    # virar root
# senha:
qwerty
```

> 💡 **Jump host (bastion):** o firewall é o **único** ponto que alcança a DMZ interna, então ele serve de "ponte" para administrar o Graylog. É um padrão comum e **mais seguro** — só o bastion fica exposto ([79.md](79.md) habilitou esse SSH).

## 🛠️ Etapa 2 — As três regras

### Regra 1 — Conexões estabelecidas/relacionadas (stateful)

```bash
iptables -A OUTPUT -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT
```

| Trecho | Significado |
|--------|-------------|
| **`-A OUTPUT`** | **Append** (adiciona ao **fim**) da chain OUTPUT |
| **`-m conntrack --ctstate RELATED,ESTABLISHED`** | Casa pacotes de conexões **já estabelecidas** ou **relacionadas** a elas |
| **`-j ACCEPT`** | Ação: **permitir** |

> 🔑 **Firewall stateful** ([66.md](66.md)): `ESTABLISHED` = a conexão já trocou tráfego nos dois sentidos; `RELATED` = uma conexão nova **ligada** a uma existente (ex.: erro ICMP, dados de FTP). Sem essa regra, as **respostas** do próprio Graylog seriam barradas — ele mandaria um pedido e não poderia receber a continuação.

### Regra 2 — Saída para o gateway

```bash
iptables -I OUTPUT -d 172.100.1.10 -j ACCEPT
```

| Trecho | Significado |
|--------|-------------|
| **`-I OUTPUT`** | **Insert** no **topo** (posição 1) da chain |
| **`-d 172.100.1.10`** | Destino = **gateway** (IP do firewall na DMZ — o `DMZ address` do [70.md](70.md)) |
| **`-j ACCEPT`** | Permitir |

> 💡 **`-A` × `-I`:** `-A` **anexa no fim**; `-I` **insere no topo** (ou numa posição). Como usamos `-I`, essa regra do gateway vai para a **posição 1**, empurrando a Regra 1 para a **posição 2**. **Ordem importa** — o iptables avalia de cima para baixo, **primeira que casar vence** ([69.md](69.md)).

### Regra 3 — Negar o resto (para a DMZ)

```bash
iptables -I OUTPUT 3 -d 172.100.1.0/24 -j DROP
```

| Trecho | Significado |
|--------|-------------|
| **`-I OUTPUT 3`** | Insere na **posição 3** — **depois** das duas regras ACCEPT |
| **`-d 172.100.1.0/24`** | Destino = **toda a subnet da DMZ** |
| **`-j DROP`** | **Descarta em silêncio** ([80.md](80.md) — DROP × REJECT) |

> ⚠️ **A posição 3 é crucial.** Se esse DROP fosse para o **topo**, bloquearia **tudo** para a DMZ — inclusive o gateway (`172.100.1.10` está dentro de `172.100.1.0/24`) — e o Graylog perderia conectividade. Ao colocá-lo **por último**, garantimos que gateway (Regra 1) e conexões estabelecidas (Regra 2) sejam aceitas **antes**, e só o **restante** seja negado.

## 📋 A chain OUTPUT resultante

Depois dos 3 comandos, a ordem final é:

```
Nº  Alvo    Destino            Ação     Por quê
──  ──────  ─────────────────  ───────  ─────────────────────────────
1   ACCEPT  172.100.1.10       permite  gateway (rota de saída)
2   ACCEPT  conntrack EST,REL  permite  respostas / conexões em curso
3   DROP    172.100.1.0/24     nega     resto da DMZ (movimento lateral)
```

> 🔑 **Leitura:** o Graylog pode falar com o **gateway** (e via ele, sair p/ internet — regras do [80.md](80.md)) e manter **conexões estabelecidas**, mas **não inicia** conexão nova para **outros hosts da DMZ**. Isso é **microssegmentação de egress** no nível de host.

## ✅ Etapa 3 — Conferir e salvar

### Conferir

```bash
iptables -nvL
```

| Flag | Significado |
|------|-------------|
| **`-n`** | Numérico (não resolve nomes/portas — mais rápido e claro) |
| **`-v`** | Verboso — mostra **contadores** de pacotes/bytes por regra |
| **`-L`** | Lista as regras |

> 💡 Os contadores do `-v` mostram **quantos pacotes** cada regra pegou — ótimo para **depurar** ("essa regra está sendo usada?").

### Salvar (persistir no reboot)

```bash
iptables-save > /etc/iptables/rules.v4
```

> 🔑 **Aqui fecha o ciclo com o [80.md](80.md):** as regras do iptables **somem no reboot**. O `iptables-save` grava o estado atual em **`/etc/iptables/rules.v4`** — e é **exatamente esse arquivo** que o `iptables-persistent` **recarrega na inicialização**. Sem esse passo, tudo se perde ao reiniciar o Graylog.

## 🐞 Troubleshooting Comum

### "Perdi conexão com o Graylog ao aplicar o DROP"

Provavelmente o DROP entrou **antes** das regras ACCEPT (ordem errada). Confira com `iptables -nvL` se a ordem é gateway → established → drop. Corrija com `-I OUTPUT <posição>` ou limpe e refaça (`iptables -F OUTPUT`, com cuidado).

### "As regras somem depois de reiniciar"

Faltou **salvar**: `iptables-save > /etc/iptables/rules.v4`. Confirme também que o `iptables-persistent` está instalado ([80.md](80.md)).

### "O Graylog não sai mais para a internet"

O DROP em `172.100.1.0/24` **não** bloqueia a internet (outra subnet) — bloqueia a **DMZ**. Se a internet parou, revise as **regras do pfSense** ([80.md](80.md)) e a regra do **gateway** (sem ela, não há rota de saída).

### "Ordem certa, mas continua bloqueando resposta"

Falta a regra **stateful** (`RELATED,ESTABLISHED`). Sem ela, o retorno das conexões é barrado.

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Chains INPUT/OUTPUT/FORWARD** | Entrada / saída / passagem no iptables |
| **Filtro de egress (OUTPUT)** | Controlar o que o host pode **enviar** |
| **`-A` (append)** | Adiciona a regra ao **fim** da chain |
| **`-I` (insert)** | Insere no **topo** (ou em posição `N`) |
| **Ordem das regras** | Avaliação **top-down**, primeira que casa vence ([69.md](69.md)) |
| **conntrack / stateful** | `ESTABLISHED` (conexão em curso) + `RELATED` (ligada a outra) ([66.md](66.md)) |
| **Gateway (`172.100.1.10`)** | IP do firewall na DMZ = rota de saída ([70.md](70.md)) |
| **DROP** | Descarta em silêncio ([80.md](80.md)) |
| **`iptables -nvL`** | Lista regras numéricas com contadores |
| **`iptables-save > rules.v4`** | Persiste as regras (lidas pelo `iptables-persistent` no boot) |
| **Microssegmentação de egress** | Impedir movimento lateral limitando saída p/ a própria subnet |

## ✅ Quiz Mental

1. Por que a regra de **DROP** foi inserida na **posição 3**, e não no topo?
2. Qual a diferença entre **`-A`** e **`-I`** no iptables?
3. Para que serve a regra **`RELATED,ESTABLISHED`**? O que quebra sem ela?
4. Por que filtrar a **saída** (OUTPUT) de um SIEM faz sentido em segurança?
5. Depois de criar as regras, por que rodar `iptables-save > /etc/iptables/rules.v4`?

<details>
<summary>Ver respostas</summary>

1. Porque o iptables avalia **de cima para baixo** e a primeira regra que casa vence ([69.md](69.md)). Se o DROP `172.100.1.0/24` viesse **primeiro**, bloquearia **tudo** para a DMZ — inclusive o **gateway** (`172.100.1.10`, que está dentro de `/24`) — derrubando a conectividade. Na posição 3, ele só pega o que as duas regras ACCEPT (gateway + established) **não** aceitaram antes
2. **`-A` (append)** adiciona a regra no **fim** da chain; **`-I` (insert)** insere no **topo** por padrão, ou numa **posição N** se especificada (`-I OUTPUT 3`). Como ordem importa, a escolha entre append/insert define a prioridade da regra
3. Autoriza a saída de pacotes de conexões **já estabelecidas** (`ESTABLISHED`) ou **relacionadas** (`RELATED`) a elas — o comportamento **stateful** ([66.md](66.md)). Sem ela, as **respostas/continuações** das conexões do próprio Graylog seriam barradas: ele mandaria um pedido e não conseguiria concluir a troca
4. Porque, se o SIEM for **comprometido**, o atacante tentaria **pivotar** dele para outros hosts. Restringir o **egress** (só gateway + established, DROP no resto da DMZ) limita esse **movimento lateral** — é microssegmentação no nível de host, defesa em profundidade
5. Porque as regras do iptables ficam **em memória** e **somem no reboot**. O `iptables-save` grava o estado em **`/etc/iptables/rules.v4`**, arquivo que o **`iptables-persistent`** ([80.md](80.md)) **recarrega na inicialização** — tornando as regras permanentes

## 💡 Reflexão Final

> *"Firewall não é só sobre **quem entra** — é também sobre **o que sai**.*
> *Um servidor que não pode iniciar conexões estranhas é um servidor que, se cair, **não arrasta os vizinhos junto**."*

Este lab levou o "deny by default" para **dentro** do host. Aprendemos que, no iptables, **ordem é tudo**: três comandos na sequência certa constroem uma política coerente; na sequência errada, você se **tranca para fora**. E aprendemos o valor de filtrar o **egress** — um ângulo que muita gente esquece, focada só em bloquear entrada.

### O que mudou

| Antes | Depois |
|-------|--------|
| Firewall só na **rede** (pfSense) | Firewall também **no host** (iptables) — 2 camadas |
| Foco em tráfego de **entrada** | Também controlamos a **saída** (egress) |
| Graylog livre para falar na DMZ | Graylog **contido** — só gateway + conexões estabelecidas |

### O que vem agora

Com host e rede endurecidos, o Curso 2 avança para **proxy reverso** e **DNS** — como o tráfego **chega** aos serviços e como **nomes viram IPs**. A base de defesa está sólida; agora entra a camada de **entrega e resolução**. 🎼🛡️
