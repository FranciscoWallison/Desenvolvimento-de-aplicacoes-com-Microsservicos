# 🧪 Laboratório Prático: Configurando o Resolvedor DNS (Cloudflare 1.1.1.1)

> **Tema:** Apontar o **servidor** para o **resolvedor DNS público da Cloudflare** (`1.1.1.1` / `1.0.0.1`) editando o `/etc/resolv.conf`
> **Pré-requisitos:** [80.md](80.md) (regra de saída **DNS/UDP 53** já liberada), [81.md](81.md) (acesso via jump host)
> **Conceitos base:** [66.md](66.md) (DNS = nomes → IPs, porta 53), [79.md](79.md) (SSH)
> **Curso 2 — Proxy Reverso, SSH e DNS**

---

## 🎯 Objetivo do Lab

No [80.md](80.md) criamos a regra de firewall que **permite** consultas DNS saindo da DMZ (UDP/53). Agora definimos **para qual resolvedor** o servidor vai perguntar: o **`1.1.1.1` da Cloudflare**.

São **as duas metades** do DNS funcionando:

| Metade | Onde | O que faz |
|--------|------|-----------|
| **Permissão** | pfSense ([80.md](80.md)) | Deixa a consulta DNS **sair** (UDP 53) |
| **Configuração** | `/etc/resolv.conf` (este lab) | Diz **quem** responde a consulta (Cloudflare) |

> 🔑 **Sem uma das duas, não resolve:** firewall liberado mas sem resolvedor configurado = não sabe **a quem** perguntar; resolvedor configurado mas firewall fechado = a pergunta **não sai**.

---

## 📖 O que é um resolvedor DNS?

> **DNS** traduz **nomes** (`deb.debian.org`) em **IPs** (`151.101.x.x`) — porta **53** ([66.md](66.md)). O **resolvedor** é o servidor que o seu host **pergunta** quando precisa dessa tradução.

O **`1.1.1.1`** da Cloudflare é um **resolvedor público recursivo**. Benefícios citados no vídeo:

| Benefício | Detalhe |
|-----------|---------|
| 🔒 **Privacidade** | Não vende seus dados nem os usa para **anúncios** |
| ⚡ **Rapidez** | Frequentemente medido como o **DNS público mais veloz** |

> 💡 **Alternativas conhecidas:** Google (`8.8.8.8`/`8.8.4.4`), Quad9 (`9.9.9.9`, com bloqueio de domínios maliciosos). O `1.1.1.1` usa **anycast** — você fala com o nó **mais próximo** geograficamente, o que ajuda na latência.

## 🔑 Etapa 1 — Acessar o servidor (via firewall = jump host)

Mesmo padrão do [81.md](81.md): o servidor (`172.100.1.100`) está na **DMZ interna**, então saltamos pelo firewall.

```bash
# 1) SSH no firewall (bastion)
ssh root@192.168.56.10
# senha:
qwerty
# menu do pfSense:
8                      # opção 8 = Shell

# 2) do firewall, SSH no servidor web (DMZ interna)
ssh user@172.100.1.100
# senha:
qwerty
su -                   # virar root
# senha:
qwerty
```

> ⚠️ **Atenção ao IP:** aqui é o **servidor web `172.100.1.100`** — diferente do **Graylog `172.100.1.101`** dos labs [80.md](80.md)/[81.md](81.md). Confira o último octeto.

## 🗑️ Etapa 2 — Remover a config de proxy do apt

```bash
rm /etc/apt/apt.conf
```

O arquivo `/etc/apt/apt.conf` continha uma configuração de **proxy** padrão para o `apt` (algo como `Acquire::http::Proxy "..."`). Removê-lo faz o `apt` ir **direto** aos repositórios — usando a **resolução DNS** que estamos configurando, em vez de passar por um proxy.

> 💡 **Por que isso importa aqui:** se o `apt` estivesse forçado a um proxy, ele **ignoraria** o resolvedor novo. Limpar o `apt.conf` garante que o servidor use o **caminho DNS → Cloudflare** que vamos definir a seguir.

---

## 📝 Etapa 3 — Editar o `/etc/resolv.conf`

```bash
nano /etc/resolv.conf
```

Cole o conteúdo:

```conf
nameserver 1.0.0.1
nameserver 1.1.1.1
```

Salve com **Ctrl + X** → **Enter**.

### O que cada linha faz

| Linha | Significado |
|-------|-------------|
| `nameserver 1.0.0.1` | Resolvedor **primário** (Cloudflare secundário) — o 1º consultado |
| `nameserver 1.1.1.1` | Resolvedor **de reserva** (Cloudflare primário) |

> 🔑 **Por que dois `nameserver`?** **Redundância/failover:** o sistema tenta o **primeiro** da lista; se ele não responder, cai para o **segundo**. Ter dois IPs da Cloudflare evita ficar sem resolução se um nó falhar.

> ⚠️ **Cuidado em produção:** o `/etc/resolv.conf` costuma ser **sobrescrito** no reboot por **DHCP**, **NetworkManager** ou **systemd-resolved**. Para tornar permanente, configura-se pela ferramenta de rede (ou trava-se o arquivo com `chattr +i`). No lab, a edição direta basta.

## ✅ Validação

Depois de salvar, teste a resolução no próprio servidor:

```bash
nslookup google.com        # ou: dig google.com
ping deb.debian.org        # resolve o nome e responde?
apt update                 # agora deve resolver e baixar (80.md + este lab)
```

> ✅ Se o `apt update` do [80.md](80.md) funcionar agora com a config nova, o **caminho completo** está fechado: firewall libera (UDP 53) → servidor pergunta ao **1.1.1.1** → nome vira IP → download.

## 🔐 Segurança do DNS (contexto)

O DNS é um ponto sensível: como lembramos no [66.md](66.md), ele **pode ser abusado** para **desviar** tráfego de um site legítimo para um **malicioso**. Por isso a **escolha do resolvedor importa**:

| Risco | Mitigação |
|-------|-----------|
| Resolvedor que **espia/vende** consultas | Usar resolvedor com política de **privacidade** (Cloudflare não vende dados) |
| **DNS spoofing** / respostas forjadas | **DNSSEC** (valida a autenticidade das respostas) |
| Consultas DNS em **texto claro** (porta 53) | **DoH** (DNS over HTTPS) / **DoT** (DNS over TLS) — o `1.1.1.1` suporta ambos |

> 💡 **Evolução natural:** a porta 53 clássica é **não criptografada** — quem está no caminho vê suas consultas. **DoH/DoT** cifram o DNS. Fora do escopo deste lab, mas é o próximo passo de maturidade.

## 🐞 Troubleshooting Comum

### "`apt update` ainda dá 'Temporary failure resolving'"

1. O `/etc/resolv.conf` tem as duas linhas `nameserver`? (Etapa 3)
2. A **regra de DNS** do [80.md](80.md) (UDP 53, Source DMZ net) está ativa?
3. Removeu o `/etc/apt/apt.conf`? (proxy antigo pode estar atrapalhando)

### "Resolvia, mas parou depois do reboot"

O `/etc/resolv.conf` foi **sobrescrito** (DHCP/systemd-resolved). Reaplique ou fixe pela ferramenta de rede.

### "Errei o servidor (fui parar no Graylog)"

Confira o IP do SSH: **servidor = `172.100.1.100`**; Graylog = `172.100.1.101`.

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Resolvedor DNS** | Servidor que responde consultas nome → IP ([66.md](66.md)) |
| **`/etc/resolv.conf`** | Arquivo Linux que lista os `nameserver` usados pelo sistema |
| **`nameserver`** | Cada linha = um IP de servidor DNS a consultar |
| **`1.1.1.1` / `1.0.0.1`** | Resolvedor público da Cloudflare (privacidade + velocidade) |
| **Redundância de DNS** | Dois `nameserver` = failover se um não responder |
| **`/etc/apt/apt.conf`** | Config do apt (podia ter **proxy**) — removida p/ ir direto via DNS |
| **Anycast** | Mesmo IP anunciado de vários locais → responde o nó mais próximo |
| **DoH / DoT** | DNS criptografado (over HTTPS / over TLS) |
| **DNSSEC** | Valida autenticidade das respostas DNS (anti-spoofing) |

## ✅ Quiz Mental

1. Quais são **as duas metades** necessárias para o servidor resolver nomes, e onde cada uma é configurada?
2. Para que serve o `/etc/resolv.conf` e o que faz cada linha `nameserver`?
3. Por que colocamos **dois** `nameserver`?
4. Por que removemos o `/etc/apt/apt.conf`?
5. Por que a **escolha do resolvedor** é uma questão de segurança/privacidade?

<details>
<summary>Ver respostas</summary>

1. (a) **Permissão de saída** — a regra de firewall do [80.md](80.md) que libera **UDP 53** na DMZ; (b) **configuração do resolvedor** — o `/etc/resolv.conf` apontando para o `1.1.1.1`. Sem a permissão, a consulta não sai; sem o resolvedor, não há a quem perguntar
2. O `/etc/resolv.conf` diz ao sistema **quais servidores DNS** consultar. Cada linha `nameserver <IP>` define um resolvedor; o sistema tenta na ordem listada
3. **Redundância/failover:** se o primeiro `nameserver` não responder, o sistema usa o segundo. Dois IPs da Cloudflare mantêm a resolução funcionando se um nó cair
4. Porque ele continha uma config de **proxy** para o `apt`. Com o proxy no caminho, o `apt` **não usaria** o resolvedor novo. Removendo o arquivo, o `apt` vai **direto** aos repositórios via DNS
5. Porque o resolvedor **vê todas as suas consultas** (pode espiar/vender) e, se malicioso ou comprometido, pode **desviar** você para sites falsos ([66.md](66.md)). Um resolvedor confiável (Cloudflare: não vende dados) — idealmente com **DNSSEC** e **DoH/DoT** — reduz esses riscos

</details>

## 💡 Reflexão Final

> *"Antes de qualquer conexão, existe uma pergunta silenciosa: **‘qual o IP desse nome?’**.*
> *Quem responde essa pergunta tem um poder enorme — por isso importa **em quem** você confia para respondê-la."*

Este lab foi curto, mas conceitualmente denso. Vimos que resolver nomes exige **duas peças** (firewall + resolvedor) e que o DNS, apesar de invisível, é uma **superfície de ataque** real — desviar o DNS é desviar **todo** o tráfego que depende dele. Escolher a Cloudflare (`1.1.1.1`) foi uma decisão de **privacidade e velocidade**.

### O que mudou

| Antes | Depois |
|-------|--------|
| Firewall **permitia** DNS, mas sem resolvedor definido | Servidor aponta para o **1.1.1.1** da Cloudflare |
| `apt` preso a um proxy | `apt` resolve **direto** via DNS |
| DNS como detalhe invisível | DNS entendido como **decisão de segurança** |

### O que vem agora

Com nomes resolvendo, a próxima grande peça do Curso 2 é o **proxy reverso** — o componente que recebe as requisições **por nome** e as encaminha ao serviço certo. DNS entrega o **nome→IP**; o proxy reverso decide **o que fazer** com a requisição que chega. 🎼🛡️
