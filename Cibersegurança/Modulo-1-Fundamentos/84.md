# 🧪 Laboratório Prático: Configurando o Proxy Reverso no WAF (`site.lab_conf`)

> **Tema:** Configurar o **WAF (nginx)** como **proxy reverso**, encaminhando as requisições para o servidor interno e **preservando o IP real** do usuário nos headers
> **Pré-requisitos:** [83.md](83.md) (Nginx no servidor `172.100.1.100`), [82.md](82.md) (jump host)
> **Conceitos base:** [67.md](67.md) (WAF **é** proxy reverso), [75.md](75.md) (WAF = nginx + ModSecurity; reload), [74.md](74.md) (logs no SIEM)
> **Curso 2 — Proxy Reverso, SSH e DNS**

## 🎯 Objetivo do Lab

Chegamos ao **coração do Curso 2**. O [67.md](67.md) já dizia que *"o WAF é tipicamente implementado como **proxy reverso** entre a internet e o servidor"*. Agora vamos **implementar isso na prática**: o **WAF** (`172.100.2.100`, DMZ externa) recebe as requisições e as **encaminha** ao servidor web interno (`172.100.1.100`, onde instalamos o Nginx no [83.md](83.md)).

```
Internet → 🛡️ WAF/nginx (172.100.2.100) ──proxy_pass──► 🖥️ Server (172.100.1.100)
```

E o ponto de segurança do lab: configurar os **headers** para que o servidor interno enxergue o **IP real** de quem fez a requisição — não o IP do WAF. Isso enriquece os **logs** (e o SIEM, [74.md](74.md)).

---

## 📖 O que é um proxy reverso?

> Um **proxy reverso** fica **na frente** dos servidores e recebe as requisições **no lugar deles**, repassando ao backend correto. O cliente fala só com o proxy; o backend fica **escondido**.

| Benefício | Como ajuda |
|-----------|-----------|
| **Ocultar o backend** | O cliente nunca vê o IP/estrutura interna |
| **Ponto único de controle** | WAF, TLS, rate limit, logs — tudo num lugar |
| **Balanceamento / roteamento** | Pode distribuir entre vários backends |
| **Camada de segurança** | O ModSecurity ([75.md](75.md)) inspeciona **antes** de chegar ao servidor |

> 🔁 **Proxy reverso × proxy (forward):** o *forward proxy* representa o **cliente** (sai para a internet por ele); o **reverso** representa o **servidor** (a internet chega por ele).

> ⚠️ **IP do WAF = `172.100.2.100`** (DMZ **externa**) — diferente do servidor `172.100.1.100` e do Graylog `172.100.1.101` (DMZ interna).

## 📝 Etapa 2 — Criar o arquivo do site

```bash
cd /etc/nginx/sites-available
rm default                 # remove a config padrão do nginx
nano site.lab_conf
```

Cole:

```nginx
server {
        listen 80;

        server_name site.lab;

        location / {
                proxy_pass http://172.100.1.100/;
                proxy_set_header Host $http_host;
                proxy_set_header X-Real-IP $remote_addr;
                proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
                proxy_set_header X-Forwarded-Proto $scheme;
        }
}
```

Salve com **Ctrl + X** → **Enter**.

### O que cada diretiva faz

| Diretiva | Significado |
|----------|-------------|
| `listen 80` | O WAF escuta na **porta 80** (HTTP) |
| `server_name site.lab` | Responde às requisições para o nome **`site.lab`** |
| `location / { … }` | Aplica a **todos** os caminhos da URL |
| `proxy_pass http://172.100.1.100/` | **Encaminha** ao servidor interno — a essência do proxy reverso |

### Os headers — o coração do lab

Sem eles, o servidor interno veria **todas** as requisições vindo do **IP do WAF** (`172.100.2.100`) — e perderíamos quem **de fato** acessou. Os `proxy_set_header` corrigem isso:

| Header | O que carrega |
|--------|---------------|
| `Host $http_host` | Preserva o **domínio** que o cliente pediu (não vira o IP do backend) |
| `X-Real-IP $remote_addr` | O **IP real** do cliente que fez a requisição |
| `X-Forwarded-For $proxy_add_x_forwarded_for` | **Cadeia** de IPs por onde a requisição passou (proxies) |
| `X-Forwarded-Proto $scheme` | Se o cliente usou **http** ou **https** |

> 🔑 **É exatamente o objetivo do vídeo:** com esses headers, os **logs** do servidor (e o SIEM, [74.md](74.md)) mostram o **IP real do usuário**, não o do proxy. Sem `X-Real-IP`/`X-Forwarded-For`, todo acesso pareceria vir do WAF — cegando a investigação forense.

## 🔗 Etapa 3 — Ativar o site (symlink)

O nginx separa **configs disponíveis** de **configs ativas**:

| Diretório | Papel |
|-----------|-------|
| `/etc/nginx/sites-available/` | Onde **moram** todas as configs (disponíveis) |
| `/etc/nginx/sites-enabled/` | **Symlinks** para as configs **ativas** |

Ativar o site = criar um **link simbólico** de `sites-enabled` → `sites-available`:

```bash
cd ../sites-enabled/
ls -l                    # veja os links atuais
rm default               # remove o site padrão ativo
ln -s /etc/nginx/sites-available/site.lab_conf /etc/nginx/sites-enabled/site.lab_conf
```

| Comando | Significado |
|---------|-------------|
| `ls -l` | Lista mostrando os **symlinks** (aparecem como `link -> alvo`) |
| `rm default` | Desativa o site padrão (só remove o **link**, a config original fica em available) |
| `ln -s <alvo> <link>` | Cria o **link simbólico** que **ativa** o `site.lab_conf` |

> 💡 **Por que esse padrão?** Você **habilita/desabilita** sites só criando/removendo o **link** — sem apagar a config. É reversível e organizado. `rm` no `sites-enabled` **não** apaga o arquivo real (que está em `sites-available`).

## 🔄 Etapa 4 — Recarregar e verificar

```bash
nginx -s reload
ss -nlt
```

- **`nginx -s reload`** — recarrega a config **sem derrubar** conexões (zero-downtime, [75.md](75.md))
- **`ss -nlt`** — confirma a **porta 80** escutando ([83.md](83.md))

> 🔑 **Boa prática ([75.md](75.md)):** rode **`nginx -t`** *antes* do reload para **validar a sintaxe**. Recarregar uma config quebrada pode derrubar o WAF. O `-t` testa sem aplicar.

## 🗺️ O fluxo pós-lab

```
                 Host: site.lab
 Cliente ───────────────────────────►  🛡️ WAF/nginx (172.100.2.100:80)
 (IP real X)                            server_name site.lab
                                        proxy_pass → 172.100.1.100
                                        + X-Real-IP: X
                                              │
                                              ▼
                                        🖥️ Server (172.100.1.100)
                                        vê o IP real X nos headers/logs
```

O ModSecurity ([75.md](75.md)) roda **no WAF** — então a inspeção contra ataques acontece **antes** de a requisição chegar ao servidor. Proxy reverso + WAF = a mesma peça.

## 🐞 Troubleshooting Comum

### "`nginx -s reload` dá erro de sintaxe"

Rode `nginx -t` para ver a linha do erro. Comum: faltou `;` no fim de uma diretiva ou `{}` desbalanceado. Corrija o `site.lab_conf` e recarregue.

### "502 Bad Gateway ao acessar"

O WAF não conseguiu falar com o backend `172.100.1.100`. Verifique:
1. O Nginx do servidor está no ar? (`ss -nlt` no `172.100.1.100`, [83.md](83.md))
2. A regra de firewall permite WAF → server na porta 80?

### "O site não responde pelo nome `site.lab`"

`server_name site.lab` só funciona se **`site.lab` resolver** para o WAF. Em lab, adiciona-se ao `/etc/hosts` do cliente (ou DNS). Por IP direto, o `location /` ainda responde.

### "O backend vê o IP do WAF, não o do cliente"

Confirme os `proxy_set_header X-Real-IP`/`X-Forwarded-For` no `site.lab_conf` e que o **backend** está configurado para **ler** esses headers (`real_ip_header X-Forwarded-For` / `set_real_ip_from`).

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Proxy reverso** | Recebe requisições no lugar do servidor e as **encaminha** ao backend ([67.md](67.md)) |
| **`proxy_pass`** | Diretiva nginx que **encaminha** ao backend |
| **`server_name`** | Nome (virtual host) a que o bloco responde |
| **`X-Real-IP` / `X-Forwarded-For`** | Headers que carregam o **IP real** do cliente ao backend |
| **`X-Forwarded-Proto`** | Informa http/https original ao backend |
| **`sites-available` × `sites-enabled`** | Configs disponíveis × ativas (via symlink) |
| **`ln -s`** | Cria **link simbólico** — ativa um site sem copiar |
| **`nginx -t`** | Valida a config **antes** do reload |
| **`nginx -s reload`** | Recarrega config sem downtime ([75.md](75.md)) |

4. Por que rodar `nginx -t` antes do `nginx -s reload`?
5. Como esse lab se conecta ao **WAF/ModSecurity** e ao **SIEM**?

<details>
<summary>Ver respostas</summary>

1. É um servidor que fica **na frente** do backend e recebe as requisições **por ele**, encaminhando-as (`proxy_pass`). O cliente só fala com o proxy; o backend (`172.100.1.100`) fica **escondido** atrás do WAF (`172.100.2.100`) ([67.md](67.md))
2. Carregam o **IP real do cliente** até o backend. Sem eles, o servidor interno veria **todas** as requisições vindo do **IP do WAF** — perdendo quem realmente acessou, o que **cega os logs** e a investigação forense (o oposto do objetivo do lab)

4. Porque `nginx -t` **valida a sintaxe sem aplicar**. Um `reload` com config quebrada pode **derrubar** o WAF ([75.md](75.md)); o `-t` evita o downtime pegando o erro antes
5. O WAF **é** o nginx com **ModSecurity** ([75.md](75.md)): ao ser o proxy reverso, ele **inspeciona** o tráfego (contra OWASP Top 10) **antes** de chegar ao servidor. E os headers `X-Real-IP`/`X-Forwarded-For` fazem os **logs** carregarem o IP real do cliente — que segue para o **SIEM** ([74.md](74.md)), enriquecendo a análise

## 📝 Quiz Oficial do Curso — Por que usar o WAF como proxy reverso?

> **Cenário:** Montamos o WAF como proxy reverso, encaminhando as requisições ao servidor na DMZ interna. Por que usar um proxy reverso em vez de deixar o servidor web lidar direto com tudo? **(Selecione 2)**

| | Alternativa | Veredito |
|--|-------------|----------|
| A | Não há motivo real para isso | ❌ |
| **B** | As conexões passam **primeiro pelo WAF**, que gera **registros/logs** relevantes p/ monitoramento | ✅ |
| C | Com o WAF na frente, "algum deles poderá ficar offline sem problemas" | ❌ |
| **D** | As conexões passam primeiro pelo WAF, que **filtra tráfego malicioso** com seu conjunto de regras | ✅ |

<details>
<summary>Ver gabarito comentado</summary>

**Respostas corretas: B e D.**

**Por quê:** ao ficar **na frente** do servidor ([84.md — este doc](84.md)), todo tráfego passa **obrigatoriamente** pelo WAF antes de chegar ao backend. Isso dá **dois ganhos**:
- **D — Filtragem de ataques:** o WAF tem um **conjunto de regras** (ModSecurity + OWASP CRS, [67.md](67.md)) e **bloqueia** tráfego malicioso — exatamente o **403** que vimos no [88.md](88.md) contra o SQLi.
- **B — Visibilidade/logs:** como ponto de passagem obrigatório, o WAF **registra** as requisições (com o **IP real**, via `X-Real-IP`) — logs que alimentam o **SIEM** ([74.md](74.md)/[75.md](75.md)) para monitoramento.

**Por que as outras estão erradas:**
- **A** — há motivos claros (segurança + observabilidade); é o oposto do correto.
- **C** — **alta disponibilidade / failover** (um servidor cair "sem problemas") é papel de **load balancer / redundância**, não do que configuramos aqui. Nosso proxy reverso tem **um único backend** (`172.100.1.100`) — se ele cai, o site cai (502). A frase confunde proxy reverso de **segurança** com **balanceamento de carga**.

> 🔑 Os **dois** motivos reais deste lab: **filtrar ataques** (D) e **registrar/monitorar** (B). *Load balancing* é um benefício **possível** de proxies reversos em geral, mas **não** é o que montamos aqui.

</details>

## 💡 Reflexão Final

> *"O proxy reverso é o **porteiro** do prédio: todo mundo fala com ele, ninguém fala direto com os apartamentos.*
> *E um bom porteiro **anota quem chegou** — por isso os headers com o IP real importam tanto."*

Este lab uniu o que estava separado: o **WAF** (que já filtrava ataques, [75.md](75.md)) agora é também o **proxy reverso** que **entrega** o tráfego ao backend. É a mesma peça exercendo dois papéis — **segurança** (ModSecurity) e **roteamento** (proxy_pass) — no ponto exato onde faz mais sentido: **na frente de tudo**.

### O que mudou

| Antes | Depois |
|-------|--------|
| WAF só inspecionava | WAF **inspeciona e encaminha** (proxy reverso) |
| Backend exposto/direto | Backend **escondido** atrás do WAF |
| Logs veriam o IP do proxy | Logs veem o **IP real** (X-Real-IP) |

### O que vem agora

Falta a última peça para o acesso **por nome**: fazer `site.lab` **resolver** para o WAF (via `/etc/hosts` ou DNS) e **testar a aplicação** ponta a ponta — cliente → `site.lab` → WAF → server. O porteiro está no posto; agora é abrir a portaria. 🎼🛡️
