## 🎯 Objetivo do Lab

Para **aprender a defender**, precisamos de algo para **atacar** — de forma **legal e controlada**. Entra o **DVWA** (*Damn Vulnerable Web Application*): uma aplicação web **propositalmente insegura**, feita para treinar ataques e defesas.

Neste lab vamos **criar a VM do DVWA** no VirtualBox e anexar o ISO. É a **5ª VM** do laboratório (firewall, WAF, server, Graylog + **DVWA**).

> 📥 **Download:** o pacote `Seguranca-Defensiva.zip` da Alura contém o `DVWA-1.0.7.iso`.

> 🔒 **Ética (sempre):** o DVWA só deve ser usado em **ambiente isolado de laboratório**. Atacar aplicações reais sem autorização é crime. Aqui o alvo é **nosso**, controlado e desconectado da internet.

---

## 📖 O que é o DVWA?

> **DVWA (Damn Vulnerable Web Application)** é uma app **PHP/MySQL** cheia de vulnerabilidades **de propósito**, com **níveis de dificuldade** ajustáveis (Low/Medium/High/Impossible).

| Recurso | Detalhe |
|---------|---------|
| **Para que serve** | Treinar **ataque e defesa** web num alvo legal |
| **Vulnerabilidades** | SQL Injection, XSS, CSRF, Command Injection, File Upload, Brute Force… ([47.md](47.md)) |
| **Níveis** | Do **Low** (sem proteção) ao **Impossible** (código seguro de referência) |
| **Concorrentes** | **bWAPP** ([62.md](62.md)), Juice Shop, WebGoat, Mutillidae |

> 🔑 **Papel no nosso lab:** o DVWA será o **alvo atrás do WAF**. Vamos atacá-lo **sem** e **com** o ModSecurity ([67.md](67.md)) para **ver o WAF bloqueando** — e tudo pousando no SIEM.

## 🛠️ Etapa 1 — Criar a VM

No **VirtualBox** → **Ferramentas** → **Novo**. Preencha:

```makefile
Nome: DVWA
Pasta da Máquina: <padrão do seu usuário>   # ex.: C:\Users\<você>\VirtualBox VMs
Tipo: Linux
Versão: Ubuntu (64-bit)
```

**Próximo (N)**.

| Campo | Valor | Por quê |
|-------|-------|---------|
| **Tipo: Linux / Ubuntu (64-bit)** | — | O DVWA roda sobre base Linux |
| **Nome: DVWA** | — | Identifica a VM na lista |

## 💾 Etapa 2 — Memória e disco

| Passo | Valor |
|-------|-------|
| **Tamanho da memória** | **512 MB** de RAM |
| **Disco rígido** | *Criar um novo disco rígido virtual agora* → **Criar** |
| **Tipo de arquivo** | **VDI** (VirtualBox Disk Image) |
| **Armazenamento** | **Dinamicamente alocado** |
| **Localização e tamanho** | **5,00 GB** → **Criar** |

### Os detalhes que importam

| Opção | Significado |
|-------|-------------|
| **VDI** | Formato **nativo** do VirtualBox (há também VHD/VMDK p/ interoperar com outros hypervisors) |
| **Dinamicamente alocado** | O arquivo **cresce conforme o uso** (até 5 GB) — economiza disco vs. tamanho fixo |
| **512 MB / 5 GB** | Suficiente para uma app pequena de treino |

> 💡 **Dinâmico × fixo:** o disco **dinâmico** ocupa pouco no início e cresce sob demanda (levemente mais lento); o **fixo** reserva todo o espaço de uma vez (um pouco mais rápido). Para lab, **dinâmico** é o padrão sensato.

---

## 💿 Etapa 3 — Anexar o ISO do DVWA

Selecione a VM **DVWA** → **Configurações** → **Armazenamento**:

1. Em **Controladora IDE**, clique no disco **"Acrescente Disco Óptico"**
2. No Explorador de Arquivos, selecione o **`DVWA-1.0.7.iso`** (do zip baixado)
3. **OK**

> 🔑 **Por que anexar o ISO?** É como **inserir um CD/DVD** na máquina — a VM vai **bootar** por esse ISO para iniciar/instalar o DVWA. Sem o disco óptico anexado, a VM sobe sem sistema.

Agora é só **iniciar** a VM DVWA.

## 🌐 Rede (próximo passo)

O lab de criação cobre CPU/disco/ISO. Para o DVWA entrar na **topologia** (ser alcançável pelo WAF/`site.lab`), a **rede** da VM precisará ser configurada como **Rede interna** na zona correta — o mesmo princípio do [68.md](68.md) (DMZ/DMZEXT). Isso vem nos próximos passos do curso.

> 💡 A ideia final: **atacante → `site.lab` → WAF (ModSecurity) → DVWA**. O DVWA fica **atrás** do WAF, para o ModSecurity ter o que proteger ([84.md](84.md)/[85.md](85.md)).

## 🐞 Troubleshooting Comum

### "A VM não boota / tela preta"

Confirme que o **ISO está anexado** na Controladora IDE (Etapa 3) e que a **ordem de boot** inclui o óptico (`Configurações → Sistema → Ordem de boot`).

### "Versão da VM: não acho Ubuntu 64-bit"

Se só aparecem opções **32-bit**, habilite a **virtualização (VT-x/AMD-V)** na BIOS/UEFI do host — sem ela, o VirtualBox esconde os perfis 64-bit.

### "Pouca RAM / travando"

512 MB é o mínimo do lab. Se travar, suba para 1024 MB nas Configurações → Sistema.

## ✅ Quiz Mental

1. O que é o DVWA e por que usamos uma app **propositalmente vulnerável**?
2. Qual o papel do DVWA na **topologia** do nosso lab?
3. Diferença entre disco **dinamicamente alocado** e **fixo**?
4. Por que anexamos o **ISO** à Controladora IDE?
5. Qual o cuidado **ético** ao usar o DVWA?

<details>
<summary>Ver respostas</summary>

1. É a *Damn Vulnerable Web Application* — uma app PHP/MySQL com vulnerabilidades **de propósito** (SQLi, XSS, CSRF…). Usamos algo vulnerável porque **não dá para aprender a defender sem ter o que atacar** — e fazer isso num alvo **legal e isolado** evita crime/impacto real
2. Será o **alvo atrás do WAF**: `atacante → site.lab → WAF (ModSecurity) → DVWA` ([84.md](84.md)/[85.md](85.md)). Atacando-o **com e sem** o WAF, veremos o ModSecurity **bloqueando** ([67.md](67.md)) e os eventos no SIEM
3. **Dinâmico** ocupa pouco disco no início e **cresce** conforme o uso (até o limite); **fixo** reserva **todo** o espaço de imediato. Fixo é levemente mais rápido; dinâmico economiza espaço — melhor para lab
4. Porque o ISO é como um **CD de boot**: a VM inicia/instala o DVWA a partir dele. Sem o disco óptico anexado, a VM sobe **sem sistema**
5. Usar **só em ambiente isolado e autorizado**. O DVWA existe para treino; aplicar as mesmas técnicas em sistemas de terceiros **sem autorização é crime**. O alvo aqui é **nosso** e controlado

</details>

> *"Não se aprende a apagar incêndio sem fogo.*

Até agora construímos toda a **infraestrutura defensiva** — firewall, IPS, WAF, SIEM, proxy reverso. Faltava o **alvo**: algo genuinamente vulnerável para provar que a defesa **funciona**. O DVWA preenche essa lacuna.

A defesa vai, enfim, encontrar o ataque. 🎼🛡️
