# 🧪 Laboratório Prático: Rede e Acesso do DVWA (IP estático + `site.lab`)

> **Tema:** Dar ao **DVWA** um **IP estático** (`172.100.1.100`, o slot do server na DMZ) para ele ficar acessível via `site.lab` — completando a integração do alvo à topologia
> **Pré-requisitos:** [86.md](86.md) (VM DVWA criada), [84.md](84.md)/[85.md](85.md) (proxy reverso → `172.100.1.100` + `site.lab`)
> **Conceitos base:** [70.md](70.md) (gateway = `172.100.1.10`), [66.md](66.md) (IP/máscara/gateway)
> **Curso 2 — Proxy Reverso, SSH e DNS**

---

## 🎯 Objetivo do Lab

A VM do DVWA existe ([86.md](86.md)), mas ainda **não tem IP** na nossa rede. Vamos dar a ela o **IP estático `172.100.1.100`** — **exatamente** o backend para onde o WAF encaminha (`proxy_pass`, [84.md](84.md)) e que a regra de firewall autoriza ([85.md](85.md)). Feito isso, `http://site.lab` **cai no DVWA**.

```
Navegador → site.lab → WAF (proxy) → 172.100.1.100 = DVWA  ✅
```

> 🔑 **Por que `172.100.1.100`?** Porque é o endereço que **todo o resto já espera**: o `proxy_pass` do WAF ([84.md](84.md)) e a regra `Acesso WAF > SITE` ([85.md](85.md)) apontam para ele. O DVWA assume o papel de **servidor de aplicação** atrás do WAF.

---

## ⚠️ Lembrete crítico: é uma imagem *live*

O DVWA roda como **live** (a partir do ISO, **na RAM**) — ou seja, **não persiste** nada em disco. Toda a config de rede deste lab **se perde ao desligar**, a menos que você **salve o estado da máquina**.

> 🔴 **Sempre que for desligar o DVWA, escolha "Salvar o estado da máquina"** (não "Desligar"). Isso **congela a RAM** em disco e preserva a config. Se você desligar normalmente, terá que **refazer** este lab toda vez.

| Opção ao fechar a VM | Efeito |
|----------------------|--------|
| **Salvar o estado** ✅ | Congela a RAM — config **preservada** |
| Enviar sinal de desligar / Desligar | Live reinicia do zero — **perde** a config |

## 🚀 Etapa 1 — Bootar e virar root

1. **Inicie** a VM DVWA
2. No menu de boot, tecle **Enter** na **primeira opção (`live`)**
3. Ao carregar, vire **root**:

```bash
sudo su
# Password:
password
```

> 💡 Credencial `password` é **padrão de lab** — o DVWA é feito para ser inseguro ([86.md](86.md)).

---

## 📝 Etapa 2 — Configurar IP estático

Edite o arquivo de interfaces (estilo Debian **ifupdown**):

```bash
nano /etc/network/interfaces
```

**Depois** da linha `auto eth0`, adicione:

```conf
iface eth0 inet static
address 172.100.1.100
netmask 255.255.255.0
gateway 172.100.1.10
```

Salve (**Ctrl + X** → **Enter**).

### O que cada linha faz

| Linha | Significado |
|-------|-------------|
| `iface eth0 inet static` | Configura a interface **eth0** com IPv4 **estático** (não DHCP) |
| `address 172.100.1.100` | O **IP** do DVWA = slot do server na DMZ (backend do WAF) |
| `netmask 255.255.255.0` | Máscara **/24** → rede `172.100.1.0/24` ([66.md](66.md)) |
| `gateway 172.100.1.10` | **Gateway** = IP do firewall na DMZ (o `DMZ address`, [70.md](70.md)) |

> 🔑 **Por que estático e não DHCP?** Porque **outras configs dependem** desse IP exato: o `proxy_pass` do WAF e a regra de firewall apontam para `172.100.1.100`. Se o IP mudasse (DHCP), o WAF falaria com o endereço errado → **502** ([85.md](85.md)).

---

## 🔄 Etapa 3 — Aplicar e acessar

Reinicie o serviço de rede para valer a config:

```bash
/etc/init.d/networking restart
```

> 💡 **`/etc/init.d/networking restart`** é o jeito **SysV init** (clássico) de reiniciar a rede. Em sistemas modernos seria `systemctl restart networking` ou **netplan** — o DVWA usa a base antiga.

No navegador da **sua máquina**:

```
http://site.lab
```

A página do **DVWA** deve aparecer — a requisição percorreu `site.lab` → WAF → `172.100.1.100` (DVWA).

```
🧑 Navegador ─ site.lab ─► /etc/hosts → IP público do WAF (85.md)
                              │
                              ▼  NAT 1:1 (76.md)
                        🛡️ WAF/nginx 172.100.2.100
                          ├─ ModSecurity (75.md)
                          └─ proxy_pass → 172.100.1.100 (84.md)
                               │  regra DMZEXT autoriza (85.md)

                        🎯 DVWA 172.100.1.100  ← ESTE LAB deu o IP
                           (app vulnerável, gateway 172.100.1.10)
```

Pela primeira vez, o fluxo termina numa **aplicação real e vulnerável** — pronta para os testes de ataque × WAF.

## 🐞 Troubleshooting Comum

### "Perdi a config depois de desligar"

Você **desligou** em vez de **salvar o estado**. Refaça as Etapas 1–3 e, ao fechar, escolha **Salvar o estado da máquina**.

### "`site.lab` dá 502 Bad Gateway"

O WAF não alcançou o DVWA. Verifique:
1. O IP do DVWA é **`172.100.1.100`**? (`ip a` ou `ifconfig` na VM)
2. A rede da VM está na **zona correta** (DMZ)? ([68.md](68.md))
3. O `networking restart` foi aplicado?

### "Sem rede / não pinga o gateway"

Confira `gateway 172.100.1.10` e a **máscara** `255.255.255.0`. Teste `ping 172.100.1.10` (o firewall) de dentro do DVWA.

### "eth0 não existe (nome diferente)"

Alguns sistemas usam `ens33`/`enp0s3`. Veja o nome real com `ip a` e ajuste o `interfaces`.

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **Imagem live** | Sistema que roda da RAM, **sem persistir** — precisa **salvar estado** |
| **Salvar estado da VM** | Congela a RAM em disco (preserva config do live) |
| **`/etc/network/interfaces`** | Config de rede estilo Debian **ifupdown** |
| **`inet static`** | IP **fixo** (≠ DHCP) |
| **address / netmask / gateway** | IP + máscara (/24) + rota de saída ([66.md](66.md)) |
| **Gateway `172.100.1.10`** | IP do firewall na DMZ (`DMZ address`, [70.md](70.md)) |
| **`/etc/init.d/networking restart`** | Reinicia a rede (SysV init) |
| **IP estático p/ backend** | Garante o endereço que WAF/regras esperam |

## ✅ Quiz Mental

1. Por que **sempre salvar o estado** da máquina DVWA ao desligar?
2. Por que o DVWA recebe o IP **`172.100.1.100`** especificamente?
3. Por que **IP estático** e não DHCP para o DVWA?
4. O que representa a linha `gateway 172.100.1.10`?
5. Depois de acessar `http://site.lab`, por qual caminho a requisição passou?

<details>
<summary>Ver respostas</summary>

1. Porque o DVWA roda como **imagem live** (na RAM, **sem persistir** em disco). Se você **desligar**, a config de rede se perde e você refaz tudo. **Salvar o estado** congela a RAM e preserva a configuração
2. Porque é o **backend que o resto já espera**: o `proxy_pass` do WAF ([84.md](84.md)) e a regra de firewall ([85.md](85.md)) apontam para `172.100.1.100`. Dando esse IP ao DVWA, ele vira o **servidor de aplicação** atrás do WAF
3. Porque configs **dependem** desse IP exato. Com DHCP, o endereço poderia **mudar**, e o WAF passaria a falar com o IP errado → **502 Bad Gateway**. Estático garante estabilidade para um servidor
4. É o **gateway** da rede DMZ — o **IP do firewall** naquela interface (`DMZ address`, [70.md](70.md)). É por ele que o DVWA envia tráfego para fora da própria subnet
5. `site.lab` → resolve (via `/etc/hosts`, [85.md](85.md)) para o **IP público do WAF** → **NAT 1:1** ([76.md](76.md)) → **WAF/nginx** (ModSecurity + `proxy_pass`, [84.md](84.md)) → regra de firewall autoriza ([85.md](85.md)) → **DVWA** `172.100.1.100`

</details>

## 💡 Reflexão Final

> *"Um alvo sem endereço é só uma ideia.*
> *Dar ao DVWA o IP que o WAF já procurava foi o **último parafuso** — e a máquina inteira, enfim, girou."*

Este lab **encaixou o alvo na engrenagem**. O DVWA deixou de ser uma VM isolada e assumiu o endereço (`172.100.1.100`) que o proxy reverso e as regras de firewall **já esperavam** desde o [84.md](84.md). Agora `site.lab` entrega uma aplicação **de verdade** — e vulnerável de propósito.

### O que mudou

| Antes | Depois |
|-------|--------|
| DVWA sem IP na rede | DVWA em **`172.100.1.100`** (backend do WAF) |
| `site.lab` → backend vazio/502 | `site.lab` → **DVWA** funcionando |
| Infra + alvo **separados** | Cadeia **completa**: navegador → WAF → DVWA |

### O que vem agora

Com o alvo publicado atrás do WAF, chega o clímax do curso: **atacar o DVWA** (SQLi, XSS…) e observar o **ModSecurity bloqueando** — com o **IP real do atacante** ([84.md](84.md)) pousando no **SIEM**. Toda a defesa em profundidade que construímos vai, finalmente, ser **posta à prova**. 🎼🛡️
