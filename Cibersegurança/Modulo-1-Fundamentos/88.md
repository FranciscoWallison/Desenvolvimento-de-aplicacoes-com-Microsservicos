# 🧪 Laboratório Prático: Testando Ataque (SQL Injection) e o WAF Bloqueando com 403

> **Tema:** Atacar o **DVWA** com um **SQL Injection** e ver o **WAF (ModSecurity)** bloquear com **HTTP 403** — a prova de que a defesa em profundidade funciona
> **Pré-requisitos:** [87.md](87.md) (DVWA acessível via `site.lab`), [84.md](84.md) (WAF proxy reverso)
> **Conceitos base:** [67.md](67.md) (WAF/ModSecurity + OWASP CRS), [47.md](47.md) (OWASP A03 — Injection), [75.md](75.md) (bloqueios no error_log → SIEM)
> **Curso 2 — Proxy Reverso, SSH e DNS**

---

## 🎯 Objetivo do Lab

Chegamos ao **clímax** de todo o projeto. Construímos firewall, IPS, SIEM, proxy reverso e WAF — e agora **atacamos de propósito** o DVWA ([86.md](86.md)/[87.md](87.md)) para ver o **WAF bloqueando** em tempo real.

O ataque: um **SQL Injection** clássico. O resultado esperado: **HTTP 403 (Forbidden)** — o WAF barrou antes de a requisição chegar ao código vulnerável.

> 🔒 **Ética (sempre):** isto é feito **só no nosso DVWA**, isolado e autorizado. As mesmas técnicas contra sistemas de terceiros **sem autorização são crime**. O objetivo aqui é **defensivo**: provar que a proteção funciona.

## 💉 Etapa 2 — Executar o SQL Injection

No campo **User ID**, digite o payload clássico e clique em **Submit**:

```sql
' OR 1=1 #
```

### Anatomia do payload

A aplicação monta, por trás, algo como:
```sql
SELECT ... FROM users WHERE user_id = '<SUA-ENTRADA>';
```

Injetando `' OR 1=1 #`, a consulta viraria:
```sql
SELECT ... FROM users WHERE user_id = '' OR 1=1 #';
```

| Parte | O que faz |
|-------|-----------|
| `'` | **Fecha** a string do parâmetro — "escapa" do valor esperado |
| `OR 1=1` | Condição **sempre verdadeira** → retornaria **todos** os registros |
| `#` | **Comenta** o resto da query (o `'` final é ignorado) |

> 🔑 É o **SQLi de manual** ([47.md](47.md) — OWASP A03 Injection): quebrar a consulta para burlar a lógica e **vazar/entrar** sem credencial válida.

---

## 🛑 Etapa 3 — Observar o 403

Ao clicar em **Submit**, em vez do vazamento de dados, você vê:

```
403 Forbidden
```

**Isso é vitória, não erro.** O **403** significa que o **WAF (ModSecurity)** **reconheceu o padrão de SQLi** e **bloqueou** a requisição **antes** de ela alcançar o DVWA.

### O que aconteceu por trás

```
Navegador → site.lab → 🛡️ WAF/nginx (ModSecurity + OWASP CRS)
                              │
                    "isto casa com regra de SQLi!" → 403 Forbidden ⛔
                              │  (a requisição NÃO segue)
                              ▼
                        🎯 DVWA  ← nunca recebe o ataque
```

> 🔑 **A grande sacada:** o DVWA **é vulnerável** — se o ataque chegasse nele, funcionaria (dumparia os usuários). Mas o **WAF na frente** ([84.md](84.md)) é um **controle compensatório**: mesmo com o app furado, o perímetro **barra o exploit**. É **virtual patching** — proteger sem corrigir o código.

---

## 🔍 O que diferencia os códigos HTTP aqui

| Código | Significado | No nosso caso |
|--------|-------------|---------------|
| **200 OK** | Sucesso | Ataque teria **passado** (sem WAF) |
| **403 Forbidden** | Servidor **entende e recusa** | **WAF bloqueou** o ataque ✅ |
| **404 Not Found** | Recurso inexistente | (não é o caso) |
| **401 Unauthorized** | Falta autenticação | (não é o caso) |

> 💡 **403 ≠ erro seu.** É o WAF dizendo "reconheci um ataque e **não deixo passar**". Numa app **sem** WAF, o mesmo payload daria **200** com dados vazados.

---

## 📊 Conectando ao SIEM

Esse bloqueio **não fica invisível**. Como configuramos no [75.md](75.md):
- Bloqueios do ModSecurity vão para o **`error_log`** do nginx
- O `error_log` é encaminhado ao **Graylog** (SIEM)
- E, graças aos headers do [84.md](84.md), o log carrega o **IP real do atacante** (`X-Real-IP`)

```
Ataque bloqueado (403) → error_log do WAF → syslog → Graylog → alerta com IP real
```

> 🔑 Fecha o ciclo completo do curso: **detectar** (IPS/WAF), **bloquear** (403), **registrar** (error_log) e **centralizar** (SIEM) — tudo com o **IP real** de quem tentou.

## 🐞 Troubleshooting Comum

### "O ataque funcionou (não deu 403)"

O WAF/ModSecurity pode não estar ativo/inline no caminho. Verifique:
1. Você acessou por **`site.lab`** (que passa pelo WAF), não direto no `172.100.1.100`?
2. O ModSecurity está **habilitado** no nginx do WAF ([75.md](75.md))?
3. O `site.lab_conf` (proxy reverso) está ativo ([84.md](84.md))?

### "Deu 502, não 403"

502 = o WAF não alcançou o backend (DVWA). Reveja a regra WAF→server e o IP do DVWA ([85.md](85.md)/[87.md](87.md)). O 502 é problema de **conectividade**, não de bloqueio.

### "Quero confirmar que era vulnerável mesmo"

Em **ambiente isolado**, acessar o DVWA **direto** (sem passar pelo WAF) mostraria o ataque funcionando — evidenciando que é o **WAF** que protege, não o app.

## 🧠 Conceitos Aprendidos

| Conceito | Definição |
|----------|-----------|
| **SQL Injection** | Injetar SQL na entrada p/ alterar a consulta ([47.md](47.md)) |
| **`' OR 1=1 #`** | Payload que fecha a string, força condição verdadeira e comenta o resto |
| **HTTP 403 Forbidden** | Servidor **recusa** — aqui, o **WAF bloqueou** |
| **ModSecurity + OWASP CRS** | Motor de regras que reconhece padrões de ataque ([67.md](67.md)) |
| **Controle compensatório** | Proteger (WAF) sem corrigir o código vulnerável |
| **Virtual patching** | Bloquear o exploit no perímetro, não no app |
| **Bloqueio → SIEM** | 403 vira log (error_log) → Graylog c/ IP real ([75.md](75.md)/[84.md](84.md)) |

## ✅ Quiz Mental

1. O que faz cada parte do payload `' OR 1=1 #`?
2. Por que o **403** é um **bom** resultado neste lab?
3. O DVWA é vulnerável — então **por que** o ataque não funcionou?
4. O que é **virtual patching** e como este lab o demonstra?
5. Como esse bloqueio chega ao **SIEM** com o IP real do atacante?

<details>
<summary>Ver respostas</summary>

1. `'` **fecha** a string do parâmetro (escapa do valor); `OR 1=1` cria uma condição **sempre verdadeira** (retornaria todos os registros); `#` **comenta** o resto da query, anulando o que viria depois. Juntos, burlam a lógica da consulta ([47.md](47.md))
2. Porque **403 Forbidden** significa que o **WAF (ModSecurity) reconheceu e bloqueou** o SQLi **antes** de chegar ao app. Sem WAF, o mesmo payload daria **200** com dados vazados — o 403 é a **prova visível** de que a defesa funcionou
3. Porque o **WAF na frente** ([84.md](84.md)) inspecionou a requisição, casou com uma regra de SQLi do **OWASP CRS** ([67.md](67.md)) e a **barrou** — a requisição **nunca chegou** ao código vulnerável do DVWA
4. **Virtual patching** é **proteger sem alterar o código**: o app continua furado, mas o **WAF** bloqueia o exploit no perímetro. É um **controle compensatório** — exatamente o que o 403 deste lab demonstra
5. O bloqueio do ModSecurity vai para o **`error_log`** do nginx ([75.md](75.md)), que é encaminhado por **syslog ao Graylog**; e, com os headers `X-Real-IP`/`X-Forwarded-For` ([84.md](84.md)), o log traz o **IP real** de quem atacou

</details>

## 💡 Reflexão Final

> *"Um app vulnerável atrás de um bom WAF é como um cofre com fechadura frágil — dentro de um bunker.*
> *A fechadura ainda é ruim; mas o atacante nem **chega** nela."*

Este lab é o **pagamento** de tudo que construímos desde o [67.md](67.md). Digitamos um SQL Injection de manual — o mesmo `' OR 1=1 #` que, num app desprotegido, entregaria o banco inteiro. E a resposta foi um **403 seco**: o WAF reconheceu, barrou e **registrou**. O ataque morreu no perímetro.

### O que a jornada provou

| Camada | Papel no bloqueio |
|--------|-------------------|
| **Firewall/NAT** ([76.md](76.md)) | Só o WAF é exposto; o DVWA fica escondido |
| **Proxy reverso** ([84.md](84.md)) | Todo tráfego passa **obrigatoriamente** pelo WAF |
| **WAF/ModSecurity** ([67.md](67.md)) | **Reconhece e bloqueia** o SQLi (403) |
| **SIEM** ([74.md](74.md)/[75.md](75.md)) | **Registra** o ataque com o IP real |

### O sentido de tudo

> **Defesa em profundidade não é ter uma parede alta — é ter várias paredes.**
> Se o app falha, o WAF pega. Se o WAF passa, o SIEM vê. Cada camada cobre a falha da anterior.

Do primeiro `Block Geral` ([69.md](69.md)) ao `403` deste lab, construímos — camada por camada — uma **postura defensiva real**. O DVWA continua vulnerável; mas, do jeito que montamos o ambiente, **isso não basta para o atacante vencer**. E essa é, no fim, a essência da **segurança defensiva**. 🎼🛡️
